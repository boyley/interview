# 08 · 调用三方支付，怎么防止错付 / 多付？

> **核心理念：防多付 = 全流程「唯一流水号 + 幂等」；防错付 = 「金额核对 + 验签 + 状态机」；支付结果不确定时绝不猜，必须「查询确认」；日终「对账」兜底。**
> 答题方法：先分清两类问题 → 防多付(幂等) → 防错付(核对/验签) → 结果确认 → 超时处理 → 对账兜底。

---

## 一、先分清两类问题（★ 分类框架）

| 类型 | 表现 | 根因 |
|---|---|---|
| **多付 / 重复扣款** | 同一笔业务被扣多次 | 重复提交、超时重试、回调重复、MQ 重投 |
| **错付** | 金额错、收款方错、串单、状态错乱 | 未核对金额/订单、回调被篡改、状态机不严 |

> 一句话：**多付靠「幂等」防，错付靠「核对 + 验签 + 状态机」防。**

---

## 二、防「多付 / 重复扣款」（★ 核心 = 幂等）

> 主线：**一笔业务一个全局唯一流水号（商户订单号 outTradeNo），贯穿全流程，处处用它去重。**

1. **唯一业务流水号**：下单即生成唯一 `outTradeNo`（雪花/号段），调三方、回调、重试都带它
2. **前端防重**：提交后按钮置灰 + 防重 token（PRG 模式），防用户狂点
3. **服务端幂等（多重保险）**：
   - **唯一索引**：`outTradeNo` 建唯一约束，重复插入直接失败
   - **状态机**：订单状态 `待支付→支付中→已支付`，只允许合法流转，**已支付不再发起**
   - **分布式锁**：按订单号加锁，防并发重复发起
4. **三方请求幂等**：调三方带同一 `outTradeNo`，**微信/支付宝对同一商户订单号天然幂等**（不会重复扣）
5. **回调幂等**：三方通知可能**重复推送** → 回调处理先查本地状态，**已处理直接返回 success**，不重复入账
6. **重试幂等**：超时重试用**同一流水号**，不会造成第二次扣款

---

## 三、防「错付」（金额 / 对象 / 串单错误）

1. **金额三方核对**：下单金额 = 请求金额 = **回调金额**，回调回来必须**核对金额一致**才置成功（防金额不符/篡改）
2. **订单归属核对**：回调里的 `outTradeNo` 必须是**自己系统存在的订单**，金额、收款方都匹配（防串单）
3. **验签防篡改**：三方回调**验证签名（RSA）**，防伪造/篡改回调（否则有人伪造"支付成功"）
4. **状态机严格流转**：防止**已退款/已关闭**的订单又被置成功
5. **金额用「分」存整数**，绝不用 float（浮点精度丢失导致错付）

---

## 四、支付结果怎么确认（★ 不能只信同步返回）

> 支付是**异步**的，调三方的**同步返回不可信**（可能网络断在半路）。三种手段确认真实结果：

| 手段 | 作用 |
|---|---|
| **① 异步回调通知**（主） | 三方主动推支付结果 |
| **② 主动查询 reconcile**（补） | 轮询查三方订单状态，**防回调丢失** |
| **③ 日终对账文件**（兜底） | 和三方逐笔对账，查漏 |

> 铁律：**「未知状态」（超时/网络异常）绝不能想当然当成功或失败 → 必须主动查三方真实状态。**

---

## 五、超时 / 异常处理（★ 最容易多付的点）

- 调三方**超时**：不知道成没成 → **绝不重复发起新支付**，而是用**同一流水号「查询」或「幂等重试」**
- 落库**支付中**状态 + 定时任务**查询补偿**，查到最终结果再流转
- **不确定时宁可挂起查询，也不重复扣**

---

## 六、对账兜底（最终一致）

- 日终拉三方**对账单**，与本地流水**逐笔核对**
- **长款(多收) / 短款(少收) / 错账** → 差错处理、自动/人工退款冲正

---

## 七、退款同样要幂等

- 退款用**唯一退款单号**，防重复退款（重复退 = 反向的"多付"）

---

## 🗺️ 一笔安全支付的链路

```
下单 → 生成唯一 outTradeNo → 调三方(带号) → 同步返回(不可信)
     → 异步回调 / 主动查询 → 验签 + 核对金额+订单 → 状态机置"已支付"(幂等)
     → 记账 → 日终对账兜底
```

---

> **核心原则：一笔业务一个唯一流水号贯穿全程；防多付靠幂等（唯一索引/状态机/锁/三方幂等/回调幂等/重试幂等），防错付靠金额核对+验签+状态机+金额用分；结果不确定必须查询确认，绝不重复扣；日终对账兜底，退款也要幂等。**
