# 03 · 分布式 ID 生成（雪花算法 Snowflake 为主）

> **核心理念：分布式 ID 要满足「全局唯一 + 趋势递增 + 高性能 + 不依赖 DB」。雪花算法用「时间戳 + 机器 ID + 序列号」拼成一个 64 位 long，本地生成、性能极高。**
> 答题方法：先讲为什么需要 → 雪花的位结构 → 生成流程 → 优缺点 → 时钟回拨(★追问) → 替代方案。

---

## 一、为什么需要分布式 ID？（先讲背景与要求）

分库分表后，**不能再用单库自增主键**（多库会冲突）。分布式 ID 的要求：

| 要求 | 说明 |
|---|---|
| **全局唯一** | 多节点生成不重复（硬要求） |
| **趋势递增** | 大致有序，利于 B+ 树索引写入（避免页分裂） |
| **高性能高可用** | 本地生成最好，不能每次查 DB |
| **信息安全** | 不易被猜测（如订单号别用纯自增暴露量级） |

---

## 二、雪花算法结构（★ 核心：64 位 long 的位拆分）

```
 0 |        41 位时间戳        |   10 位机器ID   | 12位序列号
┌─┬───────────────────────────┬───────────────┬────────────┐
│0│  当前毫秒 - 起始纪元(epoch) │ 5位机房+5位机器 │  同毫秒自增 │
└─┴───────────────────────────┴───────────────┴────────────┘
 1bit          41bit                 10bit          12bit     = 64bit
```

| 段 | 位数 | 作用 |
|---|---|---|
| 符号位 | 1 | 固定 0，保证 ID 为正数 |
| 时间戳 | 41 | 毫秒级，`当前时间 - 起始纪元`，约可用 **69 年** |
| 机器 ID | 10 | 标识节点，支持 **1024** 台（5 位机房 + 5 位机器，可自定义划分） |
| 序列号 | 12 | 同一毫秒内自增，**4096** 个/ms → 单机每秒 **约 409 万** ID |

> 记忆：**1 + 41 + 10 + 12 = 64**；同一毫秒同一机器最多 4096 个，够用。

---

## 三、生成流程（小步骤）

1. 取**当前毫秒时间戳** `now`
2. 若 `now == 上次时间戳`（同一毫秒）→ **序列号 +1**；若序列号溢出(>4095) → **自旋等到下一毫秒**
3. 若 `now > 上次时间戳`（新的一毫秒）→ **序列号归 0**
4. 若 `now < 上次时间戳`（**时钟回拨**）→ 见第五节处理
5. 拼接：`id = (时间戳差 << 22) | (机器ID << 12) | 序列号`

```java
long timestamp = System.currentTimeMillis();
if (timestamp == lastTimestamp) {
    sequence = (sequence + 1) & 4095;      // 12位掩码
    if (sequence == 0) timestamp = tilNextMillis(lastTimestamp); // 溢出，等下一毫秒
} else {
    sequence = 0;
}
lastTimestamp = timestamp;
return ((timestamp - EPOCH) << 22) | (workerId << 12) | sequence;
```

---

## 四、优缺点

**优点**
- 本地生成，**不依赖 DB / 网络**，性能极高（百万级/秒）
- **趋势递增**，对 MySQL 主键索引友好
- 位数可灵活调整（如机器位/序列位按需分配）

**缺点**
- **强依赖机器时钟** → 时钟回拨会生成重复 ID（核心痛点）

---

## 五、时钟回拨问题（★ 高频追问）

**问题**：机器时钟被 NTP 同步等回退，`now < lastTimestamp`，可能生成**重复 ID**。

**解决方案（小步骤，按回拨幅度分级）：**
1. **回拨很小**（几毫秒）→ **阻塞等待**，直到时钟追上 `lastTimestamp` 再生成
2. **回拨较大** → **抛异常 / 拒绝服务**，告警人工介入
3. **切备用**：切换到备用 workerId，或预留一段"回拨缓冲"
4. **用成熟框架**：**美团 Leaf-snowflake**（用 ZooKeeper 校验时钟、记录上报时间）、**百度 UidGenerator**（用未来时间 + RingBuffer 预生成，弱化时钟依赖）

---

## 六、workerId 怎么分配？

- 配置文件写死（节点少时）
- **ZooKeeper / Redis / DB 注册**动态分配（节点多、自动扩缩容）
- K8s 用 Pod 序号 / IP 映射

---

## 七、替代 / 对比方案

| 方案 | 唯一 | 趋势递增 | 依赖 | 缺点 |
|---|---|---|---|---|
| **雪花 Snowflake** | ✅ | ✅ | 时钟 | 时钟回拨 |
| **UUID** | ✅ | ❌ 无序 | 无 | 太长(128位)、无序伤索引 |
| **号段模式**(Leaf-segment) | ✅ | ✅ | DB | 依赖 DB(批量取一段缓存本地，降压) |
| **Redis INCR** | ✅ | ✅ | Redis | 依赖 Redis 可用性 |
| **DB 自增** | ✅ | ✅ | DB | 单点、性能差、分库冲突 |

> 工程成熟方案：**美团 Leaf**(号段 + snowflake 双模式)、**百度 UidGenerator**、**滴滴 TinyId**。

---

> **核心原则：雪花 = 时间戳 + 机器ID + 序列号 拼成 64 位 long，本地生成、趋势递增、高性能；核心坑是「时钟回拨」，用等待 / 备份 workerId / Leaf 等方案解决。**
