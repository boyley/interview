# 06 · 服务高可用设计实现指南（High Availability）

> **核心理念：消除单点 + 快速故障恢复，用「冗余」保命、用「柔性」保核心。**
> 答题方法：**先拆大步骤（是什么 → 难在哪 → 怎么解决 → 故障转移 → 架构 → 原则 → 小结），每个大步骤再拆小步骤。**

---

## 一、什么是服务高可用？

**高可用 = 保证服务稳定运行 + 故障自动恢复 + 尽量不影响用户。**
服务在出现宕机、网络、硬件或系统异常时，仍能持续可用，不中断或快速恢复。

**衡量指标——可用性等级（几个9）：**

| 等级 | 可用性 | 年停机时间 | 适用场景 |
|---|---|---|---|
| 基础 | 99.9%（三个9） | ≤ 8.77 小时 | 一般业务系统 |
| 标准 | 99.99%（四个9） | ≤ 52.6 分钟 | 重要业务系统 |
| 极高 | 99.999%（五个9） | ≤ 5.26 分钟 | 核心关键系统 |

> 公式：**可用性 = MTBF /（MTBF + MTTR）**。工程上**降 MTTR(恢复时间)比升 MTBF 更可控**。

---

## 二、高可用的主要挑战（先搞清"难在哪"）

| 挑战 | 说明 |
|---|---|
| 单点故障 | 某个组件只有一个实例，挂了整条链路断 |
| 硬件/网络故障 | 机器宕机、机房断网、磁盘损坏 |
| 服务雪崩 | 一个依赖故障，超时堆积拖垮上游（级联） |
| 数据不一致 | 主从延迟、故障切换丢数据 |
| 流量突增 | 瞬时高峰打垮系统 |

---

## 三、高可用解决方案设计（★ 核心大步骤：分层解决，每层再拆小步骤）

> 思路：从接入到数据，**每一层都去单点 + 有故障转移**；再叠加柔性策略控制爆炸半径。

### 步骤 1 · 架构层：去中心化 + 冗余部署

| 技术方案 | 说明 |
|---|---|
| 服务集群化 | 多实例部署，避免单点故障 |
| 负载均衡 | Nginx、LVS、Gateway 均衡转发 |
| 去中心设计 | 不依赖单一主节点，主从切换机制 |
| 微服务化 | 各服务独立部署，避免一个模块拖垮整体 |

### 步骤 2 · 部署层：多活 + 弹性

| 技术方案 | 说明 |
|---|---|
| 多机房部署 | 主机房宕机，备用机房接管（同城双活/异地多活） |
| 弹性伸缩 | K8s HPA、AutoScaling 自动扩缩容 |
| 容器化部署 | Docker/K8s，快速部署、快速恢复 |
| 灰度发布 | 分批上线，控制风险，快速回滚 |

### 步骤 3 · 服务治理层：限流 + 熔断 + 降级 + 隔离

| 功能 | 说明 |
|---|---|
| 限流 | 挡住流量激增（令牌桶/漏桶/滑动窗口）→ [05-rate-limiter](05-rate-limiter.md) |
| 熔断 | 失败率过高自动中断调用（Closed→Open→Half-Open） |
| 降级 | 不可用时返回默认值/缓存/兜底逻辑，保核心 |
| 超时控制 | RPC/HTTP 设超时，避免阻塞传播（雪崩起点） |

### 步骤 4 · 数据层：主从 + 备份 + 读写分离

| 技术方案 | 说明 |
|---|---|
| 数据主从同步 | 主库写、从库读，提高可用性 |
| 分库分表 | 大数据量拆分，防单点瓶颈 |
| 自动主备切换 | MySQL MHA/MGR、Redis 哨兵、TiDB |
| 异地容灾备份 | 数据异地多份，定期恢复演练 |

### 步骤 5 · 运维层：监控 + 自动恢复

| 功能 | 说明 |
|---|---|
| 健康检查 | 定期检查（Spring Boot Actuator），要检依赖不只检端口 |
| 容器自愈 | K8s Pod crash 自动重建 |
| 异常告警 | Prometheus + AlertManager 实时告警 |
| 自动化运维 | Ansible / Jenkins 持续部署与故障修复 |

---

## 四、故障转移流程（Failover）

1. 🔍 **故障检测** — 健康检查失败、响应超时
2. ⚠️ **故障确认** — 多次检测验证，避免误判
3. 🔄 **流量切换** — DNS 切换、负载均衡调整
4. 📢 **故障通知** — 告警通知、日志记录
5. 🔧 **自动恢复** — 服务重启、健康检查通过

---

## 五、架构示意图

**典型高可用架构**
```
                    [用户请求]
                        │
                   [负载均衡层]   ← Nginx / Gateway / SLB
                    /        \
             [服务实例A]    [服务实例B]   ← 多副本，K8s 部署
                 │              │
             [本地缓存]     [远程缓存]   ← Redis / Memcached 集群
                        │
              [数据库读写分离]   ← 主从 + 备份 + 自动切换
```

**高可用金字塔（自上而下）**
```
┌────────────────┐
│  运维监控       │ ← 自动恢复、健康检查、告警
├────────────────┤
│  服务治理       │ ← 熔断、降级、限流、超时
├────────────────┤
│  部署架构       │ ← 多副本、负载均衡、K8s
├────────────────┤
│  数据保障       │ ← 主从备份、容灾、读写分离
└────────────────┘
```

## 七、高可用设计原则

**核心设计思路**
1. **无单点** — 每层冗余，任何组件都不只一个实例
2. **无状态** — 会话外置，应用可随意扩缩容/摘除
3. **柔性可用** — 限流/熔断/降级/隔离，保核心弃非核心
4. **快速恢复** — 自动故障转移 + 可回滚，压低 MTTR
5. **监控演练** — 全链路监控 + 定期故障演练

**实施步骤**
1. **识别单点** — 找出系统中的单点，设计冗余方案
2. **落地冗余** — 负载均衡 + 健康检查 + 多副本
3. **建监控** — 完善监控告警体系
4. **定预案** — 制定故障处理流程和预案
5. **常演练** — 定期故障演练 + 系统优化

---

## 八、小结

高可用不依赖某个单一技术，而是端到端组合：

> **冗余去单点 + 柔性保核心 + 隔离控爆炸半径 + 监控演练兜底。**

**设计注意事项**
- ⚠️ 避免过度设计（按业务定可用性等级）
- ⚠️ 定期演练（没演练过的高可用都是纸面高可用）
- ⚠️ 成本平衡（可用性越高成本越高）
- ⚠️ 渐进式实施（从核心模块开始）

---

> **核心原则：冗余设计 + 快速检测 + 自动恢复 + 优雅降级。**
