# 06 · 服务高可用设计实现指南（High Availability）

> **高可用 = 保证服务稳定运行 + 故障自动恢复 + 尽量不影响用户**
> 服务在出现宕机、网络、硬件或系统异常时，仍能持续可用，不中断或快速恢复。
> 面试答题主线：**目标(几个9) → 五层策略 → 架构落地 → 故障转移 → 监控演练兜底**。

---

## 📊 可用性等级目标

| 等级 | 可用性 | 年停机时间 | 适用场景 |
|---|---|---|---|
| 基础 | 99.9%（三个9） | ≤ 8.77 小时 | 一般业务系统 |
| 标准 | 99.99%（四个9） | ≤ 52.6 分钟 | 重要业务系统 |
| 极高 | 99.999%（五个9） | ≤ 5.26 分钟 | 核心关键系统 |

> 面试加分：**降 MTTR（恢复时间）比升 MTBF（无故障时间）更可控**，所以自动故障转移、快速回滚是重点。

---

## 🔑 五大核心策略（分层记忆）

### 1️⃣ 架构层：去中心化 + 冗余部署

| 技术方案 | 说明 |
|---|---|
| 服务集群化 | 服务部署多个实例，避免单点故障 |
| 负载均衡 | Nginx、LVS、Gateway 等均衡转发 |
| 去中心设计 | 不依赖单一主节点，主从切换机制 |
| 微服务化 | 每个服务单独部署，避免一个模块拖垮整体 |

### 2️⃣ 部署层：多活 + 弹性

| 技术方案 | 说明 |
|---|---|
| 多机房部署 | 主机房宕机，备用机房接管（同城双活 / 异地多活） |
| 弹性伸缩 | Kubernetes HPA、AutoScaling 自动扩缩容 |
| 容器化部署 | Docker/K8s，快速部署、快速恢复 |
| 灰度发布 | 分批上线，控制风险，快速回滚 |

### 3️⃣ 服务治理层：限流 + 熔断 + 降级

| 功能 | 说明 |
|---|---|
| 限流 | 避免流量激增拖垮服务（令牌桶、漏桶、滑动窗口）→ [05-rate-limiter](05-rate-limiter.md) |
| 熔断 | 服务调用失败率过高，自动中断调用（Closed→Open→Half-Open 三态） |
| 降级 | 服务不可用时返回默认值/缓存/兜底逻辑，保核心 |
| 超时控制 | RPC/HTTP 接口设置超时，避免阻塞传播（雪崩起点） |

### 4️⃣ 数据层：主从 + 备份 + 读写分离

| 技术方案 | 说明 |
|---|---|
| 数据主从同步 | 主库写，从库读，提高可用性 |
| 分库分表 | 大数据量拆分，防止单点瓶颈 |
| 自动主备切换 | MySQL MHA / MGR、Redis 哨兵、TiDB 自动主备机制 |
| 异地容灾备份 | 数据异地多份备份，定期恢复演练 |

### 5️⃣ 运维层：监控 + 自动恢复

| 功能 | 说明 |
|---|---|
| 健康检查 | 服务定期检查（Spring Boot Actuator），要检依赖不只检端口 |
| 容器自愈 | Kubernetes Pod crash 自动重建 |
| 异常告警 | Prometheus + AlertManager 实时告警 |
| 自动化运维 | Ansible / Jenkins 持续部署和故障修复 |

---

## 🏛 典型高可用架构

```
                    [用户请求]
                        │
                   [负载均衡层]   ← Nginx / Gateway / SLB
                    /        \
             [服务实例A]    [服务实例B]   ← 多副本，K8s 部署
                 │              │
             [本地缓存]     [远程缓存]   ← Redis / Memcached 集群
                        │
              [数据库读写分离]   ← 主从 + 备份 + 自动切换
```

## ⚡ 故障转移流程（Failover）

1. 🔍 **故障检测** → 健康检查失败，响应时间超时
2. ⚠️ **故障确认** → 多次检测验证，避免误判
3. 🔄 **流量切换** → DNS 切换，负载均衡调整
4. 📢 **故障通知** → 告警通知，日志记录
5. 🔧 **自动恢复** → 服务重启，健康检查通过

## 🛠 关键技术栈推荐

| 方向 | 推荐技术 | 说明 |
|---|---|---|
| 服务注册/发现 | Nacos / Consul / Eureka | 动态服务管理，自动注册发现 |
| 熔断/限流 | Sentinel / Hystrix / Resilience4j | 服务治理，防止级联故障 |
| 容器化部署 | Docker / Kubernetes | 容器编排，自动扩缩容 |
| 负载均衡 | Nginx / Envoy / Istio | 流量分发，健康检查 |
| 弹性伸缩 | Kubernetes HPA / 云服务 ESS | 根据负载自动调整实例数 |
| 自动恢复 | K8s 自愈机制 / Serverless | 故障自动重启，快速恢复 |
| 监控告警 | Prometheus + Grafana / SkyWalking | 全方位监控，实时告警 |

## ✅ 最佳实践

**架构设计**
- 无单点故障设计
- 服务无状态化（水平扩展 / 故障转移的前提）
- 水平扩展能力
- 多机房部署

**数据管理**
- 实时数据备份
- 主从读写分离
- 数据分片策略
- 备份恢复演练

**监控告警**
- 全方位监控指标
- 实时告警机制
- 故障自动处理
- 性能趋势分析

**测试验证**
- 故障注入测试（混沌工程 ChaosBlade）
- 压力测试验证
- 灾难恢复演练
- 自动化测试

---

## ⚠️ 设计注意事项

1. **避免过度设计** — 根据业务需求合理设计可用性等级
2. **定期演练** — 定期进行故障演练和恢复测试（没演练过的高可用都是纸面高可用）
3. **成本平衡** — 可用性越高，成本越高，要找到平衡点
4. **渐进式实施** — 从核心模块开始，逐步完善

---

## 🎯 实施步骤检查清单

- ☐ 识别系统中的单点故障，设计冗余方案
- ☐ 实施负载均衡和健康检查机制
- ☐ 建立完善的监控告警体系
- ☐ 制定详细的故障处理流程和预案
- ☐ 定期进行故障演练和系统优化

## 📈 可用性计算公式

```
可用性 =（总时间 - 停机时间）/ 总时间 × 100%

MTBF = 平均故障间隔时间
MTTR = 平均修复时间
可用性 = MTBF /（MTBF + MTTR）
```
