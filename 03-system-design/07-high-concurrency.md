# 07 · 高并发系统的设计与解决方案（High Concurrency）

> **核心理念：分而治之，各个击破，系统性解决高并发挑战。**
> 答题方法：**先拆大步骤（是什么 → 难在哪 → 怎么解决 → 架构 → 优化 → 原则），每个大步骤再拆小步骤**，别一上来就堆技术点。

---

## 一、什么是高并发？

**高并发（High Concurrency）** = 系统在单位时间内处理大量请求的能力，常见于高访问量网站、秒杀、IM、交易撮合等。

**关注的核心指标：**
- **QPS**：每秒请求数（读）
- **TPS**：每秒事务数（写）
- **吞吐量**：单位时间处理的数据量
- **并发用户数**：同时在线/在处理的请求数
- **响应时间（RT）**：一次请求耗时（看 P99，不看平均）

> 记忆：**并发数 ≈ QPS × RT**（利特尔法则）。

---

## 二、高并发的主要挑战（先搞清"难在哪"）

| 挑战 | 说明 |
|---|---|
| 系统瓶颈 | CPU、内存、网络、磁盘 I/O、数据库连接数 |
| 资源争抢 | 并发访问共享资源造成的锁竞争、阻塞 |
| 数据一致性 | 缓存与数据库一致性、分布式状态同步 |
| 服务雪崩 | 某个服务失败拖垮整套系统（级联故障） |
| 可扩展性 | 单点服务或资源扩展受限 |

---

## 三、高并发解决方案设计（★ 核心大步骤：分层解决，每层再拆小步骤）

> 思路：请求从上到下，**每一层都设一道关卡，层层拦截、层层减压**。

### 步骤 1 · 前端层：限流 + CDN + 动静分离
- 静态资源(图片/JS/CSS)通过 **CDN** 缓存下发，部署到对象存储(OSS/S3)
- 动静分离：动态接口与静态资源拆开
- 网关层(Nginx/Kong/Envoy)做**限流、降级**
  ```nginx
  limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s;
  ```

### 步骤 2 · 接入层：负载均衡 + 多副本部署
- Nginx / LVS / SLB 做请求分发
- 后端服务**多实例部署**(K8s Deployment 多副本)
- **自动扩缩容**(K8s HPA)

### 步骤 3 · 应用层：异步 + 并发控制 + 服务隔离

| 手段 | 技术方案 |
|---|---|
| 异步处理 | MQ（Kafka / RabbitMQ / RocketMQ） |
| 并发控制 | 信号量 / 限流器（Guava RateLimiter） |
| 服务隔离 | 线程池、舱壁模式（Resilience4j/Hystrix） |
| 缓存策略 | 本地缓存 + Redis + 多级缓存 |
| 请求合并 | 批处理、请求合并（Batch + Future） |

```java
RateLimiter limiter = RateLimiter.create(1000); // 每秒 1000 个请求
if (!limiter.tryAcquire()) {
    // 拒绝请求
}
```

### 步骤 4 · 数据层：缓存 + 分布式 + 数据库优化

**小步骤 4.1 缓存策略**
- 热数据放入 Redis
- 缓存更新：主动更新 + 定时刷新 + 异步淘汰
- 防止缓存**击穿 / 穿透 / 雪崩** → [../01-cheatsheet/06-redis](../01-cheatsheet/06-redis.md)

**小步骤 4.2 数据库优化**
- 分库分表：Sharding-JDBC / Vitess / MyCat
- 主从复制 + 读写分离
- SQL 调优：建索引、避免全表扫描、分页优化

### 步骤 5 · 消息队列：解耦 + 削峰

用 MQ 作缓冲通道，实现"削峰填谷、异步解耦"。

| 使用场景 | MQ 优势 |
|---|---|
| 秒杀系统 | 削峰，防止流量打死数据库 |
| 注册邮件通知 | 异步处理，主链路快 |
| 数据同步 | 解耦系统 |

### 步骤 6 · 服务治理与容错
- 熔断、限流、降级（Sentinel / Resilience4j）
- 设置超时时间，避免调用链阻塞
- 业务降级兜底（如秒杀失败返回"排队中"）→ [06-high-availability](06-high-availability.md)

## 五、高并发实践优化建议

**小步骤 5.1 性能优化**
- 使用连接池（数据库、Redis、HTTP）
- 使用线程池控制并发量（自定义线程池 + 拒绝策略）
- 数据预热（热点缓存、热点库加载）

**小步骤 5.2 测试与规划**
- 压测与容量规划（JMeter / wrk / Locust）
- 分布式唯一 ID 生成器（雪花算法 / Leaf）

## 七、高并发系统设计原则

**核心设计思路**
1. **分层解决** — 每一层都要有相应的优化策略
2. **异步解耦** — 用消息队列减少同步依赖
3. **缓存优先** — 多级缓存减少数据库压力
4. **水平扩展** — 通过增加实例数量提升处理能力
5. **服务治理** — 熔断、限流、降级保证系统稳定

**实施步骤**
1. **识别瓶颈** — 通过压测找到系统瓶颈点
2. **分层优化** — 从前端到数据库逐层优化
3. **监控完善** — 建立全链路监控体系
4. **容量规划** — 根据业务增长预估容量需求
5. **持续优化** — 定期压测和性能调优

---

## 八、小结

高并发系统不依赖某个单一技术，而是端到端组合：

> **前端限流 + 接入层负载均衡 + 应用层异步解耦 + 数据层缓存优化 + 服务治理容错 + 自动弹性扩缩容**

每个环节都可能是瓶颈点，必须**端到端设计 + 压测验证 + 容量预估**。

**关键成功因素**
- ✅ 架构合理 — 分层设计，每层职责清晰
- ✅ 技术选型 — 按场景选合适技术栈
- ✅ 监控完善 — 全方位监控，及时发现问题
- ✅ 持续优化 — 随业务发展不断调整架构
