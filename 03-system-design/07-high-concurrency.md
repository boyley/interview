# 07 · 服务高并发设计实现指南（High Concurrency）

> **高并发 = 用尽量少的资源，扛住尽量多的同时请求，且响应快、不雪崩**
> 三字诀：**散(分流) · 缓(缓存) · 异(异步)**；一句话核心 —— **能挡就挡在前面、能读缓存不读库、能异步就别同步等。**
> 面试答题主线：**指标(QPS/RT) → 五层策略 → 架构落地 → 缓存三问题 → 容量估算**。

---

## 📊 核心衡量指标

| 指标 | 含义 | 面试一句话 |
|---|---|---|
| **QPS** | 每秒查询数(读) | 系统每秒能处理多少请求 |
| **TPS** | 每秒事务数(写) | 每秒完成多少笔完整业务 |
| **RT(响应时间)** | 一次请求耗时 | 越低越好，P99 比平均值更能反映体验 |
| **并发数** | 同时在处理的请求数 | 并发数 ≈ QPS × RT |
| **吞吐量** | 单位时间处理量 | 系统整体处理能力 |

> 面试加分：**QPS × RT = 并发数**(利特尔法则)。压测要看 **P99/P999**，不是平均值。

---

## 🔑 五大核心策略(分层记忆)

### 1️⃣ 接入层：分流 + 拦截

| 技术方案 | 说明 |
|---|---|
| CDN | 静态资源(图片/JS/CSS)就近分发，不回源 |
| 页面静态化 | 动态页提前生成静态 HTML，扛读(如商品详情) |
| 负载均衡 | Nginx / LVS / DNS 轮询，把流量摊到多台 |
| 接入层限流 | 把超额流量挡在最外层(Nginx / 网关)，别放进后端 → [05-rate-limiter](05-rate-limiter.md) |

### 2️⃣ 应用层：无状态 + 水平扩展 + 池化

| 技术方案 | 说明 |
|---|---|
| 无状态化 | 会话放 Redis，应用可随意加机器/摘除(扩展前提) |
| 集群 + 水平扩展 | 加机器(scale-out)比升配(scale-up)更能扛并发 |
| 线程池 / 连接池 | 池化复用，避免频繁创建销毁；隔离防拖垮 |
| 异步化 | Servlet 异步、CompletableFuture、响应式，提高吞吐 |

### 3️⃣ 缓存层：多级缓存(高并发第一利器)

| 技术方案 | 说明 |
|---|---|
| 本地缓存 | Caffeine / Guava，进程内、最快，扛热点 |
| 分布式缓存 | Redis 集群，共享、容量大 |
| 多级缓存 | 浏览器 → CDN → Nginx → 本地 → Redis → DB 层层拦截 |
| 热点探测 | 识别热 key，单独本地缓存 + 打散，防击穿 |

### 4️⃣ 数据层：读写分离 + 分库分表

| 技术方案 | 说明 |
|---|---|
| 读写分离 | 主库写、从库读，扛读多场景 |
| 分库分表 | 数据量/写压力大时水平拆分(ShardingSphere) |
| NoSQL | ES(搜索)、MongoDB、HBase 扛特定海量场景 |
| 连接池 | Druid / HikariCP，控制 DB 连接数不被打爆 |

### 5️⃣ 异步/中间件层：削峰 + 解耦

| 技术方案 | 说明 |
|---|---|
| MQ 削峰填谷 | Kafka/RocketMQ 缓冲突发流量，后端匀速消费 |
| 异步解耦 | 非核心链路(发短信/记日志)异步化，主链路快 |
| 最终一致性 | 用消息 + 补偿代替强一致事务，换吞吐 |
| 任务分片/批处理 | 大任务拆小、批量提交，降单次开销 |

## 🏛 典型高并发架构

```
                    [用户请求]
                        │
                     [CDN]        ← 静态资源就近分发
                        │
                 [Nginx/网关]     ← 负载均衡 + 限流 + 静态化
                    /       \
             [应用集群A]  [应用集群B]  ← 无状态、水平扩展、K8s
                    \       /
              [多级缓存 Caffeine + Redis 集群]  ← 挡住绝大多数读
                        │
                   [MQ 削峰]      ← Kafka/RocketMQ 缓冲写
                        │
         [DB 读写分离 + 分库分表 + 连接池]
```

---

## ⚡ 一个高并发请求的处理链路

1. 🌐 **CDN/静态化** → 静态内容直接返回，不进后端
2. 🚦 **网关限流** → 超额请求快速拒绝/排队，保护后端
3. ⚡ **多级缓存** → 命中缓存直接返回(挡住 ~90% 读)
4. 📨 **异步削峰** → 写请求进 MQ，后端匀速消费
5. 🗄 **分库分表** → 落库时按分片路由，避免单点瓶颈
6. 🛡 **熔断降级** → 依赖异常时兜底，防雪崩 → [06-high-availability](06-high-availability.md)

---

## 🔥 缓存三大问题(面试必问)

| 问题 | 现象 | 解决方案 |
|---|---|---|
| **缓存穿透** | 查不存在的 key，每次都打到 DB | 布隆过滤器 / 缓存空值 |
| **缓存击穿** | 单个热点 key 失效瞬间，大量请求压 DB | 互斥锁重建 / 逻辑过期 / 热点永不过期 |
| **缓存雪崩** | 大量 key 同时失效，DB 被压垮 | 过期时间加随机 + 多级缓存 + 高可用集群 |

> 详见 [../01-cheatsheet/06-redis](../01-cheatsheet/06-redis.md)。

## 🛠 关键技术栈推荐

| 方向 | 推荐技术 | 说明 |
|---|---|---|
| 缓存 | Redis / Caffeine | 分布式 + 本地多级缓存 |
| 消息队列 | Kafka / RocketMQ / RabbitMQ | 削峰、解耦、异步 |
| 限流熔断 | Sentinel / Resilience4j | 流量防护，防级联故障 |
| 负载均衡 | Nginx / LVS / Gateway | 流量分发 |
| 分库分表 | ShardingSphere / MyCat | 数据水平拆分 |
| 连接池 | HikariCP / Druid | 数据库连接复用 |
| 搜索/海量读 | Elasticsearch | 全文检索、聚合 |
| 压测 | JMeter / wrk / Gatling | 容量验证 |

## 🏔 高并发优化漏斗模型(自上而下层层拦截)

```
┌──────────────────────────┐
│   接入层拦截               │ ← CDN、静态化、限流(挡掉最多)
├──────────────────────────┤
│   缓存层拦截               │ ← 多级缓存，挡住绝大多数读
├──────────────────────────┤
│   异步削峰                 │ ← MQ 缓冲突发写
├──────────────────────────┤
│   数据层扩展               │ ← 读写分离、分库分表(最后兜底)
└──────────────────────────┘
   原则：流量越往下越少、越贵，尽量在上层解决
```

## ✅ 最佳实践

**流量治理**
- 前置限流(接入层就挡)
- 熔断降级保核心
- 热点隔离 + 打散

**读多优化**
- 多级缓存
- 页面/接口静态化
- 读写分离

**写多优化**
- MQ 削峰异步落库
- 批量提交 / 合并写
- 分库分表

**资源与稳定性**
- 全链路池化(线程池/连接池)
- 无状态 + 水平扩展
- 压测定容量、留余量

---

## ⚠️ 设计注意事项

1. **先压测再优化** — 用数据定位瓶颈，别凭感觉；先找 P99 慢在哪
2. **缓存一致性** — 用"更新DB + 删缓存(Cache Aside)"，接受短暂不一致
3. **异步的代价** — 换来吞吐但牺牲实时性/强一致，要能接受最终一致
4. **避免过度设计** — 按真实量级设计，DAU 1万别照百万级堆架构
5. **防雪崩闭环** — 限流+熔断+降级+隔离+超时，五件套一起上

---

## 🎯 实施步骤检查清单

- ☐ 压测摸底，定位当前 QPS/RT 瓶颈点
- ☐ 接入层加 CDN + 限流，先挡住无效/超额流量
- ☐ 引入多级缓存，把读扛在缓存层
- ☐ 写链路用 MQ 削峰、非核心异步化
- ☐ 数据层做读写分离，压力大再分库分表
- ☐ 全链路池化 + 熔断降级兜底
- ☐ 再压测验证 + 预留容量余量

---

## 📈 容量估算公式(面试常考)

```
并发数 = QPS × 平均响应时间(RT)          # 利特尔法则

峰值 QPS = (日总请求数 × 峰值集中系数) / 峰值时间秒数
  经验：80% 流量集中在 20% 时间里
  峰值QPS ≈ (DAU × 人均请求 × 0.8) / (24 × 3600 × 0.2)

所需机器数 = 峰值QPS / 单机QPS  (再留 30%~50% 余量)
```
