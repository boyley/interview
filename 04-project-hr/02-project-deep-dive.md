# 02 · 项目深挖高频问答（Project Deep Dive）

> 技术面 60% 时间在深挖你简历上的项目。面试官不看你做了什么牛逼系统，看的是**你在里面思考了多深、能不能自圆其说**。
> 本篇给每个高频深挖问题的「答题思路 + 话术模板 + 反例」。配合 [`01-star-template.md`](./01-star-template.md) 一起用。

---

## 🎯 一句话心法

> **深挖 = 面试官在验证你简历的真实性 + 技术深度。**
> 答不好不是因为项目不够大，而是：① 只会用不懂原理 ② 没讲权衡 ③ 数据编造对不上 ④ 揽了不属于你的功劳被追问穿。
> 准备的核心动作：**把简历每个项目按下面 6 段过一遍，能脱稿讲。**

```
是什么 → 我做了什么 → 难点 → 怎么解决 → 效果（带数字） → 还能怎么优化
  背景      个人贡献     深度点    技术+思考     可量化       持续思考
```

---

## 🔥 高频深挖问题 · 速查表

| 面试官问 | 真实意图 | 答题主线 | 致命坑 |
|----------|----------|----------|--------|
| 这个项目难点在哪？ | 有没有技术深度 | 找**真难点**，讲思考过程不只讲结论 | 说"需求老变""进度紧"这种非技术难点 |
| 为什么这么设计/为什么选 X？ | 有没有权衡意识 | 选型对比，讲 trade-off | "因为流行""大家都用" |
| 遇到过什么线上问题？ | 排查能力/责任心 | 现象→定位→根因→解决→复盘 完整闭环 | 没有细节、工具答不上来 |
| 你负责哪部分？ | 甄别真实贡献 | 突出个人，边界清晰 | 团队功劳全揽，一追问就穿 |
| QPS/数据量多少？怎么扛的？ | 容量意识 | 提前准备数据，能自圆其说 | 张口就来、前后对不上 |
| 流量涨 10 倍怎么办？ | 架构演进思路 | 缓存/异步/拆分/限流/扩容 分层答 | 只会说"加机器" |
| 还有什么能优化的？ | 是否持续思考 | 主动暴露 1~2 个已知短板 | "挺完美的，没啥可优化" |

## 1️⃣ "你这个项目难点在哪？"

### 怎么找「真难点」

面试官要的是**技术难点**，不是"需求经常变""排期紧""跨部门沟通难"（这些是项目管理难点，答了减分）。真难点通常长这样：

| 难点类型 | 举例 | 为什么算深度 |
|----------|------|--------------|
| 高并发下的一致性 | 秒杀扣库存超卖 / 分布式锁 | 涉及锁、原子性、CAP |
| 性能瓶颈 | 接口 P99 从 2s 优化到 200ms | 涉及定位手段 + 原理 |
| 数据量级带来的问题 | 单表 5000w 慢查询 / 大 key | 涉及索引、分库分表 |
| 分布式复杂度 | 跨服务事务 / 消息幂等/乱序 | 涉及最终一致、幂等设计 |
| 稳定性 | 依赖抖动导致雪崩 | 涉及熔断/降级/隔离 |

### 话术模板（三段式）

> **1. 难在哪（点明矛盾）**：这个模块的难点是「在 X 约束下要同时满足 Y 和 Z」——比如"在 5000 QPS 下既要保证库存不超卖，又不能让扣减接口变慢"。
> **2. 我的思考（给方案对比）**：我考虑过 A/B 两种方案。A 是数据库悲观锁，简单但并发上不去；B 是 Redis 预扣 + 异步落库，性能好但要处理一致性。我选了 B，因为……
> **3. 怎么落地 + 踩了什么坑**：落地时遇到「Redis 扣了但 MySQL 落库失败」的问题，我用……解决。

✅ **加分**：讲清楚"为什么难" > "我做了什么"。把你**权衡、试错、踩坑**的过程讲出来，这是编不出来的，最能证明是你亲手做的。

⚠️ **反例**：
- "难点就是业务逻辑复杂，if-else 特别多" —— 不是技术难点。
- "难点是用了 Redis" —— 用了不等于难，面试官立刻追问原理，答不上就穿。

## 2️⃣ "为什么这么设计 / 为什么选 X 不选 Y？"

### 核心：讲权衡（trade-off），不讲流行

任何选型都要能从 **4 个维度**说出取舍，至少覆盖 2 个：

```
性能  ←→  成本
场景匹配  ←→  团队熟悉度/维护成本
```

### 常见选型对比话术

| 问题 | 别这么答 | 该这么答（讲权衡） |
|------|----------|--------------------|
| 为什么用 Redis？ | "因为快" | "热点数据读多写少，DB 扛不住 5k QPS，用 Redis 把读挡在缓存层；接受最终一致，用双删+过期兜底" |
| 为什么用 Kafka 不用 RabbitMQ？ | "Kafka 流行" | "我们是日志/埋点场景，吞吐要求高（百万级/s）、允许消息堆积，Kafka 顺序写+分区扩展更合适；RabbitMQ 强在低延迟和复杂路由，但吞吐是短板" |
| 为什么分库分表不用单库？ | "数据大" | "单表已 8000w，B+ 树层高增加、慢查询频发，垂直拆分先分不动了才水平分；按 user_id 取模分 16 库，兼顾扩容和路由简单" |
| 为什么用 MySQL 不用 MongoDB？ | "团队会 MySQL" | "数据强关系、有事务需求（订单+库存），MySQL 的事务和成熟生态更稳；团队运维经验也在这（成本维度也算权衡）" |
| 为什么自研不用现成？ | "想练手" | 慎答，通常应该"能用轮子不造轮子"，除非现成方案确实不满足某硬约束 |

### 话术模板

> "我在 X 和 Y 之间做过对比。X 的优势是……代价是……；Y 反过来。我们的场景是【具体约束：数据量/QPS/一致性要求/团队】，所以选了 X。**如果换成【另一种场景】，我会选 Y。**"

✅ 最后那句"如果场景变了我会选另一个"是**满分点**——证明你不是背答案，是真懂各自适用边界。

⚠️ **反例**：
- "因为大厂都在用" / "网上说这个好" —— 没有独立判断。
- 只说优点不说代价 —— 面试官会追"那它有什么缺点？"，答不上就露馅。任何技术都有代价，说不出代价 = 没深入用过。

> 深度原理链接：缓存一致性见 [`../01-cheatsheet/06-redis.md`](../01-cheatsheet/06-redis.md)，分库分表见 [`../01-cheatsheet/05-mysql.md`](../01-cheatsheet/05-mysql.md)。

## 3️⃣ "遇到过什么线上问题？怎么排查的？"

### 结构：现象 → 定位 → 根因 → 解决 → 复盘（5 步闭环）

**必须有工具和数字**，否则像编的。准备 1~2 个能讲深的真实故事。

### 完整示例：接口突然变慢（P99 从 200ms → 3s）

> **① 现象**：某天上午监控告警，订单查询接口 P99 从 200ms 飙到 3s，错误率没涨（说明不是报错，是慢），CPU/内存正常。
>
> **② 定位（讲工具）**：
> - 先看监控大盘（Prometheus/Grafana）：确认是单接口慢，不是全站。
> - 看链路追踪（SkyWalking/Zipkin）：定位到耗时集中在一条 SQL。
> - `EXPLAIN` 这条 SQL：发现 `type=ALL` 全表扫描，没走索引。
>
> **③ 根因**：前一天上线加了个查询条件 `status`，但没建对应联合索引，数据量涨到 2000w 后全表扫描被放大。
>
> **④ 解决**：加联合索引 `(user_id, status)`，P99 回落到 180ms。临时先加了限流兜底防雪崩。
>
> **⑤ 复盘**：在 CI 加了慢 SQL 扫描（上线前 EXPLAIN 检查 `type` 和 `rows`），并对核心接口加了 P99 告警阈值。

### 其他常见故障 · 排查工具速记

| 故障 | 定位工具/命令 | 常见根因 |
|------|---------------|----------|
| 接口慢 | 链路追踪 → EXPLAIN / 日志耗时打点 | 慢 SQL、锁等待、下游依赖抖动、N+1 查询 |
| CPU 飙高 | `top` → `top -Hp <pid>` 定位线程 → `jstack` 打栈 → 16进制线程 ID 匹配 | 死循环、频繁 GC、正则回溯、序列化 |
| 内存 OOM | `jmap -histo` / dump → MAT 分析 → 看 GC 日志 | 内存泄漏（静态集合/连接未关）、大对象、缓存无上限 |
| 频繁 Full GC | GC 日志 / `jstat -gcutil` | 大对象进老年代、内存泄漏、堆设置不合理 |
| 线程死锁 | `jstack` 看 `deadlock` / `jconsole` 检测 | 多把锁加锁顺序不一致 |
| CPU 不高但慢 | 看线程状态 BLOCKED/WAITING | 锁竞争、线程池满、DB 连接池耗尽 |

> ⚡ CPU 飙高的**黄金四步**要能背：`top` 找进程 → `top -Hp` 找线程 → `printf %x` 转 16 进制 → `jstack <pid> | grep <hex>` 看栈。
> 深度排查手册见 [`../01-cheatsheet/11-troubleshooting.md`](../01-cheatsheet/11-troubleshooting.md)。

✅ **加分**：讲到**复盘/预防**（加监控、加 CI 检查）——证明你不是救火队员，会系统性防止复发。

⚠️ **反例**：
- "重启就好了" —— 没找根因，减分。
- 讲不出用什么工具、看什么指标 —— 一定是编的，面试官一追细节就崩。
- 现象和根因对不上（比如说 CPU 高但根因是慢 SQL）—— 逻辑不自洽。

## 4️⃣ "你负责哪部分？"

### 原则：突出个人，边界清晰，别揽全功

面试官最烦"这个系统是我做的"然后一问细节全不知道。**团队做的说团队，你做的说你**，反而显得诚实可信。

### 话术模板

> "这个项目是 X 人的团队做的。我主要负责【具体模块】，包括【设计了什么 / 写了什么核心逻辑 / 解决了什么问题】。其中【某个难点】是我独立搞定的。整体架构是和 Leader 一起定的，我参与了【某部分】的讨论。"

### 措辞对照

| 场景 | ❌ 揽功（易穿） | ✅ 诚实且亮眼 |
|------|----------------|---------------|
| 架构是 Leader 定的 | "架构是我设计的" | "架构是 Leader 主导，我负责其中 XX 模块的详细设计" |
| 参与讨论 | "缓存方案我定的" | "缓存方案是我调研提的，团队评审后采纳的" |
| 只写了部分 | "整个系统我写的" | "我负责订单和库存模块，支付是同事做的" |

✅ **加分**：即使是小模块，把你**独立解决的那个点讲深**，比揽一堆讲不清强 10 倍。深度 > 广度。

⚠️ **反例**：说"整个项目我负责"，面试官问"那消息队列的幂等你怎么做的？"——不是你做的就答不出，直接判定简历注水，全盘信任崩塌。

## 5️⃣ "QPS / 数据量多少？怎么扛的？"

### 提前准备一套能自圆其说的数字

**关键：数字之间要能互相印证，别孤立编一个。** 面试官会交叉验证。

需要提前想好的一组数：

```
日活/日请求量 → 峰值 QPS → 单机 QPS → 机器数 → 数据量(总行数/日增) → 存储大小 → 缓存命中率
```

### 自圆其说示例

> "系统日请求约 1000w，主要集中在白天，按二八原则峰值 QPS 大概 1000w × 80% ÷（8h×3600）≈ 300~400，峰值我们按 1000 QPS 做容量。单机压测扛住 500 QPS，部署了 4 台留了余量。核心表订单表日增约 50w，一年 1.8 亿，所以做了按月分表。热点数据（商品详情）走 Redis，命中率 95%+，DB 实际压力很小。"

这段里 QPS、机器数、数据量、分表、缓存**互相印证**，问哪个都接得住。

### 常用估算换算（背下来）

| 换算 | 公式 |
|------|------|
| 平均 QPS | 日请求量 ÷ 86400 |
| 峰值 QPS | 平均 QPS ×（2~10 倍，看业务波峰） |
| 二八估算 | 80% 流量集中在 20% 时间（约 5h） |
| 单表建议行数 | 阿里规范：单表 > 500w 或 2GB 考虑分表 |

⚠️ **反例**：
- "QPS 大概几万吧" 但只部署 2 台单机顶不住 —— 数字对不上，一算就穿。
- 说不清楚就老实说"具体峰值 QPS 我记不太准，量级是千级" —— 给量级比乱报精确数字安全。
- 数据量说"很大"却答不出具体表多少行 —— 显得没关注过。

## 6️⃣ "如果流量涨 10 倍怎么办？"

### 考的是架构演进思路，分层作答（从便宜到贵）

**不要只说"加机器"**。按下面顺序展开，体现你有体系：

```
读优化          写优化           架构拆分         保护兜底
─────           ─────           ─────           ─────
① 缓存          ④ 异步削峰       ⑥ 服务拆分       ⑧ 限流
  多级缓存         MQ 异步          微服务           熔断降级
  缓存预热       ⑤ 批量/合并      ⑦ 分库分表       ⑨ 隔离
② 读写分离                        水平扩展         (线程池/资源)
③ 水平扩容
```

### 话术模板（分层递进）

> "我会分几层看：
> **① 先看读还是写压力大。** 如果是读多——加多级缓存（本地 Caffeine + Redis）、读写分离，读基本能挡下来。
> **② 写压力大**——用 MQ 异步削峰，把同步写变异步，非核心链路解耦；能合并的批量写。
> **③ 单机/单库到瓶颈**——无状态服务水平扩容加机器，数据库做分库分表。
> **④ 全程加保护**——限流（防打满）、熔断降级（依赖挂了不雪崩）、资源隔离。
> 具体涨 10 倍我会先压测找到当前瓶颈在哪一环，**哪层先撑不住就先优化哪层**，而不是一上来全上。"

✅ **满分点**：最后一句"先压测定位瓶颈，按瓶颈针对性优化"——体现你不是无脑堆方案，懂**数据驱动**。

> 深度见 [`../03-system-design/07-high-concurrency.md`](../03-system-design/07-high-concurrency.md)、[`../03-system-design/06-high-availability.md`](../03-system-design/06-high-availability.md)。

## 7️⃣ "项目还有什么可以优化的？"

### 千万别说"挺完美的，没啥可优化"

这题是送分题，考的是**持续思考的习惯**。主动暴露 1~2 个已知短板，反而加分。

### 话术模板

> "有几个点我一直想优化：
> ① 【当前短板】现在 XX 还是同步调用，高峰有点勉强，可以改异步/加缓存。
> ② 【技术债】早期为了快上线，XX 用了比较糙的方案，现在可以重构成 YY。
> ③ 【可观测性】监控还不够细，想补上 XX 维度的埋点。
> 没做主要是【排期/ROI/优先级】原因，但方案我都想过。"

✅ **加分维度**（挑 1~2 个说）：性能、稳定性（监控/降级）、成本（资源利用率）、可维护性（重构/测试覆盖）、扩展性。

⚠️ **反例**：
- "没啥可优化的" —— 显得没追求、没深度思考。
- 暴露的是**致命设计缺陷**（比如"其实有超卖 bug 没修"）—— 暴露短板要选"可优化项"，不是"没做完的活"。

## ⚠️ 反例警示总集（深挖翻车四大死法）

| 死法 | 表现 | 后果 | 破解 |
|------|------|------|------|
| **吹太大被追问穿** | 简历写"负责整个高并发系统"，一问 QPS/幂等/分表全不知道 | 判定注水，全项目不可信 | 只写你真懂的，深度 > 头衔 |
| **数据编造对不上** | QPS 说几万但机器就 2 台；说 5000w 数据但没分表 | 现场心算就穿 | 数字提前算好，互相印证 |
| **只会用不懂原理** | "用了 Redis/Kafka"，问底层（持久化/分区/一致性）就哑 | 显得只是 API 调用工 | 用到的中间件至少懂核心原理一层 |
| **难点讲成流水账** | 只说"我做了 A、B、C"，没有为什么、没有权衡 | 平淡无深度 | 讲思考、试错、踩坑、trade-off |

> 🔑 底线：**简历上写的每个技术名词，都要准备好被追问到底层。** 不懂原理的就别往简历上写。

## ✅ 面试前准备清单（每个项目都过一遍）

对着简历里**每一个项目**，逐条能脱稿说出来：

- [ ] **是什么**：一句话说清业务背景 + 系统在解决什么问题。
- [ ] **我做了什么**：个人负责的模块，边界清晰（哪些是我、哪些是团队）。
- [ ] **难点**：1~2 个真·技术难点，能讲出"为什么难 + 我怎么想的"。
- [ ] **怎么解决**：方案对比 + 选型权衡 + 踩过的坑。
- [ ] **效果**：可量化数字（QPS/延迟/成本/稳定性提升 X%）。
- [ ] **一套容量数字**：QPS、数据量、机器数，能互相印证。
- [ ] **一个线上排查故事**：现象→工具→根因→解决→复盘，5 步完整。
- [ ] **可优化点**：主动想好 1~2 个短板 + 为什么没做。
- [ ] **选型能答权衡**：每个中间件为什么选它、缺点是什么、什么场景我会换。
- [ ] **每个技术名词的底层原理**：至少懂一层，经得起追问。
