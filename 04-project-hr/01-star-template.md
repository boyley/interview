# 01 · STAR 项目话术模板（STAR Method）

> 面试讲项目的万能结构：用 STAR 把「你做过什么」讲成「你解决了什么、贡献是什么、结果多好」。
> 本篇给：STAR 四段拆解 + 一个完整后端示例 + 亮点包装法 + 常见坑 + 准备清单 + 可套用骨架。

---

## 🎯 一、STAR 是什么

| 字母 | 含义 | 讲什么 | 篇幅占比 | 一句话要点 |
|------|------|--------|---------|-----------|
| **S** Situation | 背景 | 项目/业务背景、当时的处境 | ~15% | 交代清楚就行，别铺垫太久 |
| **T** Task | 任务/目标 | 你要解决的问题、目标指标 | ~15% | 把「问题」和「量化目标」说明白 |
| **A** Action | 行动 ★ | **你具体做了什么**、怎么做、为什么这么选 | ~50% | **重头戏**，突出「我」的技术决策 |
| **R** Result | 结果 | 量化收益 + 复盘沉淀 | ~20% | **必须有数字**：QPS/RT/成本/故障率 |

**为什么面试讲项目要用 STAR：**

```
不用 STAR：流水账 → "我做了个订单系统，用了 Redis、MQ、分库分表……"
                    面试官：所以呢？你到底解决了什么？你干了啥？

用 STAR：  结构清晰 → 面试官能顺着 S→T→A→R 跟上你的思路
          突出个人 → A 段全程说"我"，让"我"的贡献从团队里凸显
          可量化   → R 段甩数字，把"做过"升级成"做好了"
```

一句话：**STAR = 让面试官在 90 秒内听懂「你解决了一个有难度的问题，且效果可衡量」。**

## ⭐ 二、一个完整 STAR 示例（订单系统性能优化）

> 场景：电商订单查询接口高峰期变慢。逐段示范怎么说，右侧标注要点。

### S · 背景（约 20 秒）

> "我负责公司电商平台的订单服务。大促期间订单查询接口是核心链路，日常 QPS 约 2k，大促峰值能到 **1.5w**。当时这个接口在高峰期 **P99 从 200ms 恶化到 1.8s**，还偶发超时，直接影响用户下单后查看订单。"

- ✅ 要点：交代**你的角色**（我负责）+ **业务重要性**（核心链路）+ **量级**（QPS 有数字）
- ⚠️ 别只说"我们有个订单系统"——要让人感到「这事有分量」

### T · 任务/目标（约 15 秒）

> "我的目标是在不加机器的前提下，把这个接口 **P99 压到 300ms 以内**，同时保证大促峰值 1.5w QPS 稳定不超时。"

- ✅ 要点：**明确、可量化的目标**（P99 < 300ms，扛住 1.5w），并给约束（不加机器 = 有成本意识）
- ⚠️ 目标别虚（"让系统更快"）——给数字才显得你真扛过 KPI

### A · 行动 ★（约 60 秒，重头戏）

> "我先**定位瓶颈**：通过 Arthas + 慢 SQL 日志 + 链路追踪（SkyWalking）发现两个问题——
> ① 订单详情接口一次请求打了 **7 条 SQL**（订单主表 + 子表 + 商品 + 物流……），存在 N+1 查询；
> ② 热点商品订单反复查库，DB 单点压力大。
>
> 针对性做了三步：
> 1. **减少 DB 交互**：把 N+1 的循环查询改成批量 `IN` 查询 + 应用层内存组装，7 条 SQL 降到 2 条；
> 2. **加缓存**：引入 Redis 缓存订单详情，采用 Cache-Aside，设置差异化过期时间避免雪崩，热点数据命中率做到 ~90%；
> 3. **兜底与一致性**：更新订单时先改库再删缓存（延迟双删），并加了缓存空值 + 布隆过滤器防穿透。
>
> 中间踩过一个坑：延迟双删的第二次删除放在同步链路里拖慢了写接口，我改成用 MQ 异步删除，写接口 RT 又降回来了。"

- ✅ 要点：**排查工具具体**（Arthas/慢SQL/SkyWalking）→ **问题根因清楚**（N+1、热点）→ **方案分点 + 说明为什么这么选**（为什么 Cache-Aside、为什么延迟双删）
- ✅ 全程说"**我**发现 / **我**做了 / **我**改成"——个人贡献一眼可见
- ✅ 主动抛一个**踩坑 + 怎么修**，显得真实、有深度（面试官爱追问这里）
- ⚠️ 别只堆名词（"用了 Redis 和 MQ"），要讲**为什么用、怎么用、解决了什么**

### R · 结果（约 20 秒）

> "上线后 **P99 从 1.8s 降到 210ms**，DB QPS 下降约 **70%**，大促峰值 1.5w 稳定无超时，**没加一台机器**。之后我把这套「排查—缓存—一致性」的方法沉淀成团队的接口优化 checklist，另外两个慢接口复用后也达标了。"

- ✅ 要点：**多个量化维度**（P99、DB QPS、成本）+ **复盘沉淀**（形成方法论/被复用）
- ⚠️ 结果别只说"变快了"——`1.8s → 210ms`、`DB QPS -70%` 才有说服力

## 💡 三、如何把普通 CRUD 讲出技术含量

核心公式：**遇到什么问题 → 怎么排查 → 怎么解决（讲权衡）→ 效果（量化）**。

```
普通版（无亮点）          →   包装版（有技术含量）
─────────────────────────────────────────────────────────
"做了个订单列表查询"       →   "订单列表高峰 P99 1.8s，定位到 N+1 +
                              热点查询，用批量查询 + Redis 缓存 +
                              延迟双删治理，P99 降到 210ms，DB 压力 -70%"

"加了个缓存"              →   "为什么加缓存（DB 扛不住热点）→ 选 Cache-Aside
                              而非 Read-Through（为什么）→ 怎么防穿透/击穿/雪崩
                              → 命中率 90%，RT 从 X 降到 Y"

"用了 MQ 削峰"           →   "同步链路 3s 内要处理 1.5w 请求扛不住 →
                              MQ 异步化把核心链路缩到 3 步 → 削峰后
                              DB 写入平滑，峰值不再打挂"
```

**四步深挖模板（任何 CRUD 都能套）：**

| 步骤 | 你要说的 | 示例关键词 |
|------|---------|-----------|
| ① 问题 | 具体的性能/稳定性/一致性问题 + 量级 | "P99 1.8s"、"日增 500w 数据慢查询" |
| ② 排查 | 用了什么工具、定位到什么根因 | Arthas / 慢SQL / 链路追踪 / GC 日志 / 火焰图 |
| ③ 解决 | 方案 + **为什么这么选（权衡）** | 缓存策略、索引优化、异步化、分库分表 |
| ④ 效果 | 多维量化 + 沉淀 | QPS↑ / RT↓ / 成本↓ / 故障率↓ + 方法论 |

> 深度原理需要展开时链接深度库，例如缓存一致性详见 `../01-cheatsheet/06-redis.md`、
> 慢接口排查详见 `../01-cheatsheet/11-troubleshooting.md`。面试话术**点到为止 + 能被追问答得上**即可。

## ⚠️ 四、常见坑（踩一个掉一档）

| 坑 | 反面 | 正确做法 |
|----|------|---------|
| **只说"我们团队"** | "我们做了缓存优化" | A 段主语全用"**我**"：我定位、我设计、我实现（团队协作可以提，但贡献要归到你） |
| **堆技术名词不讲为什么** | "用了 Redis、MQ、ES、分库分表" | 每个技术都回答「解决什么问题、为什么选它、有什么代价」 |
| **结果不量化** | "性能提升明显" | 给数字：`P99 1.8s→210ms`、`QPS 2k→1.5w`、`成本 -3台机器` |
| **难点太假** | "难点是需求经常变" | 讲**技术难点**：一致性、热点、数据倾斜、分布式事务、锁竞争 |
| **背台词式流水账** | S 段铺垫 2 分钟还没到重点 | 控制节奏：A 段占一半，其他压缩 |
| **经不起追问** | 说了用布隆过滤器，追问"怎么算 bit 数组大小"答不上 | 只说你**真懂、能深挖**的点；不确定的别主动提 |

## ✅ 五、准备清单（面试前必做）

**每个写进简历的项目，提前准备 2~3 个能深挖的技术点：**

```
项目 A：订单系统
  ☐ 技术点 1（架构选型）：为什么用 Redis 而不是本地缓存？分库分表怎么分的？
  ☐ 技术点 2（难点）：缓存一致性怎么保证？延迟双删/Canal 订阅 binlog？
  ☐ 技术点 3（优化/踩坑）：N+1 治理、热点 key 打散、大促限流降级
  ☐ 能手画架构图：接入层 → 服务 → 缓存 → DB/MQ 的数据流
```

**逐项自检：**

- [ ] 每个技术点都能答出「**为什么这么选 + 有什么代价 + 换个方案会怎样**」
- [ ] 关键指标背下来：QPS、RT/P99、数据量级、机器数、优化前后对比
- [ ] 能**手画架构图**（白板/纸上 30 秒画出核心链路），并指出瓶颈点
- [ ] 准备好「**踩坑经历**」：真实、有技术含量、讲清怎么修的
- [ ] 预判追问：一致性、高可用、扩容、故障恢复、数据倾斜怎么办
- [ ] 敏感信息脱敏：QPS/数据量可模糊到量级，不泄露公司机密

## 📝 六、一页话术骨架模板（套用填空）

```
【S 背景】（20 秒）
我负责 ______（系统/模块），它在业务里是 ______（核心链路/重要性）。
当时的量级：QPS ______、数据量 ______。
遇到的问题：______（性能/稳定/一致性），表现为 ______（P99 X、故障 Y）。

【T 目标】（15 秒）
我的目标是在 ______（约束，如不加机器/两周内）把 ______ 指标做到 ______。

【A 行动】★（60 秒，重头戏，主语用"我"）
1. 定位：我用 ______（Arthas/慢SQL/链路追踪/GC日志）发现根因是 ______。
2. 方案：我做了
   ① ______（做法）——因为 ______（为什么这么选，权衡）；
   ② ______——对比 ______ 方案，我选它是因为 ______；
   ③ ______（兜底/一致性/降级）。
3. 踩坑：中间遇到 ______，我通过 ______ 解决。

【R 结果】（20 秒）
上线后 ______ 从 ______ 优化到 ______（多维量化：RT/QPS/成本/故障率）。
沉淀：我把这套方案总结成 ______（方法论/组件/规范），被 ______ 复用。
```

**背诵口诀：** 背景一句带过 → 目标给数字 → 行动讲"我怎么做+为什么" → 结果甩数据+沉淀。
