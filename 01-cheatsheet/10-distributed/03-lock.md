# 分布式锁 · Redis 锁的坑 · Redlock

> **核心理念：分布式锁要保证「互斥 + 防死锁 + 防误删 + 可靠」。Redis 锁的坑几乎都围绕这四点；单机锁有主从丢锁风险，Redlock 用多实例多数派提升可靠。**
> 答题方法：先立锁的要求 → 单机怎么做 → **有哪些坑(逐个 坑+解决)** → Redlock 改进 → 争议与选型。

## 一、分布式锁的基本要求（先立标准）

| 要求 | 说明 |
|---|---|
| **互斥** | 同一时刻只有一个客户端持有 |
| **防死锁** | 锁必须能**自动过期**（持有者宕机也能释放） |
| **防误删** | 只能删自己加的锁（用唯一 value 校验） |
| **可重入** | （可选）同一线程可重复获取 |
| **高可用高性能** | 加解锁快、Redis 挂部分仍可用 |

## 二、单机 Redis 分布式锁怎么做（基础）

1. **加锁**：`SET key uniqueValue NX PX 30000`（`NX` 保互斥、`PX` 保自动过期、一条命令原子）
2. **解锁**：Lua 脚本「比对 value 再 del」保证原子
   ```lua
   if redis.call('get', KEYS[1]) == ARGV[1] then
     return redis.call('del', KEYS[1])
   else return 0 end
   ```

> 命令层细节见 [../06-redis](../06-redis.md) 讲14「分布式锁的原子性」。

## 三、Redis 实现分布式锁有哪些坑？（★ 逐个：坑 + 解决）

| # | 坑 | 后果 | 解决 |
|---|---|---|---|
| 1 | **加锁非原子**：先 `SETNX` 再 `EXPIRE` 两条命令 | 中间宕机→没过期→**死锁** | 用 `SET NX PX` **一条命令**搞定 |
| 2 | **锁提前过期**：业务没执行完锁就到期 | 别人拿到锁→**并发进临界区** | **看门狗自动续期**（Redisson watchdog，默认 30s，每 10s 续） |
| 3 | **误删别人的锁**：A 锁过期，B 加锁，A 执行完把 B 的锁删了 | 互斥被破坏 | value 存**唯一标识**(UUID+线程)，删前**校验是自己的** |
| 4 | **删锁非原子**：先 `get` 判断再 `del`，两步之间锁过期被 B 拿到 | A 删了 B 的锁 | **Lua 脚本**把"判断+删除"合成原子 |
| 5 | **主从异步复制丢锁**：master 加锁成功但没同步就宕机，slave 升主无锁 | 两个客户端同时持锁 | **Redlock** 多实例多数派 / 改用 **ZooKeeper** |
| 6 | **不可重入**：同一线程重复获取 | **自己把自己锁死** | Redisson 可重入锁（Hash 记重入次数） |
| 7 | **时钟/GC 暂停**：长 GC(STW) 超过锁 TTL，锁已释放它却以为还持有 | 并发（Redlock 也有此问题） | **fencing token**(单调递增令牌) + 资源端校验 |
| 8 | **惊群 / 无限自旋**：大量客户端争锁疯狂重试 | CPU 飙高、Redis 压力大 | 重试**退避**、设获取超时、或用 ZK **顺序节点**避免惊群 |
| 9 | **锁粒度过大** | 并发度低 | 减小锁粒度（按 key 维度锁） |
| 10 | **异常没兜底**：加锁失败/异常没处理、解锁没放在 finally | 锁泄漏 / 逻辑错乱 | 解锁必放 `finally`；加锁失败要有降级 |

> **一句话工程建议：生产别自己造，直接用 Redisson** —— 它封装了 看门狗续期、可重入、Lua 原子解锁、Redlock，规避了上面绝大多数坑。

## 四、Redlock 算法（★ 针对坑 #5：多实例多数派）

> **前提**：部署 **N 个（通常 5 个）相互完全独立的 Redis master**（不是主从）。

**加锁步骤：**
1. 记录**当前时间 T1**
2. 用**相同 key/value + 过期时间**，依次向 N 个实例请求加锁，每次设**小超时**（防某实例卡住）
3. 统计成功数与总耗时 `T2 - T1`
4. **判定成功**：成功数 **≥ N/2+1（多数派）** 且 总耗时 **< 锁有效期**；实际有效期 = 初始有效期 − 加锁耗时
5. **失败回滚**：没拿到多数/超时 → 向**所有**实例解锁

**为什么更可靠**：不依赖主从复制，挂少数实例(5 挂 2)仍可用。

### ❓ 追问：请求打到「没有这把锁」的实例怎么办？

**先纠正前提**：Redlock **根本不用主从同步**。N 个实例**完全独立**、彼此**不复制、不同步**任何数据 —— 所以**不存在"同步没到那台"的问题**（"打到没同步的机器"是**单机主从**的坑，Redlock 正是为避开它才不用副本）。

- **锁怎么加**：不是"写一台再等同步"，而是**客户端亲自向 N 台每一台直接 `SET NX`**；"拿到锁" = 亲自在 **≥ N/2+1 台**都成功。
- **"某台没有这把锁"是正常的**：判定持锁看**"多数台拿到"**，不是"每台都有"；客户端**自己记得**在哪几台成功，解锁时向**所有 N 台**发解锁 —— 不靠"查某一台"判断持锁。
- **多数派为何保证互斥**：两个客户端不可能同时凑齐多数派 —— 两个多数派必在**至少一台重叠**，那台 `NX` 只让一个成功。
- **真正的隐患**（经典争议）：某台加锁实例**崩溃重启**（默认无 AOF fsync，锁丢失）→ 多数派被削弱 → 另一客户端借空出的实例凑齐新多数派 → **两客户端同时持锁**。**解决：延迟重启**（restart delay **>** 锁 TTL），让重启实例在锁自然过期前不参与投票；或开持久化。

## 五、Redlock 的争议（★ 加分：Martin vs antirez）

- **Martin 质疑**：① 依赖时钟(NTP 跳变→锁提前过期)；② GC/STW 长暂停→锁失效仍以为持有；结论：**正确性要求极高别用 Redlock**，要加 **fencing token + 资源端校验**。
- **antirez 回应**：工程实践可用，fencing token 也有前提。
- **实务选型**：
  - **效率锁**（偶尔重复可容忍）→ 单 Redis / Redlock 足够
  - **正确性锁**（绝不能重复，如扣款）→ fencing token + 资源端校验，或用 **ZooKeeper**

## 六、对比：Redis 锁 vs Redlock vs ZooKeeper 锁

| 方案 | 可靠性 | 性能 | 原理 | 适用 |
|---|---|---|---|---|
| **单 Redis 锁** | 中(主从会丢锁) | 高 | SET NX PX + Lua | 一般业务效率锁 |
| **Redlock** | 较高(多数派) | 中 | N 个独立实例投票 | 想用 Redis 但要更可靠 |
| **ZooKeeper 锁** | 高(共识/顺序节点) | 中低 | 临时顺序节点 + Watch | 强一致、正确性要求高 |

---

> **核心原则：Redis 锁的坑都绕着「原子性、误删、过期、主从丢锁、可重入」；单点用 `SET NX PX`+Lua+唯一value+看门狗（Redisson），要更可靠用 Redlock 多数派，强正确性用 fencing token 或 ZooKeeper。**
