# 08 · 计算机网络 速答（Computer Network）

> 面向后端/Java 面试的网络高频八股：TCP 三次握手四次挥手、HTTP/HTTPS、输入 URL 到页面全过程是三大必考。覆盖 ~25 个高频问，术语给英文，能背出口。

## 🔥 高频必背（Top 20）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | TCP 三次握手（Three-way Handshake）流程？ | ①Client→Server `SYN`(seq=x) ②Server→Client `SYN+ACK`(seq=y,ack=x+1) ③Client→Server `ACK`(ack=y+1)。目的：**双方各确认一次自己的发送和接收能力都正常**，并同步初始序列号 ISN。 |
| 2 | 为什么三次握手不能两次？ | ①**防止已失效的历史连接**（旧的延迟 SYN 到达会建立错误连接）②**同步双方初始序列号**（seq）需要一来一回各确认一次。两次无法确认 Client 的接收能力和 Server 的发送序列号。 |
| 3 | TCP 四次挥手（Four-way Wave）流程？ | ①Client→Server `FIN` ②Server→Client `ACK` ③Server→Client `FIN` ④Client→Server `ACK`。 |
| 4 | 为什么挥手要四次（握手三次）？ | TCP 是**全双工**，关闭要双向各关一次。Server 收到 FIN 先回 ACK（我知道了），但自己**可能还有数据要发**，发完才发自己的 FIN，所以 ACK 和 FIN **不能合并**，比握手多一步。 |
| 5 | TIME_WAIT 状态作用？为什么等 2MSL？ | 主动关闭方最后进入 TIME_WAIT，等 **2MSL**（Max Segment Lifetime）。作用：①**保证最后的 ACK 能到达**对方（丢了对方重发 FIN，我还能补发 ACK）②**让旧连接的残余报文在网络中消散**，不干扰用同一四元组的新连接。 |
| 6 | CLOSE_WAIT 大量堆积说明什么？ | 说明**被动关闭方（通常是我们的服务）收到 FIN 后没有 close() socket**——代码没关连接/连接池泄漏。是**程序 Bug**，不是网络问题，会耗尽 fd。 |
| 7 | TCP vs UDP？ | TCP：面向连接、可靠、有序、有流量/拥塞控制、字节流、慢，用于 HTTP/文件传输。UDP：无连接、不可靠、无序、快、开销小、报文，用于 DNS/视频/游戏/DNS/直播。 |
| 8 | TCP 靠什么保证可靠传输？ | **序列号+确认应答(ACK)**、**超时重传**、**滑动窗口**、**流量控制**、**拥塞控制**、校验和、失序重排、去重。 |
| 9 | 流量控制 vs 拥塞控制？ | **流量控制**：防止发送方压垮**接收方**，靠接收窗口 rwnd（接收方通告）。**拥塞控制**：防止压垮**网络**，靠拥塞窗口 cwnd（发送方根据丢包动态调）。实际发送窗口 = min(rwnd, cwnd)。 |
| 10 | 拥塞控制四个算法？ | **慢启动**（cwnd 指数增长）→ 到阈值 ssthresh 转**拥塞避免**（线性增长）→ 收到 3 个重复 ACK 触发**快重传**+**快恢复**（ssthresh 减半，cwnd 减半后线性增，不回到 1）。 |
| 11 | HTTP 常用方法？ | `GET`(查) `POST`(增/提交) `PUT`(全量改) `PATCH`(部分改) `DELETE`(删) `HEAD`(只要头) `OPTIONS`(预检/探测能力)。 |
| 12 | GET vs POST？ | GET 参数在 URL、有长度限制、**幂等安全**、可缓存可收藏、明文；POST 参数在 body、无长度限制、**非幂等**、不缓存、更适合提交/敏感数据。本质都是 TCP，安全性都靠 HTTPS。 |
| 13 | 常见状态码？ | 2xx 成功、3xx 重定向、4xx 客户端错、5xx 服务端错。背：`200` OK、`301` 永久重定向、`302` 临时、`304` 未改用缓存、`400` 请求错、`401` 未认证、`403` 无权限、`404` 找不到、`500` 服务器内部错、`502` 网关错、`504` 网关超时。 |
| 14 | 301 vs 302？302 vs 307？ | 301 永久（浏览器缓存、SEO 转移权重）；302 临时（不缓存）。302 历史上会把 POST 改成 GET，**307** 临时且**保持原方法**。 |
| 15 | 302 vs 304？ | **易混**：302 是重定向（换地址）；304 Not Modified 是**协商缓存命中**，让浏览器用本地缓存，无 body。 |
| 16 | HTTP/1.0 → 1.1 → 2 → 3 关键区别？ | 1.0 短连接；**1.1** 默认 `keep-alive` 长连接 + 管线化（有队头阻塞）；**2** 二进制分帧 + **多路复用**（一个连接并发多请求）+ 头压缩 HPACK + Server Push；**3** 底层换 **QUIC(基于 UDP)**，解决 TCP 层队头阻塞、0-RTT 建连。 |
| 17 | HTTP vs HTTPS？ | HTTPS = HTTP + **TLS/SSL**。加密（防窃听）、认证（证书防冒充）、完整性（防篡改）。端口 80 → 443，需 CA 证书，握手有额外开销。 |
| 18 | Cookie vs Session？ | Cookie 存**客户端**（浏览器）、随请求自动带；Session 存**服务端**、靠 Cookie 里的 `sessionId` 关联。Session 有状态、集群需共享（Redis）；Cookie 有大小限制、可被篡改。 |
| 19 | Session vs Token/JWT？ | Session 服务端存状态、扩展要共享存储；**Token/JWT** 服务端**无状态**、信息签名后放客户端、天然适合分布式/跨域/移动端。JWT 缺点：**无法主动失效**、体积大。 |
| 20 | 输入 URL 到页面展示，发生了什么？（必考） | ①**DNS 解析**域名→IP ②与 IP **建 TCP 连接**（三次握手，HTTPS 再 TLS 握手）③发 **HTTP 请求** ④服务器处理返回响应 ⑤浏览器**解析渲染**（HTML→DOM、CSS→CSSOM→渲染树→布局→绘制）⑥**关闭/复用连接**（四次挥手）。 |

## 📌 展开速答

**Q：详细讲一下 TCP 三次握手，以及为什么是三次？**
状态流转：Client 发 `SYN(seq=x)` 进入 `SYN_SENT`；Server 回 `SYN+ACK(seq=y, ack=x+1)` 进入 `SYN_RCVD`；Client 回 `ACK(ack=y+1)`，双方进入 `ESTABLISHED`。
为什么必须三次、两次不行：核心是**双方都要确认「对方能收、对方能发」，并同步初始序列号 ISN**。
- 第一次：Server 确认了「Client 能发、Server 能收」。
- 第二次：Client 确认了「Server 能发能收、Client 能收发」。
- 第三次：Server 才确认「Client 能收」。
若只有两次，Server 无法确认 Client 是否收到自己的 SYN，且**一个早已失效的历史 SYN**（旧连接延迟到达）会让 Server 单方面建立一个不存在的连接，浪费资源。三次握手让 Client 有机会用 ACK 否认历史连接。
🔗 深挖：SYN Flood 攻击就是伪造大量 SYN 占满半连接队列，防御用 SYN Cookie。

**Q：详细讲四次挥手，TIME_WAIT 为什么要等 2MSL？**
流程：主动方发 `FIN` 进 `FIN_WAIT_1`；被动方回 `ACK` 进 `CLOSE_WAIT`（主动方进 `FIN_WAIT_2`）；被动方数据发完后发 `FIN` 进 `LAST_ACK`；主动方回 `ACK` 进 **`TIME_WAIT`**，等 2MSL 后才真正 `CLOSED`；被动方收到 ACK 立即 `CLOSED`。
为什么四次：ACK 和 FIN 分开发，因为被动方收到 FIN 时**可能还有数据没发完**（半关闭），所以先 ACK、数据发完再 FIN。
TIME_WAIT 等 **2MSL** 的两个原因：①**如果最后的 ACK 丢了**，对方会重发 FIN，主动方还在 TIME_WAIT 就能重发 ACK；如果直接关了，对方会一直重发 FIN 收不到响应。②**让本连接的所有残留报文在网络中自然消亡**（一个 MSL 去、一个 MSL 回），避免它们串到用相同四元组的新连接里。

**Q：CLOSE_WAIT 和 TIME_WAIT 大量堆积分别怎么排查？**
- **TIME_WAIT 多**：出现在**主动关闭方**，高并发短连接下正常（每个连接都留 2MSL）。优化：开启 `tcp_tw_reuse`、用长连接/连接池、调 `net.ipv4.tcp_max_tw_buckets`。一般不致命。
- **CLOSE_WAIT 多**：出现在**被动关闭方**，几乎一定是**代码 Bug**——收到对方 FIN 却没调 `close()`（连接没释放、异常路径漏关、连接池配置错）。会持续占用 fd 直到耗尽导致「Too many open files」。排查方向是代码里的连接释放。

**Q：TCP 拥塞控制完整讲一遍？**
维护一个拥塞窗口 cwnd 和阈值 ssthresh：
1. **慢启动 (Slow Start)**：cwnd 从 1 开始，每个 RTT **翻倍**（指数增长），快速探测带宽。
2. 到达 `ssthresh` 后进入**拥塞避免 (Congestion Avoidance)**：每个 RTT **+1**（线性增长），谨慎试探。
3. **超时**（严重丢包）：ssthresh 设为 cwnd/2，cwnd **回到 1**，重新慢启动。
4. **收到 3 个重复 ACK**（个别丢包）：触发**快重传 (Fast Retransmit)** 立即重传丢的段，不等超时；并进入**快恢复 (Fast Recovery)**：ssthresh = cwnd/2，cwnd = 新 ssthresh（不回到 1），继续拥塞避免。
一句话记：**指数涨 → 线性涨 → 丢包就砍半**；轻微丢包砍半（快恢复），严重超时归 1。

**Q：HTTPS 的 TLS 握手过程？（高频深挖）**
目的：用**非对称加密安全地协商出一个对称密钥**，之后用对称加密传数据（快）。以 RSA 简化流程：
1. Client Hello：客户端发支持的 TLS 版本、加密套件、随机数 Random1。
2. Server Hello：服务端选定套件，发随机数 Random2 + **CA 证书（含公钥）**。
3. **客户端验证证书**：用内置 CA 公钥验签，确认服务端身份（防中间人）。
4. 客户端生成随机数 **Pre-Master**，用**证书公钥加密**后发给服务端。
5. 双方用 Random1 + Random2 + Pre-Master 各自算出**相同的对称会话密钥 (Session Key)**。
6. 之后通信全部用这个对称密钥加密。
要点：**非对称加密只用于握手交换密钥，数据传输用对称加密**（对称快得多）。TLS 1.3 简化到 **1-RTT**（甚至 0-RTT），并改用 ECDHE 前向保密。
🔗 为什么不全程非对称：非对称加解密慢，只用来安全地传对称密钥这一步。

**Q：详细讲「输入 URL 到页面展示」全过程？（几乎必考）**
1. **URL 解析**：浏览器拆出协议、域名、端口、路径。
2. **DNS 解析**：浏览器缓存 → 系统 hosts → 本地 DNS 服务器 → 根/顶级/权威域名服务器**递归+迭代**查询，拿到 IP。有 CDN 则返回最近节点 IP。
3. **建立 TCP 连接**：与目标 IP:443 做**三次握手**；HTTPS 再做 **TLS 握手**协商密钥。
4. **发送 HTTP 请求**：构造请求行 + 请求头（含 Cookie、Host）+ body。
5. **服务器处理并响应**：Nginx/网关 → 应用服务 → 返回状态码 + 响应头 + HTML。
6. **浏览器渲染**：解析 HTML 构建 **DOM 树**、解析 CSS 构建 **CSSOM**，合成**渲染树 (Render Tree)** → **布局 Layout(回流)** → **绘制 Paint** → **合成显示**；遇到 JS 会阻塞解析、遇到 img/css/js 再发资源请求。
7. **连接处理**：HTTP/1.1 keep-alive 复用连接，页面关闭/超时后**四次挥手**断开。
答题技巧：**能把 DNS → TCP → TLS → HTTP → 渲染 → 断开这条主线串下来**，每步能展开一两句就是好答案。

**Q：DNS 解析的完整过程？**
按缓存优先级逐级查：①**浏览器 DNS 缓存** ②**操作系统缓存 / hosts 文件** ③**本地 DNS 服务器（LDNS，一般是运营商）**。LDNS 没有则**迭代查询**：**根域名服务器**（返回 `.com` 顶级服务器地址）→ **顶级域 TLD 服务器**（返回权威服务器地址）→ **权威域名服务器**（返回真正 IP）。全程 UDP 53 端口为主。有 CDN 时权威服务器返回 CNAME 指向 CDN 调度，最终给最近节点 IP。

**Q：跨域 (CORS) 是什么？怎么解决？**
浏览器**同源策略**：协议+域名+端口任一不同即跨域，会拦截 AJAX 响应（不是不发请求，是拦响应）。解决：①**CORS**——服务端返回 `Access-Control-Allow-Origin` 等响应头放行（主流）；复杂请求前浏览器先发 `OPTIONS` **预检**。②网关/Nginx 反向代理让前后端同源。③开发期 `proxy`。④历史方案 JSONP（只支持 GET，已少用）。

**Q：OSI 七层 vs TCP/IP 四层，各层协议？**
| OSI 七层 | TCP/IP 四层 | 典型协议 |
|---|---|---|
| 应用层/表示层/会话层 | **应用层** | HTTP、HTTPS、DNS、FTP、SMTP |
| 传输层 | **传输层** | **TCP、UDP** |
| 网络层 | **网络层** | **IP、ICMP、ARP**、路由 |
| 数据链路层/物理层 | **网络接口层** | 以太网、MAC、PPP |
记忆口诀（自上而下）：**应表会传网数物**。面试重点是传输层（TCP/UDP）和网络层（IP）。

## ⚠️ 易错 / 反问加分

- ⚠️ **握手三次、挥手四次，别记反**；核心区别是挥手的 ACK 和 FIN 因「半关闭、还有数据要发」不能合并。
- ⚠️ **TIME_WAIT 在主动关闭方，CLOSE_WAIT 在被动关闭方**——面试爱考「线上大量 CLOSE_WAIT 怎么办」，答「代码没 close，是程序 Bug」直接命中。
- ⚠️ **302 是重定向、304 是缓存命中**，别混。304 无响应体，配合 `If-Modified-Since`/`ETag` 协商缓存。
- ⚠️ **GET/POST 的「安全」不是加密**——两者都是明文，安全靠 HTTPS；GET 更该强调「幂等、可缓存」而非「不安全」。
- ✅ **加分**：HTTPS 主动补一句「非对称加密只用来协商对称密钥，数据用对称加密，因为非对称太慢」，显示懂原理。
- ✅ **加分**：讲 HTTP/2 主动提「多路复用解决了 1.1 的队头阻塞，但 TCP 层队头阻塞还在，HTTP/3 用 QUIC(UDP) 才彻底解决」。
- ✅ **加分**：URL 全过程结尾补「CDN 就在 DNS 这步返回最近节点 IP」「静态资源走强/协商缓存」，体现工程视角。
- ✅ **加分**：分布式登录场景主动对比「Session 要共享存储(Redis)，JWT 无状态但难主动失效，可用短 Token+Refresh Token 折中」。
