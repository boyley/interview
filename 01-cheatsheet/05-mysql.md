## 二、必背核心结论（背这些就够应付大部分八股）

1. **一条查询的执行流程**：连接器 → 查询缓存(8.0删除) → 分析器 → 优化器 → 执行器 → 存储引擎。
2. **两阶段提交**：写 redo log(prepare) → 写 binlog → 提交 redo log(commit)。崩溃恢复时，redo 有 commit 直接提交；只有 prepare 则看 binlog 是否完整，完整则提交、否则回滚。目的是保证 redo 与 binlog 逻辑一致。
3. **双 1 配置**：`innodb_flush_log_at_trx_commit=1`（redo 每次事务落盘）、`sync_binlog=1`（binlog 每次事务落盘）——保证 crash 不丢数据。
4. **回表**：普通（二级）索引的叶子节点存主键值，查非索引列需拿主键再回主键索引树查一次；**覆盖索引**（索引已含所有要查的列）可避免回表。
5. **最左前缀原则**：联合索引 (a,b,c) 可命中 a、(a,b)、(a,b,c) 的前缀匹配；调整索引顺序常能少维护一个索引。
6. **MVCC**：InnoDB 通过 undo log 形成行的多版本 + 一致性视图实现可重复读，读不加锁、读写不冲突。**RR 在事务启动时建视图，RC 在每条语句开始时建视图**。
7. **当前读 vs 快照读**：普通 SELECT 是快照读（走 MVCC）；`update`/`delete`/`select ... for update`/`lock in share mode` 是当前读（读最新版并加锁）。
8. **间隙锁(gap lock) + 行锁 = next-key lock**，用于在可重复读级别下解决幻读（幻读专指当前读时读到别的事务新插入的行）。
9. **两阶段锁协议**：行锁在需要时加、事务提交时才释放 → 把最可能冲突、影响并发度的锁尽量往后放。
10. **索引失效三典型**：① 条件字段用了函数；② 隐式类型转换（字符串字段传数字）；③ 隐式字符编码转换（join 两表字符集不同）。

### 图3 · 更新语句 + 两阶段提交 ★（面试超高频）— 讲02

> 例：`update T set c=c+1 where id=2;`

```
① 找到 id=2 这行            （在内存就取，不在就从磁盘读进内存）
② 执行器把 c 加 1           → 更新内存数据页(脏页)
③ 引擎写 redo log          [redo]  状态 = prepare（准备）   ┐
④ 执行器写 binlog          [binlog] 写入磁盘                ├ 两阶段提交
⑤ 引擎提交，改 redo log     [redo]  状态 = commit（提交）✅  ┘
```
**为什么要两阶段？** redo 和 binlog 是两份独立日志。若不用两阶段提交，中间一崩溃就会「一份有、一份没有」→ 原库和用 binlog 恢复出来的库不一致（主从也不一致）。两阶段提交保证两份日志「要么都有、要么都没有」。

### 图4 · 崩溃恢复：提交还是回滚？★（你问过的那题）— 讲02/15

```
                重启，扫描 redo log 中的事务
                          │
              redo log 里有 commit 标记吗？
                 ┌────────┴─────────┐
              有 commit           只有 prepare
                 │                    │
               ✅提交         对应的 binlog 完整吗？
                              ┌───────┴────────┐
                            完整              不完整
                             │                  │
                          ✅提交              ❌回滚
                       (保证主从一致)
```
**一句话背**：「redo 到 commit 一定提交；停在 prepare，就看 binlog 全不全——**全就提交，不全就回滚**。」binlog 完整判断：statement 格式看结尾 COMMIT、row 格式看结尾 XID event；redo 的 prepare 存了同一个 XID 用来关联 binlog。

### 图5 · redo log 循环写 — 讲02

```
   逻辑上是一个环（写到末尾回到开头）：ib_logfile0~3 循环用

   ┌──────────────────────────────────────────────────┐
   │███████████████│                                    │
   └──────────────────────────────────────────────────┘
     ▲已写入·待刷盘  ▲                                  
   checkpoint      write pos              空闲·可写空间
   (擦除位置,        (当前写入位置)
    往后推=刷脏页)
```
**记忆点**：`write pos` 追上 `checkpoint` = redo 写满了 → 必须先刷一批脏页把 checkpoint 往前推、腾出空间，**此时更新被迫暂停**（讲12「抖一下」的原因之一）。

### 图6 · WAL 与刷脏页（为什么快 + 为什么抖）— 讲02/12

```
更新请求 ─→ 改内存数据页(变脏页) + 顺序写 redo log[redo] ─→ ✅返回成功
                                    （顺序写,不随机寻址,所以快）

—— 之后后台在这些时机把脏页真正刷回磁盘(flush) ——
  ❌redo写满(被迫刷)  ❌内存不足淘汰脏页   系统空闲(从容刷)   正常关机(全刷)
       ↑ 前两种会阻塞前台更新 = 「抖一下」的来源
```
**调优**：`innodb_io_capacity` 设成磁盘真实 IOPS（让 InnoDB 敢放心刷）；控制脏页比例 `innodb_max_dirty_pages_pct`。

### 图7 · B+树索引：聚簇/二级索引/回表 ★（超高频）— 讲04/05

```
   主键索引(聚簇索引)                 二级索引(如 k)
   叶子 = 整行数据                    叶子 = 主键值
        [非叶]                           [非叶]
       /  |  \                          /    \
  id=1  id=5  id=8                   k=3     k=6
  整行  整行  整行                  →id=5   →id=8
         ▲                            │
         └──────── 回表 ◀─────────────┘
      拿到主键 id=5，再回主键树查一次拿整行
```
- **回表**：二级索引只能查到主键值，要别的列就得回主键树再查一次。
- **覆盖索引**：要查的列已在索引里（如只查 id/k）→ 免回表，性能大幅提升。
- **最左前缀**：联合索引 (a,b,c) 命中 a、(a,b)、(a,b,c) 这些「从左开始」的前缀。
- **索引下推 ICP**：查找时就先过滤不满足的行，减少回表次数。

**记忆点**：二级索引叶子存主键 → 要别的列就回表 → 覆盖索引免回表。也是**推荐自增主键**的原因：主键短则二级索引小，自增避免页分裂。

### 图8 · MVCC：版本链 + 一致性视图 ★（超高频）— 讲03/08

```
   一行数据的版本链（右新左旧，靠 undo log 往回退）：

   V1(c=10,早已提交) ◀── V2(c=20) ◀── V3(c=30,事务B未提交)
                          ▲
                          │ 事务A(可重复读)启动时建视图
                          └─ 整个事务只认 V2 → 始终看到 c=20
```
- **可见性口诀**：本事务自己改的、以及**视图创建前已提交**的版本可见；**视图创建后才提交的、还没提交的**不可见，就顺 undo 往回找旧版本。
- **RR vs RC 差别只在建视图时机**：RR 在**事务启动时**建一次、整个事务复用（可重复读）；RC 在**每条 SQL 开始时**都重建（读到最新已提交）。
- **快照读 vs 当前读**：普通 select 走 MVCC 快照；`update/delete/select...for update/lock in share mode` 是当前读，读最新版并加锁。

<details>
<summary><b>📖 展开：MVCC 从零系统整理（还没懂就看这个，一层层搭）</b></summary>

> **预告一句话**：MVCC = 同一行数据保留多个历史版本，每个事务读取时，只读到"在它启动那一刻就已经定格的那份数据"。下面全是在解释这句怎么做到的。

#### 第 1 步：它要解决什么问题

一行数据 `(id=1, 余额=100)`，两个事务同时操作：

```
事务A：begin; 读余额... 干点别的... 再读余额;
事务B：            begin; 把余额改成 200; commit;   ← 插在A两次读中间
```

若数据库只存一个当前值，A 两次读就是 `100` 和 `200`（**不可重复读**）。我们想要：A 整个事务期间读到的始终是它开始那一刻的样子（100）。用加锁不让 B 改太慢——**MVCC 目标：不加锁也让读写互不打架**。

#### 第 2 步：核心思路——改数据不覆盖旧值，留一份历史

每次修改新增一个版本、旧版本还留着，一行数据变成一条**版本链**（像文档历史记录 / git 提交历史）：

```
余额=100  ←  余额=200  ←  余额=300     （每次改都往链上加新版本）
最老                       最新
```

写的人加新版本，读的人找"自己该读的那一版"，两边不冲突。剩下唯一问题：**读的人怎么知道读哪一版？**

#### 第 3 步：给事务和版本编号

| 编号 | 谁的 | 规则 |
|---|---|---|
| **事务ID（trx_id）** | 每个事务 | 启动时申请，**严格递增**（越晚开始 ID 越大） |
| **row trx_id** | 每个数据版本 | 记录"这版是哪个事务改的"（版本的作者签名） |

```
(余额=100, 作者10) ← (余额=200, 作者17) ← (余额=300, 作者22)
```

#### 第 4 步：黄金规则——你"应该"看到什么（灵魂，先记这个）

> **你这个事务应该看到的世界 = "在你启动那一瞬间，所有已经提交完成的修改"。**
> 之后发生的都与你无关：别人**没提交**的看不到、别人在你**启动后才提交**的看不到、在你**启动后才开始**的事务更看不到。
> **唯一例外：你自己做的修改，自己永远看得到。**

翻译成读版本链：**从最新往老里找，找到第一个"作者在你启动前就已提交"的版本，就读它。** 原理到这就讲完了，第 5、6 步只是"如何高效判断作者提交没提交"。

#### 第 5 步：怎么落实——拍一张"快照名单"(read-view)

事务启动时拍快照，核心就记一样东西：

> **m_ids：我启动这一刻，"已启动但还没提交"的事务ID名单**（拍下就冻结，整个事务不变）。

遇到任何版本，看作者ID：**在名单里 = 我启动时它没提交 = 看不到**；不在名单里且是我启动前就存在的事务 = 已提交 = 看得到。这就是第 4 步的落地。

#### 第 6 步：三分法——低水位 / 高水位（只是加速判断）

| 名字 | 定义 | 数值 |
|---|---|---|
| **低水位** | 名单 m_ids 里**最小**的事务ID | 小 |
| **高水位** | 系统**已创建的最大事务ID + 1**（下一个将分配的ID） | 大 |

```
  小 ←──────────────── 事务ID ────────────────→ 大
  ┌──────────┬──────────────────────┬────────────┐
  │  绿区     │        黄区           │   红区      │
  │ 一定可见  │   要查名单才知道      │  一定不可见 │
  └──────────┴──────────────────────┴────────────┘
             ▲                       ▲
           低水位                   高水位
```

| 作者ID落在 | 为什么能这样判断 | 结果 |
|---|---|---|
| **绿区**（< 低水位） | 比"最小活跃事务"还早，不可能还活着 → 必然早提交 | ✅ 可见（免查名单） |
| **红区**（≥ 高水位） | 比"下一个待分配ID"还大 → 我启动时它还没出生 = 未来事务 | ❌ 不可见（免查名单） |
| **黄区**（低水位≤ID<高水位） | 这段事务都已存在，但有的提交有的没，数值分不清 | 🔍 查名单：在名单=❌；不在=✅ |

> 低/高水位不高深，**就是"快速判断作者提交没提交"的两条捷径**，本质还是第 4 步那句话。

#### 第 7 步：完整走一遍（一个故事到底）

主角 **事务20**（只读，反复查余额）。启动瞬间快照：`m_ids={17,20}`、低水位=17、高水位=21。数轴：`绿区<17 ｜ 黄区17~20 ｜ 红区≥21`。

版本链（新→旧）：`(300,作者22) ← (200,作者17) ← (100,作者10)`

事务20 执行 `select 余额`，从新往老判断：

| 顺序 | 版本(值/作者) | 事务20 判断 | 动作 |
|---|---|---|---|
| ① | 300 / 作者22 | 22 ≥ 高水位21 → 红区 → 未来事务 | ❌ 跳过 |
| ② | 200 / 作者17 | 17 黄区且**在名单** → 没提交 | ❌ 跳过 |
| ③ | **100 / 作者10** | 10 < 低水位17 → 绿区 → 早已提交 | ✅ **停！返回 100** |

结论：事务20 读到 **100**。哪怕 17、22 改成了 200、300，事务20 整个生命周期都读到 100 → **可重复读**。（补充：若遇到作者是 20 自己的版本，直接可见。）

> **注意区分两个事务**：判断"看不到"时，主语是**正在查询的事务20**（观察者），它看不到的是**另一个事务（如17）生成的那版数据**。视图是谁建的，"看得到/看不到"就是站在谁的角度说。

#### 第 8 步：RR vs RC，只差"拍快照的时机"

| 隔离级别 | read-view 何时拍 | 效果 |
|---|---|---|
| **可重复读 RR** | 事务启动时拍一次，整事务复用 | 每次读一致（可重复读）；MySQL 默认 |
| **读提交 RC** | 每条 SQL 开始时都重拍 | 每句读到最新已提交（会不可重复读）；Oracle 默认 |

#### 第 9 步：例外——"当前读"不走快照

| 类型 | 语句 | 读哪版 |
|---|---|---|
| **快照读** | 普通 `select` | 按上面规则找可见历史版本 |
| **当前读** | `update`/`delete`/`select...for update`/`lock in share mode` | 直接读最新已提交版本并加锁 |

**为什么更新必须当前读**：`update 余额=余额+1` 必须基于当前真实最新值去加，否则覆盖别人刚提交的修改（丢更新）。

#### 总纲（背这段）

> ① **多版本**：改不覆盖，旧版本留着串成链（每版有作者 row trx_id）。② **黄金规则**：只看到"启动那一刻已提交的数据"+"自己的修改"。③ **read-view**：启动时拍快照名单 m_ids，判断"某版作者我启动时提交没"。④ **低/高水位**：切三段加速——绿区一定可见、红区一定不可见、黄区查名单。⑤ **读版本链**：从新到旧，第一个可见版本就返回它的值。⑥ **RR/RC**：只差拍快照时机。⑦ **当前读例外**：update/for update 不走快照，读最新并加锁。

### 图9 · 间隙锁 + 行锁 = next-key lock ★（可重复读专属）— 讲20/21

```
   数轴上已有记录：  id=5 ──────── id=10 ──────── id=15

   select ... where id=10 for update 在 RR 下加锁：
                    ├════════════════┤
                    (5 ────间隙──── 10]     ← next-key lock (5,10]
                    间隙锁(5,10) + 行锁10
   效果：区间内不能插入 id=7 等新行，防止「幻读」
```
- **next-key lock = 间隙锁(锁开区间) + 行锁(锁记录本身)**，前开后闭区间 `(5,10]`。
- **加锁规则（背）**：①基本单位是 next-key lock（前开后闭）；②访问到的对象才加锁；③等值查询命中唯一索引 → 退化成**行锁**；④等值查询向右遍历到第一个不满足的值 → 退化成**间隙锁**。间隙锁之间不互斥，但会阻塞插入。

### 图10 · 主从复制流程 — 讲24

```
   主库 Master                       备库 Slave
   ┌──────────────┐                 ┌──────────────────┐
   │ 执行事务       │                 │ IO线程(接收binlog) │
   │   ↓          │                 │   ↓               │
   │ 写 binlog     │ ──binlog──▶     │ relay log(中转)    │
   │   ↓          │   dump线程推送    │   ↓               │
   │ dump线程      │                 │ SQL线程(重放→改数据)│
   └──────────────┘                 └──────────────────┘
```
- **线程数**：主库 1 个 dump 线程；备库 2 个线程（IO 收 + SQL 放）。
- **binlog 三格式**：statement(记SQL,省空间但可能不一致)、row(记改动前后行,最安全,主流)、mixed(自动二选一)。
- **主备延迟**：SQL 线程单线程重放跟不上 → 用**并行复制**（按库/按表/按行/WriteSet）。防**过期读**：强制走主库、判断无延迟、等主库位点/等 GTID、semi-sync 半同步。

## 四、逐讲精要（01-08 基础篇）

## 01 | 基础架构：一条SQL查询语句是如何执行的？

**一句话核心**：MySQL 分为 Server 层和存储引擎层，一条查询语句依次经过连接器、（查询缓存）、分析器、优化器、执行器，最终调用存储引擎接口取数据。

**关键知识点**
- MySQL 逻辑架构分两层：**Server 层**（连接器、查询缓存、分析器、优化器、执行器，以及所有内置函数、存储过程、触发器、视图等跨引擎功能）和**存储引擎层**（负责数据存取，插件式，支持 InnoDB/MyISAM/Memory 等）。
- **连接器**：负责建立连接、认证身份、获取并缓存权限。连接建立后权限就固定，中途改权限不影响已有连接。长连接省去建连开销但内存易涨（临时内存挂在连接对象上），可定期断开或用 `mysql_reset_connection` 重置。
- **查询缓存**：key 是 SQL、value 是结果。弊大于利，因为一旦表有更新，该表所有查询缓存全部失效，命中率低。**MySQL 8.0 已彻底删除查询缓存功能**。
- **分析器**：做词法分析（识别关键字、表名、列名）和语法分析（是否符合语法）。表/列是否存在的校验（语义分析）也在此阶段——课后题答案即"分析器"。
- **优化器**：在多个索引时决定用哪个索引，多表 join 时决定连接顺序，确定执行方案。
- **执行器**：先做权限校验（precheck），再根据引擎接口逐行取数据、判断条件、组装结果集。慢查询日志里的 `rows_examined` 就是执行器调用引擎取行时累加的。

**面试高频问答**
- **Q：** 一条 SQL 查询语句在 MySQL 内部的完整执行流程？
  **A：** 连接器（建连+鉴权）→ 查询缓存（8.0 已删）→ 分析器（词法/语法/语义解析）→ 优化器（选索引、定连接顺序）→ 执行器（鉴权+调引擎接口逐行取数据）→ 存储引擎返回结果。
- **Q：** 为什么 8.0 要删掉查询缓存？
  **A：** 缓存失效太频繁，只要表有任意更新，该表全部查询缓存立即清空，更新频繁的库命中率极低，收益远小于维护成本；只适合极少更新的静态配置表。
- **Q：** Server 层和存储引擎层各自负责什么？
  **A：** Server 层实现连接、解析、优化、执行、内置函数及跨引擎功能；存储引擎层负责具体数据的存储与提取，可插拔（InnoDB 从 5.5.5 起为默认引擎）。
- **Q：** 长连接有什么隐患，怎么解决？
  **A：** 执行中的临时内存挂在连接对象上、连接断开才释放，长连接累积会导致内存暴涨甚至被 OOM 杀掉。解决：定期断开长连接；或 5.7+ 大操作后执行 `mysql_reset_connection` 重置连接资源（无需重连和重新鉴权）。

## 02 | 日志系统：一条SQL更新语句是如何执行的？

**一句话核心**：更新语句除了走查询那套流程外，还涉及 redo log 和 binlog 两个日志，通过 WAL 和两阶段提交保证 crash-safe 与两份日志的逻辑一致。

**关键知识点**
- **WAL（Write-Ahead Logging）**：先写日志、再写磁盘。更新时 InnoDB 先把改动写入 redo log 并更新内存即算完成，空闲时再把数据刷回磁盘，避免每次更新都随机 IO 找磁盘位置。
- **redo log（重做日志）**：InnoDB 引擎特有，**物理日志**，记录"在某个数据页上做了什么修改"。大小固定、循环写（write pos 写入位置、checkpoint 擦除位置，追上就得先刷盘腾空间）。提供 **crash-safe** 能力：异常重启后已提交的记录不丢。
- **binlog（归档日志）**：Server 层实现，所有引擎都能用，**逻辑日志**，记录语句原始逻辑（statement 记 SQL；row 记录改动前后的行内容）。**追加写**、不覆盖旧日志，可长期归档，用于备份和主从复制。
- **redo log vs binlog 三点区别**：① redo 是 InnoDB 特有，binlog 是 Server 层所有引擎可用；② redo 是物理日志，binlog 是逻辑日志；③ redo 循环写空间会用完，binlog 追加写可持续保存。
- **update 内部流程**：取行（在内存直接取，否则从磁盘读入）→ 执行器改值 → 引擎写内存并写 redo log 置 **prepare** 态 → 执行器写 binlog 落盘 → 引擎提交，redo log 改 **commit** 态。
- **两阶段提交**：redo log 的写入拆成 prepare 和 commit 两步，中间写 binlog，目的是让 redo log 与 binlog 逻辑一致。
- 关键参数：`innodb_flush_log_at_trx_commit=1`（每次事务 redo log 直接落盘）、`sync_binlog=1`（每次事务 binlog 落盘），双 1 保证崩溃不丢数据。
- **数据恢复**：全量备份 + 重放 binlog，可恢复到任意时间点，也用于扩容搭建备库。

**面试高频问答**
- **Q：** redo log 和 binlog 有什么区别？
  **A：** redo 是 InnoDB 特有的物理日志、循环写、用于 crash-safe；binlog 是 Server 层逻辑日志、追加写、用于归档/备份/主从复制。物理 vs 逻辑、循环 vs 追加、引擎 vs Server 层是三大区别。
- **Q：** 什么是两阶段提交？为什么需要它？
  **A：** 把 redo log 写入拆成 prepare 和 commit，中间夹着写 binlog。因为 redo 和 binlog 是两个独立逻辑，若不用两阶段提交（无论先写哪个），中间 crash 都会造成一份日志有、一份没有，导致原库与用 binlog 恢复出的库数据不一致。两阶段提交保证两份日志逻辑一致。
- **Q：** 崩溃恢复时怎么判断事务要提交还是回滚？
  **A：** redo log 已 commit → 直接提交；redo log 只有 prepare → 判断对应 binlog 是否完整：binlog 完整则提交该事务，不完整则回滚。
- **Q：** 什么是 WAL？带来什么好处？
  **A：** Write-Ahead Logging，先写日志再写磁盘。redo log 是顺序写、无需在磁盘找位置，把随机写变成顺序写，大幅提升更新性能，同时保证 crash-safe。
- **Q：** 只有 InnoDB 时能不能去掉 binlog？
  **A：** 不能。redo log 只有 InnoDB 有且循环写不持久保存，不具备归档能力；binlog 承担归档、主从复制功能，且可被关闭，不能只依赖它做崩溃恢复。

## 03 | 事务隔离：为什么你改了我还看不见？

**一句话核心**：InnoDB 通过 MVCC（多版本 + 回滚日志构建一致性视图）实现事务隔离，不同隔离级别对应不同的可见性规则；长事务会长期占用回滚段和锁，应尽量避免。

**关键知识点**
- 事务四大特性 **ACID**：原子性、一致性、隔离性、持久性。MySQL 中事务在**引擎层**实现，MyISAM 不支持事务（被 InnoDB 取代的重要原因）。
- 并发事务的三类问题：**脏读**（读到未提交数据）、**不可重复读**（同一事务内两次读同一行值不同）、**幻读**（同一查询条件读到别的事务新插入的行）。
- 四种隔离级别（隔离性递增、并发性能递减）：
  - **读未提交(RU)**：一个事务未提交的变更就能被别的事务看到；直接返回记录最新值，无视图。
  - **读提交(RC)**：事务提交后其变更才被别人看到；视图在**每条 SQL 开始执行时**创建。
  - **可重复读(RR)**：事务执行期间看到的数据始终与启动时一致；视图在**事务启动时**创建，整个事务用同一个视图。
  - **串行化**：读加读锁、写加写锁，读写冲突时后者等前者完成；直接用加锁避免并发。
- **MVCC（多版本并发控制）**：每次更新都同时记录一条回滚操作（undo），通过回滚操作可把最新值倒推回旧值；同一条记录在系统中可存在多个版本，不同事务按各自 read-view 读到对应版本，读写不冲突。
- **回滚日志何时删除**：当系统里没有比它更早的 read-view（没有事务再需要它）时才删除。
- **长事务危害**：① 存在很老的事务视图，需长期保留大量回滚记录，占用大量存储（5.5 及以前回滚段放在 ibdata，长事务提交后文件也不缩小）；② 长期占用锁资源，可能拖垮整个库。
- **事务启动方式**：`begin`/`start transaction`（显式，配 commit/rollback）；`set autocommit=0`（隐式，易造成意外长事务）。建议 `set autocommit=1` 并显式启动；需减少交互可用 `commit work and chain`（提交并自动开启下一个事务）。
- 查长事务：`information_schema.innodb_trx` 表（如查持续超 60s 的事务）。RC 与 RR 是最常考的两个级别；MySQL 默认 RR，Oracle 默认 RC。

**面试高频问答**
- **Q：** 四种隔离级别分别是什么，怎么区分？
  **A：** 读未提交（能看别人未提交的改动）、读提交（提交后才可见）、可重复读（事务内前后一致）、串行化（加锁串行执行）。隔离性从低到高、并发性能从高到低；RC 视图在每条 SQL 开始时建，RR 视图在事务启动时建。
- **Q：** 脏读、不可重复读、幻读各由哪个级别解决？
  **A：** 读提交解决脏读；可重复读解决不可重复读；串行化解决幻读（InnoDB 的 RR 通过间隙锁也能基本避免幻读，后续讲锁展开）。
- **Q：** MVCC 是怎么实现可重复读的？
  **A：** 每行更新时记录 undo 回滚日志，形成多版本链；事务启动时创建一致性视图（read-view），读取时通过回滚操作把最新值倒推到该视图应看到的版本，所以整个事务内读同一行结果一致，且与正在更新的事务不冲突。
- **Q：** 为什么要尽量避免长事务？
  **A：** 长事务持有很老的视图，导致它可能用到的回滚记录都不能删、长期占用大量存储；同时长期占用锁资源，可能拖垮整个库。可通过 `set autocommit=1`、监控 `innodb_trx` 表报警、限制单语句执行时间等方式避免。
- **Q：** 一致性视图（read-view）在 RC 和 RR 下有什么区别？
  **A：** RR 在事务启动时创建一个视图，整个事务期间复用，保证可重复读；RC 在每条 SQL 语句开始执行时重新创建视图，因此能读到已提交的最新变更。

## 04 | 深入浅出索引（上）

**一句话核心**：InnoDB 是索引组织表，每个索引都是一棵 B+ 树；主键索引叶子存整行（聚簇索引），二级索引叶子存主键值，查非主键列要回表。

**关键知识点**
- 三种基础索引模型：哈希表（只适合等值查询，区间查询要全扫）、有序数组（查询 O(log N) 且支持范围，但插入要挪数据，只适合静态数据）、搜索树。
- 数据库用 N 叉树（B+ 树）而非二叉树：为了减少磁盘访问。InnoDB 整数索引 N 约 1200，树高 4 就能存 17 亿行，10 亿行表查一个值最多访问 3 次磁盘（根节点常驻内存）。
- InnoDB 表按主键顺序以 B+ 树组织存放，称"索引组织表"，每个索引对应一棵 B+ 树。
- 主键索引（聚簇索引）叶子存整行数据；非主键索引（二级索引）叶子存主键值。
- 回表：普通索引查询先搜二级索引拿到主键值，再回主键索引树查整行，多扫一棵树。
- 页分裂：插入导致数据页满时，申请新页并挪动部分数据，性能下降、空间利用率约降 50%；相邻页利用率低时会页合并（分裂的逆过程）。
- 自增主键优势：追加式插入不触发页分裂；且主键越短，二级索引叶子越小越省空间（int 4 字节 vs 身份证 20 字节）。
- 适合用业务字段做主键的场景：只有一个索引且是唯一索引的 KV 场景（直接设为主键避免回表）。

**面试高频问答**
- **Q：** 为什么 InnoDB 用 B+ 树而不用二叉树或哈希？
  **A：** 索引要存磁盘，查询要尽量少读磁盘块。二叉树树高太高（百万行树高 20，要 20 次磁盘 IO）；哈希不支持范围查询。B+ 树是 N 叉、矮胖，配合磁盘读写特性，单次查询磁盘访问次数少。
- **Q：** 什么是回表？如何理解聚簇索引和二级索引？
  **A：** 聚簇索引叶子存整行，主键查询一次搜索到位。二级索引叶子只存主键值，查其他列时要拿主键值再回主键树查一次，这个过程叫回表，比主键查询多扫一棵树。
- **Q：** 为什么建议用自增主键？
  **A：** 自增插入是追加操作，不挪数据、不触发页分裂；主键占用小则所有二级索引叶子更小更省空间。
- **Q：** 重建普通索引和重建主键索引的写法有何区别？
  **A：** `drop index k; add index(k)` 重建普通索引合理，能省空间。但 drop/add primary key 都会重建整张表，第一个语句白做；应直接用 `alter table T engine=InnoDB`。

## 05 | 深入浅出索引（下）

**一句话核心**：覆盖索引、最左前缀、索引下推都是围绕"少回表、少访问资源"的优化手段。

**关键知识点**
- 覆盖索引：查询所需字段已在使用的索引上，不需要回表。二级索引叶子含主键，故 `select 主键 where 二级索引` 天然覆盖。是常用性能优化手段。
- `select * from T where k between 3 and 5`：读 k 索引 3 条记录，回表 2 次（Server 层认为扫描行数=2）。
- 最左前缀原则：B+ 树联合索引按字段定义顺序排序，只要满足最左前缀就能用上索引；可以是联合索引的最左 N 个字段，也可以是字符串索引的最左 M 个字符（如 `like '张%'`）。
- 联合索引字段顺序设计：第一原则是复用能力——若调整顺序能少维护一个索引就优先（有 (a,b) 就不用单独建 a）；第二原则是空间——需要单独 (b) 时，把大字段放联合索引、小字段单独建。
- 索引下推（ICP，index condition pushdown，MySQL 5.6 引入）：在索引遍历过程中就对索引中包含的字段做条件判断，直接过滤不满足的记录，减少回表次数。
- ICP 例子 `name like '张%' and age=10 and ismale=1`：5.6 前只用"张"定位后逐条回表比对；5.6 后在 (name,age) 索引内先判断 age=10，跳过不满足的，回表次数从 4 减到 2。

**面试高频问答**
- **Q：** 什么是覆盖索引？有什么好处？
  **A：** 查询需要的字段恰好都能从使用的索引树上取到，不需要回表。减少树搜索次数，显著提升性能。为高频查询建冗余联合索引来做覆盖索引是常见优化。
- **Q：** 什么是最左前缀原则？
  **A：** 联合索引按定义顺序排序，查询条件命中索引最左边连续的若干字段/字符时可用索引加速。所以 (a,b) 索引能支持 a 和 (a,b) 查询，但不能支持单独查 b。
- **Q：** 索引下推是什么？解决了什么问题？
  **A：** 5.6 引入的优化，在索引遍历时就用索引里已有的字段做条件过滤，跳过不满足的记录，减少无谓回表。EXPLAIN 显示 Using index condition。
- **Q：** 联合索引的字段顺序怎么定？
  **A：** 优先考虑复用（能少维护一个索引的顺序优先），其次考虑空间（大字段进联合索引、小字段单独建单列索引）。

## 06 | 全局锁和表锁

**一句话核心**：锁分全局锁、表级锁、行锁；全局锁 FTWRL 用于全库逻辑备份，表级锁尤其 MDL 会因长事务导致"给小表加字段却锁住整个库"的经典事故。

**关键知识点**
- 全局锁 FTWRL（Flush tables with read lock）：让整库只读，阻塞增删改、DDL、更新类事务提交。典型用于全库逻辑备份。
- 备份不加锁会拿到逻辑不一致的视图（不同表处于不同时间点）。一致性方案：RR 隔离级别下开事务用 MVCC 拿一致性视图。
- mysqldump --single-transaction：备份前启事务拿一致性视图，靠 MVCC 支持备份期间正常更新。但只适用于全部是事务引擎（InnoDB）的库；MyISAM 等不支持事务的引擎只能用 FTWRL。
- 为什么不用 set global readonly=true：一是 readonly 值可能被用作其他逻辑（如判断主备库），影响面大；二是异常处理不同——FTWRL 客户端断开会自动释放锁，而 readonly 客户端异常后会一直只读，风险高。
- 表级锁两种：表锁（lock tables … read/write，限制别人也限制本线程后续操作）和元数据锁 MDL。InnoDB 支持行锁，一般不用 lock tables。
- MDL（MySQL 5.5 引入，自动加）：增删改查加 MDL 读锁，改表结构加 MDL 写锁。读读不互斥，读写/写写互斥，保证变更表结构的正确性。
- MDL 事故：加字段需 MDL 写锁，若前面有长事务/未提交事务占着 MDL 读锁，写锁被阻塞，后续所有 MDL 读锁请求也全被排队阻塞，表完全不可读写。**MDL 在事务提交后才释放**。
- 安全加字段：先处理长事务（查 information_schema.innodb_trx，必要时 kill）；热点表用 alter table 的 DDL NOWAIT/WAIT n（MariaDB/AliSQL 支持）设置等待超时，拿不到锁就放弃后重试。
- Online DDL 过程：拿 MDL 写锁→降级为 MDL 读锁→真正做 DDL（占大部分时间，此时可读写）→升级 MDL 写锁→释放。第一步就可能被堵住。

**面试高频问答**
- **Q：** 全库备份为什么要加锁？有哪些方案？
  **A：** 不加锁各表备份时间点不同，逻辑不一致。方案：FTWRL 全局读锁（适用于含非事务引擎的库）；mysqldump --single-transaction 靠 MVCC 拿一致性视图、备份期间可更新（仅适用全 InnoDB 库）。
- **Q：** 为什么给小表加个字段可能把整个库搞挂？
  **A：** 加字段要 MDL 写锁，若有长事务占着 MDL 读锁未释放，写锁被阻塞，之后所有对该表的增删改查（都要 MDL 读锁）排在写锁后被阻塞，表完全不可读写；配合客户端重试会迅速打满连接。
- **Q：** MDL 锁什么时候释放？如何安全加字段？
  **A：** MDL 语句执行时申请、事务提交后才释放。加字段前先排查并处理长事务（innodb_trx 查、kill）；热点表用 ALTER TABLE ... WAIT n / NOWAIT 设超时，拿不到就放弃重试。
- **Q：** 为什么用 FTWRL 而不是 set global readonly=true？
  **A：** readonly 可能被用作主备判断等其他逻辑；且客户端异常断开后 FTWRL 会自动释放，readonly 会一直卡在只读状态风险更高。

## 07 | 行锁功过：减少行锁对性能的影响

**一句话核心**：InnoDB 行锁遵循两阶段锁协议（提交时才释放），把冲突最大的锁尽量往后放；热点行更新的性能瓶颈往往在死锁检测（O(n)）。

**关键知识点**
- 行锁在引擎层实现，MyISAM 不支持行锁（只能表锁，同表任意时刻只能一个更新），InnoDB 支持——这是 InnoDB 替代 MyISAM 的重要原因。
- 两阶段锁协议：行锁在需要时才加，但要到事务结束（commit）时才释放，而不是用完立刻放。
- 由两阶段锁推出的实践：事务中要锁多行时，把最可能造成锁冲突、最影响并发度的锁尽量往后放（如影院余额行更新放最后），缩短其被占用时间。
- 死锁：不同线程循环等待对方持有的资源，都进入无限等待。
- 两种处理策略：①直接等待直到超时，超时时间由 innodb_lock_wait_timeout 控制（默认 50s，太长；设太短又会误伤正常锁等待）；②主动死锁检测（innodb_deadlock_detect=on，默认开），发现死锁主动回滚死锁链中某个事务。
- 死锁检测的代价：每个被堵线程都要检测加入自己后是否形成环，O(n)。1000 个并发更新同一行时检测量达百万级，CPU 打满但每秒执行不了几个事务。
- 热点行更新优化：①确保不死锁时临时关死锁检测（有风险，可能大量超时）；②控制并发度，让同行更新在进入引擎前排队（中间件或改源码）；③将一行拆成逻辑多行（如账户拆 10 条记录随机加，冲突概率降为 1/10），但退款等减值逻辑需特殊处理。

**面试高频问答**
- **Q：** 什么是两阶段锁协议？对写事务有什么指导意义？
  **A：** 行锁在需要时加、事务提交时才统一释放。指导：把最可能冲突、最影响并发的行锁操作放到事务最后，减少锁被占用时间，提升并发。
- **Q：** 死锁怎么处理？两种策略各自的问题？
  **A：** 等待超时（innodb_lock_wait_timeout 默认 50s，太长影响在线服务，设短又误伤正常锁等待）；主动死锁检测（innodb_deadlock_detect=on 默认开，发现环就回滚一个事务）。正常用第二种。
- **Q：** 为什么大量并发更新同一行会 CPU 打满但 TPS 很低？
  **A：** 死锁检测是 O(n)，每个新被堵事务都要检测是否成环，n 个并发同行更新检测量达 n² 量级，消耗大量 CPU 却查不出死锁，导致 TPS 极低。
- **Q：** 如何缓解热点行更新的性能问题？
  **A：** 控制并发（进引擎前排队）、把一行拆成逻辑多行分散冲突、或确保不死锁时临时关检测（有超时风险）。
- **Q：** 删表前 10000 行，选哪种方式？
  **A：** 选"一个连接循环 20 次 delete limit 500"。单条 delete 10000 锁时间长且大事务致主从延迟；20 个连接并发则人为制造锁冲突。

## 08 | 事务到底是隔离的还是不隔离的？

**一句话核心**：InnoDB 靠 MVCC + 一致性读视图实现可重复读；快照读看版本可见性，更新用"当前读"读最新版本并加锁——这是 RR 下"查得到旧值却改不掉"的根源。

**关键知识点**
- 事务启动时机：begin/start transaction 不是起点，执行第一个操作 InnoDB 表的语句时才真正启动；`start transaction with consistent snapshot` 可立即启动并创建一致性视图。
- 两个"视图"概念：view（虚拟表）；MVCC 用的一致性读视图 consistent read view（无物理结构，定义事务能看到什么数据）。
- 多版本：每行有多个版本，每次更新生成新版本并赋 row trx_id（等于更新它的事务 transaction id，严格递增）；旧版本通过 undo log 保留，V1/V2/V3 靠当前版本+undo log 回算得到。
- 一致性视图（read-view）构成：事务启动瞬间正在"活跃"（已启动未提交）的所有事务 ID 数组；最小值为低水位，已创建事务的最大值+1 为高水位。
- 可见性规则（相对启动瞬间）：row trx_id < 低水位 → 已提交，可见；> 高水位 → 将来事务，不可见；落在低高水位之间 → 在活跃数组中则未提交不可见，不在数组中则已提交可见；自己更新的总是可见。
- 翻译版规则：除自己更新总可见外——版本未提交不可见；已提交但在视图创建后提交不可见；已提交且在视图创建前提交可见。
- 当前读（current read）：更新数据是先读后写，这个读只能读最新已提交版本。update、`select ... lock in share mode`（S 锁）、`select ... for update`（X 锁）都是当前读。
- 更新逻辑：事务 B 更新时用当前读读到最新版本（如 1,2），在此基础上 +1 生成新版本（1,3），避免丢失他人已提交更新。若最新版本行锁被别的事务占用（两阶段锁），当前读会被阻塞进入锁等待。
- RR vs RC 核心区别：**RR 在事务启动时创建一致性视图，整个事务共用**；**RC 在每个语句执行前重新创建视图**。当前读总是读最新已提交版本。

**面试高频问答**
- **Q：** 可重复读是怎么实现的？
  **A：** 核心是一致性读——事务启动时创建 read-view（活跃事务 ID 数组+高低水位），普通查询按 row trx_id 与视图对比确定版本可见性；整个事务共用这个视图，所以看到的数据始终一致。
- **Q：** RR 和 RC 的本质区别是什么？
  **A：** 视图创建时机不同。RR 在事务启动时创建一致性视图、全事务共用（承认事务启动前已提交的数据）；RC 每条语句执行前重建视图（承认语句启动前已提交的数据）。
- **Q：** 什么是当前读？为什么事务 B 更新后查到的是自己改的值？
  **A：** 更新是先读后写，读只能读最新已提交版本，叫当前读。B 更新时当前读拿到最新版本并在其上生成自己的新版本（row trx_id=自己），查询时发现最新版本是自己更新的，直接可见。select 加 lock in share mode / for update 也是当前读。
- **Q：** RR 下为什么会出现"查到旧值却改不掉"（丢失更新的异象）？
  **A：** 一致性读看到的是启动前的旧快照，但 update 是当前读读到别的事务已提交的最新值。若另一事务已把 c 或 id 改掉并提交，当前事务的 update 按当前读匹配不到行、改不动，再 select 又是旧快照，看似"值没变却改不掉"。解法：用 for update 加锁读，或每次 CAS 更新按 affected_rows 判断成败、失败则重开事务。
- **Q：** 事务里的快照真的拷贝整库数据吗？
  **A：** 不拷贝。靠"所有数据都有多版本"实现秒级快照——事务只声明一个一致性视图，之后的更新生成的新版本对它不可见，需要旧版本时用当前版本+undo log 回算。

## 09 | 普通索引 vs 唯一索引怎么选

**一句话核心**：查询性能几乎无差别，更新时唯一索引用不上 change buffer，所以业务允许时优先选普通索引。

**关键知识点**
- 查询差别微乎其微：唯一索引查到第一条即停；普通索引要多判断"下一条"，但因数据按页（16KB）读入内存，只是一次指针移动+计算，成本可忽略。
- change buffer：更新的数据页不在内存时，把变更缓存下来（不必从磁盘随机读该页），下次访问该页时再执行 merge。可持久化（存于系统表空间 ibdata1），并非纯内存。
- 为什么唯一索引不能用 change buffer：唯一性校验必须把数据页读入内存判断是否冲突；页都读进来了就直接更新，用不上缓存。因此只有普通索引能用 change buffer。
- merge：把 change buffer 中的操作应用到数据页得到最新结果。触发时机：访问该页、后台线程定期、数据库正常关闭。
- change buffer 大小由 innodb_change_buffer_max_size 控制（占 buffer pool 的百分比）。
- 适用场景：写多读少（账单、日志类）。若写完马上读，会立即触发 merge，反而增加维护代价，应关闭 change buffer。
- change buffer vs redo log：redo log 节省"随机写"磁盘 IO（转顺序写）；change buffer 节省"随机读"磁盘 IO。

**面试高频问答**
- **Q：** 普通索引和唯一索引怎么选？
  **A：** 业务已保证不重复时优先普通索引，因为它能用 change buffer 提升更新性能；若业务需要数据库做唯一性约束则必须用唯一索引。查询性能两者几乎一样。
- **Q：** change buffer 为什么唯一索引用不了？
  **A：** 唯一索引每次更新都要判重，必须先把数据页读入内存，既然读进来了就直接更新，没有缓存的意义。
- **Q：** change buffer 掉电会丢吗？
  **A：** 不会。change buffer 可持久化到系统表空间，且更新时也写了 redo log，崩溃恢复能找回。
- **Q：** change buffer 和 redo log 区别？
  **A：** redo log 把随机写变顺序写（省随机写 IO）；change buffer 减少更新时的随机读盘（省随机读 IO）。一条更新可能同时用到两者。

## 10 | MySQL 为什么有时候会选错索引

**一句话核心**：优化器靠"扫描行数（基数）"估算代价选索引，而基数是采样统计得来、可能不准，导致选错；可用 analyze table 修正或 force index 强制。

**关键知识点**
- 选索引是优化器的工作，目标是用最小代价执行；判断依据包括扫描行数、是否回表、是否用临时表、是否排序等。
- 基数（cardinality）= 索引上不同值的个数，越大区分度越好。通过 show index 查看。
- 采样统计：InnoDB 随机选 N 个页统计不同值取平均，再乘以页数得到基数——所以不精确。变更行数超过 1/M 会自动触发重新统计。
  - innodb_stats_persistent=on：统计持久化，N=20，M=10。
  - innodb_stats_persistent=off：只存内存，N=8，M=16。
- 选错原因：①统计信息（扫描行数预估）不准；②优化器会把普通索引的回表代价算进去，可能宁愿全表扫主键；③为避免排序而选了扫描行数更多的索引（如 order by 字段本身是索引）。
- 解决方法：
  - analyze table t：重新统计索引信息，解决统计不准问题。
  - force index(a)：强制走指定索引（缺点：不优雅、改名要改、迁移可能不兼容、上线不敏捷）。
  - 改写语句引导优化器：如 order by b limit 1 改成 order by b,a limit 1；或用 limit 100 诱导。
  - 新增更合适的索引，或删掉被误用的索引。

**面试高频问答**
- **Q：** MySQL 为什么会选错索引？
  **A：** 优化器按预估扫描行数估算代价，扫描行数来自采样统计的基数，采样不准（尤其长事务导致大量未purge的历史版本堆积）就会误判；另外优化器还会权衡回表代价、是否需要排序。
- **Q：** 发现 explain 的 rows 和实际差很多怎么办？
  **A：** 先用 analyze table 重新统计索引信息；仍不对再用 force index 强制索引，或改写 SQL 引导，或增删索引。
- **Q：** 为什么长事务会导致选错索引？
  **A：** 一致性读的长事务使 delete 的旧版本数据不能被 purge，索引上新旧两份数据都在，采样统计的行数被放大，误导优化器。

## 11 | 怎么给字符串字段加索引

**一句话核心**：字符串字段可用前缀索引省空间，但会增扫描行数且不能用覆盖索引；前缀区分度不够时用倒序存储或 hash 字段。

**关键知识点**
- 前缀索引：只取字符串前 N 字节建索引，如 email(6)。省空间，但前缀区分度低时会增加回表扫描次数。
- 定长度方法：用 count(distinct left(email,N)) 对比 count(distinct email)，预设可接受损失比例（如 5%），选满足区分度的最短前缀。
- 前缀索引的代价：不能使用覆盖索引——即使前缀已包含全部信息，系统也不确定是否被截断，仍要回主键表查一次。
- 身份证号等前缀重复度高（前6位地址码相同）的字段，两种优化：
  - 倒序存储：reverse(id_card) 后建前缀索引，让区分度高的后几位在前。查询用 where id_card = reverse('xxx')。
  - hash 字段：加一个整数字段存 crc32() 校验码并建索引，查询 where id_card_crc=crc32('xxx') and id_card='xxx'（防哈希冲突）。索引仅4字节。
- 倒序 vs hash 异同：都不支持范围查询（只支持等值）。区别：hash 需额外字段占空间、倒序不需；CPU 上 reverse 比 crc32 略省；查询效率 hash 更稳定（平均扫描接近1行），倒序仍是前缀索引会多扫行。

**面试高频问答**
- **Q：** 字符串字段建索引有哪些方式？
  **A：** ①完整索引（占空间）；②前缀索引（省空间但增扫描、不能覆盖索引）；③倒序存储+前缀索引（解决前缀区分度低）；④额外 hash 字段索引（性能稳定，有存储和计算开销）。后两种都不支持范围查询。
- **Q：** 前缀索引为什么用不了覆盖索引？
  **A：** InnoDB 无法确定前缀是否截断了完整值，必须回主键表验证完整字段，所以即使前缀已含全部信息也要回表。
- **Q：** 怎么确定前缀长度？
  **A：** 用 count(distinct left(字段,N)) 逐个测试，选区分度损失在可接受范围内（如不小于总区分度95%）的最短长度。

## 12 | 为什么 MySQL 会"抖"一下（刷脏页）

**一句话核心**：平时快是因为只写内存和 redo log（WAL），"抖"是在刷脏页（flush）占用了 IO 资源。

**关键知识点**
- WAL：更新只写内存+redo log 即返回。内存页与磁盘页内容不一致叫"脏页"，一致叫"干净页"，都在内存中。刷脏页=把内存写入磁盘。
- 触发 flush 的四种场景：
  - ①redo log 写满：系统停止所有更新，推进 checkpoint 刷脏页（更新数跌为0，必须避免）。
  - ②内存不够，淘汰最久未用的数据页，若是脏页必须先刷盘（常态）。
  - ③系统空闲时。
  - ④正常关闭时（把脏页全刷盘，下次启动快）。
- 明显影响性能的两种：单次查询要淘汰的脏页太多（响应变长）；redo log 写满导致更新全堵（写性能跌0）。
- innodb_io_capacity：告诉 InnoDB 磁盘 IO 能力（建议设为磁盘 IOPS，可用 fio 测）。设太小（如 SSD 却设300）会刷得太慢、脏页堆积拖慢性能。
- 刷盘速度控制：根据脏页比例 F1(M) 和 redo log 写盘速度 F2(N) 各算一个 0~100 的值，取较大者 R，按 innodb_io_capacity × R% 的速度刷。
- innodb_max_dirty_pages_pct：脏页比例上限，默认75%。平时要监控脏页比例不要接近75%（Innodb_buffer_pool_pages_dirty / total）。
- innodb_flush_neighbors：值1时刷脏页会"连坐"相邻脏页（机械盘减随机IO有益）；SSD 建议设0；MySQL 8.0 默认已是0。

**面试高频问答**
- **Q：** 一条平时很快的 SQL 偶尔变慢是什么原因？
  **A：** 很可能碰上刷脏页：要么查询需淘汰大量脏页、要么后台刷脏占用 IO，也可能是 redo log 写满被迫停更新推 checkpoint。
- **Q：** 什么情况会触发刷脏页？
  **A：** redo log 写满、内存不足淘汰脏页、系统空闲、正常关闭。前两种影响性能。
- **Q：** redo log 设太小会怎样？
  **A：** 很快写满，write pos 追着 checkpoint，系统频繁停更新推进 checkpoint，表现为磁盘压力不大但数据库间歇性性能下跌，甚至 change buffer 优化也失效。
- **Q：** innodb_io_capacity 怎么设？
  **A：** 设为磁盘实际 IOPS（fio 测），太小会让 InnoDB 误以为磁盘慢、刷脏太慢导致脏页堆积。

## 13 | 表数据删一半，文件大小为什么不变

**一句话核心**：delete 只把记录/数据页标记为"可复用"，不释放磁盘空间（形成空洞），要用 alter table ... engine=InnoDB 重建表才能收缩。

**关键知识点**
- innodb_file_per_table：ON 时每表单独存 .ibd 文件（推荐，drop 可直接删文件回收空间）；OFF 存共享表空间，删表也不回收。5.6.6 起默认 ON。
- delete 只标记可复用：记录复用受范围限制（删了ID400的位，只能插ID在该范围的）；整页删空后可复用到任意位置；相邻页利用率低会合并。磁盘文件不缩小。
- 空洞来源：delete（标记删除）、随机插入（页分裂后页尾留空洞）、update（相当于删+插）。大量增删改的表都可能有空洞。
- 重建表收缩：新建结构相同的表，按主键顺序把数据搬过去，新表紧凑无空洞。命令 alter table A engine=InnoDB（recreate）。
- Online DDL（5.6引入）流程：建临时文件→扫描原表主键生成B+树写入临时文件→过程中对表的操作记入 row log→临时文件生成后回放 row log→替换原表文件。允许期间增删改。
- MDL：alter 启动时拿 MDL 写锁，真正拷贝数据前退化为读锁（不阻塞增删改，又防止其他线程同时做DDL），拷贝完再升级。锁时间相对很短。
- Online vs inplace：inplace 指数据在 InnoDB 内部原地重建、对 server 层无需搬到临时表；Online 指不阻塞增删改。Online 一定 inplace，inplace 不一定 Online（如加全文索引/空间索引是 inplace 但非 Online）。
- 空间：inplace 重建仍需 tmp_file 临时空间，1TB 表磁盘只剩0.2TB 做不了。ALGORITHM=inplace（默认）/ ALGORITHM=copy（强制拷贝表，走 server 层临时表）。
- 三种重建对比：alter table engine=InnoDB = recreate；analyze table 只重新统计索引信息（不改数据，加MDL读锁）；optimize table = recreate + analyze。
- 大表重建耗 IO/CPU，线上推荐用 GitHub 的 gh-ost。

**面试高频问答**
- **Q：** 删了一半数据表文件为什么不变小？
  **A：** delete 只把记录/数据页标记为可复用形成空洞，不释放磁盘空间；要 alter table t engine=InnoDB 重建表才能真正收缩。
- **Q：** Online DDL 和 inplace 的区别？
  **A：** Online 指不阻塞增删改（靠 row log 记录并回放期间的DML）；inplace 指在 InnoDB 内部原地重建、不经 server 层临时表。Online 一定 inplace，反之不一定。
- **Q：** 重建表后空间反而变大是为什么？
  **A：** 表本身已很紧凑（如刚重建过），重建时每页会预留约1/16给后续更新，或 DDL 期间并发 DML 引入新空洞，导致比原来还大一点。
- **Q：** optimize table、analyze table、alter engine=InnoDB 区别？
  **A：** alter engine=InnoDB 是重建表；analyze 只重新统计索引信息不动数据；optimize = 重建+统计。

## 14 | count(*) 这么慢怎么办

**一句话核心**：InnoDB 因 MVCC 必须逐行判断可见性无法直接存总数，count(*) 已被优化器专门优化，效率 count(字段)<count(id)<count(1)≈count(*)。

**关键知识点**
- 不同引擎实现：MyISAM 把总行数存磁盘，无 where 的 count(*) 直接返回（但不支持事务）；InnoDB 需逐行读出累加。
- InnoDB 为什么不存总数：MVCC 下同一时刻不同事务"应该返回多少行"不确定，每行都要判断对当前事务是否可见。
- 优化：count(*) 会选最小的索引树遍历（普通索引树比主键树小），减少扫描量。
- show table status 的 TABLE_ROWS 是采样估算，误差可达40%~50%，不能代替 count(*)。
- 自己计数方案：
  - Redis 计数：读写快，但①异常重启可能丢更新（可用重启后重新 count 补回）；②即使正常也逻辑不精确——Redis 和 MySQL 是两个系统、不支持分布式事务，无法拿到一致视图（插入与计数顺序无论怎么调都可能不一致）。
  - 数据库单独计数表：InnoDB 崩溃恢复不丢数据；利用事务隔离性，更新事务未提交前计数值+1对其他会话不可见，查计数和查记录逻辑一致。
- count 各用法（InnoDB）原则：server 层要什么给什么，InnoDB 只给必要值，优化器只优化了 count(*)。
  - count(主键id)：遍历全表取 id 返回 server 层，判非空累加。
  - count(1)：遍历全表但不取值，server 层每行放个1累加。比 count(id) 快（省了解析行、拷贝字段）。
  - count(字段)：取字段值；not null 直接判累加，允许 null 还要取值判断非null 才加。
  - count(*)：专门优化，不取值不判空直接累加。
- 效率排序：count(字段) < count(主键id) < count(1) ≈ count(*)，建议用 count(*)。

**面试高频问答**
- **Q：** InnoDB 的 count(*) 为什么慢、为什么不像 MyISAM 存起来？
  **A：** InnoDB 支持事务，MVCC 下每个事务可见的行数不同，无法存一个固定总数，只能逐行读出判断可见性再累加。
- **Q：** count(*)、count(1)、count(id)、count(字段) 谁快？
  **A：** count(字段) < count(主键id) < count(1) ≈ count(*)。count(*) 被专门优化不取值不判空；count(字段) 要判 null 最慢。用 count(*) 即可。
- **Q：** 用 Redis 存计数有什么问题？
  **A：** 可能丢更新（重启），且 Redis 与 MySQL 无分布式事务、无法保证一致视图，计数与实际记录逻辑上不精确。放到 MySQL 计数表用事务才能保证一致。
- **Q：** 计数事务里先插记录还是先更新计数表？
  **A：** 先插入记录再更新计数表。计数表是热点行，更新加行锁到事务结束才释放，放最后能缩短持锁时间、减少锁等待、提升并发。

## 15 | 答疑（一）：日志和索引相关问题

**一句话核心**：两阶段提交（redo log prepare → binlog → redo log commit）靠 XID 关联，保证崩溃恢复时主库与 binlog（备库）数据一致。

**关键知识点**
- 两阶段提交崩溃恢复规则：
  - redo log 已有 commit 标识 → 直接提交。
  - redo log 只有完整 prepare → 判断对应 binlog 是否存在且完整：完整则提交，否则回滚。
- 时刻A（prepare 后、写 binlog 前）crash：binlog 没写、redo 没提交 → 回滚，也不传备库。
- 时刻B（binlog 写完、redo commit 前）crash：binlog 完整 → 提交事务（保证与已可能被备库使用的 binlog 一致）。
- binlog 完整性判断：statement 格式结尾有 COMMIT；row 格式结尾有 XID event；5.6.2 起有 binlog-checksum 校验。
- redo log 与 binlog 靠共同字段 XID 关联：崩溃恢复扫描 redo log，遇只有 prepare 的就拿 XID 去 binlog 找对应事务。
- 为什么要两阶段提交：为事务持久性。redo log 一旦提交就不能回滚；若 redo 先提交、binlog 写失败又回滚不了，就会与 binlog 不一致。两阶段是让双方都"OK"再一起提交。
- 只用 binlog 不行：binlog 没有恢复"数据页"的能力（WAL 靠 redo log 恢复数据页），且 binlog 是循环之外的顺序追加但不记录页级变更细节，无法崩溃恢复。
- 只用 redo log（关 binlog）崩溃恢复可以，但生产必须开 binlog：①归档（redo 循环写不留历史）；②高可用主从复制、异构系统消费 binlog 都依赖它。
- redo log 大小建议：几TB磁盘设4个文件×1GB；太小会频繁刷、WAL 失效。
- 最终落盘来自 buffer pool 不是 redo log：redo log 不记录完整数据页，无法自己更新磁盘。正常运行时落盘就是把内存脏页写盘（与 redo 无关）；崩溃恢复时才把页读入内存让 redo log 更新内容变脏页再落盘。
- redo log buffer：一块内存，事务执行中先存 redo 日志（多次写），真正写到 redo log 文件是在 commit 时（事务中一般不主动刷盘，但内存不足/其他事务提交等可能被动写入）。

**面试高频问答**
- **Q：** 两阶段提交时不同时刻 crash 怎么恢复？
  **A：** 写 binlog 前 crash → 回滚；binlog 写完但 redo 未 commit 时 crash → 只要 redo 有完整 prepare 且 binlog 完整就提交，否则回滚。
- **Q：** redo log 和 binlog 怎么关联？为什么要两阶段提交？
  **A：** 靠共同的 XID 关联。两阶段提交保证 redo 和 binlog 状态一致，避免出现"redo 提交了 binlog 没写"或反之导致的主备/数据不一致，本质是保证事务持久性和一致性。
- **Q：** 能不能只用 binlog 或只用 redo log？
  **A：** 只用 binlog 不行，它无法恢复数据页（WAL 依赖 redo log）；只用 redo log 崩溃恢复能行但生产必须开 binlog，因为归档和主从复制/异构系统都依赖它。
- **Q：** 数据最终落盘是靠 redo log 更新的吗？
  **A：** 不是。redo log 不含完整数据页。正常运行时落盘是把内存脏页直接写盘；只有崩溃恢复时才用 redo log 更新读入内存的数据页。

## 16 | order by 是怎么工作的？

**一句话核心**：order by 排序有全字段排序和 rowid 排序两种算法，用联合索引让数据天然有序（配合覆盖索引）可彻底避免排序。

**关键知识点**
- 排序在 server 层的 sort_buffer（内存）中完成，大小由 `sort_buffer_size` 控制；数据量超过它就用磁盘临时文件做外部归并排序（`number_of_tmp_files>1` 即用了临时文件，越小份数越多）。
- 全字段排序：把 select 需要的所有字段全放进 sort_buffer 排序，排完直接返回，只读一次原表；缺点是单行大时 sort_buffer 放不下多少行，临时文件多。
- rowid 排序：当单行长度超过 `max_length_for_sort_data` 时触发，sort_buffer 里只放排序字段+主键 id，排完再回表取数据；省内存但多一次回表（多磁盘读）。
- 权衡结论：InnoDB 表内存够就优先全字段排序（减少回表磁盘访问），这是 MySQL"能用内存就多用内存"的设计思想。
- 避免排序：建 (city,name) 联合索引，从索引取出的行天然按 name 有序，`Extra` 不再有 Using filesort。
- 进一步用覆盖索引 (city,name,age)，`Extra` 出现 Using index，连回表都省了，最快。
- `Rows_examined`/`@b-@a` 统计的是 server 层向引擎"取一行"的次数；rowid 排序 limit 1000 会多 1000 次回表，所以扫描行数比全字段排序多。

**面试高频问答**
- **Q：** 全字段排序和 rowid 排序的区别，MySQL 怎么选？
  **A：** 全字段排序把所有查询字段放 sort_buffer，排完直接返回、不回表；rowid 排序只放排序列和主键 id，排完要回表取值。由 `max_length_for_sort_data` 决定：单行长度超过它就用 rowid 排序。InnoDB 内存够时优先全字段排序以减少回表磁盘读。
- **Q：** 怎么让 order by 不排序？
  **A：** 建包含排序字段的联合索引让数据天然有序（如 (city,name)），消除 Using filesort；再把 select 字段也放进索引做覆盖索引，还能消除回表。
- **Q：** 怎么判断一个排序用了磁盘临时文件？
  **A：** 打开 optimizer_trace 看 `number_of_tmp_files`，等于 0 表示纯内存排序，大于 1 表示做了外部归并排序。

## 17 | 如何正确地显示随机消息？（order by rand() 的代价）

**一句话核心**：`order by rand()` 要用临时表+排序、扫描行数巨大，应把随机逻辑拆到业务代码里用主键定位（随机算法2）来实现。

**关键知识点**
- `order by rand()` 的 explain 显示 `Using temporary; Using filesort`：先建 memory 引擎内存临时表，扫全表把 (随机值R, word) 写入，再排序取前 3。
- 内存临时表排序用的是 rowid 排序：内存表回表只是按数组下标访问内存、不涉及磁盘，所以优化器选行更小的 rowid 排序（rowid 即 memory 表的数组下标）。
- 扫描行数很大：10000 行表随机取 3 个，Rows_examined ≈ 20003（写临时表 1 万 + 排序全表扫描 1 万 + 取 3 行）。
- 临时表超过 `tmp_table_size`（默认 16M）会转成磁盘临时表（默认 InnoDB）。
- 优先队列排序：只取 limit 少量最小值时，MySQL 5.6+ 用堆排序（优先队列）精确取前 N，不用把全部排好序、不用临时文件；但 limit 太大（堆超过 sort_buffer_size）就退回归并排序。
- 正确做法（随机算法2）：`select count(*)` 得 C，`Y=floor(C*rand())`，再 `limit Y,1`；严格均匀，代价约 C+Y+1，远小于 order by rand()。
- 随机算法1（用 max/min id 取中间值）代价最小(约3行)，但 id 有空洞时概率不均匀，不严格随机。

**面试高频问答**
- **Q：** 为什么 `order by rand()` 慢？
  **A：** 需要 Using temporary + Using filesort：建临时表存全表的随机值、全表扫描进 sort_buffer 排序、再取值回表，扫描行数是表行数的两倍多，资源消耗大。
- **Q：** 正确的随机取行方案是什么？
  **A：** 把逻辑拆到业务代码：先 count 得总行数 C，生成随机偏移 Y=floor(C*rand())，再用 limit Y,1 取行。避免数据库排序，代价远小。
- **Q：** 内存临时表为什么用 rowid 排序而不是全字段排序？
  **A：** 内存临时表回表只是按数组下标访问内存、无磁盘开销，优化器没有"减少回表磁盘访问"的顾虑，于是选参与排序的行更小的 rowid 排序。

## 18 | 逻辑相同性能却差异巨大（索引失效三典型）

**一句话核心**：对索引字段做函数操作会破坏索引有序性，导致优化器放弃树搜索、全索引扫描；三种典型是条件字段函数、隐式类型转换、隐式字符编码转换。

**关键知识点**
- 核心原则：**对索引字段做函数操作，可能破坏索引值有序性，优化器放弃走树搜索（快速定位）功能**，但可能仍走全索引扫描（选更小的索引遍历）。
- 案例一·条件字段函数操作：`where month(t_modified)=7` 用不上索引快速定位，要改成范围查询 `t_modified>='2016-7-1' and <'2016-8-1' or ...`。优化器还"偷懒"：`where id+1=10000` 也不会自动优化，得手写成 `id=10000-1`。
- 案例二·隐式类型转换：字段是 varchar、传入整型（`where tradeid=110717`）→ MySQL 规则是"字符串转数字"，相当于 `CAST(tradeid AS signed)=110717`，对索引字段加了函数 → 全表扫描。（判断规则：`select "10">9` 结果为 1 说明字符串转数字）
- 反向：字段是 int、传字符串（`where id="83126"`）→ 函数加在参数上（把字符串转数字），不影响字段，仍能走索引。
- 案例三·隐式字符编码转换：join 时两表字符集不同（utf8 与 utf8mb4），被驱动表字段被 `CONVERT(... USING utf8mb4)` 包裹（小字符集转大字符集）→ 被驱动表索引失效、全表扫描。
- 解法：统一字符集（`alter table modify ... CHARACTER SET utf8mb4`），或手动把驱动表字段转成被驱动表的字符集（把函数加到输入参数上而非被驱动表索引字段上）。

**面试高频问答**
- **Q：** 为什么对索引字段用函数就走不了索引？
  **A：** B+ 树的快速定位依赖同层节点的有序性，对字段做函数运算可能破坏这个有序性，优化器无法用树搜索，只能全索引/全表扫描（注意是放弃快速定位，不一定放弃这个索引）。
- **Q：** 隐式类型转换为什么会导致全表扫描？
  **A：** 字符串与数字比较时 MySQL 把字符串转数字，若索引字段是字符串就相当于在字段上加了 CAST 函数，破坏有序性，走全表扫描。字段是数字、参数是字符串则不受影响。
- **Q：** 两表 join 字段都有索引，为什么用不上被驱动表索引？
  **A：** 若两表字符集不同（如 utf8 与 utf8mb4），关联时会对被驱动表索引字段做 CONVERT 字符集转换（小转大），等于给索引字段加函数，导致被驱动表全表扫描。统一字符集或把转换放到驱动表侧即可。

## 19 | 只查一行也很慢的原因

**一句话核心**：查一行慢分两类——被锁住（等 MDL 锁/等 flush/等行锁）和查询慢（无索引全表扫、一致性读要回滚大量 undo）。

**关键知识点**
- 第一步排查：`show processlist` 看语句状态。
- 等 MDL 锁：状态 `Waiting for table metadata lock`，有线程持有表的 MDL 写锁（如 lock table / DDL）堵住了查询；查 `sys.schema_table_lock_waits` 找到持锁线程 kill 掉。
- 等 flush：状态 `Waiting for table flush`，一个 flush tables 命令被前面的慢语句堵住，又堵住了查询。
- 等行锁：`lock in share mode` 等当前读要加读锁，被别的事务持有的写锁堵住；查 `sys.innodb_lock_waits` 找罪魁，用 **KILL（断连接）** 而非 KILL QUERY——因为占锁的 update 已执行完，只有断开连接回滚事务才释放行锁。
- 查询慢·无索引：`where c=50000`（c 无索引）走主键全扫，扫 5 万行。记住"坏查询不一定是慢查询"，数据量涨上去时间线性增长。
- 查询慢·一致性读回滚 undo：一致性读（快照读）`select * from t where id=1` 若之前有事务做了 100 万次 update 未提交，当前事务的快照要从最新版本沿 undo log 回滚 100 万次才拿到旧值，扫描一行却耗时几百毫秒；而 `lock in share mode`（当前读）直接读最新值反而快。

**面试高频问答**
- **Q：** 查一行却卡住不返回，怎么排查？
  **A：** show processlist 看状态：Waiting for table metadata lock（等 MDL 写锁）、Waiting for table flush（等 flush）、等行锁三种；分别通过 sys 库定位持锁线程并 kill。
- **Q：** 为什么普通 select 比 lock in share mode 还慢？
  **A：** 普通 select 是一致性读，若数据被大量未提交 update 改过，需沿 undo log 一路回滚到事务开始时的版本，回滚次数多就慢；lock in share mode 是当前读，直接读最新已提交值，反而快。
- **Q：** 等行锁时该 KILL 还是 KILL QUERY？
  **A：** 用 KILL（断开连接），因为占锁的语句可能已执行完，KILL QUERY 只停当前语句不回滚事务、不释放行锁；断开连接才会回滚事务释放锁。

## 20 | 幻读是什么，幻读有什么问题？

**一句话核心**：幻读专指当前读下看到别的事务新插入的行，行锁锁不住"间隙"，InnoDB 用间隙锁+行锁（next-key lock）来解决。

**关键知识点**
- 幻读定义：一个事务前后两次查询同一范围，后一次看到了前一次没有的行；**只专指"新插入的行"**，且**只在当前读（for update / lock in share mode / update）下才出现**。
- 快照读（普通 select）靠 MVCC 本就看不到别的事务插入，不会幻读；别人 update 已有行被当前读看到不算幻读。
- 幻读的两个问题：①语义被破坏（声明锁住所有 d=5 却锁不住新插入/新改成 d=5 的行）；②数据与 binlog 不一致（statement 格式下 binlog 顺序执行会让主从数据不同）。
- 即使把扫描到的所有行都加写锁也挡不住幻读——新插入的行当时还不存在、加不上锁，所以幻读要单独解决。
- 间隙锁（Gap Lock）：锁两个值之间的空隙，与它冲突的是"往间隙插入记录"这个动作；**间隙锁之间互不冲突**（两个事务可对同一间隙都加间隙锁）。
- next-key lock = 行锁 + 间隙锁，**前开后闭区间**（如 (5,10]）；`select * from t for update` 会形成覆盖全表的多个 next-key lock（含 (25,+supremum]）。
- 间隙锁副作用：降低并发；两个事务对同一间隙加间隙锁后各自要插入会互相等待形成死锁。
- 间隙锁只在可重复读（RR）生效；改用读提交（RC）+ binlog_format=row 可去掉间隙锁、锁范围更小。

**面试高频问答**
- **Q：** 什么是幻读？和不可重复读什么区别？
  **A：** 幻读是当前读下两次查同一范围，后一次多出了别的事务新插入的行；只针对"新增的行"。不可重复读针对已有行被修改。RR 下快照读靠 MVCC 防不可重复读，当前读靠间隙锁防幻读。
- **Q：** 间隙锁解决幻读的原理？
  **A：** 幻读根源是行锁只能锁已存在的行、锁不住"间隙"里的新插入。间隙锁锁住值之间的空隙，与"插入"动作冲突，从而阻止新行插入，配合行锁组成 next-key lock 覆盖整个范围。
- **Q：** 间隙锁之间会冲突吗？
  **A：** 不会。间隙锁只和"插入"冲突，多个事务可对同一间隙同时加间隙锁；这也是两事务各持间隙锁再各自插入会死锁的原因。

## 21 | 只改一行却锁很多（加锁规则）

**一句话核心**：加锁规则=两个原则+两个优化+一个 bug，锁都加在索引上，用 next-key lock（前开后闭）逐个分析。

**关键知识点（规则总纲，限 RR 隔离级别，5.x≤5.7.24 / 8.0≤8.0.13）**
- 原则1：加锁基本单位是 next-key lock（前开后闭区间）。
- 原则2：查找过程中**访问到的对象**才加锁。
- 优化1：索引等值查询，**唯一索引**命中时 next-key lock 退化为行锁。
- 优化2：索引等值查询，向右遍历且**最后一个值不满足等值条件**时，next-key lock 退化为间隙锁。
- 一个 bug：唯一索引上的**范围查询**会访问到第一个不满足条件的值为止（多锁一段）。
- 补充：锁是加在索引上的；`for update` 会给主键上满足条件的行也加锁，`lock in share mode` 只锁覆盖索引（要避免覆盖索引优化可在 select 中加索引外字段）；`next-key lock 实际= 先加间隙锁再加行锁`两步。

**典型案例锁范围**
- 等值查空隙 `id=7`：退化为间隙锁 (5,10)（原则1+优化2）。
- 非唯一索引等值 `c=5`：(0,5] + 向右到 c=10 退化为间隙锁 (5,10)；覆盖索引不锁主键。
- 主键范围 `id>=10 and id<11`：id=10 退化行锁 + (10,15] next-key lock。
- 非唯一索引范围 `c>=10 and c<11`：(5,10] + (10,15]（非唯一不退化行锁）。
- 唯一索引范围 bug `id>10 and id<=15`：(10,15] + 多锁 (15,20]（扫到第一个不满足 id=20 才停）。
- delete `c=10`（有两行 c=10）：next-key lock 从 (c=5,id=5) 到 (c=15,id=15)，两端开区间；加 `limit 2` 后范围缩到 (c=5,id=5)~(c=10,id=30]，锁更小——**删数据尽量加 limit**。

**面试高频问答**
- **Q：** 加锁规则怎么概括？
  **A：** 两原则（加锁单位是前开后闭的 next-key lock；只锁访问到的对象）+两优化（唯一索引等值命中退化行锁；等值向右遍历最后一个不满足退化间隙锁）+一 bug（唯一索引范围查询会多扫到第一个不满足条件的行并加锁）。
- **Q：** `select * from t where id=10 for update` 和 `id>=10 and id<11` 加锁一样吗？
  **A：** 不一样。前者等值查唯一索引退化为 id=10 行锁；后者是范围查询，除 id=10 行锁外还会向右加 (10,15] 的 next-key lock。
- **Q：** 为什么 lock in share mode 有时锁不住更新？怎么办？
  **A：** lock in share mode 走覆盖索引时只锁二级索引、不锁主键，别的 update 可执行。要真正锁行需绕过覆盖索引优化——在 select 里加一个索引里没有的字段（如 select d ...），或用 for update。

## 22 | MySQL 有哪些"饮鸩止渴"提高性能的方法？

**一句话核心**：业务高峰临时救火手段针对短连接风暴、慢查询、QPS 突增三类问题，多数有损，是应急止血而非常规方案。

**关键知识点**
- 短连接风暴：连接数暴涨触发 `max_connections`（报 Too many connections）。调大该参数有风险（资源耗在权限验证）。
- 处理短连接：①kill connection 主动踢掉占着连接不干活的线程——优先断"事务外空闲"连接（查 `information_schema.innodb_trx` 的 trx_mysql_thread_id 判断是否在事务中），断事务中连接会回滚有损；②减少连接消耗：`--skip-grant-tables` 跳过权限验证（极高危，8.0 会同时开 skip-networking，不推荐）。
- 慢查询三类原因及应急：①索引没设计好→Online DDL 加索引（先备库执行 set sql_log_bin=off 加索引再主备切换）；②语句没写好→MySQL 5.7 的 query_rewrite 查询重写（如把 id+1=? 改写成 id=?-1）；③选错索引→force index（也可用查询重写加上）。
- 预防慢查询：上线前测试环境把 slow log 打开、`long_query_time=0` 记录所有语句，插模拟数据回归测试，重点看 `Rows_examined` 是否符合预期；可用 pt-query-digest。
- QPS 突增：①白名单机制下从 DB 端去掉该业务白名单；②新功能用独立账号则删账号断连接使其 QPS 归零；③与主体部署在一起只能查询重写成 `select 1` 返回（高危、可能误伤，优先级最低）。
- 方案1、2 依赖规范运维体系（虚拟化、白名单、业务账号分离）——更多准备意味着更稳定的系统。

**面试高频问答**
- **Q：** 短连接风暴导致连接数打满怎么紧急处理？
  **A：** 不建议盲目调大 max_connections。可 kill connection 踢掉空闲连接（优先事务外空闲，查 innodb_trx 判断是否在事务中，事务内的断开会回滚有损）；极端情况用 skip-grant-tables 跳过权限验证（高危不推荐）。
- **Q：** 慢查询打挂数据库，紧急怎么救？
  **A：** 分三种：索引没建好就 Online DDL 加索引（可备库先加再切换）；语句写得差用 query_rewrite 重写；选错索引加 force index。根本上靠上线前 slow log + Rows_examined 回归测试预防。
- **Q：** 某语句 QPS 突增打爆数据库怎么办？
  **A：** 优先让业务下线该功能；DB 端可去白名单、删独立业务账号断连接；实在不行用查询重写把该语句重写成 select 1 止血（风险高、可能误伤，最后选项）。

## 23 | MySQL是怎么保证数据不丢的？

**一句话核心**：只要 redo log 和 binlog 完整持久化到磁盘，异常重启就能恢复数据；"双1"配置保证不丢，组提交（group commit）保证性能。

**关键知识点**
- binlog 写盘：事务执行时写 binlog cache（每线程一个，`binlog_cache_size` 控制大小，超限暂存磁盘），提交时一次性写入 binlog 文件并清空 cache（一个事务的 binlog 不可拆分）。
- write（写到文件系统 page cache，快）与 fsync（持久化到磁盘，占 IOPS）是两步。`sync_binlog`：0=只 write 不 fsync；1=每次提交都 fsync；N=每 N 个事务 fsync 一次（掉电最多丢 N 个事务 binlog）。
- redo log 三种状态：redo log buffer（进程内存）→ page cache（write）→ 磁盘（fsync）。`innodb_flush_log_at_trx_commit`：0=只留 buffer；1=每次提交直接持久化磁盘；2=每次提交只写 page cache。后台线程每秒轮询把 buffer 刷盘。
- 未提交事务的 redo log 也可能被刷盘：后台每秒轮询、buffer 达到一半、并行事务提交时"搭便车"一起刷。
- 两阶段提交时序：redo log prepare（`=1` 时此阶段就 fsync）→ 写 binlog → redo log commit（commit 阶段不需再 fsync，因崩溃恢复靠 prepare 的 redo log + binlog）。
- "双1"配置 = `sync_binlog=1` 且 `innodb_flush_log_at_trx_commit=1`，一个事务完整提交前刷盘两次（redo prepare + binlog）。
- 组提交（group commit）：靠 LSN（日志逻辑序列号，单调递增）。一组 leader 写盘时带上最大 LSN，小于等于该 LSN 的 redo log 都被持久化，组员越多越省 IOPS。MySQL 故意把 redo log fsync 拖到 binlog write 之后，让 binlog 也能组提交。
- WAL 提速两点：顺序写快于随机写；组提交降低 IOPS。

**面试高频问答**
- **Q：** "双1"是什么？为什么 TPS 两万不等于每秒四万次磁盘写？
  **A：** 双1 = sync_binlog 和 innodb_flush_log_at_trx_commit 都设 1，每个事务提交刷两次盘（redo prepare + binlog）。靠组提交，一次 fsync 可以帮多个并发事务一起刷盘，所以磁盘写次数远少于事务数×2。
- **Q：** IO 出现瓶颈，如何临时提升性能（及风险）？
  **A：** ①设 `binlog_group_commit_sync_delay`/`sync_no_delay_count`（故意等待攒组，只增加响应时间，不丢数据）；②`sync_binlog` 设 100~1000（掉电丢 binlog）；③`innodb_flush_log_at_trx_commit=2`（掉电丢数据）。不建议设 0，因 MySQL 进程重启也丢，且性能与 2 差不多。
- **Q：** 为什么 binlog cache 每线程独享，而 redo log buffer 全局共享？
  **A：** binlog 一个事务必须连续、不可被打断，要整个事务写完再一起写；redo log 无此要求，中间日志可先写 buffer，还能被其他事务提交时搭便车一起刷盘。

## 24 | MySQL是怎么保证主备一致的？

**一句话核心**：备库回放主库 binlog 达到最终一致；binlog 有 statement/row/mixed 三种格式，靠 server_id 断循环复制。

**关键知识点**
- 主备切换：状态1 客户端读写 A、B 是备库；状态2 切到 B。建议备库设 `readonly`（防误操作、防切换 bug 双写、判断角色）；readonly 对 super 权限无效，同步线程有 super 权限，不影响同步。
- 主备同步流程：备库 `change master` 设置主库 IP/端口/账号密码/起始位点 → `start slave` 启动 io_thread 和 sql_thread → io_thread 连主库，主库有专门线程（dump 线程）按位点读本地 binlog 发给备库 → 备库写入中转日志（relay log）→ sql_thread 读 relay log 解析执行。
- binlog 三格式：**statement** 记 SQL 原文（省空间，但带 limit 等可能主备用不同索引导致不一致，会报 unsafe warning）；**row** 记真实操作行的主键和字段值（不会主备删错行，占空间大，但便于数据恢复）；**mixed** 折中，MySQL 自动判断有风险用 row、否则用 statement。
- statement 隐患例子：`delete ... limit 1` 主备可能选不同索引删不同行。
- row 恢复数据：delete 可反转成 insert 恢复误删；insert 反转成 delete；update 前后两行对调即可回滚。MariaDB Flashback 基于此原理。
- statement 的上下文：`now()` 会被记录 `SET TIMESTAMP` 保证主备时间一致，所以恢复数据要用 mysqlbinlog 解析后整体喂给 mysql，不能直接拷贝 statement 执行。
- 循环复制：生产多用双 M 结构（A、B 互为主备）。靠 server_id 断环——两库 server_id 必须不同；备库回放生成的 binlog 保留原 server_id；收到日志发现 server_id 与自己相同就丢弃。建议 `log_slave_updates=on`（备库执行 relay log 后生成 binlog）。

**面试高频问答**
- **Q：** binlog 三种格式区别，生产用哪种？
  **A：** statement 记原文省空间但可能主备不一致；row 记行数据准确、便于误操作恢复但占空间；mixed 是两者折中自动切换。生产至少用 mixed，越来越多场景要求用 row（数据恢复、并行复制等需要）。
- **Q：** 主备同步的完整流程？io_thread 和 sql_thread 各做什么？
  **A：** 备库 change master 设位点后 start slave 启动两线程：io_thread 连主库、接收主库 dump 线程发来的 binlog 写入 relay log；sql_thread 读 relay log 解析执行，达到与主库一致。
- **Q：** 双 M 结构如何避免循环复制？server_id 相同的场景会怎样？
  **A：** 靠判断 binlog 里的 server_id，收到与自己 server_id 相同的日志就丢弃。若中途 `set global server_id` 改了 id，或三节点迁移场景，可能导致 server_id 判断失效引发死循环，可用 `CHANGE MASTER TO IGNORE_SERVER_IDS` 忽略。

## 25 | MySQL是怎么保证高可用的？

**一句话核心**：高可用依赖主备延迟大小；切换有可靠性优先（保数据、有不可用时间）和可用性优先（保可用、可能数据不一致）两种策略。

**关键知识点**
- 三个时间点：T1 主库写完 binlog、T2 备库收完 binlog、T3 备库执行完。主备延迟 = T3-T1，即 `show slave status` 的 `seconds_behind_master`（精度秒）。
- seconds_behind_master 算法：备库拿当前执行事务 binlog 里的时间字段与本地系统时间求差；建连接时用 `SELECT UNIX_TIMESTAMP()` 校准主备时间差，所以时钟不同步不影响（但连接建立后再改时间会不准）。
- 主备延迟来源：①备库机器性能差（现多用对称部署）；②备库压力大（读查询耗 CPU，解法：一主多从分担、binlog 导到 Hadoop 等外部系统）；③大事务（如一次性 delete 大量数据、大表 DDL，主库执行 10 分钟备库就延迟 10 分钟；解法：拆分删除、DDL 用 gh-ost）；④备库并行复制能力不足。
- 可靠性优先切换：①判断备库 SBM 小于阈值（如5秒）否则重试 → ②主库 A 设 readonly → ③等备库 SBM=0 → ④备库 B 设可读写 → ⑤切业务到 B。步骤2~5 之间系统不可写（不可用时间），所以步骤1先判断延迟。
- 可用性优先切换：不等同步直接切到 B 并可读写，几乎无不可用时间，但可能数据不一致。mixed/statement 格式会悄悄不一致；row 格式会报 duplicate key error 停止同步、更易发现。
- 主库掉电异常切换：可靠性优先下必须等 SBM=0 才切，此时系统完全不可用；不能直接切只读因为 relay log 没应用完客户端会看到"数据丢失"。
- 结论：多数场景用可靠性优先（数据可靠性优于可用性）；只有日志类可容忍不一致的库才用可用性优先。

**面试高频问答**
- **Q：** 可靠性优先和可用性优先策略的区别？分别用于什么场景？
  **A：** 可靠性优先要等备库 SBM=0 才切换，保证数据一致但有系统不可写的窗口期；可用性优先不等同步直接切，几乎无不可用时间但可能主备数据不一致。绝大多数用可靠性优先，只有能容忍短暂不一致、且不可写会阻塞业务的日志类库才用可用性优先。
- **Q：** seconds_behind_master 是怎么算的？主备时钟不一致会不会不准？
  **A：** 备库取正在执行事务 binlog 的时间字段与本地时间求差得 T3-T1。建连接时通过 SELECT UNIX_TIMESTAMP() 校准主备时钟差，不影响；但连接建立后再改系统时间会不准。
- **Q：** 主备延迟有哪些来源？
  **A：** 备库机器性能差、备库查询压力大、大事务（大批量 delete、大表 DDL）、备库并行复制能力不足。

## 26 | 备库为什么会延迟好几个小时？

**一句话核心**：单线程复制追不上高并发主库，需多线程（并行）复制；核心约束是同一行/同一事务必须分到同一 worker，各版本策略在此基础上提升并行度。

**关键知识点**
- 单线程复制瓶颈：主库多线程写、备库 sql_thread 单线程回放，高 TPS 下延迟越来越大（5.6 前只支持单线程）。多线程模型：coordinator（原 sql_thread）只读日志分发，worker 线程真正执行，`slave_parallel_workers` 控制 worker 数（建议 8~16）。
- 分发两大原则：①更新同一行的事务必须分到同一 worker（防更新覆盖）；②同一事务不能拆开，必须同一 worker（防破坏事务隔离性/原子性）。
- **按表分发**（作者 5.5 版）：hash key = 库名.表名，两事务改不同表可并行；遇热点表退化为单线程。
- **按行分发**（作者 5.5 版）：hash key = 库名+表名+唯一键值（含主键和唯一索引），改不同行可并行，并行度更高；要求 binlog 为 row、表有主键、无外键；大事务耗内存耗 CPU，超行数阈值退化单线程。
- **MySQL 5.6 按库并行**（DATABASE）：hash key=库名。构造快、不要求 row 格式；但表都在一个库或热点集中时无效。
- **MariaDB 策略**：利用组提交——同组（相同 commit_id）事务不会改同一行、主库能并行则备库能并行。缺点：备库要等第一组全执行完才能执行第二组，吞吐不够，且易被大事务拖后腿。
- **MySQL 5.7**（`slave-parallel-type`）：DATABASE 或 LOGICAL_CLOCK。LOGICAL_CLOCK 优化——不用等到 commit，只要到 redo log prepare 状态就表示通过锁冲突检验，可并行；同时处于 prepare 的事务、prepare 与 commit 之间的事务都可并行。可调 `binlog_group_commit_sync_delay` 制造更多同时 prepare 的事务提升并行度。
- **MySQL 5.7.22 WriteSet**（`binlog-transaction-dependency-tracking`）：COMMIT_ORDER（同 5.7）、WRITESET（每行算 hash 组成 writeset，无交集即可并行）、WRITESET_SESSION（额外要求同线程事务在备库保持先后顺序）。writeset 在主库写入 binlog、备库无需解析 binlog 内容、省内存、支持 statement 格式。

**面试高频问答**
- **Q：** 事务为什么不能轮询分发给 worker？同一事务多语句为什么不能拆给不同 worker？
  **A：** 轮询可能导致同一行两个事务在备库执行顺序与主库相反，造成主备不一致；同一事务拆开则中间可能被查到"更新一半"的结果，破坏隔离性/原子性。所以同行事务、同一事务都要在同一 worker。
- **Q：** MySQL 5.7 LOGICAL_CLOCK 靠什么判断可并行？
  **A：** 靠两阶段提交。事务到达 redo log prepare 状态即已通过锁冲突检验，所以同时处于 prepare 的事务、以及 prepare 与 commit 之间的事务在备库可并行。可用 binlog_group_commit_sync_delay 制造更多同时 prepare 的事务。
- **Q：** 主库单线程插入很多数据后搭备库追数据，选哪个 dependency-tracking 参数？
  **A：** 选 WRITESET。因主库单线程每个事务 commit_id 不同，COMMIT_ORDER 会退化单线程；WRITESET_SESSION 要求同线程保持先后顺序也退化单线程；只有 WRITESET 靠行冲突判断可以并行。

## 27 | 主库出问题了，从库怎么办？

**一句话核心**：一主多从切换的痛点是从库找新主库的同步位点；基于位点需跳错，GTID 彻底解决找位点问题。

**关键知识点**
- 一主多从切换：A' 成新主，从库 B/C/D 都要改指向 A'，比一主一备多了"重新指向"的复杂度。
- 基于位点切换：`CHANGE MASTER TO ... MASTER_LOG_FILE, MASTER_LOG_POS`。B 原来记的是 A 的位点，同一日志在 A 和 A' 位点不同，需"找同步位点"——取一个稍靠前的位置（用 mysqlbinlog 解析 A' 在故障时刻 T 的 end_log_pos），再跳过已执行事务。
- 跳错方法：①`set global sql_slave_skip_counter=1; start slave` 主动跳过一个事务（跳过一个 event 会跳到事务末尾）；②`slave_skip_errors` 设 "1032,1062"（1062=唯一键冲突 insert，1032=删除找不到行），切换稳定后要清空以免掩盖真不一致。
- GTID（Global Transaction Identifier）= `server_uuid:gno`，事务提交时生成的唯一标识（gno 提交时分配、从1递增，比 transaction_id 更连续易懂）。启动加 `gtid_mode=on`、`enforce_gtid_consistency=on`。
- gtid_next：automatic 时分配 server_uuid:gno；指定具体 GTID 时，若已在实例 GTID 集合中则忽略该事务（可提交空事务把 GTID 加入集合来跳过冲突），否则用该 GTID 且 gno 不加1。
- 基于 GTID 切换：`CHANGE MASTER TO ... master_auto_position=1`，无需指定 file/pos。B 把自己 GTID 集合 set_b 发给 A'，A' 算 set_a 与 set_b 差集，从第一个不在 set_b 的事务开始发；差集日志已被删则直接报错。GTID 协议保证主库发给备库的日志必须完整（由主库判断），位点协议则由备库指定不做完整性判断。
- GTID + 在线 DDL：备库先做 DDL，用 `set gtid_next=` 提交空事务把该 GTID 加到主库集合，让主库跳过执行但保留 binlog 记录，避免数据与日志不一致。

**面试高频问答**
- **Q：** GTID 是什么？相比位点切换的好处？
  **A：** GTID = server_uuid:gno，事务提交时的全局唯一标识。位点切换要人工找位点、跳错很复杂；GTID 下 master_auto_position=1，找位点工作由主库自动完成（算 GTID 集合差集），一主多从切换非常方便。
- **Q：** 基于 GTID 主备关系里，主库发现需要的 binlog 已被删除怎么办？
  **A：** ①业务允许不一致：主库查 gtid_purged，从库 reset master 后 set global gtid_purged=该集合，再 start slave（缺失部分数据会丢）；②要求一致：重新搭建从库；③接到有全量 binlog 的从库做级联；④有 binlog 备份则先应用缺失 binlog 再 start slave。
- **Q：** 主备切换跳过 1032/1062 错误安全吗？
  **A：** 仅在切换找不到精确位点时无损（唯一键冲突 insert 已存在、删除找不到行说明已删），可临时用 slave_skip_errors 跳过；同步稳定后必须清空，否则会掩盖真正的主备不一致。

## 28 | 读写分离有哪些坑？

**一句话核心**：读写分离的坑是"过期读"（从库读到旧数据），有从强制走主库到等 GTID 等多种精度递增的方案。

**关键知识点**
- 两种架构：客户端直连（性能好、架构简单，但主备切换/迁移时客户端要感知调整，常配 Zookeeper 管理）；带 proxy 中间层（对客户端友好，但 proxy 需高可用、整体复杂）。趋势是 proxy 方向。
- 过期读：主从有延迟，更新后马上查从库可能读到旧状态。处理方案共六种：
- **强制走主库**：请求分类，必须最新的走主库、可容忍旧数据的走从库。用得最多，但金融类"全都不能过期读"时等于放弃读写分离。
- **sleep 方案**：查询前 sleep 一下（如 Ajax 先展示输入内容再等一会儿查）。不精确：0.5秒能拿到的也等1秒，超1秒仍过期读。
- **判断主备无延迟**：①`seconds_behind_master=0`（精度秒）；②对比位点（Master_Log_File/Read_Master_Log_Pos = Relay_Master_Log_File/Exec_Master_Log_Pos）；③对比 GTID 集合（Retrieved_Gtid_Set = Executed_Gtid_Set）。仍不精确——"备库收到的都执行完"，但有已回复客户端却还没传到备库的日志（trx3 未到从库）。
- **配合 semi-sync**：主库把 binlog 发从库→从库回 ack→主库收 ack 才返回客户端"事务完成"。保证已确认事务备库都收到。但一主多从时只等一个从库 ack，查其他从库仍可能过期读；持续延迟时位点等值判断可能一直不成立导致过度等待。
- **等主库位点**：`select master_pos_wait(file, pos, timeout)`（从库执行）。更新完在主库 `show master status` 拿 File/Position，在从库执行 master_pos_wait，返回≥0 就在该从库查，否则退化到主库查。只需保证 trx1 已执行完，不必等完全同步。
- **等 GTID**：`select wait_for_executed_gtid_set(gtid_set, 1)`，包含则返回0、超时返回1。更新事务后直接从返回包拿 GTID（需 `session_track_gtids=OWN_GTID` + `mysql_session_track_get_first` API，5.7.6+），比等位点少一次 show master status。
- 实际可混用：先分类，不能过期读的语句再用等 GTID/等位点，超时退化到主库（需业务限流，防全落主库打挂）。

**面试高频问答**
- **Q：** 什么是过期读？有哪些解决方案？
  **A：** 主从延迟下更新后查从库读到旧数据。方案（精度递增）：强制走主库、sleep、判断主备无延迟（SBM/位点/GTID）、配合 semi-sync、等主库位点（master_pos_wait）、等 GTID（wait_for_executed_gtid_set）。各有取舍，实际混用。
- **Q：** 判断主备无延迟为什么还会过期读？semi-sync 如何改善？
  **A：** "备库收到的都执行完"不代表主库所有已提交事务都到了备库，存在已回复客户端却未传到从库的日志。semi-sync 让主库收到从库 ack 才回客户端，保证已确认事务备库都已收到，配合位点判断可避免过期读（但一主多从只等一个 ack 仍有隐患）。
- **Q：** 等主库位点和等 GTID 方案的流程？
  **A：** 等位点：更新后 show master status 拿 File/Pos，从库 select master_pos_wait(File,Pos,1)，返回≥0 就查该从库否则查主库。等 GTID：更新后从返回包直接拿 gtid1，从库 select wait_for_executed_gtid_set(gtid1,1)，返回0就查该从库否则查主库，省一次 show master status。

## 29 | 如何判断一个数据库是不是出问题了？

**一句话核心**：select 1 只能证明进程活着；健康检测从 select 1 →查表→更新→performance_schema 内部统计逐级演进。

**关键知识点**
- **select 1 不够**：只说明进程还在。设 `innodb_thread_concurrency`（并发线程上限，建议64~128）后，并发查询占满时 select 1 仍成功但真实查询被堵，检测不出问题。
- 并发连接 ≠ 并发查询：show processlist 里几千个是并发连接（多占内存无妨）；"正在执行"的才是并发查询（CPU 杀手），innodb_thread_concurrency 限制的是后者。
- 进入锁等待（行锁/间隙锁）的线程并发计数减1（不吃 CPU），否则热点行更新会让线程数占满导致系统整体锁死。真正执行查询（如 select sleep）仍计入。
- **查表判断**：查 `mysql.health_check`（一行数据），能检出并发线程过多。但 binlog 磁盘满时更新被堵、读仍正常，查表检测不出。
- **更新判断**：`update mysql.health_check set t_modified=now()`。主备都要检测，双 M 结构下备库更新会传回主库可能行冲突导致同步停止——解法：表存多行、用 server_id 做主键，`insert ... values(@@server_id, now()) on duplicate key update`。
- 更新判断的"判定慢"问题：外部检测有超时时间 N 且是定时轮询（随机性）。日志盘 IO 利用率100%时系统很慢，但 update 需资源少可能超时前就返回，检测认为正常。
- **内部统计**：`performance_schema.file_summary_by_event_name` 统计每次 IO 时间（单位皮秒），如 `wait/io/file/innodb/innodb_log_file`（redo log）、`wait/io/file/sql/binlog`。用 MAX_TIMER_WAIT 设阈值（如单次 IO>200ms 异常）判断。打开全部统计约损耗10%性能，建议按需开启，异常后 truncate 表重新累积。
- 作者倾向：优先 update 系统表 + 配合 performance_schema 内部统计。MHA 默认用 select 1。

**面试高频问答**
- **Q：** 为什么 select 1 不能判断数据库是否正常？
  **A：** 它只证明进程还在。并发查询占满 innodb_thread_concurrency 后 select 1 仍成功但真实查询被堵；binlog 磁盘满时更新被堵但 select 仍正常。所以要逐级用查表、更新、performance_schema 检测。
- **Q：** 更新判断为什么会"判定慢"（外部检测的局限）？
  **A：** 外部检测定时轮询有随机性，且 update 资源需求小，即使日志盘 IO 100%、业务 SQL 很慢，检测 update 也可能在超时前返回被判为正常。所以要配合 performance_schema 内部 IO 时间统计。
- **Q：** 双 M 结构下用更新做健康检测要注意什么？
  **A：** 主备用相同更新语句会因行冲突导致主备同步停止。要让 health_check 表存多行、用各自 server_id 做主键（server_id 主备必须不同），用 insert...on duplicate key update 避免冲突。

## 30 | 答疑文章（二）：用动态的观点看加锁

**一句话核心**：加锁是执行过程中一行一行动态加的，必须配合语句执行逻辑（索引搜索过程）分析；"间隙"由其右边的记录定义。

**关键知识点**
- 加锁规则回顾（两原则两优化一 bug）：①加锁基本单位是 next-key lock（前开后闭）；②只给查找过程访问到的对象加锁；优化1：索引等值查询命中唯一索引，next-key lock 退化为行锁；优化2：索引等值查询向右遍历最后一个值不满足等值条件时，退化为间隙锁；bug：唯一索引范围查询会访问到不满足条件的第一个值为止。
- 不等号里的等值查询：`id>9 and id<12 order by id desc for update` 加锁 (0,5]、(5,10]、(10,15)。索引搜索本质是"找第一个值再遍历"，找 id=12（找不到落在(10,15)间隙）就是等值查询，故 id=15 用优化2 退化间隙锁。
- in 的等值查询：`c in(5,20,10)` 逐个 B+ 树定位，一个一个加锁（c=5 加 (0,5]+(5,10)，c=10 加 (5,10]+(10,15)，c=20 加 (15,20]+(20,25)），顺序是 5→10→20。
- 死锁成因：`in(5,20,10) lock in share mode` 按 5→10→20 加锁，`in(5,20,10) order by c desc for update` 按 20→10→5 加锁，加锁顺序相反并发即死锁。避免死锁：对同一组资源按尽量相同顺序访问。
- 看死锁：`show engine innodb status` 的 LATEST DETECTED DEADLOCK。关键字段：TRANSACTION（各事务）、WAITING FOR THIS LOCK（在等的锁）、HOLDS THE LOCK（持有的锁）、lock mode S/X、Record lock、WE ROLL BACK TRANSACTION（回滚谁——选回滚成本小的，for update 持锁多故回滚 lock in share mode）。
- 看锁等待：`lock_mode X locks gap before rec insert intention waiting` = 插入意向锁被间隙锁堵。delete 掉 id=10 后 (5,10)、(10,15) 合并成 (5,15)，间隙由右边记录（id=15）定义。
- update 拆解：update 改索引值 = 删除旧记录 + 插入新记录，会改变间隙范围，进而动态改变别的 session 的加锁范围（如 c=5 改 c=1 使 (5,10) 间隙左移）。

**面试高频问答**
- **Q：** 为什么 where 是 `id>9 and id<12` 也会有"等值查询"和优化2？
  **A：** 加锁发生在执行过程中，索引搜索本质是"先定位第一个值再遍历"。找 id=12 这个定位动作就是等值查询，找不到落在间隙，因最后一个值不满足等值条件，next-key lock 退化为间隙锁。所以分析加锁必须结合索引搜索逻辑。
- **Q：** 两条 in 语句为什么会死锁？如何避免？
  **A：** 锁是一个个加的，lock in share mode 按 c=5→10→20 加、order by c desc for update 按 20→10→5 加，加锁顺序相反并发就互相等待成死锁。避免办法是对同一组资源按尽量相同的顺序访问。
- **Q：** "间隙由右边记录定义"是什么意思？delete 一行会怎样？
  **A：** 间隙锁锁的范围由间隙右边那条记录决定。delete 掉 id=10 后，原来 (5,10)、(10,15) 两个间隙合并成 (5,15)，导致别的 session 什么都没做但加锁范围突然变大。

## 31 | 误删数据后除了跑路，还能怎么办？

**一句话核心**：按误删类型分治——delete 用 binlog + Flashback 闪回，drop/truncate/drop database 用全量备份 + 增量 binlog，核心是预防加定期演练。

**关键知识点**
- 误删分四类：delete 删行、drop/truncate 删表、drop database 删库、rm 删实例。
- 误删行（delete）：用 Flashback 闪回，原理是反转 binlog event 并回放（insert↔delete，update 前后 image 对调），多事务需倒序执行。前提是 `binlog_format=row` 且 `binlog_row_image=FULL`。
- 恢复时不要直接在主库操作，应先恢复到临时库/从库确认后再回主库，避免二次破坏。
- drop/truncate/drop database 无法用 Flashback：即使 row 格式，这三类 DDL 记的 binlog 仍是 statement，只有一句语句，恢复不出数据。
- 误删库/表恢复：全量备份 + 应用增量 binlog。用 GTID 或 --stop-position/--start-position 跳过误操作语句；可用 `--database` 只恢复指定库。
- 加速恢复：把临时库接成线上备库的从库，用 `change replication filter replicate_do_table` 只同步误删表，可用上并行复制。
- 延迟复制备库（MySQL 5.6+）：`CHANGE MASTER TO MASTER_DELAY=N`，让备库持续延迟 N 秒，误删后在延迟窗口内 stop slave 跳过误操作，缩短恢复时间。
- 事前预防：`sql_safe_updates=on`（无 where 或 where 不含索引字段则报错）、SQL 审计、账号权限分离（开发只给 DML）、删表先改名加后缀观察。

**面试高频问答**
- **Q：** delete 误删和 drop/truncate 误删恢复方式为何不同？
  **A：** delete 是 DML，row 格式下 binlog 记录了每行数据，可 Flashback 反转恢复；drop/truncate 是 DDL，binlog 只记 statement 语句本身，无数据行，只能靠全量备份 + binlog 重放。
- **Q：** Flashback 恢复的原理和前提？
  **A：** 解析 binlog，把 Write_rows 改 Delete_rows、Delete_rows 改 Write_rows、Update 前后 image 对调，多事务倒序回放。前提 `binlog_format=row`、`binlog_row_image=FULL`。恢复到临时库确认后再回主库。
- **Q：** 如何缩短核心业务的误删恢复时间？
  **A：** 搭建延迟复制备库（MASTER_DELAY），保持 N 秒延迟，误删在窗口内发现即可 stop slave 跳过；最多再追 N 秒即得到临时实例。
- **Q：** 忘写 where 全表误更新怎么预防？
  **A：** 开启 `sql_safe_updates=on`，delete/update 无 where 或 where 不含索引字段会报错；真要删全表可加 `where id>=0`，但性能上优先用 truncate/drop。

## 32 | 为什么还有 kill 不掉的语句？

**一句话核心**：kill 不是立即停止，而是设置线程状态 + 发信号，被 kill 线程必须执行到判断状态的“埋点”才进入终止逻辑；线程卡在不判断状态的循环或终止逻辑耗时长时就“kill 不掉”。

**关键知识点**
- 两个命令：`kill query + 线程id` 只停当前语句（事务不回滚）；`kill connection + 线程id` 断开连接（本质仍先走 kill query 停语句，再关网络连接）。
- kill 的实现：把目标线程的 killed 状态设为 KILL_QUERY/KILL_CONNECTION，并发一个信号唤醒它。不能直接终止，否则如持有的 MDL 读锁无法释放。
- 语句执行中有多处“埋点”判断线程状态，只有执行到埋点且状态为 KILL 才进入终止逻辑；等待必须是可被唤醒的等待。
- kill 无效第一类——线程没执行到判断逻辑：如 `innodb_thread_concurrency` 不够，线程每 10ms 循环判断能否进 InnoDB，进不去就 nanosleep，循环中不判断 killed 状态；或 IO 压力大读写 IO 函数一直不返回。
- kill 无效第二类——终止逻辑本身耗时长：超大事务回滚、大查询回滚删大临时文件、DDL 收尾删临时文件（都可能受 IO 影响）。
- show processlist 里 Command 显示 Killed：因为 KILL_CONNECTION 状态的线程有特殊逻辑显示成 Killed；客户端已断开但服务端语句仍在执行。
- Ctrl+C 不能直接终止服务端线程：客户端只能操作本地线程，Ctrl+C 实际是另起连接发一个 kill query 命令。
- 处理 Killed 状态只能改善环境让它尽快结束：调大 innodb_thread_concurrency、停别的线程、减少 IO 压力；回滚中的事务即使重启 MySQL 也仍要回滚，从恢复速度看应让它自己执行完（有锁/IO 影响则先主备切换）。

**面试高频问答**
- **Q：** kill query 为什么不是立即生效？
  **A：** kill 只设置线程 killed 状态并发信号，被 kill 线程要执行到判断状态的“埋点”才进终止逻辑，同时终止逻辑（如回滚）本身耗时；所以是“通知停止”不是“立即停止”。
- **Q：** 什么情况下语句 kill 不掉？
  **A：** 两类：一是线程没执行到判断状态的逻辑（如卡在等待进 InnoDB 的 sleep 循环、IO 阻塞导致函数不返回）；二是终止逻辑耗时长（大事务回滚、大查询/DDL 删临时文件）。
- **Q：** 客户端 Ctrl+C 能杀掉服务端线程吗？
  **A：** 不能直接杀。客户端只能操作本地线程，Ctrl+C 是另起一个连接发送 kill query 命令，后端仍需走完整套 kill 逻辑。
- **Q：** 回滚中的事务，应重启 MySQL 还是等它跑完？
  **A：** 重启后该做的回滚仍要做，从恢复速度看应等它自己完成；若它占锁/占 IO 影响其他语句，则先主备切换到新主库，再等它回滚完。

## 33 | 我查这么多数据，会不会把数据库内存打爆？

**一句话核心**：不会——server 层边读边发（net_buffer + socket send buffer），InnoDB 层用改进的 LRU（young/old 分区）防止全表扫描冷数据淘汰热数据。

**关键知识点**
- server 层“边读边发”：取一行写 net_buffer（`net_buffer_length`，默认 16k），写满就调网络接口发出去，发送成功清空再取下一行；socket send buffer 写满则暂停取数据。整个查询占用 server 内存最大就是 net_buffer_length，不会到 200G。
- 客户端接收慢会拖慢服务端查询（发不出去），可能造成长事务；正常业务建议 `mysql_store_result`（本地缓存结果），大结果集才用 `mysql_use_result`（读一行处理一行）。
- 状态区分：`Sending to client` 表示服务端网络栈（socket send buffer）写满、在等客户端接收；`Sending data` 只表示“正在执行”（任意执行阶段），二者常被混淆。
- Buffer Pool 作用：既配合 WAL 加速更新，也加速查询；命中率（`Buffer pool hit rate`）一般要 99% 以上；`innodb_buffer_pool_size` 建议为物理内存 60%~80%。
- 基本 LRU 问题：全表扫描一个冷表会把整个 Buffer Pool 热数据全淘汰，命中率骤降。
- 改进 LRU：链表按 5:3 分 young（前 5/8）和 old（后 3/8）区。新读入的数据页放 old 区头部（LRU_old），old 区数据页再次访问时，若在链表中存在超过 1 秒才移到 young 区头部，短于 1 秒位置不变（`innodb_old_blocks_time` 默认 1000ms）。
- 效果：全表扫描的数据页第一次到最后一次访问间隔 < 1 秒，始终留在 old 区很快被淘汰，不污染 young 区，保护正常业务命中率。

**面试高频问答**
- **Q：** 全表扫描 200G 大表会把内存打爆吗？
  **A：** 不会。server 层边算边发，只占 net_buffer_length + socket send buffer 大小；InnoDB 层有 LRU 淘汰，且改进的 LRU 保证冷数据扫描不冲垮热数据。但仍耗 IO，高峰期不宜在主库执行。
- **Q：** InnoDB 改进的 LRU 算法怎么防止全表扫描污染 Buffer Pool？
  **A：** 链表分 young(5/8)/old(3/8) 区，新页进 old 头部，只有在链表存在超 1 秒再被访问才升到 young。全表扫描的页访问间隔 < 1 秒，留在 old 很快淘汰，不影响 young 区热数据。
- **Q：** `Sending to client` 和 `Sending data` 有何区别？
  **A：** Sending to client 是服务端网络栈写满、在等客户端接收结果；Sending data 只表示语句正在执行（可能在等锁等任意阶段），不代表在发数据。
- **Q：** 大量线程处于 Sending to client 怎么办？
  **A：** 让业务优化查询结果集大小、改用 mysql_store_result；应急可调大 net_buffer_length（最大 1G），让结果缓存在 net_buffer 中，执行器视为“已发出”，释放锁资源。

## 34 | 到底可不可以使用 join？

**一句话核心**：能用上被驱动表索引的 Index Nested-Loop Join（NLJ）可以放心用；用不上索引退化成 Block Nested-Loop Join（BNL）则尽量避免；无论哪种都应让小表做驱动表。

**关键知识点**
- Index Nested-Loop Join（NLJ）：被驱动表关联字段有索引时使用。遍历驱动表每行，拿字段值走被驱动表索引树查找。驱动表 N 行、被驱动表 M 行，复杂度约 N + N*2*log₂M。
- 能不能用 join：NLJ 比拆成多条单表 SQL 好——同样扫描行数但少了多次网络交互和客户端拼接。
- NLJ 驱动表选择：N 对扫描行数影响远大于 M（N 扩大 1000 倍扫描扩大 1000 倍，M 扩大 1000 倍扫描扩大不到 10 倍），应让小表做驱动表。
- Simple Nested-Loop Join：被驱动表无索引时每次全表扫描，M*N 次扫描太笨重，MySQL 实际不用它。
- Block Nested-Loop Join（BNL）：被驱动表无索引时使用。把驱动表数据读入 join_buffer（内存无序数组），再扫描被驱动表逐行与 join_buffer 对比。总扫描 M+N，内存判断 M*N 次；虽和 Simple NLJ 时间复杂度同，但内存判断快很多。
- join_buffer 放不下时分段（Block 由来）：`join_buffer_size`（默认 256k）不够就分 K 段，被驱动表被扫 K 次，扫描行数 N+λ*N*M；join_buffer_size 越大分段越少。
- 判断是否用了 BNL：explain 的 Extra 出现 `Block Nested Loop`。
- “小表”的准确定义：两个表各自按 where 条件过滤后，参与 join 的字段总数据量小的那个才是小表，应作驱动表（不是单纯看行数或表大小）。

**面试高频问答**
- **Q：** 到底能不能用 join？
  **A：** 看 explain 的 Extra——若能用上被驱动表索引（NLJ）就可以用，性能不比拆分单表差；若出现 Block Nested Loop（BNL，用不上索引），大表 join 会多次扫描被驱动表、占大量资源，尽量不用。
- **Q：** NLJ 和 BNL 有什么区别？
  **A：** NLJ 被驱动表走索引树搜索，逐行匹配；BNL 被驱动表无索引，把驱动表放 join_buffer 无序数组，全表扫描被驱动表逐行内存比对，join_buffer 放不下要分段多次扫描被驱动表。
- **Q：** join 应该用大表还是小表做驱动表？
  **A：** 总是用小表做驱动表。NLJ 里驱动表行数 N 影响最大；BNL 里 join_buffer 不够时小表驱动分段少、扫描少。“小表”指按条件过滤后参与 join 字段总数据量小的表。
- **Q：** join 很慢怎么快速优化（BNL 场景）？
  **A：** 给被驱动表关联字段加索引转成 NLJ；或调大 join_buffer_size 减少被驱动表扫描次数。

## 35 | join 语句怎么优化？

**一句话核心**：核心是让 join 用上被驱动表索引触发 BKA 算法——NLJ 靠 MRR+BKA 批量顺序读优化，BNL 靠加索引或临时表转 BKA，MySQL 不支持 hash join 可在业务端模拟。

**关键知识点**
- MRR（Multi-Range Read）：目的是把回表的随机读变顺序读。将索引查到的主键 id 放入 read_rnd_buffer 排序后再按序回主键索引查，`read_rnd_buffer_size` 控制大小。需 `set optimizer_switch="mrr_cost_based=off"` 稳定启用。核心前提是范围/多值查询能拿到足够多主键才体现顺序性优势。
- BKA（Batched Key Access，5.6+）：对 NLJ 的优化。从驱动表一次多取若干行放 join_buffer，批量传给被驱动表，配合 MRR 顺序回表。启用：`set optimizer_switch='mrr=on,mrr_cost_based=off,batched_key_access=on'`。
- BNL 三大影响：多次扫描被驱动表占磁盘 IO；M*N 次对比占大量 CPU；多次扫描冷表（执行超 1 秒）把冷表数据页移到 young 区、或使正常访问页无法进 young 区，污染 Buffer Pool 影响命中率且持续时间长。
- BNL 优化方向：给被驱动表 join 字段加索引，转成 BKA。
- 临时表方案（不适合直接建索引时）：把被驱动表满足条件的数据 `insert into` 一张带索引的临时表，再 join 触发 BKA。文中例子从 1 分 11 秒（10 亿次判断）优化到不到 1 秒。
- hash join：MySQL 不支持（官方 roadmap 未排期）。若 join_buffer 用哈希表而非无序数组，10 亿次判断可降为 100 万次 hash 查找；可在业务端用 set/数组模拟，理论比临时表更快。

**面试高频问答**
- **Q：** MRR 优化原理是什么，为什么能提速？
  **A：** 普通索引查到主键 id 后回表是随机读，性能差。MRR 先把主键 id 攒到 read_rnd_buffer 排序，再按主键递增顺序回表，把随机读变接近顺序读。前提是范围查询能拿到足够多主键。
- **Q：** BKA 是什么，和 NLJ、MRR 什么关系？
  **A：** BKA 是 NLJ 的优化，用 join_buffer 从驱动表一次批量取多行传给被驱动表，从而能利用 MRR 顺序回表。BKA 依赖 MRR，需同时开启 mrr 和 batched_key_access。
- **Q：** BNL 算法慢且不适合建索引时怎么优化？
  **A：** 用临时表：把被驱动表过滤后的小结果集插入带索引的临时表，再让驱动表和临时表 join 触发 BKA；或业务端模拟 hash join。
- **Q：** MySQL 支持 hash join 吗？
  **A：** （8.0.18 之前）不支持。可在应用端把小表存成 hash 结构，再拿大表过滤结果逐行匹配，理论上比临时表方案更快。

## 36 | 为什么临时表可以重名？

**一句话核心**：临时表只对创建它的 session 可见、可与普通表/其他 session 临时表重名、session 结束自动删除——因其 table_def_key 和磁盘文件名都带了 server_id+thread_id 加以区分。

**关键知识点**
- 临时表 ≠ 内存表：内存表是 `engine=memory`（数据在内存、重启清空但表结构在）；临时表是 `create temporary table`，可用任意引擎（InnoDB/MyISAM 写磁盘，也可用 Memory）。
- 临时表特性：只能被创建它的 session 访问，对其他线程不可见；可与普通表同名（同名时 show create、增删改查都访问临时表）；show tables 不显示；session 结束自动删除。
- 适合 join 优化/复杂查询：不同 session 临时表可重名，多会话并发不冲突；自动回收，异常断开/重启无需手动清理中间表。
- 典型场景：分库分表的跨库查询——在汇总库建临时表，各分库把结果插入临时表，再统一 order by/group by/limit。
- 为什么能重名：磁盘上 frm/数据文件名前缀是 `#sql{进程id}_{线程id}_序列号`（5.7+ 用临时表空间存数据，不再单独建 ibd）；内存中普通表 table_def_key = 库名+表名，临时表额外加 server_id+thread_id，故不冲突。每个线程维护自己的临时表链表，操作时先查链表优先用临时表，session 结束对链表逐个 DROP TEMPORARY TABLE。
- 临时表与 binlog：仅 `binlog_format=statement/mixed` 时才记录临时表操作（row 格式不记录）；statement 下需把 create/drop temporary table 同步给备库。备库应用线程共用，靠 binlog 里记录的主库线程 id 构造不同 table_def_key，区分同名临时表。
- drop table 记 binlog 会被改写成 `DROP TABLE `t` /* generated by server */`（因一次可删多表且要剔除备库不存在的临时表）。

**面试高频问答**
- **Q：** 临时表和内存表是一回事吗？
  **A：** 不是。内存表是 engine=memory，数据存内存、重启清空、对所有 session 可见；临时表是 create temporary table，可用任意引擎（可写磁盘），只对创建它的 session 可见、session 结束自动删除。
- **Q：** 为什么不同 session 能创建同名临时表？
  **A：** 磁盘文件名带进程id+线程id前缀，内存 table_def_key 在库名+表名基础上加了 server_id+thread_id，所以互不冲突，也能和普通表重名。
- **Q：** 临时表操作会写 binlog 吗？
  **A：** binlog_format=row 时不记录；statement/mixed 时会记录，因为要同步到备库（否则备库执行依赖临时表的语句会报表不存在），主库线程退出时还要写 DROP TEMPORARY TABLE 传给备库。
- **Q：** 为什么临时表用来做 join 优化很合适？
  **A：** 可重名，多会话并发执行同一优化逻辑不会建表冲突；session 结束自动回收，异常断开或重启无需手动清理中间表。

## 37 | 什么时候会使用内部临时表？

**一句话核心**：需要“暂存中间结果并用到二维表特性（唯一约束、累加统计）”时用内部临时表——典型是 union（去重）、group by（分组统计）、部分 order by。

**关键知识点**
- 三种中间数据结构：sort_buffer（有序数组，排序用）、join_buffer（无序数组，join 用）、内部临时表（二维表结构）。用到二维表特性就用临时表。
- union 执行：创建带唯一索引的内存临时表，第一个子查询结果插入，第二个子查询逐行插入，靠主键唯一约束去重，最后读出返回并删表。explain 见 `Using temporary`。
- union all 不去重，直接依次把子查询结果发客户端，不需要临时表（explain 可能 Using index 覆盖索引）。
- group by 执行：创建内存临时表（分组字段为主键），扫描并对每个分组累加计数，遍历后按分组字段排序返回。explain 常见 Using index + Using temporary + Using filesort。
- 内存临时表大小由 `tmp_table_size`（默认 16M）控制，超限则转成磁盘临时表（默认 InnoDB 引擎）。
- group by 优化一——索引：若分组字段有序（如用 generated column 加索引让 id%100 有序），可顺序扫描累加，无需临时表和排序。确认标准：explain 无 Using temporary、无 Using filesort。
- group by 优化二——直接排序：加 hint `SQL_BIG_RESULT` 提示数据量大，优化器直接用 sort_buffer 排序（不够用则借磁盘临时文件），得到有序数组后统计，跳过临时表。
- 不需要排序时加 `order by null` 可跳过 group by 默认的最后排序阶段。

**面试高频问答**
- **Q：** MySQL 什么时候会用内部临时表？
  **A：** 无法边读边出结果、需暂存中间结果且要用到二维表特性时，比如 union 需要唯一约束去重、group by 需要额外字段累加计数。可从 explain 的 Using temporary 判断。
- **Q：** group by 为什么需要临时表，怎么优化？
  **A：** 因为分组字段无序，需临时表记录并统计各分组计数。优化：①让分组字段走有序索引（generated column + 索引），免临时表和排序；②数据量大时用 SQL_BIG_RESULT 直接走排序算法；③不需排序加 order by null；④调大 tmp_table_size 避免转磁盘临时表。
- **Q：** union 和 union all 在临时表使用上有何区别？
  **A：** union 要去重，需建带唯一索引的临时表靠唯一约束过滤重复；union all 不去重，直接把子查询结果拼接发给客户端，不用临时表。
- **Q：** 内存临时表什么时候转磁盘临时表？
  **A：** 内存临时表大小超过 tmp_table_size（默认 16M）时转为磁盘临时表，默认引擎 InnoDB（索引组织表，按主键有序）。

## 38 | 都说 InnoDB 好，那还要不要使用 Memory 引擎？

**一句话核心**：Memory 引擎是堆组织表、数据无序存放、只有表锁、重启丢数据，生产环境不建议用普通内存表；仅内存临时表场景可用。

**关键知识点**
- 数据组织：InnoDB 是索引组织表（数据放主键 B+ 树、有序），Memory 是堆组织表（数据按写入顺序单独存数组，索引存数据位置）。所以 select * 时 InnoDB 主键 0 在第一行，Memory（hash 索引无序、按写入顺序全表扫描）0 在最后一行。
- 引擎差异：InnoDB 数据总是有序、有空洞要在固定位置插入，Memory 找到空位就插；数据位置变化 InnoDB 只改主键索引，Memory 要改所有索引；InnoDB 主键查一次索引/普通索引查两次，Memory 所有索引地位相同；Memory 不支持 Blob/Text，varchar(N) 当 char(N) 定长存储。
- 索引类型：Memory 默认 hash 索引，范围查询用不上需全表扫描；也支持 B-Tree 索引（`using btree`），范围查询优化器会选 B-Tree。
- 不建议生产用的两大原因：①锁粒度——不支持行锁只有表锁，一有更新就阻塞该表所有读写，并发差；②持久化——重启数据全部清空。
- 高可用下的隐患：备库重启内存表清空 → 应用 update 报“找不到行” → 主备同步停止；MySQL 重启后会往 binlog 写 `DELETE FROM t1`，双 M 结构下备库重启的 delete 会传回主库把主库内存表数据也删空。
- 建议：普通内存表用 InnoDB 代替（InnoDB 有行锁并发好、数据安全，热数据也缓存在 Buffer Pool，读性能不差）。
- 例外——内存临时表适合用 Memory：不被其他线程访问（无并发问题）、重启本就要删（无持久化问题）、备库临时表不影响主库；且不写磁盘更快、hash 索引查找快、数据量小占内存有限。

**面试高频问答**
- **Q：** InnoDB 和 Memory 引擎的数据组织有何不同？
  **A：** InnoDB 是索引组织表，数据存主键 B+ 树上有序，二级索引存主键；Memory 是堆组织表，数据按写入顺序单独存数组，索引（默认 hash）只存数据位置、无序。
- **Q：** 为什么生产环境不建议用 Memory 内存表？
  **A：** 两点：只支持表锁不支持行锁，一更新就阻塞全表读写，并发差；重启数据全丢，高可用下备库重启会导致主备同步停止，双 M 结构下 delete 还会传回删空主库数据。
- **Q：** Memory 表支持哪些索引，范围查询能用索引吗？
  **A：** 默认 hash 索引（等值快、范围查询用不上要全表扫描），也支持 B-Tree 索引；范围查询优化器会选 B-Tree 索引。
- **Q：** 既然不建议用内存表，为什么内存临时表又推荐？
  **A：** 内存临时表刚好规避了内存表两大缺点：只被本 session 访问无并发/表锁问题、重启本就要删无持久化顾虑、备库临时表不影响主库；还不写磁盘、hash 索引查找快，join 优化中比 InnoDB 临时表更快。

## 39 | 自增主键为什么不是连续的？

**一句话核心**：自增值不保证连续，只保证递增；唯一键冲突、事务回滚、批量申请浪费都会造成空洞。

**关键知识点**
- 自增值保存位置：MyISAM 存在数据文件；InnoDB 8.0 之前存在内存（重启后按 max(id)+1 重建，可能回退），8.0 起用 redo log 持久化，重启后不变。
- 自增值修改机制：插入时 id 指定为 0/null/未指定则用当前 AUTO_INCREMENT；若指定值 X≥当前自增值 Y，则把自增值改为大于 X 的第一个值（步长 auto_increment_increment，初值 auto_increment_offset，默认都为 1）；X<Y 则不变。
- 自增值在真正插入前就已修改，且申请后不回退——所以唯一键冲突、事务回滚都会留下空洞（不连续原因 1、2）。
- 不回退的原因是性能：若允许回退会导致后续申请到已存在的 id 引发主键冲突，避免它需要查主键树或扩大锁范围，都会降低并发。
- 自增锁不是事务锁，申请完即释放。参数 innodb_autoinc_lock_mode：0=语句级锁（5.0 老策略）；1（默认）=普通 insert 申请后即释放，insert…select 等批量语句持锁到语句结束；2=全部申请后即释放。
- 批量插入（insert…select/replace…select/load data）申请自增 id 采用「2 的倍数」策略：每次申请数是上次的两倍（1、2、4…），用不完就浪费，造成空洞（不连续原因 3）。
- 生产建议：innodb_autoinc_lock_mode=2 且 binlog_format=row，兼顾并发与一致性。

**面试高频问答**
- **Q：** 自增主键为什么会不连续？
  **A：** 三种原因：①唯一键冲突（自增值已加、插入失败不回退）；②事务回滚（同样不回退）；③批量插入按 2 的倍数一次性申请 id，多申请的被浪费。
- **Q：** 为什么唯一键冲突/回滚后 MySQL 不把自增值改回去？
  **A：** 为了性能。回退会导致后面事务申请到已被其他事务用掉的 id，产生主键冲突；避免它要么查主键树判重、要么扩大自增锁粒度，都会严重影响并发，所以干脆不回退。
- **Q：** innodb_autoinc_lock_mode 为什么默认是 1 而不是 2？insert…select 为何要语句级锁？
  **A：** 为数据一致性。statement 格式 binlog 下，若批量插入申请一个释放一个，主库生成的 id 不连续但 binlog 按语句串行到备库会生成连续 id，导致主备不一致。默认 1 让批量语句持锁到结束、生成连续 id 规避此问题；若设为 2 必须配合 binlog_format=row。

## 40 | insert 语句的锁为什么这么多？

**一句话核心**：普通 insert 很轻量，但 insert…select、唯一键冲突等特殊 insert 会给其他资源加读锁甚至造成死锁。

**关键知识点**
- insert…select 在可重复读 + binlog_format=statement 下，会对 select 的源表扫描到的记录和间隙加 next-key lock（读锁），目的是保证 binlog 与数据一致（防止并发 insert 先执行后写 binlog 造成主备不一致）。
- insert…select 只锁需要访问的资源，不锁全表；但若源表与目标表相同（循环写入），会全表扫描并用临时表。
- 循环写入需临时表原因：一边遍历一边写回原表，可能读到自己刚插入的记录参与计算，与语义不符（RR 下事务能看到自己刚修改的数据）。优化：先写入用户临时表再插回。
- insert 唯一键冲突：并不是简单报错，会在冲突的唯一索引值上加共享 next-key lock（S 锁），作用是防止该行被删。主键冲突和唯一索引冲突加的都是 next-key lock（官方文档描述有误，作者已提 bug 被 verified）。
- insert 死锁经典场景：三个 session 对同一唯一值 c=5，session A 先插入持记录锁；B、C 发现冲突各加 S 锁（间隙锁加成功、记录锁等待）；A 回滚后 B、C 都要加写锁（插入意向锁），互相等待形成死锁。
- insert…on duplicate key update：冲突时直接加排他 next-key lock（写锁）；多个唯一约束冲突时按索引顺序改第一个冲突行；affected rows 返回 2（insert 和 update 各计 1）但实际只更新一行。

**面试高频问答**
- **Q：** insert…select 为什么要对源表加锁？
  **A：** RR + statement binlog 下，为保证日志与数据一致。若不加锁，并发 insert 可能先执行后写 binlog，备库按 binlog 顺序重放会把本不该出现的行也写进目标表，造成主备不一致。
- **Q：** insert 唯一键冲突时加什么锁？有什么建议？
  **A：** 在冲突的唯一值上加共享 next-key lock（S 锁）。所以遇唯一键冲突报错后应尽快提交/回滚，避免持锁时间过长阻塞他人。
- **Q：** 描述一个 insert 唯一键冲突导致的死锁。
  **A：** A 插入 c=5 持记录锁；B、C 同插 c=5 发现冲突各申请 S 锁并加上间隙锁、记录锁等待；A 回滚后 B、C 都要升级为写锁插入，彼此等待对方行锁形成死锁，一方被回滚。

## 41 | 怎么最快地复制一张表？

**一句话核心**：三种复制方法——mysqldump、select…into outfile+load data、物理拷贝表空间；物理拷贝最快但限制多。

**关键知识点**
- 方法一 mysqldump：导出为一组 INSERT 语句。常用参数 --single-transaction（一致性快照不加表锁）、--add-locks=0、--no-create-info（不导表结构）、--set-gtid-purged=off、--where 过滤、--skip-extended-insert（一行一条 insert）。用 source 导入（source 是客户端命令，实际执行的是 INSERT，binlog/slowlog 记录的也是 INSERT）。
- 方法二 select…into outfile 导 CSV + load data infile 导入：文件生成在服务端，受 secure_file_priv 限制（empty=不限、路径=限定目录、NULL=禁止）；不覆盖同名文件；不生成表结构（可用 mysqldump --tab 同时导结构和数据）。
- load data 分两种：不加 local 读服务端文件（受 secure_file_priv 限制）；加 local 读客户端文件（客户端先传给服务端）。statement binlog 下 load data 会把 csv 内容写进 binlog，备库重放时用带 local 的语句避免受 secure_file_priv 影响。
- 方法三 物理拷贝（可传输表空间，MySQL 5.6+）：create table r like t → alter table r discard tablespace（删 r.ibd）→ flush table t for export（生成 t.cfg，此时 t 只读）→ 拷贝 t.cfg/t.ibd 为 r.cfg/r.ibd（注意文件权限归 MySQL 进程）→ unlock tables → alter table r import tablespace。
- 直接 cp .frm/.ibd 不行，因为数据字典没注册这个表。
- 三种对比：物理拷贝最快（尤其大表）但只能全表拷贝、需登录服务器、源和目标都必须 InnoDB；mysqldump 可 where 过滤但不支持复杂 join 条件；outfile 最灵活支持所有 SQL 但一次只能一张表且表结构需另导。后两者是逻辑备份、可跨引擎。

**面试高频问答**
- **Q：** 复制表有哪几种方法，怎么选？
  **A：** mysqldump 生成 INSERT（可 where 过滤）、select…into outfile 导 CSV 再 load data（最灵活支持所有 SQL）、物理拷贝表空间（最快、适合大表但只能全表且限 InnoDB）。大表整表拷贝选物理，需过滤/跨引擎选逻辑方式。
- **Q：** 为什么不能直接拷贝 .frm 和 .ibd 文件到新库？
  **A：** InnoDB 表除物理文件外还要在数据字典中注册，直接拷文件系统识别不了。正确做法是用可传输表空间（discard/import tablespace）流程。
- **Q：** 物理拷贝流程中 flush table t for export 的作用？
  **A：** 生成 t.cfg 元信息文件并把表置为只读，保证拷贝期间 .ibd 数据一致；执行 unlock tables 后释放读锁并删除 cfg。

## 42 | grant 之后要跟着 flush privileges 吗？

**一句话核心**：规范使用 grant/revoke 不需要 flush privileges，因为它们同时更新磁盘表和内存、即时生效。

**关键知识点**
- 权限数据双存：磁盘在 mysql 库的权限表，内存在数组/hash 结构。grant/revoke 会同时改磁盘和内存，权限判断用的是内存数据。
- 权限按范围从大到小：全局（mysql.user 表 / 内存 acl_users 数组）、库级 db（mysql.db / acl_dbs）、表级（mysql.tables_priv）、列级（mysql.columns_priv，表列合存于内存 column_priv_hash）。
- 对已存在连接的影响不同：全局权限在连接建立时拷贝到线程对象，grant/revoke 影响不到已有连接（如 revoke super 后老连接仍能 set global）；db/表/列权限用全局数组/hash，revoke 立即影响已有连接。特例：已 use 某库的会话，库权限存在会话变量中，切库前一直有效。
- flush privileges 作用：清空内存权限数组，从磁盘表重新加载重建，使内存与磁盘一致。
- 只有当内存与磁盘不一致时才需要 flush，而这种不一致通常源于直接用 DML（delete/insert/update）操作系统权限表这类不规范操作——应避免。
- 用户标识是 user+host，ua@ip1 与 ua@ip2 是不同用户。identified by 写法会顺带改密码，不建议。

**面试高频问答**
- **Q：** grant 之后必须 flush privileges 吗？
  **A：** 不需要。grant/revoke 同时更新磁盘表和内存并即时生效，内存磁盘本就同步。flush privileges 只在两者不一致（如直接用 DML 改权限表）时才需要，用来以磁盘为准重建内存。
- **Q：** revoke 一个正在连接用户的权限，会马上生效吗？
  **A：** 看权限类型。全局权限在连接时已拷贝到线程对象，对已有连接不生效；db/表/列权限走全局数组，立即生效（但已 use 该库的会话因权限缓存在会话变量中，切库前仍有效）。
- **Q：** 什么情况会出现权限内存与磁盘不一致？
  **A：** 直接用 delete/insert 等 DML 操作 mysql.user 等系统表。比如 delete 删了用户但内存还在，导致删后仍能登录、重新创建同名用户又报已存在，需 flush privileges 修复。

## 43 | 要不要使用分区表？

**一句话核心**：分区表对引擎层是多个表、对 server 层是一个表；主要坑是第一次访问要打开所有分区、以及所有分区共用一个 MDL 锁。

**关键知识点**
- 组织形式：一个 .frm + 每个分区一个 .ibd。对引擎层是 N 个表（各自独立加锁），对 server 层是 1 个表。
- 引擎层行为：加锁按分区隔离。InnoDB 间隙锁只落在命中的那个分区（相邻分区间隙不连续，跨分区插入不被挡）；MyISAM 表锁只锁住命中的分区。
- 第一次访问分区表需打开所有分区文件。MyISAM 用通用分区策略（generic partitioning），实现粗糙，分区多 + open_files_limit 默认 1024 时会「打开文件过多」报错。
- InnoDB 5.7.9 起用本地分区策略（native partitioning），文件过多时会淘汰已打开句柄，不会报错（受 innodb_open_files 控制），但第一次访问仍要打开所有分区。
- 5.7.17 起 MyISAM 分区表标记弃用，8.0 起禁止创建，只有 InnoDB、NDB 支持本地分区策略。
- server 层行为：整个分区表共用一个 MDL 锁，任一 session 持有会阻塞对其他分区的 DDL——所以分区表做 DDL 影响更大（普通分表则无此冲突）。
- 分区裁剪：按 where 条件结合分区规则只访问必要分区；where 不含分区 key 则访问所有分区。
- 应用场景优势：对业务透明、代码简洁；可用 alter table…drop partition 快速删历史数据（直接删分区文件，比 delete 快、影响小）。分区主键必须包含分区字段。
- 实务建议：分区别太多、别提前预留太多（用前再建），无数据历史分区及时 drop。

**面试高频问答**
- **Q：** 为什么很多公司规范不让用分区表？
  **A：** 两个绕不开的问题：①第一次访问需打开所有分区（MyISAM 甚至可能因文件过多报错）；②整表共用一个 MDL 锁，DDL 会阻塞所有分区的操作，影响大。分区过多还会引发性能/主从延迟问题。
- **Q：** 分区表对引擎层和 server 层分别是几个表？
  **A：** 引擎层是 N 个表（各分区独立加锁、独立 .ibd 文件）；server 层是 1 个表（共用一个 MDL 锁）。MDL 锁之后才按分区规则只访问必要分区。
- **Q：** 分区表相比手工分表有什么优劣？
  **A：** 优势：对业务透明、代码简洁，可 drop partition 快速清历史数据。劣势：共用 MDL 锁使 DDL 影响大，第一次访问打开所有分区。引擎层看两者无本质差别，区别在 server 层由谁决定路由。

## 44 | 答疑文章（三）：说一说这些好问题

**一句话核心**：汇总 left join 驱动表、Simple Nested Loop 性能、distinct 与 group by、备库自增 id 一致性四个好问题。

**关键知识点**
- left join 驱动表不一定是左表：`on` 里放全部等值条件（Q1）会保留左表未匹配行（补 NULL），用 BNL、左表驱动；把被驱动表字段放 `where` 做等值判断（Q2）会因 NULL 与任何值比较结果为 NULL 而过滤掉未匹配行，语义等价于 join，优化器改写成 join 并用有索引的表当被驱动表（NLJ）。
- 结论：需要 left join 语义时，被驱动表字段的判断条件必须写在 on 里，不能写 where；否则会被优化成 join。普通 join 则 on 与 where 放法无区别（都被改写成 join+where）。
- Simple Nested Loop Join 比 BNL 慢的原因：①被驱动表全表扫描要多次从磁盘读数据、污染 Buffer Pool 命中率；②多次访问被驱动表数据页易被放到 Buffer Pool 头部；③即便全在内存，找「下一条记录」是指针操作，而 join_buffer 是数组遍历成本更低。
- distinct 与 group by（无聚合函数、字段无索引）性能相同：`select distinct a from t` 与 `select a from t group by a order by null` 语义与执行流程一致——都建临时表在字段 a 上建唯一索引，遍历插入、冲突跳过，最后返回。
- 备库自增 id 一致性：statement binlog 下每条 insert 前有 `SET INSERT_ID=n`，指定该语句用的自增值。即使 binlog 写入顺序与主库申请 id 顺序不同，SET INSERT_ID 也跟着对应语句，备库用的值与主库一致，不会错乱（row 格式本就直接记录每行值，无此问题）。

**面试高频问答**
- **Q：** left join 左边一定是驱动表吗？
  **A：** 不一定。如果把被驱动表字段放在 where 里做等值判断，left join 会被优化器改写成 join，可能用另一张有索引的表做驱动表（NLJ）。要保住 left join 语义，条件必须写在 on 里。
- **Q：** BNL 和 Simple Nested Loop Join 都要判断 M*N 次，为什么 BNL 快得多？
  **A：** Simple NLJ 每轮对被驱动表全表扫描，反复读磁盘、污染 Buffer Pool，且逐条找记录是指针操作；BNL 把驱动表读进 join_buffer 数组，被驱动表只遍历一次、数组遍历成本更低。
- **Q：** statement binlog 下，主备 insert 顺序不同会不会导致自增 id 不一致？
  **A：** 不会。每条 insert 前有 SET INSERT_ID=n 固定该语句的自增值，随语句一起写入 binlog，备库用的值与主库相同。

## 45 | 自增 id 用完怎么办？

**一句话核心**：不同自增 id 到达上限行为各异——表自增 id 报主键冲突，row_id 归零覆盖，trx_id 有理论脏读 bug，thread_id 处理得最好。

**关键知识点**
- 表定义自增 id：默认 int unsigned 上限 2³²-1。达到上限后再申请值不变，导致下次插入报主键冲突。可能被用完，建议用 8 字节 bigint unsigned。
- InnoDB 系统 row_id：无主键表用全局 dict_sys.row_id，实现是 8 字节但只写低 6 字节，写入范围 0 到 2⁴⁸-1。达到上限后归 0 循环，新数据会覆盖已有相同 row_id 的行——数据丢失。所以应主动建自增主键（报冲突比覆盖数据好，可靠性优先于可用性）。
- Xid：server 层维护，来自全局 global_query_id（8 字节，纯内存重启清零，每条语句 +1，事务第一条语句时赋给 Xid）。要求只在同一个 binlog 文件内不重复即可；重启会新建 binlog 文件保证唯一。理论上 global_query_id 绕回 2⁶⁴ 才会重复，概率可忽略。
- InnoDB trx_id：来自全局 max_trx_id，申请一个 +1，持久化存储、重启不清零。只读事务不分配 trx_id（好处：减小一致性视图活跃事务数组、减少 trx_id 申请锁冲突）；显示的很大的 trx_id 是「事务指针地址 + 2⁴⁸」临时算出来的（保证只读事务显示值大于读写事务）。update/delete 因标记删除放入 purge 队列会额外 +1，一个事务至少加 2；后台内部事务也会占用 trx_id。
- trx_id 脏读 bug：max_trx_id 达 2⁴⁸-1 后从 0 重新计数，导致新事务 trx_id 变小、被判定为「低水位以下可见」，产生脏读；且重启不清零，是必现 bug。TPS 50 万约 17.8 年后触发，理论存在但现实几乎不会到。
- thread_id：全局 thread_id_counter，4 字节，达 2³²-1 后重置为 0。但通过唯一数组（insert_unique）保证 show processlist 里不会出现重复 thread_id，是处理得最好的一个。

**面试高频问答**
- **Q：** 表自增主键 id 用完了会怎样？
  **A：** 再申请时值不变，下次插入报 Duplicate entry 主键冲突。int unsigned 上限约 42 亿，频繁增删的表可能用完，应改用 bigint unsigned。
- **Q：** row_id 和表自增 id 用完的行为有何不同？为什么建议建自增主键？
  **A：** 无主键表的 row_id（6 字节，上限 2⁴⁸-1）到顶后归 0 循环，覆盖已有行导致数据丢失；表自增 id 到顶只是报主键冲突（插入失败）。覆盖影响可靠性、冲突只影响可用性，可靠性优先，所以应主动建自增主键。
- **Q：** 为什么只读事务不分配 trx_id？trx_id 用完会怎样？
  **A：** 只读事务不影响可见性判断，不分配 trx_id 可减小一致性视图活跃数组、减少 trx_id 申请的锁冲突。max_trx_id 持久化、达上限后从 0 重新计数会导致新事务 id 变小被误判可见，产生必现脏读 bug（但需实例跑约 17.8 年@50 万 TPS，现实几乎不会到）。

### 01 | 存储：一个完整的数据存储过程

**一句话核心**：MySQL 数据存储分「建库 → 定字段 → 建表 → 插数据」四步，层级从大到小是「数据库服务器 → 数据库 → 表 → 行列」。

**关键知识点**
- 数据库是最大存储单元，必须先建库再建表：`CREATE DATABASE demo;`，删库 `DROP DATABASE demo;`，查库 `SHOW DATABASES;`。
- 建表要指明数据库、最后一个字段后不能加逗号：`CREATE TABLE demo.test (barcode text, goodsname text, price int);`。
- 常用查看语句：`USE demo;` 选库、`SHOW TABLES;` 查表、`DESCRIBE demo.test;` 查表结构（返回 Field/Type/Null/Key/Default/Extra 六列）。
- 空值（NULL）不等于空字符串：空字符串长度为 0，空值长度为「空」且占用空间。
- 主键三特征：唯一、非空、能唯一标识记录；一个表只能有一个主键。任何业务字段都可能重复/重用，推荐加自增字段做主键：`ALTER TABLE demo.test ADD COLUMN itemnumber int PRIMARY KEY AUTO_INCREMENT;`。
- 插入建议写全字段名，可读性好且不易错：`INSERT INTO demo.test (barcode,goodsname,price) VALUES ('0001','本',3);`。每条 SQL 必须以分号结尾。

**面试高频问答**
- **Q：** MySQL 的存储层级结构是怎样的？
  **A：** 从大到小是数据库服务器 → 数据库 → 数据表 → 表的行与列。数据库是最大存储单元，所以要先建库才能建表。
- **Q：** MySQL 里 NULL 和空字符串一样吗？
  **A：** 不一样。空字符串长度为 0，空值（NULL）长度是「空」，且 NULL 在 MySQL 中是占用存储空间的。

### 02 | 字段：字段类型该怎么定义？

**一句话核心**：类型选型的第一原则是「先保证不超范围再省空间」；金额必用 DECIMAL，浮点数不精准，字符串非主键用 TEXT，时间优先 DATETIME。

**关键知识点**
- 整数 5 种：TINYINT / SMALLINT / MEDIUMINT / INT / BIGINT，占用字节递增（1/2/3/4/8 字节）、取值范围递增。选型权衡「存储空间 vs 可靠性」，优先保证不超范围——系统故障成本远高于几字节存储成本。SMALLINT 无符号上限仅 65535，商品编号这类会持续增长的用 INT。
- 浮点数 FLOAT（4 字节）/ DOUBLE（8 字节）/ REAL（默认等于 DOUBLE）**不精准**：底层用二进制存小数，尾数非 0/5 无法精确表达只能近似。求和会出误差（如 0.47+0.44+0.19 得 1.0999999999999999）。
- 金额、对精度要求高的场景必须用定点数 `DECIMAL(M,D)`：M 是总位数（≤65），D 是小数位数（D<M）。它把整数和小数部分拆开分别转十六进制存储，所以精准无误差。例：`DECIMAL(10,2)` 存金额，`DECIMAL(10,3)` 存称重数量。
- 字符串类型：`CHAR(M)` 定长（不足补空格，取多少存多少）；`VARCHAR(M)` 变长（按实际长度存，但需预定义最大长度）；`TEXT` 系统按实际长度存、无需预定义长度。TEXT 4 种：TINYTEXT(255)/TEXT(65535)/MEDIUMTEXT(16M)/LONGTEXT(4G)。**TEXT 不能做主键**（长度不确定），主键场景用 CHAR/VARCHAR。ENUM 单选、SET 多选，需预先知道所有取值。
- 时间类型：YEAR、TIME、DATE、DATETIME、TIMESTAMP。DATETIME 信息最完整、取值范围最大，优先用它。TIME 取值范围超过 24 小时是因为它还能表示时间间隔。
- BLOB 是二进制字符串，可存图片等二进制数据，缺点是占用存储资源大。
- 快捷选型口诀：整数→INT，小数→DECIMAL，字符串（非主键）→TEXT，日期时间→DATETIME。可靠性优先，若要进一步优化效率再考虑 CHAR/VARCHAR 替换 TEXT。

**面试高频问答**
- **Q：** 存金额为什么必须用 DECIMAL 而不是 FLOAT/DOUBLE？
  **A：** 浮点数底层用二进制存小数，尾数非 0/5 时无法精确表达、只能近似，运算会累积误差，导致金额对不上、按数值等值查询查不到。DECIMAL 是定点数，整数和小数部分分开精准存储，金额一分不差。（补充：也可用 INT 存「分」，但只适合纯加减，涉及乘除仍有精度损失。）
- **Q：** CHAR 和 VARCHAR 有什么区别，怎么选？
  **A：** CHAR(M) 定长，不足位补空格，存取快但短了会截断、长了浪费空间，适合长度固定的串（如身份证、手机号、MD5）；VARCHAR(M) 变长，按实际长度存储省空间，适合长度不固定的串。TEXT 最灵活但效率低、且不能做主键，主键场景只能用 CHAR/VARCHAR。
- **Q：** date、datetime、timestamp 有什么区别？
  **A：** DATE 只有日期，TIME 只有时间（且可表示 >24 小时的间隔），DATETIME 含完整日期+时间、范围最大、与时区无关；TIMESTAMP 也含日期时间但范围小（到 2038 年）、占字节少、会随时区转换。作者建议优先 DATETIME 因其最完整；实际中需要时区自动转换/时间戳时用 TIMESTAMP。

### 03 | 表：怎么创建和修改表？

**一句话核心**：建表核心是用「约束」保证数据的唯一、合法、完整；改表全靠 `ALTER TABLE`（ADD/CHANGE/MODIFY）。

**关键知识点**
- 建表语法：`CREATE TABLE 表名 (字段名 数据类型 [字段级约束] [默认值], ... [表级约束]);`。字段名要避开系统关键字（如 `value`、`order`）。
- 六种约束：主键约束（PRIMARY KEY）、外键约束（FOREIGN KEY）、非空约束（NOT NULL）、唯一性约束（UNIQUE）、默认约束（DEFAULT）、自增约束（AUTO_INCREMENT）。
  - `DEFAULT`：不赋值时自动填默认值，如 `importtype INT DEFAULT 1`。
  - `NOT NULL`：字段必须有值。
  - `UNIQUE`：值不能重复，但可为空、且可有一条记录为空；一个表可有多个唯一约束字段，主键只能一个。
  - `AUTO_INCREMENT`：只能用于整数字段，每插一条自动 +1；手动赋值会重置自增基数，下次以「当前最大值 +1」赋值。
- 主键 vs 唯一约束的关键区别：主键字段自动满足非空，唯一约束字段可以为空。
- 复制表结构（只结构不数据）：`CREATE TABLE demo.new LIKE demo.old;`。
- 改表三大操作：
  - 加字段：`ALTER TABLE 表 ADD 字段 类型;`，可指定位置 `... ADD 字段 类型 AFTER 某字段;` 或 `FIRST`。
  - 改名+改类型：`ALTER TABLE 表 CHANGE 旧字段名 新字段名 新类型;`。
  - 只改类型：`ALTER TABLE 表 MODIFY 字段名 新类型;`。
- 改存储引擎：`ALTER TABLE 表 ENGINE = INNODB;`。

**面试高频问答**
- **Q：** 主键约束和唯一约束（UNIQUE）有什么区别？
  **A：** ① 一个表只能有一个主键，但可以有多个唯一约束字段；② 主键自动非空，唯一约束字段允许为 NULL（且可有一条为空）；③ 主键唯一标识记录，语义更强。
- **Q：** ALTER TABLE 里 CHANGE 和 MODIFY 有什么区别？
  **A：** CHANGE 能同时改「字段名 + 字段类型」（要写旧名和新名）；MODIFY 只改字段类型、不改字段名。都可配 FIRST/AFTER 调整字段位置。
- **Q：** AUTO_INCREMENT 有什么限制？
  **A：** 只能定义在整数类型字段上；手动给它赋值会重置自增基数，之后按「已有最大值 +1」继续，所以删记录后再插会出现值不连续。

### 04 | 增删改查：如何操作表中的数据？

**一句话核心**：CRUD 四语句 INSERT/DELETE/UPDATE/SELECT；删改务必带 WHERE 防误操作，SELECT 有固定子句顺序。

**关键知识点**
- 插入：`INSERT INTO 表 (字段列表) VALUES (值列表);`。可只给部分字段赋值，前提是未赋值字段能被 MySQL 处理（可为空/有默认值/自增），否则报错。
- 插入查询结果（批量搬数据高效）：`INSERT INTO 表A (字段) SELECT 字段 FROM 表B WHERE 条件;`。
- 主键/唯一键冲突处理：`INSERT ... ON DUPLICATE KEY UPDATE 字段=值;`，遇重复则更新而非报错（数据整合常用）。
- 删除：`DELETE FROM 表 WHERE 条件;`。Workbench 安全模式要求删改必须带 WHERE 且用主键/唯一键，防误删。
- 修改：`UPDATE 表 SET 字段=值 WHERE 条件;`。**不要修改主键值**，否则破坏数据完整性和关联。
- 查询完整语法及书写顺序：`SELECT 字段 FROM 数据源 WHERE 条件 GROUP BY 字段 HAVING 条件 ORDER BY 字段 LIMIT 起始,行数;`。
  - `DISTINCT` 去重；`ORDER BY 字段 ASC|DESC`（升/降序，可多字段）。
  - `LIMIT m,n`：从第 m+1 条（起点为 0）开始取 n 条，如 `LIMIT 1,2` 取第 2、3 条。
  - FROM 后可跟子查询（派生表），必须用 AS 起别名。

**面试高频问答**
- **Q：** DELETE、TRUNCATE、DROP 三者有什么区别？（本讲讲了 DELETE，此为面试必配对比）
  **A：** DELETE 是 DML，逐行删数据、可带 WHERE 删部分、可回滚、不重置自增；TRUNCATE 是 DDL，清空整表、不能带条件、不可回滚、速度快且重置自增；DROP 是 DDL，直接删掉整张表（结构+数据+索引全没）。删数据留表用 DELETE/TRUNCATE，连表都不要了用 DROP。
- **Q：** 为什么强调删除、更新一定要带 WHERE？
  **A：** 不带 WHERE 会操作全表所有记录，极易误删/误改。MySQL/Workbench 的安全模式甚至强制删改必须带 WHERE 且用主键或唯一键作条件，就是为防止误操作波及无关数据。
- **Q：** LIMIT 2,3 是什么意思？
  **A：** 起始位置从 0 计，2 表示跳过前 2 条从第 3 条开始，3 表示取 3 条，即返回第 3、4、5 条记录。分页常用，建议配 ORDER BY 保证顺序稳定。

### 05 | 主键：如何正确设置主键？

**一句话核心**：不要用业务字段做主键（会重复/重用导致历史数据错乱），单机用自增主键，分布式多点写入用手动赋值/全局唯一 ID。

**关键知识点**
- 三种设主键思路：业务字段做主键、自增字段做主键、手动赋值字段做主键。
- **业务字段做主键的坑**：会员卡号、手机号会被回收重发，身份证号涉隐私顾客不愿给。一旦主键被 UPDATE，之前通过该主键关联的历史流水（JOIN 查询）就会张冠李戴——张三的消费记到王五头上。所以尽量不用业务字段做主键。
- **自增字段做主键**：加一个 `id INT PRIMARY KEY AUTO_INCREMENT`，唯一非空、稳定不变，历史关联用不变的 id 而非会卡号，卡号重用也不影响。单机系统最佳选择。
  - 删主键约束（保留字段）：`ALTER TABLE demo.membermaster DROP PRIMARY KEY;`
  - 加自增主键：`ALTER TABLE demo.membermaster ADD id INT PRIMARY KEY AUTO_INCREMENT;`
- **手动赋值做主键**：多门店/多服务器各自写入时，自增 id 会冲突（A、B 店都产生 id=101）。解决办法是取消自增，改由总部维护一个「当前最大编号」字段，各门店取值 +1 后赋值并回写，保证全系统唯一。
- 多表更新（给 trans 表回填 memberid）：`UPDATE demo.trans AS a, demo.membermaster AS b SET a.memberid=b.id WHERE a.transactionno>0 AND a.cardno=b.cardno;`——比删表重建更适合生产（不丢历史数据）。

**面试高频问答**
- **Q：** 为什么推荐用自增主键而不是业务字段（如会员卡号、身份证）做主键？
  **A：** 业务字段无法保证整个生命周期不重复/不重用：会员卡号会回收重发、手机号会被收回、身份证涉隐私拿不到。而主键一旦变化，历史关联数据会全部错乱，改主键成本极高。自增 id 与业务无关、稳定不变、唯一非空，能彻底规避这个问题。
- **Q：** 自增主键在分布式/多服务器场景有什么问题，怎么解决？
  **A：** 多个节点各自独立自增会产生相同 id（都从 100 增到 101），合并数据时主键冲突。解决方案：改为手动赋值（总部集中维护全局最大编号，取号需用事务防并发重复），或用 UUID、雪花算法（Snowflake）等全局唯一 ID。注意不要用「门店编号+时间」这类业务信息拼主键，因为业务字段仍可能重复重用。

### 06 | 外键和连接：如何做关联查询？

**一句话核心**：外键（FOREIGN KEY）建关联+保一致性，连接（JOIN）做多表查询；内连接只返回匹配行，左/右连接返回主表全部行。

**关键知识点**
- 外键定义在**从表**，是从表中引用主表主键的字段。作用：MySQL 监控删除，主表记录被从表引用时不允许删，防数据缺失。
- 定义外键约束（建表时或改表时）：
  - 建表：`CREATE TABLE 从表 (..., CONSTRAINT fk_名 FOREIGN KEY (字段) REFERENCES 主表 (字段));`
  - 改表：`ALTER TABLE 从表 ADD CONSTRAINT 约束名 FOREIGN KEY 字段 REFERENCES 主表 (字段);`
- 有外键约束时，插入必须先插主表再插从表；删除必须先删从表再删主表，否则违反约束报错。
- 连接（JOIN）分两类：
  - **内连接** INNER JOIN（= JOIN = CROSS JOIN）：只返回两表都满足连接条件的记录。`SELECT ... FROM a JOIN b ON (a.字段=b.字段);`
  - **外连接** OUTER JOIN，含左连接和右连接：
    - **LEFT JOIN**：返回左表全部记录 + 右表匹配的记录，右表不匹配处填 NULL。
    - **RIGHT JOIN**：返回右表全部记录 + 左表匹配的记录。
    - `a LEFT JOIN b` 与 `b RIGHT JOIN a` 等价（谁全保留看方向）。
- **外键约束的利弊**：优点是数据库层面保证数据一致性、防误删；缺点是有系统开销、大并发/大型网站可能拖慢。结论：内部管理系统建议用外键；高并发场景可不用外键，但必须在应用层（如事务）实现同等的一致性检查。
- 外键约束不是关联查询的必要条件——没有外键也能 JOIN 查询，外键只负责保一致性。

**面试高频问答**
- **Q：** 内连接、左连接、右连接有什么区别？
  **A：** 内连接（INNER JOIN）只返回两表都满足连接条件的交集记录；左连接（LEFT JOIN）返回左表全部记录，右表没匹配的补 NULL；右连接（RIGHT JOIN）返回右表全部记录，左表没匹配的补 NULL。左连接和右连接互换表的位置即可等价。
- **Q：** 外键约束有什么利弊？实际项目该不该用？
  **A：** 利：数据库层自动保证从表引用的主表数据不被误删、维护两表数据一致性。弊：每次增删改都要做外键检查，有系统开销，大并发/大型网站可能明显变慢。所以内部管理系统（访问量小）建议用外键；面向大量用户的高并发系统可不用外键约束，但一定要在应用层用事务等手段实现等价的一致性保证。
- **Q：** 没有外键能做关联查询吗？
  **A：** 能。外键约束不是 JOIN 的必要条件，JOIN 只要两表有意义相同的公共字段就能连接。外键只负责保证数据一致性、防误删，和能否查询无关。

### 07 | 条件语句：WHERE 与 HAVING 有什么不同？

**一句话核心**：WHERE 先筛选后连接（用表字段、不能用聚合函数、更高效）；HAVING 配 GROUP BY 用、先连接分组后筛选（能用聚合函数，对分组结果过滤）。

**关键知识点**
- 执行顺序差异：WHERE **先筛选再连接**——用筛后的小数据集去 JOIN，占资源少、效率高；HAVING **先连接分组再筛选**——对完整大数据集分组后过滤，开销大。所以关联查询中 WHERE 比 HAVING 高效。
- 语法定位：查询中 WHERE 在 GROUP BY 之前，HAVING 在 GROUP BY 之后。因此 WHERE 无法对分组结果筛选，HAVING 可以。
- **WHERE 不能用聚合函数**（如 `WHERE SUM(price)=1.1` 是错的），只能用表中真实字段；HAVING 必须与 GROUP BY 配合，能把分组字段和聚合函数（SUM/MAX/COUNT 等）作筛选条件。
- 只有 HAVING 能完成「先分组再按聚合结果筛选」的需求，例：查哪个收银员哪天卖了 2 单——`... GROUP BY transdate,operatorname HAVING count(*)=2;`。
- 两者可同用、应拆分条件各展所长：普通字段条件放 WHERE（先高效筛选），含聚合函数的条件放 HAVING（对分组结果筛选）。例：`... WHERE transdate IN (...) GROUP BY ... HAVING SUM(salesvalue)>100;`。
- 补充：GROUP BY 对数据分组、常配聚合函数统计；DISTINCT 去重。

**面试高频问答**
- **Q：** WHERE 和 HAVING 的区别是什么？
  **A：** ① 作用对象：WHERE 直接作用于表中原始字段（先筛选后连接），HAVING 作用于分组后的结果集（先连接分组后筛选）；② 聚合函数：WHERE 不能用聚合函数，HAVING 可以用聚合函数和分组字段；③ 依赖：HAVING 必须配 GROUP BY，WHERE 不需要；④ 效率：关联查询中 WHERE 先缩小数据集再连接，比 HAVING 高效。
- **Q：** 为什么 WHERE 里不能用聚合函数，而 HAVING 可以？
  **A：** 因为执行/语法顺序上 WHERE 在 GROUP BY 之前执行，此时还没分组、聚合结果尚未产生，所以用不了聚合函数；HAVING 在 GROUP BY 之后执行，分组和聚合已完成，因此能对 SUM/COUNT/MAX 等聚合结果做筛选。
- **Q：** HAVING 后面必须跟聚合函数吗？
  **A：** 不必须。HAVING 也可以用普通字段/分组字段做条件（如 `HAVING goods_price<100` 与 WHERE 等效），只是这种情况用 WHERE 更高效。最佳实践是：普通条件放 WHERE，含聚合函数的条件放 HAVING，二者可在同一查询中配合使用。

### 08 | 聚合函数与分组统计

**一句话核心**：SUM/AVG/MAX/MIN/COUNT 五个聚合函数配合 `GROUP BY` 做分组统计，聚合只统计非空值。

**关键知识点**
- 五大聚合函数：`SUM()` 求和、`AVG()` 平均、`MAX()`/`MIN()` 极值、`COUNT()` 计数。都忽略 NULL（COUNT(*) 除外）。
- 分组统计模板：`SELECT 分组字段, SUM(x) FROM t GROUP BY 分组字段 ORDER BY 分组字段`。多字段分组用逗号分隔，注意字段间的层次关系。
- `COUNT(*)` vs `COUNT(字段)`：`COUNT(*)` 统计总行数（含 NULL 行）；`COUNT(字段)` 只统计该字段非空的行数。例如某字段有 2 个 NULL，`COUNT(字段)` 就少 2。
- `MAX(字段1)` 和 `MAX(字段2)` 是**分别独立计算**的，返回值不一定来自同一行——易错点，别想当然。
- 聚合可嵌套函数/关键字，如按日期分组常用 `GROUP BY LEFT(transdate,10)` 截取年月日；但对分组字段套函数会导致索引失效、性能下降，生产慎用。
- 去重：`COUNT(DISTINCT 字段)` 统计不重复的非空值个数；`SUM(DISTINCT 字段)` 同理去重后求和。

**面试高频问答**
- **Q：** `COUNT(*)`、`COUNT(1)`、`COUNT(字段)` 有什么区别？
  **A：** `COUNT(*)` 和 `COUNT(1)` 都统计总行数、不忽略 NULL，语义等价（InnoDB 下性能也基本一致，优化器会选最优索引扫描）；`COUNT(字段)` 只统计该字段非 NULL 的行数。想统计"某字段有值的记录数"用 `COUNT(字段)`。
- **Q：** `WHERE` 和 `HAVING` 区别？
  **A：** `WHERE` 在分组前过滤原始行、不能用聚合函数；`HAVING` 在分组后过滤分组结果、可用聚合函数（如 `HAVING COUNT(*)>1`）。查"哪种商品名重复了几次"就是 `GROUP BY goodsname HAVING COUNT(*)>1`。

### 09 | 时间函数

**一句话核心**：用 EXTRACT/YEAR/HOUR 等抽取时间分量，DATE_ADD/DATEDIFF 做时间计算，DATE_FORMAT 做格式化。

**关键知识点**
- 抽取分量：`EXTRACT(HOUR FROM date)` 或直接用 `YEAR()/MONTH()/DAY()/HOUR()/MINUTE()/SECOND()`。DATETIME 默认格式 `YYYY-MM-DD HH:MM:SS`。
- 时间加减：`DATE_ADD(date, INTERVAL n type)`，n 为正向后、负向前，type 是 YEAR/MONTH/DAY 等。同义/反向：`ADDDATE()`（同 DATE_ADD）、`DATE_SUB()`/`SUBDATE()`（做减法）。
- 取月末：`LAST_DAY(date)` 返回当月最后一天。求某月第一天的技巧：`DATE_ADD(LAST_DAY(上个月某天), INTERVAL 1 DAY)`。
- 当前时间：`CURDATE()` 返回当前日期（`YYYY-MM-DD`）；`NOW()` 返回当前日期时间；`CURTIME()` 返回当前时间。
- 星期：`DAYOFWEEK(date)` 返回 1=周日、2=周一…7=周六（注意从周日算起）。
- 天数差：`DATEDIFF(date1, date2)` 返回 date1 减 date2 的天数（date1 在后则为正）。
- 格式化：`DATE_FORMAT(date, 格式)`，如 `%T`=24 小时制时间、`%r`=12 小时制、`%Y-%m-%d`=年月日、`%W`=英文星期、`%M`=英文月份。
- 时间戳：`UNIX_TIMESTAMP(date)` 转 Unix 时间戳（秒），`FROM_UNIXTIME(ts)` 反向转回日期时间。

**面试高频问答**
- **Q：** 查"7 天前到今天"的数据条件怎么写？
  **A：** `WHERE d >= DATE_ADD(CURDATE(), INTERVAL -7 DAY) AND d < CURDATE()`，或用 `DATE_SUB(CURDATE(), INTERVAL 7 DAY)`。
- **Q：** 为什么不建议在 WHERE 的时间字段上套函数（如 `DATE(create_time)='2021-01-01'`）？
  **A：** 对索引列使用函数会导致索引失效、全表扫描。应改写成范围条件 `create_time >= '2021-01-01' AND create_time < '2021-01-02'`。（详见 45 讲索引部分）

### 10 | 数学、字符串与条件判断函数

**一句话核心**：取整/舍入用 FLOOR/CEIL/ROUND，拼接/截取用 CONCAT/SUBSTR，条件判断用 IF/IFNULL/CASE。

**关键知识点**
- 取整：`FLOOR(x)` 向下取整、`CEIL(x)`/`CEILING(x)` 向上取整、`ROUND(x,d)` 四舍五入保留 d 位（`ROUND(x)` 保留 0 位）；`TRUNCATE(x,d)` 直接截断不进位。
- 数学：`ABS(x)` 绝对值、`MOD(x,y)` 求余。坑：`ROUND(-1.5)` = -2（负数"五入"往更小方向）。
- 字符串处理：`CONCAT(s1,s2,...)` 拼接（任一参数为 NULL 结果为 NULL）；`SUBSTR/SUBSTRING(s,pos,len)` 或 `MID()` 截取；`LEFT(s,n)`/`RIGHT(s,n)` 取左/右 n 字符；`REPLACE(s,from,to)` 替换。
- 长度：`CHAR_LENGTH(s)` 按字符数（一个汉字算 1），`LENGTH(s)` 按字节数（UTF-8 一个汉字算 3）。
- 其他：`CAST(表达式 AS CHAR)` 类型转换、`SPACE(n)` 生成 n 个空格、`TRIM/LTRIM/RTRIM` 去空格、`UPPER/LOWER` 大小写。
- 条件判断：
  - `IF(表达式, v1, v2)`：真返回 v1，否则 v2。
  - `IFNULL(v1, v2)`：v1 非空返回 v1，否则返回 v2。
  - `COALESCE(v1,v2,...)`：返回第一个非 NULL 的值（IFNULL 的多参版）。
  - `CASE WHEN`：`CASE WHEN 条件1 THEN 结果1 WHEN 条件2 THEN 结果2 ELSE 结果m END`；也有简单形式 `CASE 表达式 WHEN 值1 THEN ... END`。
- 判空要用 `ISNULL(x)` 或 `x IS NULL`，不能用 `x = NULL`（永远不成立）。

**面试高频问答**
- **Q：** `CASE WHEN` 常见用法？
  **A：** 行转列/分档统计。例：`SUM(CASE WHEN status=1 THEN 1 ELSE 0 END)` 统计满足条件的行数；`CASE WHEN score>=60 THEN '及格' ELSE '不及格' END` 做分档。面试极高频。
- **Q：** `IFNULL` 和 `COALESCE` 区别？
  **A：** `IFNULL` 只接两个参数；`COALESCE` 接多个参数、返回第一个非 NULL 值，更灵活，是 SQL 标准函数。

### 11 | 索引（用法层）

**一句话核心**：给常用作筛选条件的字段建索引，多条件常一起出现就建组合索引并遵循最左匹配。

**关键知识点**
- 创建单字段索引三种写法：
  - 直接建：`CREATE INDEX idx_name ON 表名(字段)`
  - 建表时：表定义里加 `INDEX|KEY idx_name(字段)`
  - 改表时：`ALTER TABLE 表名 ADD INDEX|KEY idx_name(字段)`
- 主键约束/唯一约束会**自动创建**主键索引/唯一索引——所以建表务必定义主键。
- 组合（联合）索引：`CREATE INDEX idx ON 表名(字段1,字段2,字段3)`，最多 16 个字段。查询同时命中多列时比多个单列索引更快、更精准。
- **该建索引的字段**：经常用作 `WHERE`/`JOIN`/`ORDER BY` 筛选条件、区分度高的字段。
- 组合索引遵循**最左前缀**原则：条件从最左字段开始连续命中才有效，中间断开则断点之后用不上索引。如索引 `(a,b,c)`，条件 `b=1 AND c=1`（缺 a）完全用不上；范围查询也会"中断"，如 `a>10 AND b=1` 只能用到 a 部分。（B+树 / 最左前缀底层原理详见 45 讲索引部分）
- 索引失效常见情况（用法层）：对索引列使用函数/运算（如 `LEFT(col,10)`、`col+1`）、隐式类型转换、`LIKE '%xx'` 前导通配、`OR` 连接非索引列、组合索引未从最左列开始。
- 用 `EXPLAIN SQL` 看执行计划：关注 `type`（range/ref 好、ALL 差）、`possible_keys`（候选索引）、`key`（实际用的索引）、`rows`（预估扫描行数）。多个可用索引时优化器选扫描行数最少的。
- 删除索引：`DROP INDEX idx_name ON 表名`；主键索引须用 `ALTER TABLE 表名 DROP PRIMARY KEY`。
- 索引成本：额外占存储空间 + 增删改需同步维护索引，故并非越多越好。
- 回表、聚簇索引/二级索引、覆盖索引等底层机制**详见 45 讲索引部分**。

**面试高频问答**
- **Q：** 组合索引 `(a,b,c)`，哪些查询能用上、用到哪一部分？
  **A：** `a=1`、`a=1 AND b=1`、`a=1 AND b=1 AND c=1` 都能用（逐级命中）；`a=1 AND c=1` 只能用到 a；`b=1 AND c=1` 完全用不上；`a=1 AND b>1 AND c=1` 用到 a 和 b（b 是范围，c 中断）。这就是最左前缀。（详见 45 讲）
- **Q：** 什么字段适合建索引，什么不适合？
  **A：** 适合：常作查询/连接/排序条件、区分度高（如唯一编号、时间）；不适合：区分度低（如性别）、频繁更新、很少查询的字段——收益小且拖慢写入。
- **Q：** 索引为什么会失效？
  **A：** 索引列被函数/表达式包裹、隐式类型转换、`LIKE '%x'`、组合索引不满足最左前缀、优化器判断全表扫描更划算等。可用 `EXPLAIN` 验证 key 是否为空。

### 12 | 事务（用法层）

**一句话核心**：`BEGIN/COMMIT/ROLLBACK` 把一组 DML 打包成原子操作，保证要么全成、要么全败。

**关键知识点**
- 语法：`START TRANSACTION`（或 `BEGIN`）开启 → 一组 DML → `COMMIT` 提交生效 / `ROLLBACK` 回滚撤销。
- ACID 四特性：原子性（不可分割）、一致性（不破坏数据完整性）、隔离性（并发事务互不干扰）、持久性（提交后永久有效）。
- **坑：事务不会自动处理 SQL 错误**。某条 DML 报错后若不回滚继续 COMMIT，仍会造成数据不一致。需程序判断：出错就 `ROLLBACK`，全部成功才 `COMMIT`。
- 判断操作是否失败：`ROW_COUNT()` 返回 -1 表示失败，否则为影响行数。
- 存储过程中常用异常自动回滚：`DECLARE EXIT HANDLER FOR SQLEXCEPTION ROLLBACK;`（配合 `DELIMITER //` 改结束符创建）。
- DDL（建表/删库/删表等）不能回滚，且部分会隐式提交事务——操作前先备份。
- 隔离级别（从低到高）与设置语法：
  - `READ UNCOMMITTED`（读未提交，有脏读）
  - `READ COMMITTED`（读已提交，Oracle 默认）
  - `REPEATABLE READ`（可重复读，**MySQL 默认**）
  - `SERIALIZABLE`（串行化，最严格）
  - 设置：`SET [GLOBAL|SESSION] TRANSACTION ISOLATION LEVEL REPEATABLE READ;` 用完记得恢复默认，避免拖慢整体效率。
- MVCC 多版本并发控制、锁（行锁/间隙锁/next-key lock）、脏读/不可重复读/幻读的底层实现**详见 45 讲事务与锁部分**。

**面试高频问答**
- **Q：** "事务就是保证操作要么全成要么全败"这句话对吗？
  **A：** 不完全对。事务提供原子性能力，但执行中遇到错误是继续还是回滚要由程序处理；不主动判错回滚，仍可能提交出不一致的数据。
- **Q：** 四种隔离级别分别解决什么问题？
  **A：** 读未提交→都不防；读已提交→防脏读；可重复读→防脏读+不可重复读（MySQL 默认，InnoDB 靠 MVCC + 间隙锁还基本解决了幻读）；串行化→全部串行、最安全但最慢。（原理详见 45 讲）
- **Q：** 怎么设置隔离级别？
  **A：** `SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;`（GLOBAL 影响后续所有连接，SESSION 只影响当前会话）。

### 13 | 临时表

**一句话核心**：`CREATE TEMPORARY TABLE` 存查询中间结果，连接隔离、连接结束自动删除，用来拆解复杂查询。

**关键知识点**
- 两种临时表：**内部临时表**（系统为排序/分组/去重等自动生成、用户不可见，属性能优化）；**外部临时表**（用户用 SQL 显式创建、可用）。本讲讲的是外部临时表。
- 创建语法：`CREATE TEMPORARY TABLE 表名(字段 类型, ...)`，关键字是 `TEMPORARY`。
- 三个特征：需 `TEMPORARY` 关键字；**只当前连接可见**（连接隔离，不同连接可同名不冲突，适合并发）；连接结束自动删除。
- 用查询结果直接建表最便捷：`CREATE TEMPORARY TABLE demo.t SELECT ... FROM ... GROUP BY ...`，无需预先设计表结构。
- 用途：把复杂查询拆成多个前后关联的简单步骤，中间结果落到临时表，把面向集合的 SQL 变成面向过程的编程，降低难度和维护成本。
- 存储引擎：默认 InnoDB 存磁盘；`ENGINE=MEMORY` 存内存，查询更快（示例中 <10ms vs 210ms），但断电丢数据（临时结果无所谓）。
- 多个临时表汇总时，若各表都可能缺商品，用"全集表（如商品表）LEFT JOIN 各临时表 + `IFNULL(x,0)`"避免丢数据，再用 `HAVING` 过滤全 0 行。
- 与 45 讲内部临时表的区别：45 讲讲的是**内部临时表**——优化器在 `GROUP BY`/`DISTINCT`/`UNION`/无法用索引排序等场景自动创建（内存不够转磁盘），是性能话题、用户无法直接操作；本讲的**外部临时表**是用户主动 `CREATE TEMPORARY TABLE` 显式使用的表。

**面试高频问答**
- **Q：** 临时表和普通表、视图有什么区别？
  **A：** 临时表是真实存中间数据的表、仅当前连接可见、连接结束自动删；普通表持久化、所有连接共享；视图是虚拟的、不存数据、只是保存的查询定义。
- **Q：** 上面汇总查询为什么用 `HAVING` 而不用 `WHERE` 过滤？
  **A：** `salesquantity` 等是 `IFNULL(...)` 计算出的派生列（SELECT 别名），`WHERE` 在取列阶段之前执行、拿不到这些别名，须用分组/筛选后阶段的 `HAVING` 才能引用。
- **Q：** 生产环境用临时表要注意什么？
  **A：** 临时表占用数据库资源、开销较大，高并发在线系统慎用；更适合离线备份库、数据分析场景，且注意连接池复用连接时临时表未随连接销毁可能带来的影响。

### 14 | 视图：简化查询

**一句话核心**：视图是一张"虚拟表"，本质是存起来的一段 SELECT，不存数据、只存查询语句，用来简化复杂查询、隔离权限、屏蔽表结构变化。

**关键知识点**
- 定义：把查询语句存到库里当表用。创建 `CREATE [OR REPLACE] VIEW v AS SELECT ...`；改 `ALTER VIEW`；看 `DESCRIBE v`；删 `DROP VIEW v`。
- 不存数据：每次查视图都是实时执行底层 SELECT 拿原始表数据，因此不占存储、数据永远和底表一致。
- 三大作用：①简化/模块化复杂查询（把子查询、多表 JOIN 封装）；②隔离性/权限控制（用户只能查视图，看不到底表，提升安全）；③屏蔽表结构变化（底表改字段时只改视图定义，对外查询字段不变）。
- 可更新视图有严格限制：只有视图字段与实际表字段**一一对应**时才能 INSERT/UPDATE/DELETE，且这些操作实际作用在底表上；含 `GROUP BY`、聚合函数、`DISTINCT`、`UNION` 的视图**无法更新**（MySQL 无法精确定位底表记录）。作者建议视图只用于查询，不要更新。
- 缺点：底表结构变更需同步维护视图；**嵌套视图（视图上再建视图）逻辑复杂、可读性差，不建议**。很多公司直接禁用视图（维护成本 > 封装收益）。

**面试高频问答**
- **Q：** 视图和临时表有什么区别？
  **A：** 视图只存 SQL、不存数据，每次查询实时读原始表，操作数据对应真实表；临时表是会话级的、把数据真正缓存到内存或磁盘的一份拷贝（原数据的子集），随会话结束销毁，常用于缓存中间结果加速查询。
- **Q：** 什么样的视图不能更新？
  **A：** 创建语句里用了分组（GROUP BY）、聚合函数、DISTINCT、UNION，或视图字段与底表字段不能一一对应时，都不能通过视图更新数据。

### 15 | 存储过程：提高性能和安全性

**一句话核心**：存储过程是预先存在 MySQL 服务器端的一组 SQL，客户端一条 `CALL` 命令即可执行整段逻辑，减少网络往返、提升执行效率与安全性。

**关键知识点**
- 创建：`CREATE PROCEDURE 名(参数) BEGIN ... END`；需先用 `DELIMITER //` 改分隔符（否则遇到语句内第一个 `;` 就以为结束），程序体用 `BEGIN...END` 包裹，末尾 `//` 结束后再 `DELIMITER ;` 改回。
- 参数三种：`IN` 输入（默认，只读）、`OUT` 输出（把计算结果回传给调用方）、`INOUT` 既输入又输出。参数还要声明数据类型。
- 相关操作：查看 `SHOW CREATE PROCEDURE 名`；调用 `CALL 名(参数)`；删除 `DROP PROCEDURE 名`；改则需删了重建（或在 Workbench 里改）。
- 优点：执行效率高、减少 SQL 暴露在网络上的风险（更安全）、把复杂重复计算（如日结统计）放服务端。缺点：**开发调试成本高**（无专业 IDE，只能用 SELECT 输出中间变量逐步调试）、**难维护、无移植性、版本控制困难**——阿里 Java 规约明确禁止使用存储过程。
- 调试技巧：用 `SELECT` 把中间变量/结果打印出来，逐段推进。

**面试高频问答**
- **Q：** 存储过程和存储函数（function）有什么区别？
  **A：** ①函数必须 `RETURNS` 返回值/表，过程可以不返回；②过程用 `CALL` 调用，函数不能；③函数可嵌在 SELECT 等查询语句中使用，过程不行；④过程功能更强，能建表/删表、做事务操作，这些函数不具备。
- **Q：** 互联网公司为什么不爱用存储过程？
  **A：** 难调试、难扩展、无移植性、版本控制困难，业务逻辑藏在数据库里不易维护，因此多把逻辑放应用层。传统软件/商用数据库场景仍有价值。

### 16 | 游标：逐条处理结果集

**一句话核心**：普通 SQL 面向结果集整体操作，游标（cursor）提供在结果集中像指针一样逐行定位、逐条处理记录的能力，只能在存储程序（存储过程/函数）内使用。

**关键知识点**
- 四步法：①定义 `DECLARE 游标名 CURSOR FOR 查询语句`；②打开 `OPEN 游标名`；③取值 `FETCH 游标名 INTO 变量列表`（结果集字段数必须与变量数一致）；④关闭 `CLOSE 游标名`。
- **用完必须及时关闭**：游标占用系统资源，不关会一直保持到存储过程结束，影响性能。
- 结束循环靠条件处理语句：`DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;`——游标取到末尾没记录（NOT FOUND）时置标志位，配合循环判断结束。处理方式 `CONTINUE`（继续）或 `EXIT`（退出）。
- 常配合流程控制语句：循环 `LOOP`/`WHILE`（先判断后执行）/`REPEAT`（先执行后判断）；条件 `IF`/`CASE`；跳转 `ITERATE`（重开循环）/`LEAVE`（跳出）。
- 优点：把逐行处理放服务端，避免应用层写循环、多次与服务器交互，代码简洁、效率相对更高、更安全。缺点：**消耗系统资源**，逐行处理性能不如集合操作，实际开发少用。

**面试高频问答**
- **Q：** 游标为什么在实际开发中少用/要谨慎用？
  **A：** 游标逐行处理，会占用较多系统资源，性能不如面向集合的批量 SQL；且只能在存储程序内使用、可维护性差。只在确实需要对每行单独、且行间有依赖关系的处理时才用，用完立即关闭释放资源。
- **Q：** 游标取数时字段数和变量数不一致会怎样？
  **A：** FETCH 要求结果集字段数与 INTO 后变量数严格一致，否则存储过程执行时 MySQL 报错。

### 17 | 触发器：数据修改自动触发关联操作

**一句话核心**：触发器（trigger）绑定在表上，由 INSERT/UPDATE/DELETE 事件自动触发执行，用于保证数据一致性/完整性，无需应用层手动维护关联操作。

**关键知识点**
- 创建：`CREATE TRIGGER 名 {BEFORE|AFTER} {INSERT|UPDATE|DELETE} ON 表 FOR EACH ROW 表达式`。三要素：**监控的表、触发事件（增/改/删）、触发时机（前/后）**。查看 `SHOW TRIGGERS\G`；删除 `DROP TRIGGER 名`。
- `NEW` / `OLD` 伪记录：INSERT 用 `NEW`（新值）；DELETE 用 `OLD`（旧值）；UPDATE 两者都有（`OLD.x` 改前、`NEW.x` 改后）。示例：`IF (NEW.memberdeposit <> OLD.memberdeposit) THEN ...`。
- 原子性：触发器操作失败会导致触发它的那条数据操作也失败（不会一成一败），起到类似事务的保证；可用 `ROW_COUNT()` 返回 -1 捕获触发器错误。
- 三大用途：①保证数据一致性（如明细表增删改后自动重算头表合计）；②记录操作日志（如储值金额变动写历史表）；③操作前做数据合法性检查（BEFORE 触发，拦截错误数据）。
- 最大缺点——**可读性差、隐蔽性强**：逻辑藏在数据库里、由事件驱动、不受应用层控制，报错时排查困难（可能误以为是更新语句或表结构问题）；表结构变更易导致触发器出错。建议维护完整的数据库设计文档。

**面试高频问答**
- **Q：** 触发器和事务/存储过程有什么关系？
  **A：** 触发器由数据变更事件自动触发，能起到类似事务的一致性保证（触发器内操作失败则原操作一并回滚）；相比用存储过程或应用层写多步关联操作再包事务，触发器更简洁高效、资源消耗更少，但隐蔽性强、难维护。
- **Q：** 触发器最大的坑是什么？
  **A：** 隐蔽性/可读性差。逻辑藏在库里不易被开发者察觉，一旦触发器出错会导致看似无关的数据操作失败，排查成本高；大量使用会让维护在应用代码和触发器语法间反复横跳。

### 18 | 权限管理：控制数据库访问

**一句话核心**：通过用户 + 角色 + 授权，按"能看哪些表（横向）、能操作到什么程度（纵向）"控制访问，遵循最小权限原则消除安全隐患。

**关键知识点**
- 角色（MySQL 8.0 引入）= 权限的集合，用于批量管理相同权限的用户。角色名 = 角色名 + 主机名，省略主机名默认 `%`（任意主机）。操作：`CREATE ROLE`、`DROP ROLE`。
- 用户：`CREATE USER 'u' IDENTIFIED BY '密码'`（不设密码不安全，不推荐）；`DROP USER`。
- 授权 `GRANT`：给角色 `GRANT SELECT,INSERT,UPDATE,DELETE ON 库.表 TO 角色`；给用户可直接授权 `GRANT 权限 ON 库.表 TO 用户`，也可把角色赋给用户 `GRANT 角色 TO 用户`。收回用 `REVOKE`。查看 `SHOW GRANTS FOR 用户/角色`。
- 权限粒度：可精确到 `库.表` 级、精确到 SELECT/INSERT/UPDATE/DELETE 等操作级；新建角色自带 `USAGE`（仅登录权限，无实际操作权限）。
- **常见坑**：MySQL 中角色创建后默认未激活，把角色赋给用户后用户仍无权限；需 `SET global activate_all_roles_on_login=ON;` 激活所有角色。删除角色会同时收回用户经该角色获得的所有权限。
- 最小权限原则：不要图省事全用 root、把权限控制全放应用层。root 密码写代码里一旦泄露数据库彻底失守；应尽量用数据库自身的角色/用户机制按职责最小化授权。

**面试高频问答**
- **Q：** 给用户授了角色却没权限，为什么？
  **A：** MySQL 角色创建后默认未激活，赋给用户也不生效。需执行 `SET global activate_all_roles_on_login=ON;` 永久激活，用户才真正拥有角色的全部权限。
- **Q：** 什么是最小权限原则，为什么不建议用 root 连库？
  **A：** 每个用户只授予其职责所需的最小权限，缩小攻击面。root 是超级用户，密码写在代码里一旦泄露数据库完全失去保护；应按角色分权、精确到表和操作级别授权。

### 19 | 日志（上）：及时发现问题

**一句话核心**：MySQL 日志众多，通用查询日志、慢查询日志、错误日志三类用于"发现问题、定位故障、还原现场"。

**关键知识点**
- MySQL 日志全家桶：通用查询日志、慢查询日志、错误日志（本讲），二进制日志 binlog、中继日志 relay log、重做日志 redo log、回滚日志 undo log（下讲）。
- **通用查询日志（general log）**：记录所有连接的起止时间和发给服务器的**所有 SQL 指令**，用于还原操作现场、审计。默认关闭（开销大、占磁盘）。`SET GLOBAL general_log='ON'`，文件由 `general_log_file` 指定；文本格式可直接看。高并发生产环境不建议开。
- **慢查询日志（slow query log）**：记录执行时间超过阈值的查询，用于发现并优化慢 SQL。由配置文件 `my.ini` 控制：`slow_query_log=1` 开启、`slow_query_log_file` 文件名、`long_query_time=10`（秒，阈值）、`min_examined_row_limit`（最少扫描行数）。**只有执行时间 > long_query_time 且扫描行数 ≥ min_examined_row_limit 才记录**；日志含执行时间、锁表时长、Rows_examined 等。改 my.ini 需重启生效。
- **错误日志（error log）**：记录服务器启动/运行/停止的诊断信息（错误、警告、提示，含 InnoDB 启停）。**默认开启**，`my.ini` 中 `log-error` 指定文件。服务器故障排查首选。

**面试高频问答**
- **Q：** 怎么定位慢 SQL？慢查询日志判定条件是什么？
  **A：** 开启慢查询日志（`slow_query_log=1`，设 `long_query_time` 阈值），DB 变慢/阻塞时查该日志找耗时长的 SQL 再针对性优化。判定需同时满足：执行时间超过 long_query_time，且扫描行数不少于 min_examined_row_limit。
- **Q：** 数据库宕机/服务起不来，先看什么日志？
  **A：** 错误日志（默认开启），里面有启动/运行/停止的诊断信息和 InnoDB 状态，是排查系统故障的首选；若要还原某次数据操作细节则看通用查询日志。

### 20 | 日志（下）：故障后恢复数据

**一句话核心**：binlog（二进制日志）是数据恢复和主从复制的利器；relay log 服务从库复制；undo log 管事务回滚与一致性读；redo log 保证已提交数据不丢（崩溃恢复）。

**关键知识点**
- **二进制日志 binlog**：记录所有数据库更新事件（建表、增删改数据等），用途①无损失数据恢复、②主从数据同步。操作：`SHOW MASTER STATUS`（当前写入的文件与位置）、`SHOW BINARY LOGS`（所有）、`SHOW BINLOG EVENTS IN 文件`（事件明细）、`FLUSH BINARY LOGS`（切新文件）、`RESET MASTER`/`PURGE MASTER LOGS TO`（删除）。
- **binlog 恢复到时间点**：`mysqlbinlog --start-position=xxx --stop-position=yyy 文件 | mysql -u 用户 -p`，执行从 xxx 到 yyy 之间的更新操作。实战难点是**起止位置不好定位**（早了数据重复冲突、晚了丢数据）——建议每次全量备份后记录当时 binlog 位置。
- 典型恢复流程（全量 + 增量）：mysqldump 全量备份 → `FLUSH BINARY LOGS` 让新事件进新文件 → 故障后先 `mysql < 备份.sql` 恢复到备份点 → 再用 `mysqlbinlog` 回放备份之后的 binlog 增量 → 数据全部找回。
- **中继日志 relay log**：只存在于主从架构的**从库**；从库读主库 binlog 写入本地 relay log，再重放 relay log 完成同步。坑：文件名含从服务器名，重装系统改了主机名会导致读不到旧 relay log（改回原名即可）。
- **回滚日志 undo log**：记录事务每次更新前的旧值，用于事务回滚、以及让其他事务读到修改前的值（MVCC 一致性读，不受本事务影响）。
- **重做日志 redo log**：物理落盘日志，保证持久性——更新先写内存后批量刷盘，宕机/断电时用 redo log 把未落盘的已提交更新补写到磁盘，防止数据丢失（崩溃恢复）。
- binlog 与 redo log 的**两阶段提交、crash-safe 原理详见《MySQL实战45讲》日志部分**，此处不展开。

**面试高频问答**
- **Q：** 怎么把数据库恢复到某个时间点？
  **A：** 先用最近一次全量备份（mysqldump 的 .sql）恢复到备份时刻，再用 `mysqlbinlog --start-position/--stop-position` 回放备份之后到目标时间点之间的 binlog 增量。关键是准确定位 binlog 起止位置，故建议备份后立即记录 binlog 位置。
- **Q：** redo log 和 undo log 各自作用？
  **A：** redo log 保证持久性，宕机后把已提交但未落盘的更新重做到磁盘（崩溃恢复）；undo log 保存更新前的旧值，用于事务回滚和 MVCC 一致性读。（binlog 与 redolog 的两阶段提交详见45讲。）
- **Q：** binlog 有什么用？
  **A：** 记录所有数据更新事件，两大用途：数据的无损失恢复（全量备份 + binlog 增量回放）、主从复制的数据同步。

### 21 | 数据备份：确保数据安全

**一句话核心**：备份分物理备份（拷数据文件，需收费工具，少用）和逻辑备份（保存库结构+数据的 SQL，免费常用）；mysqldump 做逻辑备份，配合 binlog 实现全量+增量恢复到任意时间点。

**关键知识点**
- 物理备份 vs 逻辑备份：物理=直接复制数据文件（快、需收费工具，用得少）；逻辑=把库/表结构和数据导成 SQL 脚本（文本、免费、通用），本讲重点。
- **mysqldump 三种粒度**：备份表 `mysqldump -u root -p 库 表1 表2 > f.sql`；备份库 `--databases 库1 库2 > f.sql`；备份整个服务器 `--all-databases > f.sql`。备份文件是含 `CREATE TABLE` + `INSERT` 的 SQL 脚本。`-h` 服务器（默认 localhost）、`-u` 用户、`-p` 密码。
- **恢复两种方式**：命令行 `mysql -u root -p 库 < f.sql`（只备份了表时需指定库；备份含库结构的则不需指定）；或客户端内 `USE 库; SOURCE f.sql`（先 USE 选对库，否则恢复错位）。
- 全量 vs 增量：mysqldump 做全量备份；备份之后的变化靠 binlog 记录增量，恢复时全量 + binlog 回放（见20讲），即可恢复到时间点。建议全量后记录 binlog 位置。
- 表数据导出/导入（只要数据本身、按格式）：导出 `SELECT ... INTO OUTFILE '文件' FIELDS TERMINATED BY ',' LINES TERMINATED BY '\n' FROM 表`；导入 `LOAD DATA INFILE '文件' INTO TABLE 表 FIELDS/LINES TERMINATED BY ...`。**LOAD DATA 导入极快**（400万行几分钟），远快于 Workbench。
- **secure-file-priv 坑**：`SELECT ... INTO OUTFILE` / `LOAD DATA` 只能读写 `my.ini` 中 `secure-file-priv` 指定的目录，否则报错；要换目录得改配置并重启。

**面试高频问答**
- **Q：** 物理备份和逻辑备份的区别？mysqldump 属于哪种？
  **A：** 物理备份直接复制底层数据文件，快但依赖收费工具、可移植性差；逻辑备份把库表结构和数据导成 SQL 脚本，文本可读、免费通用但恢复较慢。mysqldump 是逻辑备份，输出可直接执行的 CREATE + INSERT 脚本。
- **Q：** 如何做全量+增量备份并恢复到故障前一刻？
  **A：** 定期用 mysqldump 全量备份并记录当时 binlog 位置；恢复时先导入全量 .sql 到备份点，再用 mysqlbinlog 从记录的位置回放到目标点的 binlog 增量。
- **Q：** 大批量数据导入用什么最快？
  **A：** `LOAD DATA INFILE`，MySQL 提供的快速读入方式，百万级几分钟完成，比逐条 INSERT 或 Workbench 导入快几个数量级。

### 22 | 范式：如何消除冗余和高效存取？

**一句话核心**：范式是设计表结构消除冗余的规范，1NF→2NF→3NF逐级递进；但工程上要遵循“业务优先”，可容忍适度反范式换取性能与业务需求。

**关键知识点**
- **为什么要范式**：表结构设计不合理会导致数据大量重复，表变庞大、内存不足、CPU飙升，甚至项目失败；范式能设计出简洁高效的表，减少冗余，配合外键约束防止增删改时的数据不一致。
- **第一范式（1NF）**：所有字段都是不可再拆分的基本数据项（原子性）。例：`property` 拆成 `specification(规格)` + `unit(单位)`。
- **第二范式（2NF）**：满足1NF基础上，每条记录可唯一标识，且所有非主键字段必须**完全依赖全部主键**，不能只依赖主键的一部分（消除部分依赖）。判断步骤：先定主键（如 `listnumber`+`barcode` 联合主键），把只依赖主键一部分的字段拆出去成新表（商品属性只依赖 barcode → 拆出商品信息表）。
- **第三范式（3NF）**：满足2NF基础上，不能存在依赖于非主键字段的字段，即不能有由其他非主键字段派生出来的字段（消除传递依赖）。例：`suppliername` 依赖非主键 `supplierid` → 拆出供货商表。
- **BCNF（BC范式，3NF的加强）**：在3NF基础上，要求**每个决定因素都必须是候选键**。即当表中存在多个候选键、且候选键之间有部分/传递依赖时，3NF 满足但 BCNF 不满足，需进一步拆分消除“主属性对候选键的部分依赖”。3NF之后还有第四范式（消除多值依赖）、第五范式。
- **业务优先原则（反范式化）**：一切以业务需求为主，技术服务业务，完全按理论不一定最优。可**故意保留冗余字段**换性能/满足业务：
  - `quantity * importprice = importvalue` 三者留其一即可（严格3NF），但保留 `importvalue`——按金额促销的进货单只有金额没有单价，且四舍五入会累积误差，影响可靠性；`importprice` 是成本核算核心指标，取消会大幅提高运算/开发/运维成本。
- **反范式化换性能的典型场景**：数据量大且频繁连接时，增加冗余字段减少 JOIN（详见25讲）；一般 MySQL 设计满足 3NF 即足够。

**面试高频问答**
- **Q：** 请说清三大范式的定义和判断。
  **A：** 1NF 字段原子不可拆；2NF 在1NF上非主键字段完全依赖全部主键（消除部分依赖，针对联合主键场景）；3NF 在2NF上非主键字段不依赖其他非主键字段（消除传递依赖）。判断口诀：原子性→部分依赖→传递依赖。
- **Q：** 3NF 和 BCNF 有什么区别？
  **A：** 3NF 只约束“非主属性”不传递依赖候选键；BCNF 更严，要求任何决定因素（能决定其他字段的字段）都必须是候选键，消除主属性对候选键的部分依赖。多候选键相互重叠时才体现差异。
- **Q：** 为什么有时要故意反范式化（违反范式）？
  **A：** 范式减少冗余但会增加表数量和 JOIN。当查询频繁、连接成本高，或业务本身需要该“冗余”字段（如成本、促销金额、精度）时，适度冗余能显著提升性能、简化 SQL、满足业务。代价是要自己保证数据一致性，且只对“不常改动、查询必需”的字段做冗余。

### 23 | ER模型：如何理清数据库设计思路？

**一句话核心**：ER（实体关系）模型用实体/属性/关系描述业务全貌，先建模理清逻辑再转成表；实体→表、多对多关系→中间表、1对1/1对多→外键、属性→字段。

**关键知识点**
- **为什么用ER模型**：只按范式局部拆分会“越拆越多、支离破碎”，还可能数据缺失；ER 模型能提前看到数据库全貌（需要哪些表、字段、表间关系与连接字段），从整体设计。
- **三要素**：
  - 实体（矩形）：可独立存在的数据对象。**强实体**不依赖其他实体（供货商、商品、门店）；**弱实体**强依赖另一实体（仓库、员工、会员、收银台都依赖门店）。
  - 属性（椭圆）：实体的特性，不可再分、不含其他属性。
  - 关系（菱形）：实体间联系。
- **实体 vs 属性 判断**：从系统整体看，可独立存在的是实体，不可再分的是属性。
- **三种关系**：1对1（人↔身份证，用外键）；1对多（超市→收款机，用外键）；多对多（供货商↔超市、超市↔顾客，必须建中间表）。
- **ER 转表四原则**：
  1. 一个实体 → 一张表；
  2. 一个多对多关系 → 一张表（**中间表**，需包含关联各实体表的主键 + 关系自有属性如进货时间/数量/价格）；
  3. 一个1对1或1对多关系 → 通过**外键**表达，不新建表；
  4. 属性 → 表的字段。
- **弱实体建表**：要包含所依赖强实体的主键作为外键。例 `stock` 表含 `branchid` 并 `FOREIGN KEY (branchid) REFERENCES branch(branchid)`。
- **最小示例（多对多→中间表拆分）**：进货多对多关系拆成 `importhead`（单头，含 supplierid/stockid/operatorid 外键）+ `importdetails`（明细，含 itemnumber 外键，主键 `PRIMARY KEY(importid,itemnumber)`）。

**面试高频问答**
- **Q：** 多对多关系在数据库里怎么落地？
  **A：** 建一张中间（关联）表，表中放两端实体表的主键作外键，再加关系自身的属性；这样把一个多对多拆成两个一对多。
- **Q：** 强实体和弱实体的区别？建表有何不同？
  **A：** 强实体独立存在；弱实体依赖某强实体存在。弱实体建表时必须包含其依赖强实体的主键作为外键关联。
- **Q：** ER 建模的意义是什么？
  **A：** 建模不是目的，是为了理清业务逻辑、把控项目整体，避免局部拆分导致的碎片化和数据缺失，从而设计出优秀数据库。小应用可省，规模应用初期必做。

### 24 | 查询有点慢，语句该如何写？

**一句话核心**：用 EXPLAIN 看执行计划定位慢查询，核心是让 WHERE 条件用上索引；注意 LIKE 前置通配符、OR、函数等会导致索引失效，子查询尽量改连接查询。

**关键知识点**
- **查询分析语句**：`{ EXPLAIN | DESCRIBE | DESC } 查询语句;`——不能直接优化，但能显示执行计划，帮你分析慢的原因。
- **EXPLAIN 关键字段**：
  - `type`：连接/扫描类型。`ALL`=全表扫描（最差）；`range`=索引范围扫描；`ref`/`eq_ref`=用索引等值查；`index_merge`=索引合并。
  - `possible_keys`：可能用到的索引；为空说明没有合适索引可用。
  - `key`：优化器**最终实际使用**的索引；`key_len`：索引使用的字节长度。
  - `rows`：预计扫描行数（越小越好）；`filtered`：筛选出记录占比。
  - `Extra`：`Using where`、`Using index condition`（索引条件下推）、`Using temporary`（用临时表，需警惕）、`Using filesort` 等。
  - `select_type`：SIMPLE（简单查询）、PRIMARY（最外层）、UNION、DEPENDENT SUBQUERY/UNION（依赖外查询的子查询）。
- **典型优化案例**：400万行全表扫描（type=ALL, rows=4157166）耗时6.54s → 对 WHERE 中 `itemnumber`、`transdate` 建索引 → type 变 range、扫描降到552行、0.09s。`CREATE INDEX transdate_trans ON demo.trans(transdate);`
- **LIKE 与索引**：通配符 `%` 在**后面**能用索引（`LIKE 'aa%'` → range）；在**前面**不能用（`LIKE '%aa'`、`LIKE '%aa%'` → 全表扫描）。
- **OR 与索引**：只有 OR 前后表达式的字段**都建了索引**，才能用索引（走 index_merge）；只要有一边字段没索引，就全表扫描。
- **子查询 vs 连接查询**：子查询效率不高——MySQL 会创建临时表保存子查询结果再查，多了建临时表开销；建议把子查询改写成连接查询。
  - 例：`SELECT a.* FROM trans a JOIN goodsmaster b ON a.itemnumber=b.itemnumber WHERE b.goodsname LIKE '%书%';`
- **补充（面试常考的索引失效场景）**：WHERE 中对索引列用函数/运算（如 `WHERE YEAR(transdate)=2020`）、隐式类型转换、`!=`/`<>`、`IS NULL` 有时、前置通配符、OR 一端无索引、不满足最左前缀的联合索引，均可能失效。
- **补充（分页优化）**：`LIMIT 1000000,10` 会先扫描并丢弃前100万行，很慢。优化：用覆盖索引/延迟关联 `SELECT * FROM t JOIN (SELECT id FROM t ORDER BY id LIMIT 1000000,10) x USING(id);`，或用“上次最大id”游标翻页 `WHERE id > ? ORDER BY id LIMIT 10`。

**面试高频问答**
- **Q：** 如何定位并优化一条慢 SQL？
  **A：** ①开慢查询日志/用 Performance Schema 找出慢语句；②`EXPLAIN` 看 type、key、rows、Extra；③若 type=ALL 说明全表扫描，检查 WHERE 字段是否可建索引；④消除索引失效（函数、前置%、OR、类型转换）；⑤子查询改连接；⑥必要时改表结构（见25讲）。
- **Q：** `LIKE '%x%'` 为什么走不了索引，怎么办？
  **A：** B+树索引按前缀有序，前置通配符无法定位前缀。可用后缀通配符、或引入全文索引（FULLTEXT）/搜索引擎（如 ES）解决模糊搜索。
- **Q：** in 和 exists 哪个快？
  **A：** 原则是“小表驱动大表”。子查询结果集小用 IN（先查子查询再匹配外表）；外表小、子查询表大用 EXISTS（对外表逐行判断子查询是否存在）。现代 MySQL 优化器已能改写，差异变小，仍建议能改连接就改连接。

### 25 | 表太大了，如何设计才能提高性能？

**一句话核心**：大表优化从表设计入手——优化数据类型（用小整型/整型代文本）、合理增加冗余字段减JOIN、垂直拆分冷热字段、尽量非空约束；再叠加分库分表与读写分离应对海量数据。

**关键知识点**
- **背景**：MySQL 按固定大小的“页”读写数据，字段占用无谓空间会导致一次读入的有效数据变少，光改语句无法解决，需优化表设计。
- **数据类型优化**：
  - 整数尽量用小类型：INT(4字节) → MEDIUMINT(3字节，范围0~16777215)可省25%空间；
  - 能用整型不用文本：18位流水号 TEXT(20字节) → BIGINT(8字节)省60%。实测400万行导入快约21s、`COUNT` 由5.18s降到3.86s。
- **合理增加冗余字段（反范式）**：数据量大且频繁 JOIN 时，把常用字段（如商品名 goodsname）冗余进流水表，把连接查询变单表查询。条件：①该冗余字段**不常修改**；②查询时**不可或缺**。代价：占空间、需保证一致性（改商品名要同步改流水表）。
- **拆分表（垂直拆分）**：把一个宽表按字段使用频率拆成多个窄表——高频字段一表、低频字段一表（都保留原主键关联）。例：流水单头拆成表1（会员号/收款机号/整单折扣折让）+ 表2（微信/支付宝/现金金额，仅财务统计用）。每次读记录更小，查询更快。
- **使用非空约束**：业务允许就设 NOT NULL——省去判空开销、便于建索引、每字段还能省约1 bit 存储。
- **补充：垂直拆分 vs 水平拆分**
  - **垂直拆分**：按列拆。垂直分表=拆字段（冷热分离，见上）；垂直分库=按业务模块把不同表分到不同库（用户库/订单库/商品库）。
  - **水平拆分**：按行拆。水平分表=同一库内按规则（如 `user_id % N`、按时间范围、按范围）把行分散到多张结构相同的表；水平分库=分散到多个库/实例，突破单机存储与并发瓶颈。
- **补充：分片键与路由**：选择区分度高、查询常用的分片键（如 user_id）；常用取模（均匀但扩容需迁移）、范围（易扩容但易热点）、一致性哈希、时间分片。跨分片的 JOIN/聚合/排序/分页变复杂，需中间件（ShardingSphere、MyCat）或应用层处理，全局唯一 ID 用雪花算法/号段。
- **补充：主从复制与读写分离**：主库写、从库读，通过 binlog 同步。读多写少时分摊读压力、提升可用性。注意主从延迟导致的读旧数据（强一致读走主库）。

**面试高频问答**
- **Q：** 单表数据量太大怎么优化？（分库分表思路）
  **A：** 循序渐进：①先优化 SQL 和索引、优化数据类型；②垂直拆分冷热字段、必要时加冗余字段减 JOIN；③引入读写分离分摊读压力；④仍不够则水平分表/分库（按分片键 hash 或范围），配合分库分表中间件。一般单表建议控制在千万级或2GB以内考虑拆分。
- **Q：** 垂直拆分和水平拆分的区别？
  **A：** 垂直=按列/业务拆（拆字段或拆库），解决表宽和业务耦合；水平=按行拆（同结构多表/多库），解决单表行数过大。海量数据通常两者结合。
- **Q：** 分库分表带来哪些问题？
  **A：** 跨分片 JOIN/聚合/排序/分页困难、分布式事务、全局唯一主键、扩容数据迁移、分片键选择不当导致热点。需用中间件、雪花ID、避免跨分片查询等手段缓解。

### 26 | 如何充分利用系统资源？

**一句话核心**：通过调整 InnoDB 关键系统参数（刷盘策略、缓冲池大小与实例数）平衡性能与安全，用 Performance Schema 监控定位真正瓶颈（常是缺索引），而非盲目堆硬件。

**关键知识点**
- **优化思路**：CPU 是最核心且获取成本高的资源，解决 CPU 不足两条路——①疏通瓶颈缩短等待；②开拓新通道增加并行。
- **三个关键 InnoDB 参数（配置文件 my.ini）**：
  - `innodb_flush_log_at_trx_commit`：日志刷盘策略。**1**（默认，最安全）每次提交都写日志并刷盘；**2** 每次提交写日志、但日志每秒刷盘一次（折中，宕机丢失小、大并发下效率高）；**0** 每秒才写日志并刷盘（最快但最不安全）。案例改成 2 降低了 CPU 使用率。
  - `innodb_buffer_pool_size`：InnoDB 用来缓存索引和数据的缓冲池大小，越大缓存越多、磁盘读写越少。专用库服务器可设为物理内存的 50%~70%（案例128G内存设64G）。
  - `innodb_buffer_pool_instances`：把缓冲池分成几个区，允许多进程并行处理不同分区，提高并行能力（案例设64）。
  - 坑：my.ini 保存编码必须用 ANSI，用记事本默认 UTF-8 保存会导致 MySQL 起不来。
- **为什么 0 比 2 更不安全**（思考题）：0 是“写日志+刷盘”都每秒做一次，二者都可能丢；2 是每次提交就写入日志（OS 缓存），只有刷盘每秒一次，丢失窗口更小、可靠性更高。
- **Performance Schema 监控**：专门监控服务器执行情况的存储引擎，数据存在 `performance_schema` 库。
  - 开启监控两张配置表：`setup_instruments`（哪些对象的事件被捕获/生产者）、`setup_consumers`（保存哪些事件信息/消费者）。`UPDATE ... SET ENABLED='YES'` 开启。
  - 事件记录表：`events_statements_current`（当前每进程查询）、`events_statements_history`（每进程最近，默认10条）、`events_statements_history_long`（全局最近，默认10000条）。
  - 定位慢查询：按 `TIMER_WAIT` 排序查 `sql_text`（TIMER_WAIT 单位微微秒/万亿分之一秒）找出最耗时语句。案例发现高频大表 `clientdeposit` 忘建索引 → 建索引后 CPU 从99%降到20%。
- **重要理念**：CPU 使用率失控导致崩溃，首先应想“应用本身有缺陷”，找自身问题（如缺索引）而非盲目加硬件——资源永远有限。

**面试高频问答**
- **Q：** `innodb_flush_log_at_trx_commit` 三个取值怎么选？
  **A：** 1 最安全（默认，金融等强一致场景）；2 折中（宕机最多丢1秒 OS 缓存里的日志，主机不宕基本不丢，高并发推荐）；0 最快但最不安全（MySQL 进程崩溃即可能丢1秒数据）。常与 `sync_binlog` 配合调优（双1最安全）。
- **Q：** buffer pool 有什么用，设多大合适？
  **A：** InnoDB 的核心缓存，缓存数据页和索引页，命中则不读磁盘。专用数据库服务器一般设物理内存的 50%~70%。
- **Q：** 怎么找出到底哪条 SQL 拖慢了系统？
  **A：** 慢查询日志（long_query_time）、`SHOW PROCESSLIST` 看当前执行、Performance Schema 的 events_statements_* 按耗时排序，再对 top 语句 EXPLAIN 优化。

### 特别 | MySQL8新特性

**一句话核心**：MySQL 8 引入窗口函数和 CTE 公用表表达式，用更简单的语句实现分组内计算与递归查询；并带来降序索引、原子 DDL、默认 utf8mb4 等一批增强。

**关键知识点**
- **窗口函数**：类似分组，但**不会把多行聚合成一行**，而是把统计结果附加到每一条记录上（保留原行数），适合“用分组统计结果对每行做计算”的场景，且不用临时表、效率更高。
  - 语法：`函数 OVER (PARTITION BY 字段 [ORDER BY ...])` 或 `函数 OVER 窗口名 ... WINDOW 窗口名 AS (...)`。
  - 例（每行同时算市/全国销售额与占比，一步完成）：
    `SUM(salesvalue) OVER(PARTITION BY city) AS 市销售额, salesvalue/SUM(salesvalue) OVER() AS 总比率`
  - 排序函数（`OVER(ORDER BY points DESC)`）：`RANK()` 计并列且并列占位（1,1,3）；`DENSE_RANK()` 计并列但不占位（1,1,2）；`ROW_NUMBER()` 不计并列，纯序号（1,2,3）。
- **CTE 公用表表达式**：命名的临时结果集，存在于单条查询中，可**代替子查询且能被多次引用**（子查询不能被多次引用）。
  - 普通 CTE：`WITH cte AS (子查询) SELECT ... FROM t JOIN cte ON ...;`
  - 递归 CTE：`WITH RECURSIVE cte AS (种子查询 UNION ALL 递归查询) SELECT ...;`——种子查询给初始值只跑一次，递归查询反复执行直到无新数据。适合查询树形/层级结构（如上下级、菜单、徒子徒孙），不受层级限制。
- **其他新特性（面试常提）**：
  - **降序索引**：`INDEX(a DESC)` 真正按降序存储，优化 ORDER BY 混合升降序。
  - **原子 DDL**：DDL（建表/改表/删表）具备原子性，`CREATE TABLE ... SELECT` 等要么全成功要么全回滚，提高数据安全。
  - **默认字符集 utf8mb4**（排序规则 utf8mb4_0900_ai_ci），真正支持 4 字节 emoji。
  - **隐藏字段/隐藏索引**：`INVISIBLE` 索引可先设不可见观察影响再决定删除，不影响优化器。
  - **默认存储引擎 InnoDB**；系统表/数据字典改为 InnoDB 事务表（原子数据字典）。
  - 完善空间数据（GIS）处理；支持 `SELECT ... FOR UPDATE SKIP LOCKED / NOWAIT`；通用表达式默认值、函数索引、角色（ROLE）权限管理、资源组等。

**面试高频问答**
- **Q：** 窗口函数和 GROUP BY 有什么区别？
  **A：** GROUP BY 把每组聚合成一行、行数减少；窗口函数保留原始行数，把聚合/排序结果附加到每一行，适合“每行 + 组内统计/排名”的需求，一步完成、无需临时表。
- **Q：** RANK、DENSE_RANK、ROW_NUMBER 区别？
  **A：** 遇并列时：RANK 并列同名次且跳号（1,1,3）；DENSE_RANK 并列同名次不跳号（1,1,2）；ROW_NUMBER 不管并列，唯一连续序号（1,2,3）。取 TopN/分组第N名常用。
- **Q：** CTE 相比子查询有什么好处？
  **A：** 可命名、可读性好、可被同一查询多次引用；递归 CTE 能优雅处理树形/层级递归查询，是子查询做不到的。

### 特别 | 经典面试题(一)

**一句话核心**：索引概念 + 排序/LIMIT 分页取第N名 + 外连接保全 + 自连接分组 HAVING + 关联子查询取分组 TopN 的实战 SQL。

**关键知识点 / 高频问答**
- **Q（索引描述判断）：** 关于索引下列哪项正确？
  **A：** “一个表可建一个或多个索引”正确。辨析：①建索引不能减少冗余数据、反而占额外空间（只提查询速度）；②组合索引不一定比单列索引高效，遵循**最左对齐（最左前缀）原则**，查询不含最左字段则整个组合索引用不上，用全字段建组合索引成本高利用率低；③索引不是越多越好，建索引有存储和维护（改数据要改索引）成本，很少作筛选条件的字段不必建。
- **Q：** 查出分数排名第三的成绩（成绩各不相同）。
  **A：** `SELECT * FROM demo.test1 ORDER BY score DESC LIMIT 2,1;`——LIMIT 两参数：起始位置（从0开始）和返回条数，`LIMIT 2,1` 即从第3条起返回1条。
- **Q：** 关联查询，人员信息必须全部包含（有无地址都要）。
  **A：** 用左外连接：`SELECT * FROM person a LEFT JOIN address b ON a.id=b.id;`——LEFT JOIN 保证左表记录全部保留，右表无匹配则为 NULL。
- **Q：** 查询至少有 2 名学生的老师姓名（自连接 + HAVING）。
  **A：** 先分组统计老师编号：`SELECT teacherid FROM teach WHERE teacherid IS NOT NULL GROUP BY teacherid HAVING COUNT(*)>=2;`（WHERE 先过滤行、HAVING 再过滤分组）；再自连接取姓名：`SELECT a.id,a.fname FROM teach a JOIN (上面子查询) b ON a.id=b.teacherid;`
- **Q：** 查出每个班级前三名（关联子查询）。
  **A：** 用“比我分高的不同分数少于3个即为前三”：
    `SELECT c.classname,a.stdname,a.points FROM student a JOIN class c ON a.classid=c.id WHERE 3 > (SELECT COUNT(DISTINCT b.points) FROM student b WHERE b.points>a.points AND b.classid=a.classid) ORDER BY c.classname,a.points DESC;`（MySQL 8 也可用 `ROW_NUMBER()/RANK() OVER(PARTITION BY classid ORDER BY points DESC)` 更简洁）。
- **补充常识**：WHERE 与 HAVING 区别——WHERE 在分组前过滤行、不能用聚合函数；HAVING 在分组后过滤组、可用聚合函数。

### 特别 | 经典面试题(二)

**一句话核心**：日志与 WAL、事务与存储引擎、视图可更新性、存储过程、游标、触发器等进阶概念的辨析题。

**关键知识点 / 高频问答**
- **Q（日志）：** 关于日志文件描述哪项正确？
  **A：** “检查点主要作用是提高故障后恢复效率”正确。辨析：①日志严格按操作执行顺序记录；②遵循 **WAL（Write-Ahead Logging）先写日志后写数据**（日志顺序写磁盘比随机写数据快，写日志成功即回复客户端成功）；③检查点标记哪些脏页已刷盘，恢复时从检查点开始重放而非全量日志；④事务故障恢复也需日志——回滚日志（undo，回到故障前）、重做日志（redo，找回内存丢失的已提交更新）、二进制日志（binlog，主从同步恢复）。
- **Q：** MySQL 支持事务吗？
  **A：** 取决于存储引擎：InnoDB 支持、MyISAM 不支持。MySQL 8 默认 InnoDB。默认 AUTOCOMMIT，每条 SQL 自成事务；可用 `START TRANSACTION`/`BEGIN` 开启、`COMMIT` 提交、`ROLLBACK` 回滚包含多条语句的事务。查引擎：`SHOW CREATE TABLE demo.test;`
- **Q（视图）：** 哪项错误？
  **A：** “所有视图都可查询和更新”错误——所有视图可查，但含分组/聚合函数、DISTINCT、UNION 等无法定位到底层记录的视图**不可更新**（DML）。视图是存储的查询语句、不存实际数据、提供逻辑独立性，对视图的操作最终都转成对底层表的操作。
- **Q（视图 DML 能否执行）：** 视图定义含 `AVG()` 子查询，UPDATE/DELETE/INSERT/SELECT 哪些能执行？
  **A：** 只有 SELECT 能。含聚合函数的视图无法精确定位底层记录，不能做 DML。
- **Q（存储过程）：** 哪项错误？
  **A：** “存储过程与数据库无关”错误——存储过程是数据库的一部分，不能独立于数据库存在。它是预编译存放在服务器上的一组 SQL，可完成特定业务逻辑，用 `CALL` 调用。
- **Q：** 什么是游标？如何知道走到最后？
  **A：** 游标是对结果集的扩展，允许逐行处理结果集。特点：一次一行、只能单向不能跳跃、只读。判断到末尾用 `DECLARE CONTINUE HANDLER FOR NOT FOUND`。
- **Q（触发器）：** 哪项错误？
  **A：** “触发器通过 CALL 调用”错误——触发器由数据操作**事件驱动**（INSERT/UPDATE/DELETE），不能用 CALL；它是特殊存储程序，创建在特定表上，有助于实现完整性、安全性和主动性。

**补充：其他超高频经典八股（讲义未列，面试常问）**
- **char vs varchar：** char 定长（0~255，不足补空格、取出去空格），存取快、适合定长如身份证/MD5；varchar 变长（最大65535字节），省空间、适合长度不定，需额外1~2字节记录长度。定长且频繁改用 char，长度波动大用 varchar。
- **drop / delete / truncate：** delete 是 DML、逐行删可带 WHERE、走事务可回滚、不释放空间、触发触发器；truncate 是 DDL、清空整表、快、不可回滚、重置自增、不触发触发器；drop 是 DDL、连表结构一起删除。删部分行用 delete，清空大表用 truncate，删表用 drop。
- **count(\*) / count(1) / count(字段)：** count(\*) 和 count(1) 统计所有行（含 NULL 行），InnoDB 下性能基本相同、官方推荐 count(\*)；count(字段) 只统计该字段非 NULL 的行。
- **UNION vs UNION ALL：** UNION 去重并排序（有开销）；UNION ALL 不去重、直接拼接、更快。确定无重复用 UNION ALL。
- **主键 vs 唯一索引：** 主键唯一且非空、一张表一个、聚簇索引；唯一索引唯一但可为 NULL、可多个。
- **truncate 后自增、外键**等细节亦是延伸考点。
