# 09 · 操作系统 速答（Operating System）

> 面向后端/Java 岗的操作系统八股：进程/线程/协程、IPC、同步、死锁、虚拟内存、调度、用户/内核态、IO 模型。覆盖 ~20 个高频问，全部「能背出口」。重点章节（★ 死锁 / 虚拟内存缺页 / IO 多路复用 epoll）在展开区细讲。

## 🔥 高频必背（Top 18）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | 进程 vs 线程？ | **进程是资源分配的最小单位**（独立地址空间、文件描述符表）；**线程是 CPU 调度的最小单位**，同进程内线程**共享地址空间/堆/全局变量**，各自独占栈和寄存器。 |
| 2 | 为什么线程比进程"轻"？ | 线程共享进程资源，**创建/销毁/切换开销小**，通信直接读共享内存即可；进程切换要换页表、刷 TLB，还得走 IPC 通信，代价大。 |
| 3 | 协程（Coroutine）是什么？ | **用户态**的轻量"线程"，由程序自己调度而非内核，切换不陷入内核、成本极低；一个线程可跑成千上万协程。适合高并发 IO（Go goroutine、Java 虚拟线程）。 |
| 4 | 进程/线程/协程调度者分别是谁？ | 进程、线程由**内核调度**（抢占式）；协程由**用户态运行时/程序**协作式调度（遇 IO 主动让出）。 |
| 5 | 上下文切换（Context Switch）开销来自哪？ | 保存/恢复寄存器 + 内核态用户态切换；**进程切换额外要换页表、刷 TLB、CPU 缓存失效**，所以最贵。协程切换只换栈和寄存器、不进内核，最便宜。 |
| 6 | ★ IPC 有哪几种？ | 管道 pipe、命名管道 FIFO、消息队列 message queue、**共享内存 shared memory**、信号量 semaphore、信号 signal、Socket。 |
| 7 | 哪种 IPC 最快？ | **共享内存**——直接读写同一块物理内存，不需内核拷贝；但要配**信号量**做同步防竞争。 |
| 8 | 线程同步手段有哪些？ | **互斥锁 mutex**（临界区互斥）、**信号量 semaphore**（控制资源计数）、**条件变量 condition variable**（等待某条件成立，配锁用）、**读写锁 rwlock**（读共享写独占）。 |
| 9 | 互斥锁 vs 信号量区别？ | 互斥锁值只有 0/1、**谁加锁谁解锁**（有归属）；信号量是计数器，可 >1、可由不同线程 P/V，用来控制"同时最多 N 个"访问。 |
| 10 | ★ 死锁的四个必要条件？ | **互斥、持有并等待、不可剥夺、循环等待**——四个同时成立才死锁，破坏任一即可预防。 |
| 11 | 死锁怎么解决？ | 四类：**预防**（破坏四条件之一）、**避免**（银行家算法，动态判断是否进入不安全状态）、**检测+解除**（画资源分配图找环，杀进程/剥夺资源）、**鸵鸟策略**（不管，重启）。 |
| 12 | ★ 为什么要虚拟内存？ | ①每个进程有**独立连续地址空间**、互相隔离更安全；②可运行**大于物理内存**的程序（换页到磁盘）；③方便内存管理与共享（如 fork COW）。 |
| 13 | 页表 / 缺页中断（Page Fault）？ | 页表把**虚拟页→物理页**映射；访问的页不在内存时触发**缺页中断**，OS 从磁盘把页调入（必要时先置换一页出去），再重执行指令。 |
| 14 | 常见页面置换算法？ | **OPT**（最优，淘汰未来最久不用，理论上限不可实现）、**LRU**（最近最久未使用，近似 OPT，工程用）、**FIFO**（先进先出，可能 Belady 异常）、Clock（LRU 的低成本近似）。 |
| 15 | 进程调度算法有哪些？ | **FCFS** 先来先服务、**SJF** 短作业优先、**时间片轮转 RR**、**优先级调度**、**多级反馈队列 MLFQ**（综合，Linux/Windows 采用）。 |
| 16 | ★ 用户态 vs 内核态？ | 内核态可执行特权指令、访问全部硬件/内存；用户态受限。二者靠**系统调用/中断/异常**切换，切换要保存上下文、有开销，所以要减少系统调用次数。 |
| 17 | ★ IO 多路复用 select/poll/epoll？ | 一个线程用一次系统调用同时监听大量 fd，就绪才处理。**epoll 用红黑树管理 fd + 就绪链表回调**，O(1) 拿就绪、无 fd 数限制，远优于 O(n) 轮询的 select/poll。 |
| 18 | 零拷贝（Zero-Copy）是什么？ | 减少数据在**内核态↔用户态**间的多余拷贝和上下文切换（如 `sendfile`/`mmap`），Kafka/Netty 大文件传输靠它提吞吐。 |

## 📌 展开速答

**Q：进程、线程、协程一句话对比怎么答？**
一句话：**进程是资源分配单位，线程是调度单位，协程是用户态调度单位**。进程独立地址空间、隔离最好但最重；线程共享进程内存、轻量但需加锁防竞争、由内核抢占式调度；协程比线程更轻，切换在用户态不进内核，由运行时协作式调度，一个线程能承载海量协程，专治 IO 密集高并发（Go goroutine、Java 21 虚拟线程）。追问"高并发为何用协程"就答：线程数受内核限制、每个线程栈占 MB 级内存、切换要陷内核；协程栈按需增长（KB 级）、切换纯用户态，能开百万级。

**Q：★ 死锁的四个必要条件和处理策略，详细讲一下？**
四条件缺一不可，记忆口诀"**互持不循**"：
- **互斥**：资源同一时刻只能被一个进程占用。
- **持有并等待**：已持有资源，又在等别的资源不释放。
- **不可剥夺**：资源只能被持有者主动释放，不能被抢。
- **循环等待**：存在一条进程—资源的环形等待链。

四种应对：
1. **预防**——破坏其中一个条件。最实用是破坏"循环等待"：给所有资源**编号，按序申请**（如统一先锁小 id 再锁大 id）；或破坏"持有并等待"：一次性申请全部资源。
2. **避免**——**银行家算法**：每次分配前先试算，若分配后系统仍处于"存在一个安全序列能让所有进程都跑完"的**安全状态**就分配，否则拒绝。
3. **检测 + 解除**——允许死锁发生，定期跑**资源分配图**找环检测，发现后**杀进程 / 剥夺资源 / 回滚**解除。
4. **鸵鸟策略**——发生概率低就不处理，靠重启（很多 OS 实际做法）。

Java 场景加分：数据库死锁、`synchronized` 嵌套锁顺序不一致都是"循环等待"，**统一加锁顺序**是最常用的工程解法；也可用 `tryLock(timeout)` 破坏"不可剥夺/持有等待"。

**Q：★ 虚拟内存和缺页中断的完整流程？**
CPU 发出的是**虚拟地址**，经 MMU 查**页表**转成物理地址；页表项记录该虚拟页是否在内存（present 位）、物理页号、权限位。命中则直接访问（TLB 缓存加速常用映射，命中率极高）。若 present=0 → 触发**缺页中断**，流程：
1. 陷入内核，OS 检查地址是否合法（非法→段错误 SIGSEGV）。
2. 内存有空闲帧则分配；**没有空闲帧就用置换算法（LRU 等）选一个牺牲页**换出——若该页被改过（dirty）需先写回磁盘。
3. 把缺的页从磁盘（swap / 文件）读进内存，更新页表。
4. 重新执行触发缺页的那条指令。

关键概念：**多级页表**省空间（不给未用的地址分配页表项）、**TLB** 缓存映射减少查表、**局部性原理**是虚拟内存高效的前提。追问 **LRU 怎么实现**：`HashMap + 双向链表` O(1)（手撕高频，见 [`../02-coding`](../02-coding)）；追问 **Belady 异常**：FIFO 可能出现"增加物理页反而缺页更多"，LRU/OPT 属栈算法不会。

**Q：★ IO 多路复用 select / poll / epoll 的区别？**

| 维度 | select | poll | epoll |
|---|---|---|---|
| fd 组织 | 固定大小 bitmap | 数组（无上限） | **红黑树**（内核维护） |
| fd 上限 | 通常 1024 | 无硬上限 | 无（系统内存限制） |
| 找就绪 fd | O(n) 遍历全部 | O(n) 遍历全部 | **O(1)**：回调把就绪 fd 放入就绪链表 |
| 每次拷贝 fd 集 | 每次全量拷贝到内核 | 每次全量拷贝 | **一次注册（epoll_ctl），不重复拷贝** |
| 触发模式 | 只有 LT | 只有 LT | **LT 水平 / ET 边缘** 两种 |

核心：select/poll 每次调用都要把全部 fd 拷进内核并**线性扫描**，fd 多了性能线性下降；epoll 用 `epoll_ctl` 一次注册到红黑树，fd 就绪时由中断回调放入**就绪链表**，`epoll_wait` 直接返回就绪的，**只关心活跃连接**，所以 C10K/海量连接场景碾压 select/poll。**LT（水平触发）**：只要缓冲区还有数据就一直通知（默认、好用不易漏）；**ET（边缘触发）**：只在状态变化时通知一次，必须一次读干（配非阻塞 IO），效率更高但难写。

面试串联加分：**Redis 单线程高性能**靠 epoll 做 IO 多路复用（一个线程管所有连接）；**Netty/Nginx** 基于 epoll 的 Reactor 模型；对上 Java NIO 的 `Selector` 底层在 Linux 就是 epoll。

**Q：五种 IO 模型怎么区分？**
按"**是否阻塞**"和"**谁负责数据拷贝**"分：
1. **阻塞 IO（BIO）**：调用后一直等到数据就绪且拷贝完，线程全程阻塞。
2. **非阻塞 IO**：没数据立即返回错误，需**轮询**，浪费 CPU。
3. **IO 多路复用**：select/poll/epoll，一个线程管多个 fd，就绪才处理（Redis/Netty）。
4. **信号驱动 IO**：注册信号，数据就绪时内核发 `SIGIO` 通知。
5. **异步 IO（AIO）**：真正的异步，内核把**数据都拷贝好**再通知你，全程不阻塞（Linux io_uring/Windows IOCP）。

前四种都是**同步 IO**（数据从内核拷到用户态这步要自己等），只有 AIO 是真异步。

**Q：用户态、内核态与系统调用是怎么回事？**
CPU 有特权级（x86 的 Ring 0~3），**内核态（Ring 0）**能执行特权指令、直接操作硬件和任意内存；**用户态（Ring 3）**受限，要访问硬件/文件/网络必须通过**系统调用（syscall）**请求内核代劳。触发切换的三种方式：**系统调用（主动，软中断）**、**硬件中断**（如网卡收包、时钟）、**异常**（缺页、除零）。切换要保存/恢复上下文、有开销，所以"减少系统调用"是性能优化方向（如 IO 缓冲批量写、零拷贝减少内核↔用户拷贝）。

## ⚠️ 易错 / 反问加分

- ⚠️ **别说"进程是调度单位"**——进程是**资源分配**单位，**线程**才是 CPU 调度单位。这是最常见口误，答反直接扣分。
- ⚠️ **协程不是内核调度**——协程在**用户态**协作式调度，遇 IO 主动让出；说成"内核抢占"就错了。Java 虚拟线程本质是把阻塞点交给 JVM 调度。
- ⚠️ **共享内存最快但不自带同步**——它只提供"共享"，防竞争要自己配**信号量/互斥锁**，别漏答同步。
- ⚠️ **死锁四条件是"必要"不是"充分"**——四个都满足才**可能**死锁，破坏任一即可预防；别背成"满足就一定死锁"。
- ⚠️ **LRU ≠ LFU**——LRU 按"最近是否用过"（时间），LFU 按"用的次数"（频率），别混。
- ✅ **加分**：把 epoll 和 Redis 单线程、Netty/Nginx Reactor、Java NIO Selector 串起来，显示你懂底层落地，不只背概念。
- ✅ **加分**：讲 fork 时带 **COW（Copy-On-Write）**——`fork` 后父子进程共享物理页，只在**某方写入时才复制该页**，省内存也加速；**Redis RDB/AOF-rewrite 靠 fork 子进程做快照**，写时复制保证主进程照常服务。
- ✅ **加分**：**僵尸进程 vs 孤儿进程**——子进程结束但父进程没 `wait()` 回收其退出状态 → **僵尸进程**（占 PID，危害在于耗尽进程表，需父进程 wait 或杀父让 init 收养）；父进程先死、子进程被 **init/systemd（PID 1）收养** → **孤儿进程**（无害，会被正常回收）。
- ✅ **加分**：谈线程同步主动提"**锁的粒度**"和"**乐观 vs 悲观锁 / CAS 无锁**"，把 OS 层同步接到 Java 并发（`synchronized`/`ReentrantLock`/`AQS`/CAS），见 [`03-concurrency`](03-concurrency.md)。
