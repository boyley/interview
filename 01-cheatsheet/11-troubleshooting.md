# 11 · 线上问题排查 速答（Troubleshooting）

> 线上高频排查题。核心方法：**沿"请求链路"分层定位，哪一段耗时高就查哪段**。
> 已收录：接口响应慢。待补：CPU 100%、内存溢出 OOM、频繁 GC、死锁、磁盘打满。

---

# 🐢 线上接口响应很慢，一般是哪些问题导致的？

> **核心理念：接口慢 = "请求进来 → 响应出去"这条链路上某一段耗时高。排查不靠猜，靠「分层链路 + 监控下钻」定位到具体那一环。**
> 答题方法：先给**分层链路框架**，再**逐层列原因**，最后给**排查工具 + 方法论步骤**。

---

## 一、先建立分层链路框架（★ 分类框架，别一上来乱猜）

一次请求的完整链路，慢一定出在其中某一环：

```
客户端 → 网络/DNS → 网关/负载均衡 → 应用服务 → 中间件 → 数据库 → 外部依赖
                                    (JVM/线程/锁/代码)  (缓存/MQ)        (RPC/第三方)
```

> 排查主线：**看 APM 链路追踪，定位是"哪一段"慢，再下钻到那一层的具体原因。**

---

## 二、逐层排查可能原因（大步骤 → 每层小步骤）

### 层 1 · 网络 / 接入层
- 带宽打满、跨机房/跨地域调用、DNS 解析慢
- TLS 握手开销、Nginx/网关配置不当、连接数耗尽
- 客户端到服务端网络抖动

### 层 2 · 应用层-JVM（偶发卡顿高发区）
- **频繁 GC / Full GC**，STW 停顿导致接口偶发变慢
- 堆内存不足、内存泄漏 → GC 越来越频繁
- 大对象分配、元空间不足

### 层 3 · 应用层-线程
- **线程池满**，请求排队等待
- **锁竞争 / 死锁**，`synchronized`/分布式锁阻塞
- 同步阻塞调用（本该异步/并行的串行了）

### 层 4 · 应用层-代码
- 慢算法、循环里调 RPC/查库（**N+1 问题**）
- 大对象序列化/反序列化、大 JSON
- 日志打太多（尤其同步日志、大量 stack trace）
- 本地缓存未用、重复计算

### 层 5 · 数据库（最常见根因）
- **慢 SQL**：无索引 / 索引失效 / 全表扫描
- **锁等待**：行锁/表锁、大事务长时间持锁
- **深分页**：`limit 100000,10` 扫大量行
- 连接池不够、连接泄漏
- 主从延迟（读到慢从库）

### 层 6 · 缓存
- 缓存未命中率高、**缓存穿透/击穿/雪崩** → 压到 DB
- **大 key / 热 key**：单 key 过大或访问过热
- Redis 慢命令（`keys *`、大范围操作）、序列化慢

### 层 7 · 外部依赖
- 下游 RPC / 第三方接口慢或超时
- 多个依赖**串行调用**，没并行（可 `CompletableFuture` 并行）
- 没设超时，被下游拖死

### 层 8 · 系统资源
- **CPU 打满**（死循环、频繁 GC、正则回溯）
- 内存不足、频繁 swap
- 磁盘 I/O 高、句柄/连接数耗尽

---

## 三、排查工具（小步骤：按层选工具）

| 层面 | 工具 | 看什么 |
|---|---|---|
| 全链路 | APM：SkyWalking / Zipkin / 鹰眼 | 每段耗时，一眼定位慢在哪 |
| JVM-GC | GC 日志 / `jstat` / arthas `dashboard` | GC 频率、STW 时长 |
| JVM-线程 | `jstack` / arthas `thread` | 线程阻塞、死锁、热点线程 |
| 方法耗时 | arthas `trace`/`watch`/`profiler` | 定位到具体慢方法 |
| 数据库 | `EXPLAIN` / 慢查询日志 / `show processlist` | 慢 SQL、锁、执行计划 |
| 系统 | `top` / `vmstat` / `iostat` / `netstat` | CPU/内存/IO/连接数 |

---

## 四、排查方法论（编号步骤）

1. **看监控/APM** — 先定位是哪个接口、哪一段慢（网络？应用？DB？依赖？）
2. **分偶发 vs 持续** — 偶发多是 GC/网络抖动/锁竞争；持续多是慢 SQL/慢算法/资源瓶颈
3. **分整体 vs 个别** — 整体慢查资源/依赖瓶颈；个别请求慢查热点 key/大参数/特定数据
4. **下钻根因** — 慢 SQL 用 EXPLAIN，线程用 jstack，方法用 arthas trace，GC 看日志
5. **优化 + 压测验证** — 改完必须压测确认，别只看单次

---

## 五、高频根因 TOP + 对应优化

| 根因 | 优化对策 |
|---|---|
| 慢 SQL / 无索引 | 加索引、避免全表扫描、改写 SQL、深分页优化 |
| 频繁 Full GC | 调堆/GC 参数、查内存泄漏、减少大对象 |
| 缓存击穿/穿透 | 互斥锁重建、布隆过滤器、缓存空值、多级缓存 |
| 线程池满/锁竞争 | 扩容/隔离线程池、减小锁粒度、无锁化 |
| 下游依赖慢 | 加超时熔断、串行改并行、结果缓存 |
| 无连接池/池太小 | 引入/调大连接池（DB、Redis、HTTP） |
| 深分页 | 游标分页 / `where id > ?` 代替大 offset |

---

> **核心原则：接口慢排查 = 分层链路定位（APM 看哪段）→ 下钻根因（SQL/GC/线程/依赖）→ 对症优化 → 压测验证。经验值：线上接口慢，八成在「慢 SQL」和「外部依赖」，其次是「GC 和锁」。**

- 频繁 GC / 内存泄漏
- 死锁排查（jstack 找 deadlock）
