## 二、必背核心结论（背这些应付大部分 JVM 八股）

1. **Java 运行流程**：`.java` →(javac 编译)→ `.class` 字节码 →(类加载)→ JVM 中**先解释执行**，热点代码被**即时编译(JIT)**成机器码。一次编译到处运行靠的是字节码 + 各平台 JVM。
2. **运行时数据区**：**堆**（对象实例，线程共享，GC 主战场）、**方法区/元空间**（类元信息、运行时常量池，JDK8 起元空间用本地内存）、**虚拟机栈**（每方法一个栈帧：局部变量表+操作数栈，线程私有）、**程序计数器 PC**、**本地方法栈**。
3. **类加载过程**：加载 → 链接（**验证 → 准备**[给静态变量赋默认零值] **→ 解析**[符号引用转直接引用]）→ 初始化（执行 `<clinit>`，真正赋初值/静态块）。
4. **双亲委派**：类加载先委派给父加载器，父加载不了才自己加载。加载器层级：启动 Bootstrap → 扩展/平台 → 应用 AppClassLoader → 自定义。目的：安全 + 避免类重复加载。
5. **方法调用 5 指令**：`invokestatic`(静态)、`invokespecial`(构造器/私有/super)、`invokevirtual`(虚方法)、`invokeinterface`(接口)、`invokedynamic`(动态,Lambda)。前两个是**静态绑定**，中间两个靠**虚方法表 vtable/itable 动态绑定**。
6. **重载 vs 重写**：重载(overload)编译期按参数静态确定；重写(override)运行期按实际对象动态分派。
7. **对象内存布局**：对象头（Mark Word[哈希/GC 年龄/锁标志] + 类型指针 Klass Pointer）+ 实例数据 + 对齐填充(8 字节对齐)。开启压缩指针可省内存。
8. **判活用可达性分析**（不是引用计数，因为解决不了循环引用）：从 **GC Roots**（栈局部变量、静态字段、常量、JNI 引用等）出发能到达的对象存活。
9. **四种引用**：强（永不回收）> 软（内存不足才回收，做缓存）> 弱（下次 GC 必回收）> 虚（仅用于回收通知）。
10. **分代收集**：新生代（Eden + 2 个 Survivor，占比 8:1:1，用**复制算法**）+ 老年代（**标记-整理/标记-清除**）。对象优先 Eden 分配，熬过多次 Minor GC(默认 15)晋升老年代。**Minor GC** 收新生代（频繁快），**Full GC** 收整堆（慢，要尽量避免）。
11. **volatile**：保证**可见性** + **禁止指令重排序**（内存屏障），但**不保证复合操作原子性**（如 i++ 仍需锁/原子类）。
12. **synchronized 锁升级**：无锁 → **偏向锁**（只一个线程，记录线程 ID）→ **轻量级锁**（少量竞争，CAS 自旋）→ **重量级锁**（竞争激烈，走操作系统 monitor 互斥，线程阻塞）。只能升级不能降级。
13. **即时编译分层**：解释执行(快启动) + C1(client,快编译) + C2(server,深度优化)。靠**方法调用计数器 + 回边计数器**探测热点；优化过度激进出错时会**去优化(deoptimization)**退回解释执行。
14. **逃逸分析**衍生 3 大优化：**栈上分配**（对象不逃逸就分配在栈，免 GC）、**标量替换**（把对象拆成基本类型字段）、**锁消除**（锁对象不逃逸就去掉锁）。

                                      │
                                 ①类加载器加载
                                      ▼

   │  ②解释器逐条执行字节码（启动快）             │

**记忆点**：Java「一次编译，到处运行」= 字节码跨平台 + 每个平台有自己的 JVM。JVM 内部**解释执行 + 即时编译**混合模式：先解释保证快速启动，热点再编译成机器码保证峰值性能。

### 图2 · JVM 运行时数据区 ★ — 讲01/10

```
   ┌─────────────────────── 线程共享 ───────────────────────┐
   │   堆 Heap（对象实例，GC 主战场）                          │
   │   ┌─────新生代─────┐   ┌────老年代────┐                  │
   │   │ Eden │ S0 │ S1 │   │  Old Gen     │                  │
   │   └───────────────┘   └──────────────┘                  │
   │   方法区/元空间 Metaspace（类元信息、运行时常量池）        │
   └─────────────────────────────────────────────────────────┘
   ┌─────────────────────── 线程私有 ───────────────────────┐
   │  虚拟机栈(栈帧:局部变量表+操作数栈) │ 程序计数器PC │ 本地方法栈 │
   └─────────────────────────────────────────────────────────┘
```
**记忆点**：**共享**（堆、方法区）会 GC、要考虑线程安全；**私有**（栈、PC、本地方法栈）随线程生灭、天然线程安全。JDK8 起「永久代」被「元空间(Metaspace，本地内存)」取代。栈深度超限 → `StackOverflowError`；堆放不下 → `OutOfMemoryError`。

### 图3 · 类加载过程 + 双亲委派 ★ — 讲03

```
  加载 ─▶ 链接[ 验证 ─▶ 准备 ─▶ 解析 ] ─▶ 初始化
   读字节码  校验安全  静态变量  符号引用→   执行<clinit>
                     赋"零值"   直接引用    (静态块/静态变量真正赋值)

  双亲委派（加载请求逐级上抛，父加载不了才自己加载）：
     自定义加载器 ──▶ AppClassLoader(应用) ──▶ 扩展/平台 ──▶ Bootstrap(启动)
                                                              │找不到才往下退回
     ◀────────────────────────────────────────────────────────┘
```
**记忆点**：准备阶段给 `static int a` 赋默认值 **0**，初始化阶段才赋你写的 **123**。双亲委派目的：①安全（防止用户自定义 `java.lang.String` 覆盖核心类）②避免类被重复加载。打破它的场景：SPI(线程上下文加载器)、Tomcat、OSGi、热部署。

### 图4 · GC 分代与对象晋升 ★（面试重中之重）— 讲11/12

```
  new 对象 ──▶ Eden 区分配
                 │ Eden 满 → 触发 Minor GC（复制算法）
                 ▼
        存活对象复制到 Survivor(S0/S1)，年龄+1
                 │ 每熬过一次 Minor GC 年龄+1
                 ▼ 年龄≥阈值(默认15) 或 Survivor放不下(大对象)
             晋升到老年代 Old Gen
                 │ 老年代满 → Full GC（标记-整理/清除，慢，STW长）
                 ▼
              回收或 OOM
```
**记忆点**：新生代用**复制算法**（Eden:S0:S1 = 8:1:1，只损失 10% 空间），老年代对象存活率高、用**标记-整理/标记-清除**。**Minor GC**（只收新生代，频繁但快）；**Full GC**（收整个堆含老年代，慢、STW 长，要尽量避免）。大对象直接进老年代。

### 图5 · 三种 GC 基础算法 — 讲11

```
 标记-清除 Mark-Sweep：  标记存活 → 清除死对象   → 快，但产生内存碎片
 标记-整理 Mark-Compact：标记存活 → 存活对象挪到一端 → 无碎片，但移动开销大
 复制算法   Copy：       存活对象复制到另一半空间 → 无碎片,但浪费一半空间(用于新生代)
```
**记忆点**：三选一看场景——新生代对象死得多用**复制**，老年代对象活得久用**标记-整理/清除**。这也是分代收集思想的由来（不同代用不同算法）。

### 图6 · 可达性分析（判断对象死活）— 讲12

```
   GC Roots（栈局部变量 / 静态字段 / 常量 / JNI引用 ...）
     │  ├──▶ 对象A ──▶ 对象B      （可达 = 存活）
     │  └──▶ 对象C
     ✗       对象D ⇄ 对象E         （互相引用但从 Roots 不可达 = 死亡,可回收）
```
**记忆点**：Java 用**可达性分析**不用**引用计数**——因为引用计数解决不了 **循环引用**（D、E 互引会导致计数永不为 0，内存泄漏）。从 GC Roots 出发遍历，到不了的就是垃圾。

### 图7 · synchronized 锁升级 ★（并发高频）— 讲14

```
  无锁 ──▶ 偏向锁 ──▶ 轻量级锁 ──▶ 重量级锁
          只1个线程   少量线程交替   多线程激烈竞争
          记录线程ID   CAS自旋抢锁    走OS monitor,线程阻塞挂起
          (几乎无开销)  (不阻塞,自旋)  (有内核态切换开销)
        —— 只能升级，不能降级 ——
```
**记忆点**：锁信息存在对象头的 **Mark Word**（锁标志位）。竞争越激烈锁级别越高、开销越大。偏向锁赌"只有一个线程"，轻量级锁赌"竞争很短可自旋等"，都赌输了才膨胀成重量级锁（挂起线程）。

### 图8 · 即时编译：解释 + 分层编译 — 讲16/17

```
   字节码 ──▶ 解释执行（启动快，逐条翻译，慢）
                │  方法调用计数器 + 回边计数器 → 探测"热点代码"
                ▼
          C1(client)编译：快，做基础优化
                │  更热
                ▼
          C2(server)编译：慢，做激进深度优化(内联/逃逸分析...)
                │  激进优化的假设不成立时
                ▼
          去优化 Deoptimization：退回解释执行
```
**记忆点**：解释器保证**快速启动**，JIT 保证**峰值性能**，分层编译兼顾两者。热点探测靠**计数器**（方法调用计数 + 循环回边计数）。C2 做的激进优化（如假设某方法不被重写）一旦被打破，就**去优化**退回解释器重新来。

### 图9 · 逃逸分析衍生的 3 大优化 — 讲23

```
   逃逸分析：判断一个对象会不会"逃出"当前方法/线程
     │
     ├─ 不逃逸 → 栈上分配：对象直接分配在栈帧，随方法返回自动销毁，免 GC
     ├─ 不逃逸 → 标量替换：把对象拆成一个个基本类型字段，压根不创建对象
     └─ 不逃逸出线程 → 锁消除：这个锁对象只有当前线程用，直接把锁去掉
```
**记忆点**：面试常问"对象一定分配在堆上吗？"——**不一定**！逃逸分析后，不逃逸的对象可能栈上分配甚至标量替换（不创建对象）。锁消除解释了为什么 `StringBuffer` 局部使用时 synchronized 开销可被优化掉。

### 图10 · 对象内存布局 — 讲10

```
  ┌──────────── 对象头 Header ────────────┐┌── 实例数据 ──┐┌─ 对齐填充 ─┐
  │  Mark Word         │  Klass Pointer    ││ 各字段值      ││ 补齐到8字节 │
  │  (哈希/GC年龄/锁标志) │  (指向类元信息)    ││(可能重排序压缩)│           │
  └────────────────────┴───────────────────┘└──────────────┘└───────────┘
```
**记忆点**：对象头两部分——**Mark Word**（存哈希码、GC 分代年龄、锁状态标志，锁升级就改这里）+ **类型指针 Klass Pointer**（指向方法区的类元信息）。开启**压缩指针(compressed oops)** 可把 64 位指针压成 32 位省内存。字段会被 JVM 重排以节省空间、满足对齐。

## 四、逐讲精要（模块一：JVM 基本原理 01-12）

## 01 | Java 代码是怎么运行的？

**一句话核心**：Java 源码先被 javac 编译成平台无关的字节码（.class），再由 JVM 加载后解释执行 + 即时编译（JIT）翻译成机器码运行，从而实现"一次编写，到处运行"。

**关键知识点**
- 运行流程：`.java` --javac--> `.class` 字节码 --类加载--> 方法区 --解释执行 / JIT--> 机器码。字节码指令的操作码（opcode）固定为一个字节，故称"字节码"。
- 为什么用虚拟机：① 可移植性（字节码可在各平台的 JVM 实现上运行）；② 提供托管环境（Managed Runtime），自动内存管理与 GC、数组越界/动态类型/安全权限等动态检测。
- JRE = JVM + Java 核心类库（运行必需）；JDK = JRE + 开发诊断工具。C++ 直接编译成机器码，无需运行时。
- JVM 运行时数据区分五部分：**方法区**（存放加载后的类）、**堆**（对象）、**PC 寄存器**（各线程执行位置）、**Java 方法栈**（Java 方法）、**本地方法栈**（native/C++ 方法）。方法区与堆线程共享，PC、两种栈线程私有。
- 每调用一个 Java 方法就在当前线程 Java 方法栈中压入一个**栈帧**（存局部变量 + 字节码操作数），栈帧大小提前算好、内存不要求连续；方法返回（正常或异常）即弹出并丢弃当前栈帧。
- 字节码不能直接执行，需翻译成机器码，HotSpot 有两种方式：**解释执行**（逐条翻译，无需等待编译，省内存）与**即时编译 JIT**（整个方法编译后再执行，运行更快、吃内存）。HotSpot 默认**混合模式**：先解释执行，把反复执行的热点代码以方法为单位 JIT。
- JIT 基于二八定律（20% 代码占 80% 计算）。JIT 有运行时信息，理论上可优化到超过 C++（如根据虚方法实际只调用一个目标来规避开销）。
- HotSpot 内置多个 JIT：**C1（Client，编译快、优化简单、面向启动性能）**、**C2（Server，编译慢、优化复杂、面向峰值性能）**、Graal（Java 10 实验性）。Java 7 起默认**分层编译**：热点先 C1 编译，其中的热点再 C2 编译。编译在独立编译线程中进行，C1:C2 约 1:2。

**面试高频问答**
- **Q：** 简述一段 Java 代码从编写到运行的完整流程。
  **A：** 源码 `.java` 经 javac 编译成字节码 `.class`；JVM 通过类加载器把类加载到方法区；执行时对字节码解释执行，同时把热点方法 JIT 编译成机器码直接跑在硬件上。
- **Q：** JVM 运行时数据区分哪几块？哪些线程共享、哪些线程私有？
  **A：** 方法区、堆、PC 寄存器、Java 方法栈、本地方法栈。方法区和堆线程共享；PC 寄存器、Java 方法栈、本地方法栈线程私有。
- **Q：** 解释执行和即时编译的区别？为什么要混合？
  **A：** 解释执行逐条翻译字节码、无需等待、不占额外内存；JIT 把整个方法编译成机器码、运行快但占内存并有编译开销。混合模式对占大头的冷代码解释执行省内存，对少量热点代码 JIT 提速，兼顾启动和峰值性能。
- **Q：** C1 和 C2 编译器有什么区别？
  **A：** C1（Client）优化简单、编译时间短，面向启动性能；C2（Server）优化复杂、编译时间长但生成代码执行效率高，面向峰值性能。Java 7 起默认分层编译，热点先 C1 再 C2。

## 02 | Java 的基本类型

**一句话核心**：JVM 中 boolean 被映射为 int（true=1、false=0），Java 共 8 种基本类型，它们在栈（局部变量区）与堆（字段/数组）中占用大小不同，存取时伴随掩码与扩展操作。

**关键知识点**
- Java 引入 8 个基本类型是出于工程考虑：执行效率和内存使用两方面提升性能（对比 Smalltalk 万物皆对象）。
- **boolean 在 JVM 中被映射为 int**：true→1、false→0。逻辑运算和条件跳转都用整数字节码实现：`if(flag)` 编译成 `ifeq`（值为 0 则跳转），`if(flag==true)` 编译成 `if_icmpne`（与 1 不等则跳转）。所以把 boolean 改成 2：`if(flag)` 成立（2≠0），`if(flag==true)` 不成立（2≠1）。
- 8 种基本类型：整数型 byte / short / char / int / long，浮点型 float / double，外加 boolean。值域 byte<short<int<long<float<double 依次扩大且被包含，故小到大无需强转；默认值在内存中都是 0。
- **boolean 和 char 是唯二的无符号类型**。char 取值 [0, 65535]，通常为非负，适合做数组索引；boolean 取值 0 或 1。
- 浮点采用 IEEE 754：有 +0.0F 和 -0.0F（内存不同但 `==` 为真）；正无穷 0x7F800000、负无穷 0xFF800000；NaN 有一大片编码，标准 NaN 为 0x7FC00000。**NaN 特性：除 `!=` 恒返回 true 外，其余比较（含 `==`、`<`、`>=`）全返回 false**。
- **栈上大小（解释栈帧局部变量区）**：局部变量区是数组，用正整数索引。long、double 占两个单元，其余基本类型和引用类型都占一个单元。即 boolean/byte/char/short 在栈上和 int 一样大——32 位 HotSpot 占 4 字节，64 位占 8 字节。
- **堆上大小（字段/数组元素）**：与值域吻合——byte 1 字节，char/short 2 字节。boolean 字段占 1 字节，boolean 数组直接用 byte 数组实现。
- **存储时掩码**：把 int 存入 byte/char/short 字段是隐式截断（掩码）；boolean 存储时 HotSpot 显式掩码，只取最后一位。**加载时扩展**：boolean/char 无符号→零扩展（高位补 0）；byte/short 有符号→符号扩展（高位补符号位）。

**面试高频问答**
- **Q：** JVM 是如何处理 boolean 类型的？
  **A：** JVM 规范把 boolean 映射为 int，true=1、false=0。所有逻辑运算和条件跳转都用整数字节码实现。boolean 字段/数组存储时会做掩码只取最低位；boolean 数组底层用 byte 数组。
- **Q：** Java 有哪 8 种基本类型？char 有什么特别？
  **A：** byte、short、int、long、float、double、char、boolean。char 是无符号类型，取值 [0,65535]，通常非负，常用作数组索引；boolean 和 char 是仅有的两个无符号类型。
- **Q：** 各基本类型占多少字节？为什么栈上和堆上不一样？
  **A：** 堆中按值域：byte 1、char/short 2、int/float 4、long/double 8 字节，boolean 1 字节。栈中（局部变量区）除 long/double 占两个单元外，其余（含 boolean/byte/char/short）都和 int 等大（32 位 4 字节、64 位 8 字节），因为变长数组不好按下标定位地址，索性统一浪费空间换取访问速度。
- **Q：** 浮点数比较要注意什么？
  **A：** 注意 +0.0F==-0.0F 为真但内存表示不同；尤其 NaN，任何与 NaN 的比较除 `!=` 返回 true 外都返回 false，`NaN==NaN` 也是 false，需用 `Float.isNaN` 判断。

## 03 | JVM 是如何加载 Java 类的？

**一句话核心**：类从 class 文件到可用的类要经过加载、链接（验证/准备/解析）、初始化三大步；加载由类加载器完成并遵循双亲委派模型，类的唯一性由"类加载器实例 + 类全名"共同决定。

**关键知识点**
- 引用类型分类、接口、数组类、泛型参数；泛型编译期被擦除，数组类由 JVM 直接生成，只有类和接口需从字节流加载。字节流可来自 class 文件、程序内部生成或网络。
- **加载**：查找字节流并据此创建类。除数组类外都需类加载器。加载后在方法区（元空间）生成 class 结构（标记为未链接暂不可用）。
- **类加载器 + 双亲委派**：收到加载请求先转发给父加载器，父加载器找不到时自己才尝试加载。层级（Java 9 前）：
  - **启动类加载器（Bootstrap）**：C++ 实现，无对应 Java 对象（Java 中用 null 指代），加载 JRE/lib 下核心类（及 -Xbootclasspath）。
  - **扩展类加载器（Extension）**：父为启动类加载器，加载 JRE/lib/ext（及 java.ext.dirs）。Java 9 改名为**平台类加载器（Platform）**。
  - **应用类加载器（Application）**：父为扩展类加载器，加载应用类路径（-cp/-classpath、CLASSPATH）。默认加载应用程序的类。
- 还可自定义类加载器（如加密 class 加载时解密）。类加载器提供命名空间：**类的唯一性 = 类加载器实例 + 类全名**，同一字节流经不同加载器加载得到两个不同类（可用于同一类多版本共存）。
- **链接 = 验证 + 准备 + 解析**：
  - **验证**：确保被加载类满足 JVM 约束（javac 生成的类通常天然满足）。
  - **准备**：为静态字段分配内存（此时是零值，真正赋值在初始化阶段）；部分 JVM 此阶段还构造方法表（用于虚方法动态绑定）。
  - **解析**：把符号引用（类名+方法名+描述符等文本指代）解析成实际引用。解析可能触发被指向类的加载（但未必链接/初始化）。解析非必须发生在链接期，规范只要求执行用到符号引用的字节码前完成解析。
- **初始化**：为标记为 ConstantValue 的常量字段赋值、执行 `<clinit>` 方法（编译器把静态字段直接赋值语句和静态代码块汇总而成）。被 final 修饰且为基本类型/字符串的直接赋值静态字段 → 编译器标记为 ConstantValue，由 JVM 直接完成。**JVM 加锁保证 `<clinit>` 只执行一次**（→ 单例延迟初始化线程安全）。
- **触发初始化的情形**：启动时初始化主类；new 目标类；调用静态方法；访问静态字段；子类初始化触发父类初始化；接口有 default 方法时其实现类初始化触发该接口初始化；反射调用；初次调用 MethodHandle 指向方法所在类。（新建数组只加载元素类、不初始化不链接。）

**面试高频问答**
- **Q：** 类加载的完整过程分哪几步？
  **A：** 加载 → 链接（验证、准备、解析）→ 初始化。加载查找字节流建类结构；验证查合规；准备给静态字段分配内存（零值）；解析把符号引用换成实际引用；初始化执行 `<clinit>` 给静态字段真正赋值。
- **Q：** 什么是双亲委派模型？有什么好处？
  **A：** 类加载器收到请求先委派给父加载器，父加载器加载不了才自己加载。层级：启动→扩展(平台)→应用。好处：避免核心类被篡改、保证同一类不被重复加载、保证类型安全。
- **Q：** 判断两个类是否"同一个类"的依据？
  **A：** 类加载器实例 + 类的全限定名，两者都相同才是同一个类。同一字节流被不同加载器加载会得到不同的类，故可实现同一类多版本共存。
- **Q：** 准备阶段和初始化阶段对静态字段的处理有何区别？
  **A：** 准备阶段只分配内存并赋"零值"（如 static int a=5 此时 a=0）；初始化阶段执行 `<clinit>` 才把 a 赋为 5。被 final 修饰的基本类型/字符串常量在准备阶段就被赋为 ConstantValue。
- **Q：** 类的初始化在什么时候触发？如何实现线程安全的懒加载单例？
  **A：** new、访问静态方法/字段、反射、子类初始化、初始化主类等会触发。利用静态内部类持有实例（LazyHolder 模式），只有首次调用 getInstance 才加载并初始化内部类，JVM 保证 `<clinit>` 仅执行一次，天然线程安全。

## 04 | JVM 是如何执行方法调用的？（上）——重载/重写与静态/动态绑定

**一句话核心**：重载在编译期由参数的声明类型决定（静态绑定），重写在运行期由调用者的动态类型决定（动态绑定）；JVM 用 invokestatic/special/virtual/interface/dynamic 五条指令区分调用方式。

**关键知识点**
- **重载（Overload）**：同类中方法名相同、参数类型不同（也可跨父子类）。编译期完成识别。编译器按传入参数的**声明类型**分三阶段选重载方法：① 不考虑自动装拆箱、不考虑变长参数；② 允许装拆箱、不允许变长参数；③ 允许装拆箱且允许变长参数。同阶段多个适配时按形参继承关系选最贴切的（如 null 优先匹配 String 而非 Object）。
- **重写（Override）**：子类定义与父类非私有、非静态、同名同参方法 → 重写（多态核心，按调用者动态类型选目标方法）。若都是静态方法则是**隐藏**（非重写）；参数类型不同则是重载。
- **JVM 识别方法的关键**：类名 + 方法名 + **方法描述符**（参数类型 + 返回类型）。故 JVM 允许同类中名字、参数相同但返回类型不同的方法（Java 语言不允许，可用字节码工具绕开）。JVM 判定重写也基于描述符（参数和返回类型都一致）；Java 层重写但 JVM 层不算重写（返回类型协变、泛型擦除）时，编译器生成**桥接方法**保证重写语义。
- **五条方法调用字节码指令**：
  - `invokestatic`：调用静态方法。
  - `invokespecial`：调用私有实例方法、构造器、super 调父类实例方法/构造器、所实现接口的 default 方法。
  - `invokevirtual`：调用非私有实例方法。
  - `invokeinterface`：调用接口方法。
  - `invokedynamic`：调用动态方法（lambda 等）。
- **静态绑定 vs 动态绑定**：invokestatic、invokespecial 可直接识别目标方法（静态绑定，实际引用是方法指针）；invokevirtual、invokeinterface 通常需运行期按动态类型确定目标（动态绑定，实际引用是方法表索引）。例外：目标方法被 final 标记且唯一时，虚方法调用也可静态绑定。
- 因为重载在编译期已区分，可认为 **JVM 中不存在重载**。但重载方法可能被子类重写，故编译器把所有非私有实例方法调用都编译成需动态绑定的类型。"静态绑定=编译时多态""动态绑定=运行时多态"的说法在 JVM 语境下并不完全准确——准确说静态绑定指解析时即能识别目标方法，动态绑定指需运行时按动态类型识别。
- **符号引用解析**：class 常量池存符号引用（类/接口名 + 方法名 + 描述符），分接口符号引用与非接口符号引用。非接口解析：先在类 C 中找 → 父类直到 Object → C 实现的接口（须非私有非静态）。接口解析：接口 I 中找 → Object 公有实例方法 → I 的超接口。

**面试高频问答**
- **Q：** 重载和重写的区别？分别在什么时候确定？
  **A：** 重载是同名不同参，编译期根据参数**声明类型**确定（静态绑定/编译时多态）；重写是子类同名同参覆盖父类非私有非静态方法，运行期根据调用者**动态类型**确定（动态绑定/运行时多态）。
- **Q：** JVM 的五条方法调用指令分别用于什么？
  **A：** invokestatic→静态方法；invokespecial→私有方法/构造器/super 调用/接口 default；invokevirtual→非私有实例方法；invokeinterface→接口方法；invokedynamic→动态方法（如 lambda）。
- **Q：** 哪些是静态绑定、哪些是动态绑定？
  **A：** invokestatic、invokespecial 是静态绑定（解析时直接确定目标，实际引用为方法指针）；invokevirtual、invokeinterface 是动态绑定（运行时按动态类型查方法表，实际引用为索引）；被 final 标记且唯一的虚方法调用可退化为静态绑定。
- **Q：** JVM 识别方法比 Java 语言多考虑什么？桥接方法是干什么的？
  **A：** JVM 靠类名+方法名+方法描述符（含返回类型），而 Java 语言重载不看返回类型。当 Java 层构成重写但 JVM 层因返回类型协变或泛型擦除不算重写时，编译器生成带 ACC_BRIDGE 的桥接方法来维持重写语义。

## 05 | JVM 是如何执行方法调用的？（下）——虚方法表与内联缓存

**一句话核心**：虚方法调用（invokevirtual/invokeinterface）通过每个类的虚方法表（vtable）以"空间换时间"实现动态绑定，JIT 再用内联缓存和方法内联进一步优化，实际开销并不大。

**关键知识点**
- **虚方法调用**：非私有实例方法编成 invokevirtual、接口方法编成 invokeinterface，二者属虚方法调用，绝大多数需按调用者动态类型动态绑定，比静态绑定更耗时。若目标方法被 final 标记则可静态绑定。
- **方法表（vtable）**：类加载**准备阶段**为每个类构造，本质是数组，每个元素指向当前类及祖先类的一个非私有实例方法（可能是抽象方法）。两大特质：① 子类方法表包含父类方法表的所有方法；② **子类重写方法在表中的索引值与被重写的父类方法索引值相同**。因此可用固定索引跨类型定位目标。
- **动态绑定过程**：解析时记录目标方法在方法表中的索引；运行时取调用者实际类型 → 找其虚方法表 → 按索引取目标方法。相比创建栈帧，这几次内存解引用开销可忽略。invokeinterface 用接口方法表（itable），原理类似但更复杂。
- **内联缓存（inlining cache）**：JIT 加速动态绑定的技术，缓存虚调用中调用者的动态类型及对应目标方法；命中缓存类型则直接调用，未命中退回方法表动态绑定。
- 多态术语：**单态 monomorphic**（1 种状态）、**多态 polymorphic**（有限种，二态 bimorphic 是其一）、**超多态 megamorphic**（超过阈值的更多种）。实践中大部分虚调用是单态，**HotSpot 只用单态内联缓存**以省内存。
- 内联缓存未命中的两种处理：① 替换记录（类似 CPU 数据缓存，要求数据局部性，若两类型交替调用则每次都替换，只有写开销无收益）；② HotSpot 实际做法——**劣化为超多态状态**，放弃优化直接走方法表，省去写缓存开销。
- **内联缓存 ≠ 方法内联**：内联缓存并不内联目标方法体。任何方法调用除非被内联，都有固定开销（保存执行位置、新建/压入/弹出栈帧）。对 getter/setter 这种极简方法，固定开销甚至超过方法本身。**方法内联（method inlining）**才能消除该开销并打开进一步优化空间。

**面试高频问答**
- **Q：** 什么是虚方法表（vtable）？如何用它实现动态绑定？
  **A：** vtable 是类加载准备阶段为每个类构造的数组，每项指向一个非私有实例方法，子类表包含父类全部方法且重写方法索引与父类一致。运行时取调用者实际类型的 vtable，按解析时记录的固定索引取目标方法，即完成动态绑定，只多几次内存解引用，开销极小。
- **Q：** 虚方法调用性能开销大吗？JVM 做了哪些优化？
  **A：** 开销并不大甚至可消除。除 vtable（空间换时间）外，JIT 用内联缓存缓存调用者动态类型与目标方法，命中即直接调用；更进一步用方法内联消除调用固定开销并开放后续优化。
- **Q：** 内联缓存的单态、多态、超多态是什么？HotSpot 怎么处理未命中？
  **A：** 单态只缓存一种类型，多态缓存有限多种，超多态超过阈值。HotSpot 为省内存只用单态内联缓存；当出现新类型未命中时，直接把内联缓存劣化为超多态状态，之后走方法表动态绑定，避免反复写缓存的开销。
- **Q：** 内联缓存和方法内联是一回事吗？
  **A：** 不是。内联缓存只缓存"类型→目标方法"映射，并不把方法体嵌入调用处，方法调用的栈帧固定开销仍在；方法内联才是把被调方法体直接展开到调用点，消除固定开销并带来进一步优化机会。

## 06 | JVM是如何处理异常的？

**一句话核心**：异常处理靠每个方法附带的"异常表"驱动控制流转移，finally 通过编译器复制代码块到所有出口来保证必定执行。

**关键知识点**
- 抛异常分显式（`throw`）和隐式（JVM 遇到无法执行的状态自动抛，如数组越界 ArrayIndexOutOfBoundsException）。
- 异常继承体系：`Throwable` → `Error`（不可恢复，不该捕获）和 `Exception`；`Exception` 下有特殊子类 `RuntimeException`。`Error` + `RuntimeException` = 非检查异常（unchecked）；其余是检查异常（checked），必须显式 catch 或用 throws 声明。
- 构造异常实例很昂贵：因为要生成栈轨迹（stack trace），逐一遍历当前线程的 Java 栈帧记录方法名/类名/文件名/行号。生成时忽略异常构造器和 `fillInStackTrace`，从新建异常处算起。所以不要缓存异常实例复用（栈轨迹会指向 new 的位置而非 throw 的位置）。
- 异常表（exception table）：每个方法一张表，每个条目 = from 指针、to 指针、target 指针、捕获的异常类型。from/to 标示监控范围（try 块），target 指向处理器起点（catch 块），指针值都是字节码索引 bci。
- 捕获流程：触发异常时从上到下遍历异常表，若 bci 在某条目监控范围内且异常类型匹配，就把控制流转移到 target；遍历完仍不匹配则弹出当前栈帧，在 caller 中重复，最坏遍历整个调用栈的所有异常表。
- finally 实现：编译器把 finally 代码块内容复制多份，放到 try-catch 所有正常执行出口 + 异常执行出口。异常路径会额外生成监控整个 try-catch、捕获任意类型（javap 中显示 `any`）的条目，target 指向复制的 finally，末尾重新抛出所捕获的异常。
- 易错点：catch 块中又抛异常，finally 捕获重抛的是后者，原异常被吞掉；finally 里有 return 会吞掉待抛异常。
- Java 7 新特性：Suppressed 异常（一个异常附加到另一个上）、try-with-resources（语法糖，AutoCloseable 自动 close + 自动用 Suppressed 避免原异常丢失）、多异常捕获 `catch (A | B e)`（实现上就是生成多个异常表条目）。

**面试高频问答**
- **Q：** try-catch-finally 中 finally 一定会执行吗？字节码层面怎么实现的？
  **A：** 编译器把 finally 内容复制到每条正常路径和异常路径的出口，所以无论走哪条路径都执行。异常路径由一个捕获 `any` 类型、监控整个 try-catch 的异常表条目负责，末尾重抛异常。若 finally 里有 return/break/continue，会覆盖待返回值或吞掉待抛异常。
- **Q：** 为什么说异常处理性能开销大？
  **A：** 开销主要在构造异常实例时生成栈轨迹，要遍历当前线程所有 Java 栈帧记录调试信息（类名、方法名、行号），很耗时。真正走"抛出-捕获"的异常执行路径也慢，因此不应把异常当正常业务控制流用。
- **Q：** 检查异常和非检查异常的区别？
  **A：** 非检查异常 = Error + RuntimeException，编译器不强制处理；检查异常 = 其余 Exception，必须显式 catch 或 throws 声明。检查异常概念只存在于源码层面，异常表不区分二者。

## 07 | JVM是如何实现反射的？

**一句话核心**：`Method.invoke` 委派给 MethodAccessor，先用本地实现、调用达 15 次后 Inflation 切换为动态生成字节码的实现，反射慢主要在装箱、变长参数数组和无法内联。

**关键知识点**
- `Method.invoke` 实际委派给接口 `MethodAccessor`，它有两个实现：本地实现（NativeMethodAccessorImpl，native 方法）和委派实现（DelegatingMethodAccessorImpl）。调用链：invoke → 委派实现 → 本地实现 → 目标方法。
- 三种实现方式：本地实现（C++）、动态实现（运行时生成字节码 GeneratedMethodAccessor，直接用 invoke 指令调目标方法）。委派实现是中间层，目的是能在本地实现和动态实现之间切换。
- 性能对比：动态实现比本地实现快约 20 倍（省去 Java→C++→Java 切换），但生成字节码耗时，只调一次时本地实现反而快 3~4 倍。
- Inflation 机制：反射调用次数 < 15 用本地实现；达到 15（从 0 数）时动态生成字节码，把委派对象切换成动态实现。阈值可调 `-Dsun.reflect.inflationThreshold=`，可用 `-Dsun.reflect.noInflation=true` 关闭 Inflation 一开始就用动态实现。
- 反射慢的三大原因：① 变长参数方法自动生成 Object 数组；② 基本类型自动装箱/拆箱（Object 数组存不了基本类型）；③ 最重要的——方法内联难，`Method.invoke` 里对 `MethodAccessor.invoke` 的调用是内联瓶颈。
- 优化手段：扩大 Integer 缓存范围（`-Djava.lang.Integer.IntegerCache.high`）、`setAccessible(true)` 关闭权限检查、关闭 Inflation 直用动态实现、循环外缓存 Class/Method（getMethod 返回的是结果拷贝，避免热点用 getMethods）。
- 类型 profile 污染：生产环境多个反射调用对应多个 GeneratedMethodAccessor，JVM 类型 profile 默认只记 2 个类型（`-XX:TypeProfileWidth`），记不下就无法内联、逃逸分析失效，性能骤降。

**面试高频问答**
- **Q：** 反射为什么慢？如何优化？
  **A：** 三个原因：变长参数生成 Object 数组、基本类型装箱拆箱、方法内联受阻（最关键）。优化：缓存 Class/Method 对象、`setAccessible(true)` 关权限检查、循环外复用参数、扩大 Integer 缓存、必要时关 Inflation。
- **Q：** 什么是 Inflation（膨胀）机制？
  **A：** 反射默认先用 native 本地实现，因为多数反射只调一次，本地实现启动快。当某 Method 调用次数达 15 次，JVM 动态生成字节码（GeneratedMethodAccessor）切换到动态实现，之后调用快约 20 倍。这个切换过程叫 Inflation。
- **Q：** 反射调用经过哪些层？
  **A：** Method.invoke → DelegatingMethodAccessorImpl（委派实现）→ NativeMethodAccessorImpl（本地实现）或 GeneratedMethodAccessorN（动态实现）→ 目标方法。委派层是为了支持本地/动态实现的切换。

## 08 | JVM怎么实现invokedynamic（上）：方法句柄

**一句话核心**：方法句柄 MethodHandle 是强类型、可直接执行的方法引用，只关心参数和返回类型（MethodType），权限检查在创建时完成，是 invokedynamic 的基石。

**关键知识点**
- 背景：Java 方法调用编译为 invokestatic/invokespecial/invokevirtual/invokeinterface 四条指令，都要求提供目标方法类名，难以实现动态语言的"鸭子类型"调用。Java 7 引入 invokedynamic 和方法句柄来解决。
- 方法句柄（MethodHandle）：强类型的、能被直接执行的引用，可指向静态/实例方法、构造器或字段（指向字段时等价于虚构的 getter/setter）。
- 方法句柄类型 MethodType：由参数类型 + 返回类型组成，是判断句柄是否适配的唯一关键，不关心类名和方法名。
- 创建：通过 `MethodHandles.Lookup`。findStatic（对应 invokestatic）、findVirtual（invokevirtual/invokeinterface）、findSpecial（invokespecial）、unreflect（从 Method 转）。调用绑定方式与原指令一致（virtual 动态绑定、special 静态绑定）。
- 权限检查在句柄创建阶段完成，调用时不再检查（比反射省去重复权限检查开销）；权限取决于 Lookup 对象的创建位置，不取决于句柄创建位置。
- 两种调用：`invokeExact`（参数类型严格匹配，不匹配运行时抛异常，需显式强转）；`invoke`（自动适配，内部调 `asType` 生成适配器句柄）。二者是签名多态方法（`@PolymorphicSignature`），编译器按传入参数的声明类型生成方法描述符。
- 增删改参数：改=asType；删=dropArguments（丢弃部分参数）；增=bindTo（插入参数，可实现柯里化，Java 8 捕获型 Lambda 也靠它）。
- 底层实现：invokeExact/invokeBasic/linkToStatic 都被 JVM 特殊处理，通过 LambdaForm 适配器，最终经 MemberName（存方法地址/方法表索引）跳转目标方法。调用次数超阈值（默认 127，`CUSTOMIZE_THRESHOLD`）会生成定制适配器优化。
- 方法句柄和反射一样是间接调用、面临无法内联问题；但其内联瓶颈在于即时编译器能否把句柄识别为常量（如赋给 static final 字段）。

**面试高频问答**
- **Q：** 方法句柄和反射有什么区别？
  **A：** ① 方法句柄权限检查在创建时一次完成，调用不再检查，反射每次调用都检查（除非 setAccessible）；② 方法句柄只关心 MethodType（参数+返回类型），不关心类名方法名；③ 二者都是间接调用、都可能无法内联，但方法句柄能通过被识别为常量（static final）来内联优化。
- **Q：** invokeExact 和 invoke 的区别？
  **A：** invokeExact 要求实参类型与句柄描述符严格匹配，不匹配运行时抛异常（传 String 给 Object 参数需显式 `(Object)` 强转）；invoke 会自动通过 asType 适配参数类型再调用。两者都是签名多态（@PolymorphicSignature）方法。
- **Q：** 什么是 MethodType？
  **A：** 方法句柄的类型，由参数类型和返回类型构成，是判断两个方法句柄是否适配的唯一依据，与方法所在类和方法名无关。

## 09 | JVM怎么实现invokedynamic（下）：调用点与Lambda

**一句话核心**：invokedynamic 把调用点 CallSite 抽象成 Java 对象，首次执行时用启动方法（Bootstrap Method）生成并绑定调用点；Lambda 就是靠它在运行时生成实现函数式接口的适配器类。

**关键知识点**
- invokedynamic（Java 7 引入，支持动态语言）：把调用点 CallSite 抽象成 Java 类，将方法调用/链接暴露给应用程序。每条指令捆绑一个调用点，调用其链接的方法句柄。
- 启动方法（Bootstrap Method）：首次执行 invokedynamic 时调用，生成调用点并绑定；之后直接调已绑定调用点的句柄。启动方法用方法句柄指定，返回 CallSite，必须接收三个固定参数：Lookup 实例、目标方法名字符串、调用点能链接的方法句柄类型 MethodType（还可接收额外参数）。
- 调用点类型：ConstantCallSite（不可更改链接对象）；MutableCallSite、VolatileCallSite（可重链接，区别类似普通字段 vs volatile 字段）；可自定义。
- Lambda 实现：Java 8 Lambda 借助 invokedynamic 生成实现函数式接口（仅一个非 default 方法的接口，通常 @FunctionalInterface）的适配器。编译期把 Lambda 体解语法糖（desugar）成一个方法（参数含 Lambda 参数 + 捕获的变量）；方法引用（如 `Horse::race`）不额外生成方法。
- 首次执行 invokedynamic，启动方法用 ASM 生成实现函数式接口的适配器类，返回 ConstantCallSite（链接返回适配器实例的句柄）。
- 捕获 vs 非捕获 Lambda：非捕获（上下文无关）→ 始终返回同一个适配器实例（单例）；捕获变量的 Lambda → 每次执行 invokedynamic 都新建适配器实例（保存捕获变量为实例字段，多一个 `get$Lambda` 静态方法）以保证线程安全和变量正确。
- 性能：捕获型和非捕获型 Lambda 性能上限都能达到直接调用——非捕获靠内联优化为空操作，捕获型靠即时编译器的逃逸分析消除新建实例（关闭逃逸分析 `-XX:-DoEscapeAnalysis` 后约慢 2.5 倍）。逃逸分析生效需句柄能内联 + accept 调用能内联，因此应尽量用非捕获 Lambda。
- 方法句柄要内联，需即时编译器识别为常量：赋给 static final 字段即可达到与直接调用一致的性能（局部变量句柄约慢 3.5 倍无法内联）。

**面试高频问答**
- **Q：** Java 8 的 Lambda 表达式是怎么实现的？
  **A：** 编译期把 Lambda 体解语法糖成一个私有方法，调用处生成 invokedynamic 指令；首次运行时其启动方法用 ASM 动态生成一个实现对应函数式接口的适配器类，返回 ConstantCallSite。非捕获 Lambda 复用单个适配器实例，捕获变量的 Lambda 每次新建实例（靠逃逸分析消除开销）。注意不是匿名内部类那种编译期生成 class。
- **Q：** invokedynamic 的执行流程？
  **A：** 首次执行时调用其绑定的启动方法（Bootstrap Method，接收 Lookup、方法名、MethodType 三个固定参数），生成一个 CallSite 调用点并绑定到该指令；之后再执行直接调用已绑定调用点链接的方法句柄。
- **Q：** 捕获变量的 Lambda 为什么可能有性能开销？如何避免？
  **A：** 捕获变量的 Lambda 每次执行 invokedynamic 都要新建适配器实例保存捕获值。通常即时编译器的逃逸分析会把该实例优化掉（栈上分配/标量替换），但需要句柄和 accept 调用都能内联才生效；否则会持续新建实例。建议尽量使用非捕获（无捕获）的 Lambda。

## 10 | Java对象的内存布局

**一句话核心**：对象 = 对象头（标记字段 markword + 类型指针）+ 实例数据 + 对齐填充；64 位 JVM 用压缩指针把指针压到 32 位省内存，并通过字段重排列实现内存对齐。

**关键知识点**
- 创建对象方式：new（invokespecial 调构造器初始化）、反射（调构造器）、Object.clone、反序列化（后两者直接复制已有数据）、Unsafe.allocateInstance（不初始化实例字段）。
- 构造器约束：无构造器时编译器自动加无参构造器；子类构造器必调父类构造器（父类有无参则隐式，否则须显式 super/this 且作为第一条语句），链式调用直到 Object。因此对象内存涵盖所有父类的实例字段（即使私有/被隐藏也分配内存）。
- 对象头（object header）= 标记字段（mark word，存哈希码、GC 信息、锁信息等运行数据）+ 类型指针（指向该对象的类）。64 位下各占 64 位 = 16 字节额外开销（Integer 只有 4 字节数据，额外开销至少 400%，这也是引入基本类型的原因之一）。
- 压缩指针（`-XX:+UseCompressedOops`，默认开）：把 64 位对象指针压成 32 位，对象头从 16 字节降到 12 字节（标记字段不压缩，仍 8 字节；类型指针压到 4 字节）。也作用于引用类型字段和引用数组。
- 压缩原理：类比"每辆车占两个车位、指针存车号而非车位号"，解引用时左移 3 位（×8）再加固定偏移；配合内存对齐（默认对齐 8 字节）可寻址 2^35 = 32GB；超 32GB 则关闭压缩指针。`-XX:ObjectAlignmentInBytes` 可调对齐进一步扩大寻址但可能增加填充。
- 内存对齐：对象起始地址对齐至 8 的倍数，不足部分为对象间填充（padding）。字段间也对齐（long/double、非压缩引用字段地址须为 8 的倍数），原因是让字段落在同一 CPU 缓存行内，避免跨缓存行读写。
- 字段重排列（`-XX:FieldsAllocationStyle`，默认 1）两规则：① 占 C 字节的字段其偏移量对齐到 C 的倍数（NC）；② 子类继承字段偏移量与父类保持一致（压缩指针下子类第一个字段对齐 4N，非压缩对齐 8N）。会把小字段填进大字段对齐留下的缺口。
- @Contended（Java 8）：解决字段间的虚共享（false sharing）——两线程访问同对象不同 volatile 字段若落在同一缓存行会造成实质共享。让被注解字段独占缓存行（`-XX:-RestrictContended` 生效），代价是浪费空间。

**面试高频问答**
- **Q：** Java 对象的内存布局是怎样的？
  **A：** 三部分：① 对象头 = 标记字段（mark word，存哈希码/GC 分代年龄/锁状态）+ 类型指针（指向类元数据）；② 实例数据（本类及所有父类的实例字段）；③ 对齐填充（补齐到 8 字节倍数）。64 位默认开压缩指针时对象头 12 字节。
- **Q：** 什么是压缩指针？原理和限制？
  **A：** 把 64 位对象指针压缩到 32 位以省内存（对象头从 16→12 字节）。原理是指针存"对象编号"而非字节地址，解引用时左移 3 位再加偏移；配合 8 字节对齐可寻址 32GB。堆超过 32GB 时自动失效。只压类型指针、引用字段、引用数组，不压标记字段。
- **Q：** 为什么要字段重排列/内存对齐？
  **A：** 为满足内存对齐：占 C 字节字段偏移对齐到 C 的倍数，把小字段填入大字段对齐产生的空隙以节省空间。对齐能保证字段落在同一 CPU 缓存行，避免跨缓存行读写降低效率。@Contended 则反向让字段独占缓存行以避免虚共享。

## 11 | 垃圾回收（上）：可达性分析与回收算法

**一句话核心**：主流 JVM 用可达性分析（从 GC Roots 出发标记存活对象）判定存活，靠安全点实现 Stop-the-world，回收方式有清除/压缩/复制三种。

**关键知识点**
- 引用计数法：每个对象记引用数，为 0 即死。缺点：额外空间和更新开销、无法处理循环引用（互相引用的死对象计数不为 0 导致内存泄露）。
- 可达性分析（主流）：以 GC Roots 为初始存活集，从中出发标记（mark）所有可达对象，未被标记的即死亡可回收。能解决循环引用。
- GC Roots（大致是"由堆外指向堆内的引用"）包括：Java 方法栈帧中的局部变量、已加载类的静态变量、JNI handles、已启动未停止的 Java 线程。
- 并发问题：多线程下其他线程更新已访问对象的引用会造成误报（引用置 null，只损失回收机会，无害）或漏报（引用指向未访问对象，可能回收仍被引用的对象导致 JVM 崩溃，危险）。
- Stop-the-world（STW）：暂停所有非 GC 线程直到回收完成，造成 GC 暂停（GC pause）。通过安全点（safepoint）机制实现——等所有线程到达安全点，堆栈稳定不变，GC 才能安全做可达性分析。
- 安全点检测（safepoint poll）插入位置：解释执行——每条字节码之间；即时编译代码——方法出口、非计数循环的循环回边（back-edge）。不是每条机器码都插，因为①检测有开销（HotSpot 简化为一次内存访问，有请求时把该页设为不可读用 segfault 挂起线程）；②要记录哪些寄存器/栈内存存着对象引用以便枚举 GC Roots，占空间。计数循环（counted loop）默认不插安全点，可能造成长暂停。
- 线程状态与安全点：JNI 本地代码（不访问 Java 对象则整段可作为一个安全点，仅在 JNI API 入口检测）、阻塞线程本身处于安全点；解释执行、执行机器码是运行状态需保证可预见时间内进入安全点。
- 三种基础回收方式：① 清除（sweep）——标记为空闲记入 free list，缺点内存碎片 + 分配效率低（需遍历空闲列表）；② 压缩（compact）——存活对象聚到起始位置，消碎片但有压缩性能开销；③ 复制（copy）——内存分两半 from/to，存活对象复制到 to 再交换指针，消碎片但堆利用率仅一半。

**面试高频问答**
- **Q：** 如何判断一个对象是否可以被回收？引用计数和可达性分析的区别？
  **A：** 主流用可达性分析：从 GC Roots 出发标记所有可达对象，未标记的可回收。引用计数为每个对象记引用数为 0 即回收，但无法处理循环引用（互相引用的死对象计数不为 0），可达性分析能解决。
- **Q：** GC Roots 有哪些？
  **A：** 大致是由堆外指向堆内的引用：Java 方法栈帧中的局部变量、已加载类的静态变量、JNI handles、已启动且未停止的 Java 线程。
- **Q：** 什么是 Stop-the-world 和安全点（safepoint）？
  **A：** STW 是 GC 时暂停所有非 GC 线程，保证可达性分析期间堆栈不变；通过安全点实现——线程只在到达安全点后才停下，此时堆栈状态稳定、能准确枚举 GC Roots。安全点检测插在字节码间、方法出口、循环回边等位置，兼顾开销与暂停时间。
- **Q：** 有哪几种基础的垃圾回收算法？各自优缺点？
  **A：** 清除（sweep）——简单但产生碎片、分配慢；压缩（compact）——无碎片但有移动开销；复制（copy）——无碎片、分配快但浪费一半堆空间。现代收集器综合使用。

## 12 | 垃圾回收（下）：分代、Minor GC、卡表与收集器

**一句话核心**：基于"弱分代假设"把堆分新生代（Eden+两 Survivor）和老年代，新生代用标记-复制做 Minor GC，靠卡表 + 写屏障避免全堆扫描跨代引用。

**关键知识点**
- 分代假设：大部分对象存活时间很短，少部分存活很久。据此分新生代（存新对象，频繁用耗时短的算法）和老年代（存活久的，回收即全堆扫描、不计成本）。
- 堆划分：新生代 = Eden + 两个大小相同的 Survivor（from/to，其中一个恒为空）；老年代。默认动态调整 Eden/Survivor 比例（`-XX:+UsePSAdaptiveSurvivorSizePolicy`），也可用 `-XX:SurvivorRatio` 固定。
- TLAB（Thread Local Allocation Buffer，`-XX:+UseTLAB` 默认开）：每个线程私有一段连续内存（如 2048 字节），new 时用指针加法（bump the pointer）分配，无需同步；TLAB 用完再加锁申请新的。解决多线程在共享堆上分配的竞争。
- Minor GC：Eden 耗尽触发，只回收新生代，用标记-复制算法。Eden + from Survivor 的存活对象复制到 to Survivor，然后交换 from/to（保证 to 恒空）。
- 晋升（promote）到老年代条件：① 对象被复制次数达 15（`-XX:MaxTenuringThreshold`，对象头分代年龄只有 4 位故上限 15）；② 单个 Survivor 被占用超 50%（`-XX:TargetSurvivorRatio`）时较高年龄对象也提前晋升。
- 跨代引用问题：Minor GC 只扫新生代，但老年代对象可能引用新生代对象，须把这类引用也作为 GC Roots，否则要全堆扫描。
- 卡表（Card Table）：把堆分成一个个 512 字节的卡，卡表存每张卡一个标识位，标记该卡是否可能存有指向新生代的引用（可能则为"脏卡"）。Minor GC 时只扫脏卡中的对象加入 GC Roots，扫完清零标识位，避免全扫老年代。
- 写屏障（write barrier）：即时编译代码中，引用型字段写操作后注入 `CARD_TABLE[address >> 9] = DIRTY`（右移 9 位=除以 512 映射到卡索引）标记脏卡。宁可错杀不放过，不判断是否真指向新生代。带来虚共享问题（64 字节缓存行覆盖 64 张卡=32KB），`-XX:+UseCondCardMark` 先判断再写以减少写操作。
- 收集器（附录）：
  - 新生代（标记-复制）：Serial（单线程）、Parallel New（Serial 多线程版）、Parallel Scavenge（注重吞吐率，不能与 CMS 搭配）。
  - 老年代：Serial Old、Parallel Old（均标记-压缩，前者单线程后者多线程）、CMS（标记-清除、并发，除少数操作外可与应用并发，并发失败时回退到压缩型收集器；Java 9 已废弃）。
  - G1（Garbage First）：横跨新生代和老年代，把堆分成极多小区域（region），每个可充当 Eden/Survivor/老年代；标记-压缩、可并发；优先回收死亡对象最多的区域（名字由来）。
  - ZGC（Java 11 引入）：宣称暂停时间不超过 10ms。

**面试高频问答**
- **Q：** 为什么要分代回收？新生代和老年代怎么划分？
  **A：** 基于分代假设——多数对象朝生夕灭、少数长期存活。新生代频繁用耗时短的标记-复制快速回收大量短命对象；老年代对象存活率高，用标记-压缩/清除、回收频率低。新生代 = Eden + 两个 Survivor（一个恒空），老年代存长寿对象。
- **Q：** Minor GC 的过程？对象何时晋升老年代？
  **A：** Eden 满触发 Minor GC，用标记-复制：把 Eden 和 from Survivor 的存活对象复制到 to Survivor，交换 from/to。对象复制次数达 15 次晋升老年代；或单个 Survivor 占用超 50% 时较高年龄对象提前晋升。
- **Q：** 什么是卡表（Card Table）？解决什么问题？
  **A：** Minor GC 只扫新生代，但老年代可能引用新生代对象。卡表把堆按 512 字节分卡，标记每张卡是否含指向新生代的引用（脏卡），Minor GC 只把脏卡对象加入 GC Roots，避免全扫老年代。脏卡靠写屏障在引用写操作时设置。
- **Q：** 什么是写屏障？和卡表什么关系？
  **A：** 写屏障是引用型字段写操作后注入的一小段代码 `CARD_TABLE[addr>>9]=DIRTY`，把对应卡标记为脏。它保证所有可能产生跨代引用的卡都被标脏，供 Minor GC 扫描。为简洁不判断是否真指向新生代（宁可错杀）。高并发下会有卡表虚共享，可用 `-XX:+UseCondCardMark` 缓解。
- **Q：** 常见的垃圾回收器有哪些？各有什么特点？
  **A：** 新生代（标记-复制）：Serial（单线程）、Parallel New（多线程）、Parallel Scavenge（重吞吐、不能配 CMS）；老年代：Serial Old、Parallel Old（标记-压缩）、CMS（标记-清除、并发低停顿，Java 9 废弃）；G1 横跨两代、按 region 划分、优先回收垃圾最多区域、可并发；ZGC（Java 11）超低停顿(<10ms)。
- **Q：** TLAB 是什么，解决什么问题？
  **A：** Thread Local Allocation Buffer，每个线程私有的一段连续堆内存。堆是共享的，直接分配需同步；TLAB 让线程在自己的缓冲区内用指针加法无锁分配对象，用完再加锁申请新的，减少分配时的竞争。

## 13 | Java 内存模型（JMM）

**一句话核心**：JMM 通过一系列 happens-before 关系约束重排序，为多线程提供内存可见性保证；底层由内存屏障实现，volatile/锁/final 是其关键应用。

**关键知识点**
- 重排序的三个来源：即时编译器的重排序、处理器的乱序执行、内存系统（缓存/写缓存）的重排序。
- as-if-serial：编译器和处理器必须保证单线程执行结果与顺序执行一致；有数据依赖的操作不能重排序。多线程下无 happens-before 约束时会出现数据竞争（如经典例子中出现看似不可能的 (r1,r2)=(1,2)）。
- happens-before：描述两操作的内存可见性——若 X happens-before Y，则 X 的结果对 Y 可见。注意它不等于"时间先执行"，只要 Y 不依赖 X 的结果，二者仍可能被重排序。
- happens-before 规则：①程序顺序（同线程内前面字节码 hb 后面）；②解锁 hb 之后对同一把锁的加锁；③volatile 写 hb 之后对同一字段的读；④Thread.start() hb 线程首个操作；⑤线程最后操作 hb 其终止（join/isAlive）；⑥中断操作 hb 被中断线程收到中断；⑦构造器最后操作 hb 析构器首操作；⑧传递性。
- 底层实现：JMM 靠内存屏障（读读/读写/写读/写写）禁止重排序。编译器针对每个 hb 关系插入屏障，再按体系架构替换为 CPU 指令。X86_64 上只有"写读屏障"需要真实指令（HotSpot 用 `lock add DWORD PTR [rsp],0x0`），其余三种是 no-op。
- volatile 语义：可见性 + 禁止重排序（写前的访问不能重排到写后，读后的访问不能重排到读前）；不保证原子性；每次访问直接读写内存（不能分配到寄存器）；X86 上只有写操作强制刷新缓存，故应"多读少写、单线程写"。
- final 字段：编译器在 final 写之后插入写写屏障，防止对象发布被重排到 final 写之前，保证其他线程只看到已初始化的 final 字段（X86 上写写屏障是 no-op）。这属于安全发布（safe publication）问题。
- 锁的可见性：解锁时强制刷新缓存使修改对其他线程可见；若逃逸分析证明锁只被单线程持有可消除加解锁（如 `synchronized(new Object()){}` 可能等价空操作）。

**面试高频问答**
- **Q：** 什么是 happens-before？它和"时间先后"是一回事吗？
  **A：** happens-before 描述内存可见性：X hb Y 则 X 结果对 Y 可见。它不代表执行的时间先后；只有当后者观测（数据依赖）前者结果时才不能重排序，否则仍可重排。
- **Q：** volatile 的作用和原理？
  **A：** 保证可见性和禁止重排序（借助内存屏障），不保证原子性（如 i++ 仍不安全）。写 volatile 会强制刷新写缓存到主内存，并无效化其他 CPU 缓存行，从而实现可见性；且不能被缓存到寄存器。
- **Q：** volatile 为什么不保证原子性？
  **A：** 它只保证单次读/写直接作用于主内存和内存屏障顺序，但复合操作（读-改-写，如自增）由多条指令组成，中间可被其他线程插入，需用锁或原子类。
- **Q：** final 字段有什么内存语义？
  **A：** 编译器在 final 写后插入写写屏障，防止"新对象的引用发布"被重排到 final 写之前，确保其他线程通过该引用看到的一定是已正确初始化的 final 字段。
- **Q：** 举例说明重排序导致的多线程问题及解决办法。
  **A：** 两线程分别执行 `r2=a;b=1;` 和 `r1=b;a=2;`，无同步时可能出现 (1,2)。将 a 或 b 声明为 volatile，通过 volatile 写读的 hb 关系加传递性，即可禁止该重排序。

## 14 | JVM 如何实现 synchronized

**一句话核心**：synchronized 基于对象监视器（monitor），字节码用 monitorenter/monitorexit 实现可重入加锁；HotSpot 按代价从低到高分为偏向锁 → 轻量级锁 → 重量级锁并逐级膨胀。

**关键知识点**
- 字节码实现：同步代码块编译为 monitorenter + 多个 monitorexit（正常路径和异常路径都要解锁，靠异常表保证）；同步方法用访问标记 ACC_SYNCHRONIZED，进入/退出时隐式做 monitor 操作。锁对象：实例方法是 this，静态方法是所在类的 Class 实例。
- monitor 抽象：每个锁对象有一个计数器 + 一个持有线程指针。monitorenter 时计数器为 0 则占有并置 1；持有者是当前线程则计数器 +1（可重入）；否则等待。monitorexit 计数器 -1，减到 0 则释放。计数器机制实现锁的可重入。
- 锁状态标志在对象头 mark word 最后两/三位：00 轻量级锁，01 无锁/偏向锁，10 重量级锁，11 与 GC 标记有关。偏向锁在 01 基础上用最后三位 101 区分。
- 重量级锁：最基础实现，阻塞/唤醒加锁失败的线程，依赖操作系统（pthread mutex），涉及用户态↔内核态切换，开销大。为减少阻塞，JVM 引入自旋（空跑轮询）+ 自适应自旋（根据历史自旋是否成功动态调整自旋次数）。副作用：不公平（自旋线程可能优先抢到锁）。
- 轻量级锁：针对"多线程在不同时间段申请同一把锁、无实际竞争"。加锁时在当前栈帧划出锁记录，复制 mark word，用 CAS 把锁对象 mark word 替换为指向锁记录的指针（末两位 00）。CAS 成功即获锁；若发现是自己重复获取则锁记录清零（可重入）；若被他人持有则膨胀为重量级锁并阻塞。锁记录呈栈结构，解锁用 CAS 还原 mark word。
- 偏向锁：针对"始终只有一个线程持有锁"。首次加锁用 CAS 把线程地址记入 mark word 并置末三位 101，之后同一线程加锁只需比对（线程地址 + epoch 值）即可直接返回，无 CAS。
- 偏向锁撤销：其他线程来竞争时需撤销，要求持有线程到达安全点，再替换为轻量级锁，代价大。epoch 用于批量重偏向：某类撤销数超阈值（BiasedLockingBulkRebiasThreshold 默认 20）则类 epoch +1 宣布上一代偏向锁失效；超过更高阈值（BiasedLockingBulkRevokeThreshold 默认 40）则该类彻底不再使用偏向锁，直接上轻量级锁。
- 锁膨胀单向不可逆（针对对象整个生命周期）：偏向 → 轻量级 → 重量级。

**面试高频问答**
- **Q：** synchronized 底层怎么实现的？
  **A：** 同步块编译为 monitorenter/monitorexit 指令，同步方法用 ACC_SYNCHRONIZED 标记。每个锁对象关联一个 monitor（计数器 + 持有线程指针），计数器机制支持可重入。HotSpot 实现分偏向锁、轻量级锁、重量级锁三级。
- **Q：** 锁升级（膨胀）过程是怎样的？
  **A：** 无竞争时先用偏向锁（只记线程地址）；出现其他线程交替获取时撤销为轻量级锁（CAS + 自旋）；真正并发竞争、CAS 失败时膨胀为重量级锁（OS 互斥量，阻塞唤醒）。单向不可逆。
- **Q：** 偏向锁、轻量级锁、重量级锁各针对什么场景？
  **A：** 偏向锁——始终一个线程持有；轻量级锁——多线程不同时间段交替持有、无实际竞争；重量级锁——多线程同时竞争。
- **Q：** 为什么 synchronized 是可重入的？
  **A：** monitor 记录持有线程和计数器，持有线程再次加锁只需计数器 +1，避免自己和自己死锁；轻量级锁用锁记录清零、偏向锁直接比对线程地址实现重入。
- **Q：** 自旋锁的作用和缺点？
  **A：** 在阻塞前先在 CPU 上空转轮询，若锁很快释放就无需昂贵的阻塞/唤醒；缺点是白白消耗 CPU，锁长时间不释放时浪费大。JVM 用自适应自旋权衡。
- **Q：** mark word 里存了什么？
  **A：** 对象头 mark word 存 hashCode、GC 分代年龄、锁标志位（末 2 位区分锁状态）、以及轻量级锁指向锁记录的指针 / 偏向锁的线程地址和 epoch。

## 15 | Java 语法糖与 Java 编译器

**一句话核心**：语法糖是编译器为提高开发效率做的"脱糖"处理，在字节码层面还原为等价的底层代码，运行时并无特殊之处。

**关键知识点**
- 自动装箱/拆箱：装箱通过 `Integer.valueOf(int)`，拆箱通过 `Integer.intValue()`。`valueOf` 对 [-128,127]（IntegerCache.low/high）范围内的值返回缓存对象，范围外 new 新对象。high 可通过 `java.lang.Integer.IntegerCache.high` 或 `-XX:+AggressiveOpts` 调大，但 low 不可改。
- 泛型类型擦除：泛型信息在 JVM 中被擦除（为兼容泛型前的旧代码）。无限定的擦除为 Object，有限定（`T extends Number`）则擦除为限定的最高层类（Number）。字节码方法签名 Signature 中仍保留泛型信息，供编译期类型检查使用。取元素时编译器插入 checkcast 向下转型。
- 泛型仍有用：编译器据泛型做静态类型检查（如向 `ArrayList<Integer>` 加 String 会编译报错）。
- 桥接方法（bridge method）：因泛型擦除后子类重写方法的描述符与父类不一致（如子类 (LVIP;)D vs 父类 (LCustomer;)D），编译器生成桥接方法在字节码层面重写父类方法并转调子类实际方法。协变返回类型（返回类型是父类方法返回类型的子类）也会生成桥接方法。桥接方法带 ACC_BRIDGE + ACC_SYNTHETIC（源码不可见），说明 class 文件允许同名同参不同返回类型的方法共存。
- foreach 循环：对数组脱糖为下标 for 循环（先取 length 再遍历）；对 Iterable 脱糖为 iterator + hasNext/next 的 while 循环。
- 字符串 switch：脱糖为两层——先比较字符串的 hashCode（转成 int switch，本质哈希桶），再用 String.equals 逐个比较相同哈希值的字符串（防哈希碰撞）。
- 其他语法糖：变长参数（编译为数组）、try-with-resources、同一 catch 捕获多种异常、内部类、条件编译等，都由编译器脱糖。

**面试高频问答**
- **Q：** Java 泛型的类型擦除是什么？带来什么问题？
  **A：** 泛型信息编译后被擦除（无限定→Object，有限定→限定类），JVM 运行时无泛型类型。带来的问题：不能 new 泛型数组、不能对泛型做 instanceof、重写需桥接方法、静态方法/字段无法用类泛型等。
- **Q：** 什么是桥接方法？为什么需要它？
  **A：** 泛型擦除后子类重写方法的描述符与父类不同，不满足 JVM 的重写定义。编译器生成一个描述符与父类一致的桥接方法（ACC_BRIDGE/ACC_SYNTHETIC），内部强转参数后转调子类真实方法，从而保留多态语义。协变返回类型也会生成桥接方法。
- **Q：** `Integer a=127,b=127; a==b` 和 128 时结果为何不同？
  **A：** 自动装箱调 `Integer.valueOf`，[-128,127] 返回同一缓存对象，故 127 时 == 为 true；128 超出缓存范围各自 new，== 比较引用为 false。
- **Q：** 字符串 switch 底层怎么实现？
  **A：** 先对字符串取 hashCode 转成 int switch（哈希桶定位），再用 String.equals 精确比较以处理哈希碰撞，最后跳转到对应 case。
- **Q：** foreach 是怎么实现的？
  **A：** 编译器脱糖：数组转下标 for 循环；实现 Iterable 的对象转为调用 iterator()，用 hasNext()/next() 的 while 循环遍历。

## 16 | 即时编译（上）：分层编译与触发

**一句话核心**：HotSpot 用分层编译综合 C1 启动快与 C2 峰值高的优势，由方法调用计数器和循环回边计数器触发即时编译（含以循环为单位的 OSR 编译）。

**关键知识点**
- 编译器：C1（-client，编译快）、C2（-server，代码质量高，峰值性能通常高 30%+）、Graal（实验性，`-XX:+UnlockExperimentalVMOptions -XX:+UseJVMCICompiler` 启用替换 C2）。
- 分层编译（-XX:+TieredCompilation，Java 8 默认开启）5 个层次：0 解释执行；1 无 profiling 的 C1；2 仅方法调用+回边 profiling 的 C1；3 带全部 profiling 的 C1；4 C2。开启分层后 -client/-server 失效；关闭时直接用 C2；只用 C1 可 `-XX:TieredStopAtLevel=1`。
- 1 层和 4 层是终止状态（编译后不再重新请求编译，除非失效/去优化）。C1 三态执行效率 1 层 > 2 层 > 3 层（profiling 越多开销越大，3 层比 2 层慢约 30%）。
- 典型路径：解释执行 → 3 层 C1 → 4 层 C2。特例：字节码很少（getter/setter）且 3 层无 profile 可收集时，3 层后直接转 1 层 C1 终止；C1 忙则解释执行时 profiling 后直接 4 层；C2 忙则走 2 层→3 层。
- 触发依据：方法调用次数 + 循环回边次数（往回跳转指令）。计数器不做同步，是近似值。不分层时和超过 -XX:CompileThreshold（C1=1500，C2=10000）触发；分层时该参数失效，改用动态阈值（阈值乘以系数 s，s 与待编译方法数正相关、与编译线程数负相关）。
- 编译线程数：64 位默认按 CPU 核数调整（-XX:+CICompilerCountPerCPU），C1:C2 按 1:2 分配（各至少 1 个）。
- OSR（On-Stack-Replacement，栈上替换）：以循环为单位的编译，由循环回边计数器触发，可在方法非入口处（热循环中）动态替换栈帧、在解释执行与编译代码间切换。用于优化"单次调用但含热循环"的场景，基准测试常见。去优化也用 OSR 技术。
- -XX:+PrintCompilation 可打印编译情况；去优化时旧编译被标 "made not entrant"，所有线程退出后标 "made zombie" 可回收代码空间。

**面试高频问答**
- **Q：** 为什么要分层编译？
  **A：** C1 编译快但代码慢，C2 代码快但编译慢。分层编译让不太热的代码快速被 C1 编译获得较好性能，很热的代码再用 C2 深度优化，兼顾启动性能和峰值性能。
- **Q：** 即时编译如何被触发？
  **A：** 靠方法调用计数器和循环回边计数器。不分层时二者之和超过 CompileThreshold（C1=1500/C2=10000）触发；分层时用动态阈值，阈值随待编译方法数和编译线程数动态调整。
- **Q：** 为什么要维护两个计数器？OSR 是什么？
  **A：** 方法调用计数器用于以方法为单位的编译，循环回边计数器用于 OSR（栈上替换）——以循环为单位的编译，能在方法执行中途（热循环处）从解释执行切换到编译代码，解决单次调用含热循环的性能问题。
- **Q：** 什么是热点代码？为什么不全部编译？
  **A：** 反复执行的方法/循环即热点。动态编译很耗 CPU 和时间，若编译出的代码只跑几次就浪费了，所以只对达到阈值的热点代码编译，其余解释执行。

## 17 | 即时编译（下）：Profiling、投机优化与去优化

**一句话核心**：C2 基于 C1 收集的分支/类型 profile 做激进的投机优化（剪枝、去虚化内联），假设失败时通过去优化退回解释执行并重新收集 profile。

**关键知识点**
- Profiling 分级：0/2/3 层都收集方法调用次数和循环回边次数（用于触发编译）；0 层和 3 层额外收集分支 profile（条件跳转的跳转/不跳转次数）和类型 profile（invokevirtual、checkcast、instanceof、aastore 的接收者类型）。这部分收集开销大，是 3 层比 2 层慢 30% 的主因。
- 通常只在 C1 编译后的 C1 代码中收集分支/类型 profile（认为该方法可能被 C2 编译时才收集）；只有极端情况（待 C1 编译方法太多）才在解释执行时收集。
- 基于分支 profile 的优化：若某分支从未跳转，C2 假设今后仍不跳转，剪掉该 false 分支（不编译），精简控制流/数据流并触发更多优化（如后续条件被直接优化掉）；即使双向都走，也可据概率优先处理高频路径。
- 基于类型 profile 的优化：若某调用点/instanceof 的接收者类型 profile 只含某一类型，则假设今后仍是该类型，生成"先测试动态类型是否匹配"的代码，匹配则走内联后的实质代码（条件去虚化内联，把虚方法调用变成直接内联）。（instanceof 对非 final 目标类型开销大，需沿类层次逐级比较；有了类型 profile 可简化为单次动态类型比较。）
- 投机优化的核心都是"假设"：分支 profile 假设只走某分支；类型 profile 假设动态类型只是那几个。
- 去优化（deoptimization）：假设失败时从编译机器码切回解释执行。编译器在假设失败处插入陷阱（trap，一条 call 指令），触发时调用 JVM 去优化方法，改写返回地址不再回到机器码；需将机器码执行状态映射还原为某字节码前的解释执行状态（如标量替换掉的对象要重新构造出来），再用 OSR 替换栈帧、从目标字节码开始解释执行。
- 去优化后的三种处理：Action_None（原因与优化无关，保留机器码下次继续用）；Action_Recompile（原因与静态分析如类层次分析有关，丢弃机器码但可不重新 profile 直接重编）；Action_Reinterpret（原因是基于 profile 的激进优化失败，丢弃机器码且需重新收集 profile，因为程序执行状态已变）。

**面试高频问答**
- **Q：** 什么是去优化（deoptimization）？何时发生？
  **A：** 从执行 JIT 机器码切回解释执行。当基于 profile 的投机优化假设失败（如出现了被剪掉分支的路径、遇到未预期的动态类型）触发编译器埋下的陷阱，还原执行状态后回到解释器。
- **Q：** JIT 的投机优化基于什么？举例说明。
  **A：** 基于 profiling 收集的分支 profile 和类型 profile 做假设。例如某分支从没走过就剪掉不编译；某调用点接收者一直是同一类型就做条件去虚化内联，把虚调用变直接内联。假设失败再去优化。
- **Q：** 为什么 3 层 C1 代码比 2 层慢？
  **A：** 3 层要收集完整的分支 profile 和类型 profile，这部分 instrumentation 开销大，约使性能低 30%，换来的是给 C2 提供优化数据。
- **Q：** 去优化后机器码会被复用还是重编？
  **A：** 视原因：与优化无关的传 Action_None 保留复用；与静态分析有关的传 Action_Recompile 直接重编；与 profile 激进优化有关的传 Action_Reinterpret 丢弃并重新收集 profile。

## 18 | 即时编译器的中间表达形式（IR）

**一句话核心**：即时编译器把字节码转成基于 SSA 的 IR（C2/Graal 用 Sea-of-Nodes），在 IR 图上以一系列图算法做优化。

**关键知识点**
- 两轮编译：javac 把源码编译成字节码，JIT 把字节码编译成机器码。字节码基于栈、剥离了高级语法，易建模，JIT 直接把它当一种 IR，无需重做词法/语法/语义分析。
- 字节码不适合直接优化，现代编译器用 SSA（静态单赋值）IR：每个变量只赋值一次、赋值后才能用。SSA 便于识别冗余赋值（找"赋值但未使用"的变量），也利于常量折叠、常量传播、强度削减、死代码删除等优化。
- Phi 函数：不同执行路径给同一变量赋不同值时，用 Phi 函数按控制流路径选择对应的值（如 `y3 = Phi(y1, y2)`）。
- Sea-of-Nodes（C2 的 Ideal、Graal 的 IR 都属此类）：去除变量概念，直接用"值"（节点）参与运算，变量名不复存在。因此常量传播在其中变成 no-op。Graal IR 可视为 C2 IR 的精简版，二者优化大同小异。
- IR 图：控制流（红色粗线）连接固定节点，浮动节点位置不定；顺序执行的节点归入同一基本块（B0/B1…）。基本块是仅一个入口一个出口的指令序列。节点调度（node scheduling）据控制/数据/内存依赖计算浮动节点排布位置（C2 无固定节点概念，全是浮动节点）。
- 内存依赖：对同一内存地址的读写有时序依赖，不能任意调度。C2 记录内存依赖供调度用；Graal 把内存读写变成固定节点，靠先后顺序天然表达，无需额外记录。
- GVN（Global Value Numbering）：发现并消除等价计算。在 Sea-of-Nodes 中很简单——无内存副作用的浮动节点，若类型相同且输入节点一致就归并为一个。可理解为 IR 图上的公共子表达式消除（CSE），区别是 GVN 直接比较值，CSE 靠词法分析且常需配合常量传播。
- 可视化工具：IGV（Ideal Graph Visualizer）。

**面试高频问答**
- **Q：** 什么是 SSA？有什么好处？
  **A：** 静态单赋值，每个变量只被赋值一次、赋值后才可用。好处：轻松识别冗余赋值和死代码，便于常量折叠/传播、强度削减、死代码删除等优化。分支合并处用 Phi 函数选择不同路径的值。
- **Q：** Sea-of-Nodes 是什么？
  **A：** C2 和 Graal 采用的一种 SSA IR，去掉变量概念、直接用值（节点）运算并相互依赖。它把控制流和数据流统一在一张 IR 图里，每个优化就是一个接收 IR 图、输出 IR 图的图算法。
- **Q：** 什么是 GVN？和 CSE 有何区别？
  **A：** 全局值编号，发现并消除等价计算（如重复的相同乘法归并成一次）。可看作 IR 图上的公共子表达式消除；区别是 GVN 直接比较值节点是否等价，CSE 借助词法分析判断表达式相同、常需配合常量传播。
- **Q：** IR 图里固定节点和浮动节点的区别？
  **A：** 固定节点被控制流边连接、位置固定；浮动节点位置不定，编译器据控制/数据/内存依赖做节点调度确定其排布。C2 全用浮动节点，Graal 把内存读写等作为固定节点。

## 19 | Java字节码（基础篇）

**一句话核心**：Java字节码是基于栈的指令集，方法栈桢由操作数栈和局部变量区组成，程序需将变量从局部变量区加载到操作数栈运算后再存回。

**关键知识点**
- 操作数栈：执行指令前操作数必须已压栈，执行时弹出操作数、压入结果。如 `iadd` 弹出栈顶两个 int，压入其和。只消耗自己需要的元素，不关心更深的栈内容。
- 操作数栈专用指令：`dup`（复制栈顶）、`pop`（舍弃栈顶）、`swap`（交换栈顶两元素）；long/double 占两个栈单元，需用 `dup2`/`pop2`。`dup` 常用于复制 new 出的未初始化引用以调用构造器；`pop` 常用于舍弃不使用的方法返回值。
- 常量加载指令：`iconst`（-1~5）、`bipush`（一字节）、`sipush`（两字节）、`ldc`（加载常量池常量，含 int/long/float/double/String/Class）。
- 局部变量区：被当成数组，依次存放 this 指针（仅非静态方法）、传入参数、局部变量。long/double 占两个单元，其余占一个。生命周期不重合的局部变量可复用同一单元。
- 加载/存储指令区分类型：int 用 `iload`/`istore`，引用用 `aload`/`astore`，需指明单元下标。唯一直接作用于局部变量区的是 `iinc M N`（第M单元 int 值加 N），常用于 for 循环自增。
- 字节码分类：加载常量、操作数栈专用、局部变量区访问、Java相关指令（new/instanceof/checkcast/athrow/monitorenter/monitorexit + 字段访问 getstatic/putstatic/getfield/putfield）、方法调用（invokestatic/special/virtual/interface/dynamic）、数组相关（newarray/anewarray/multianewarray/arraylength + iaload/iastore 等）、控制流（goto/条件跳转/tableswitch/lookupswitch/ireturn 等）、计算相关。
- 异常抛出是操作数栈压入弹出的唯一例外：JVM 清空操作数栈，再压入异常实例。

**面试高频问答**
- **Q：** JVM 为什么采用基于栈的计算模型？
  **A：** 主要是实现简单、可移植（不依赖具体硬件寄存器数量）。代价是不高效，因为没利用底层寄存器。只有解释器完整模拟该栈模型；即时编译器解析字节码时用虚拟栈模型，后续优化和生成机器码就改用寄存器了。
- **Q：** `new` 一个对象为什么字节码里有 `dup`？
  **A：** `new` 压入未初始化引用 r0，`dup` 复制出 r1；调用构造器（invokespecial）会消耗掉 r1，构造器无返回值不用 pop，栈上仍留 r0 供后续 astore 存入变量。若不 dup，引用在调用构造器时就用掉了，程序再也拿不到该对象引用。
- **Q：** 局部变量区第 0 个单元存什么？long/double 怎么存？
  **A：** 非静态方法第 0 单元存 this 指针，之后依次是参数和局部变量。long 和 double 各占两个连续单元，其余类型占一个单元。
- **Q：** 返回指令、数组指令为什么有很多种？
  **A：** 它们区分类型。返回 int 用 `ireturn`、引用用 `areturn`；int 数组加载/存储用 `iaload`/`iastore`。正常路径以 return 系列结束，异常路径以 `athrow` 结束。

## 20 | 方法内联（上）

**一句话核心**：方法内联把目标方法体纳入编译范围取代原调用，既消除调用开销又触发更多优化，是编译优化里最重要的一环，但受机器码体积和编译时间制约不能无限内联。

**关键知识点**
- 内联定义：编译时遇到方法调用，将目标方法的方法体纳入编译范围，取代原方法调用。
- 双重收益：一是消除调用本身开销（保存执行位置、创建/压入/弹出栈帧、恢复执行），如 getter/setter 内联后仅剩字段访问；二是内联后能进一步触发常量折叠、死代码消除等优化。
- C2 在解析字节码过程中完成内联；Graal 除此之外还有独立优化阶段，把指代方法调用的 IR 节点替换为目标方法的 IR 图。
- 内联三项额外操作：①被调用方法的传入参数节点替换为调用者实际传入参数对应节点；②所有指向原调用节点的数据依赖重新指向被调用方法返回节点（多返回点则生成 Phi 节点聚合）；③被调用方法的异常抛出路径与调用者的异常处理器相连。
- final 常量的作用：字段标 final，Java 编译器（非即时编译器）会把它编译成常量值直接内联进字节码；内联时用常数替换参数，配合死代码消除可把整个方法优化成一个常数节点。
- 内联的代价：内联越多编译时间越长、峰值性能推迟；生成机器码越长越易填满 Code Cache（`-XX:ReservedCodeCacheSize` 控制），满了会报 "CodeCache is full. Compiler has been disabled"，即时编译被关闭。
- 内联规则：`-XX:CompileCommand` 的 inline 指令或 `@ForceInline`（仅 JDK 内部）强制内联；dontinline/exclude 或 `@DontInline` 强制不内联；符号引用未解析、目标类未初始化、native 方法都无法内联；C2 不内联超过 9 层调用（`-XX:MaxInlineLevel`）和 1 层直接递归（`-XX:MaxRecursiveInlineLevel`）。自动拆箱总被内联，Throwable 的方法不能被其他类内联。
- 最终由程序路径热度、目标方法调用次数及大小、当前 IR 图大小共同决定能否内联。内联算法青睐小方法。

**面试高频问答**
- **Q：** 方法内联发生在哪个阶段？为什么重要？
  **A：** 只发生在即时编译器中（不是解释执行）。重要在于它不仅消除调用开销，更重要的是让被调用方法的代码进入当前编译单元，从而触发常量传播、死代码消除、逃逸分析等一连串后续优化。
- **Q：** 为什么即时编译器不无限制内联？
  **A：** 内联越多编译时间越长、达到峰值性能越晚；机器码越长越容易填满 Code Cache，一旦 Code Cache 满，即时编译会被整体关闭。所以内联算法偏好小方法，并有层数、大小、热度等限制。
- **Q：** 哪些方法无法被内联？
  **A：** 符号引用未解析、目标类未初始化、native 方法、被 dontinline/exclude/@DontInline 标记的方法、超过 9 层的调用层级、超 1 层的直接递归、以及体积过大的方法（callee is too large）。
- **Q：** final 对内联有什么帮助？
  **A：** final 静态字段被 Java 编译器编译成常量直接嵌入字节码，内联时可用常量替换参数并触发折叠与死代码消除。此外 final 方法/类可省去去虚化时的假设注册（见下篇）。

## 21 | 方法内联（下）：去虚化

**一句话核心**：虚方法调用要先去虚化（转成一个或多个直接调用）才能内联；手段包括完全去虚化（类型推导 / 类层次分析，证明唯一目标）和条件去虚化（借助类型 Profile 加类型比较）。

**关键知识点**
- 虚方法（invokevirtual/invokeinterface）需动态绑定，即时编译器必须先去虚化为直接调用才能内联。分两大类：完全去虚化、条件去虚化（guarded devirtualization）。
- 完全去虚化：证明目标方法唯一，转为直接调用。两种途径：基于类型推导、基于类层次分析（CHA）。
- 基于类型推导：靠数据流分析推导调用者动态类型。Sea-of-Nodes IR 中变量被替换为具体值，类型往往比声明类型更精确（如 `new Add()` 或强转后的 Pi 节点直接是 Add）。但若需额外数据流分析才能确定（如 `if(op instanceof Add)`），C2/Graal 都不做，因为类型推导是全局优化太费时，且有 CHA 和条件去虚化兜底。
- 基于类层次分析（CHA）：分析所有已加载类，判断抽象/接口方法是否只有一个实现；若是则直接内联该实现。因为后续可能加载新子类，JVM 会注册"假设"（assumption，如某抽象方法仅一个实现/某类无子类），每次类加载重新验证；假设失效则触发去优化（deoptimization）。
- effective final：即便声明类型是具体类（如 Add），因为可能有重写子类，仍需注册"Add 无子类"假设。目标方法事实上 final（effective final）即可去虚化内联，无需显式 final 修饰符；但显式 final 可省去假设、精简编译结果、减少类加载时的验证。
- 接口方法调用（invokeinterface）：CHA 完全去虚化不能移除动态类型检测，因为验证器把接口当 Object，可能出现声明类型是接口但实际类型未实现该接口的情况。所以 C2 对接口调用不做 CHA 去虚化，改用条件去虚化。
- 条件去虚化：把调用者动态类型依次与类型 Profile 记录的类型比较，命中就直接调用对应目标方法（生成 if/TypeSwitch）。这就是单态/多态内联缓存的思路。
- 匹配不到 Profile 类型时：若 Profile 完整（所有出现过的类型都记录了）则去优化重新收集；若不完整则改走原虚调用（内联缓存或方法表动态绑定）。C2 在 Profile 不完整时干脆不做条件去虚化，直接用内联缓存或方法表。每条类型相关字节码的 type profile 有数量上限（默认只存两个动态类型），超过就变不完整。

**面试高频问答**
- **Q：** 虚方法调用能内联吗？怎么做？
  **A：** 能，但要先去虚化。要么完全去虚化（用类型推导或类层次分析证明目标唯一，转直接调用），要么条件去虚化（用类型 Profile 加类型比较，即内联缓存，命中就调对应实现）。去虚化后才做内联。
- **Q：** 基于类层次分析去虚化，后来又加载了新子类怎么办？
  **A：** JVM 编译时注册"假设"（如该抽象方法仅一个实现）。每次类加载都重新验证这些假设；若新子类使假设失效，就对相关编译结果进行去优化，回退到解释执行并重新编译。
- **Q：** 单态、多态内联缓存是什么？
  **A：** 即条件去虚化。虚拟机收集调用点的类型 Profile；只有一种类型是单态（一次比较+直接调用），少数几种是多态（多次比较）。匹配失败时按 Profile 是否完整选择去优化重收集或退回方法表/内联缓存的常规虚调用。
- **Q：** 为什么接口方法调用不做类层次分析去虚化？
  **A：** invokeinterface 必须运行时校验调用者是否真的实现了该接口（验证器把接口当 Object，存在声明为接口但实际未实现的情况），这个类型测试无法省去，所以 C2 直接改用条件去虚化。
- **Q：** effective final 是什么？final 有必要吗？
  **A：** 方法事实上没有被重写即 effective final，即时编译器就能去虚化内联，不强制写 final。写 final 的好处是省去"无子类/唯一实现"假设的注册，编译结果更精简、类加载验证更少。

## 22 | HotSpot虚拟机的intrinsic（内建函数）

**一句话核心**：intrinsic 是被 `@HotSpotIntrinsicCandidate` 标注的方法，HotSpot 用基于特定 CPU 指令的高效实现替换其调用，忽略原方法字节码，从而远超同样字节码的普通实现。

**关键知识点**
- 定义：被 `@HotSpotIntrinsicCandidate` 标注的方法即 HotSpot intrinsic。对它们的调用会被 HotSpot 替换成高效指令序列，原方法实现被忽略。虚拟机为这些方法额外维护一套高效实现，改动 JDK 原实现时需同步修改以保证语义一致。
- 为什么不直接在源码里写高效实现：高效实现依赖具体 CPU 指令，Java 源码难以表达；且换个体系架构可能没有对应指令，就无法 intrinsic。因此不同虚拟机、不同版本 HotSpot 的 intrinsic 数量差异很大，越新版本越多（Java 9 后大量引入，该注解也是 Java 9 引入）。
- CPU 指令例子：`StringLatin1.indexOf` → X86_64 SSE4.2 的 `PCMPESTRI`；`Math.addExact` 溢出检测 → 直接读状态寄存器溢出标志位 `jo`；`Integer.bitCount` → 一条 `popcnt` 指令；`Thread.currentThread` → 读 R13 寄存器；CAS（Unsafe.compareAndSwap）→ `lock cmpxchg` 原子更新指令。
- 两种实现方式：①独立的桩程序（stub）——可被解释器或即时编译器利用，数量较少，主要是 Math 类的一些方法；②特殊的编译器 IR 节点——仅即时编译器可用，大部分 intrinsic 用此方式，替换 IR 节点后参与优化，最终后端生成指定 CPU 指令。
- intrinsic 替换发生在方法内联过程中：碰到调用节点先查目标是否 intrinsic，是则插入特殊 IR 节点，否则走常规内联。即时编译器直接忽略原方法字节码，甚至不在乎是否有字节码——native 方法只要标为 intrinsic 也能"内联"进来，从而省掉巨大的 JNI 开销。
- 已有 intrinsic（最新 HotSpot 300+ 个）：三成以上是 Unsafe 类方法（通过 java.util.concurrent 间接使用，如 CAS）；StringBuilder/StringBuffer（减少内存复制）；String/StringLatin1/StringUTF16/Arrays（用 SIMD 单指令多数据批量处理，如 `Arrays.equals` 放进 XMM/ZMM 寄存器批量比较）；包装类、Object、Math、System、反射、MethodHandle、压缩加密相关方法。

**面试高频问答**
- **Q：** 为什么 String.indexOf 比字节码几乎相同的自实现快很多？
  **A：** String.indexOf 底层的 StringLatin1.indexOf 被标注为 intrinsic，HotSpot 用基于 SSE4.2 的 PCMPESTRI 指令的高效实现替换掉原方法，字节码相同但实际执行的是专用 CPU 指令，所以快得多。
- **Q：** intrinsic 和方法内联什么关系？
  **A：** intrinsic 的替换正是在方法内联过程中进行的。即时编译器遇到调用先判断目标是否 intrinsic，是就插入特殊 IR 节点（忽略原字节码），否则做常规内联。native 方法标为 intrinsic 也能被"内联"，省掉 JNI 开销。
- **Q：** 为什么不直接把高效实现写进 Java 源码？
  **A：** 高效实现依赖具体 CPU 指令，Java 源码难以表达，且换体系架构可能没有对应指令。用 intrinsic 由虚拟机按平台适配，既保持源码可移植又能在支持的平台上用专用指令。
- **Q：** `@HotSpotIntrinsicCandidate` 注解本身有性能作用吗？
  **A：** 没有。它只是标记/提醒 JDK 开发者"虚拟机里有对应的高效实现"，改动此方法需同步通知修改。真正的替换由虚拟机内部按方法签名匹配完成，其他虚拟机可以直接用原低效 Java 实现。

## 23 | 逃逸分析

**一句话核心**：逃逸分析判断新建对象是否逃逸（被存入堆 / 传入未知代码），基于它可做锁消除、栈上分配、标量替换三种优化，HotSpot 实际采用的是标量替换。

**关键知识点**
- 逃逸分析定义：确定指针动态范围的静态分析。即时编译语境下判断"新建"对象是否逃逸。逃逸依据两条：①对象是否被存入堆（静态字段或堆中对象的实例字段，存入后其他线程可获引用）；②对象是否被传入未知代码（未内联的方法调用视为未知代码，其调用者和参数视为逃逸）。
- 逃逸分析放在方法内联之后，靠内联消除"未知代码"入口，才能证明更多对象不逃逸（如 ArrayList 的 Itr 迭代器内联后可证明不逃逸）。
- 三种优化：锁消除、栈上分配、标量替换。
- 锁消除（lock elision）：若锁对象不逃逸，其他线程无法获得该锁，加锁/解锁无意义可消除。传统编译器只需证明不逃逸出线程，JVM 因即时编译限制强化为不逃逸出当前编译方法。`synchronized(new Object()){}` 会被完全优化掉；`synchronized(escapedObject){}` 不行，逃逸对象可能被其他线程加锁构造 happens-before，至少要生成刷新缓存的内存屏障。实际锁消除不多见。
- 栈上分配：不逃逸对象可分配在栈上，方法退出弹栈自动回收，减轻 GC 压力。但需改动大量"对象只能堆分配"的假设代码，HotSpot 并未采用，改用标量替换。
- 标量替换（scalar replacement）：标量是只存一个值的变量（局部变量），聚合量能存多个值（Java 对象）。把对对象字段的访问替换为一个个局部变量访问，对象不再实际分配，字段可存栈上或直接放寄存器，对象头也消失。相比栈上分配，不要求字段内存连续，甚至可全在寄存器中不占内存。同样减轻 GC 压力。
- 部分逃逸分析（partial escape analysis，Graal）：C2 的逃逸分析与控制流无关较简单；Graal 引入与控制流相关的部分逃逸分析，处理对象仅在部分路径逃逸的情况，把新建操作推延到真正逃逸的分支，使只走非逃逸路径时不再新建对象。能优化更多情况但编译更慢。

**面试高频问答**
- **Q：** 什么是逃逸分析？判断逃逸的依据是什么？
  **A：** 判断即时编译中新建对象的动态范围是否逃出当前方法。两条依据：对象是否被存入堆（静态字段或堆对象实例字段），对象是否作为未内联方法调用的调用者或参数（传入未知代码）。满足任一即逃逸。
- **Q：** 基于逃逸分析有哪三种优化？
  **A：** 锁消除（不逃逸锁对象去掉加解锁）、栈上分配（不逃逸对象分配到栈上随栈桢回收）、标量替换（把对象拆成局部变量/寄存器，不再分配对象）。HotSpot 未实现栈上分配，实际用的是标量替换。
- **Q：** 栈上分配和标量替换有什么区别？HotSpot 用哪个？
  **A：** 栈上分配仍把对象整体放栈上，需字段内存连续；标量替换把对象彻底拆成独立字段（局部变量），可放寄存器、不要求连续、对象头也没了。HotSpot 因栈上分配要改动太多假设，实际采用标量替换。
- **Q：** `synchronized(new Object()){}` 会怎样？为什么？
  **A：** 会被锁消除完全优化掉。因为该锁对象不逃逸，其他线程拿不到，无法基于它构造 happens-before，加解锁没有意义。而 `synchronized(escapedObject)` 不能消除，逃逸对象可能被别的线程加锁，至少要保留内存屏障。
- **Q：** 为什么"避免热点代码用 foreach 减少 GC"这种说法常常没必要？
  **A：** 因为方法内联+逃逸分析能把 ArrayList.iterator 新建的迭代器判定为不逃逸并标量替换掉，实际不产生堆分配，也就没有 GC 压力，无需手工改成 size/get 循环。

## 24 | 字段访问相关优化

**一句话核心**：逃逸分析失效（对象本就逃逸或内联不彻底）时，即时编译器沿控制流缓存字段读写值，消除冗余的字段读取/存储，并配合死代码消除减少内存访问和机器码体积。

**关键知识点**
- 背景：逃逸分析不是银弹，对象本身逃逸或方法内联不彻底会被当成逃逸而无法标量替换，此时字段访问优化很重要。传入参数不属于逃逸分析范围（逃逸分析只针对新建对象）。
- 字段读取优化（read elimination）：沿控制流缓存字段存储将写入的值、或读取得到的值。遇到对同一字段的读取，缓存未失效就用缓存值替换该读取，省一次内存访问。存储节点会更新缓存值。
- 缓存失效条件（保守舍弃所有缓存）：遇到方法调用节点（视为执行未知代码可能改字段）、内存屏障节点（其他线程可能异步改字段）、加解锁操作。
- 读取值被替换为常量可触发连锁优化：如 `o.a=1; if(o.a>=0)...` 中 `o.a` 读取被替换为常量 1，比较直接得 true，else 变不可达被删除。
- volatile 的影响：即时编译器在 volatile 字段访问前后插入内存屏障节点，阻止把屏障前缓存值用于屏障后读取，强制每次用内存读指令取最新值。即使 X86_64 上 volatile 读屏障是 no-op，屏障节点仍会阻止字段读取优化（解释了 while(a) 非 volatile 时会被优化成死循环、volatile 时每次重读）。
- 字段存储优化（write/dead store elimination）：一个字段先后存两次、其间无读取（也无方法调用、内存屏障），第一次冗余存储被消除。即使中间有读取，借助字段读取优化把读转成常量后，第一次存储仍可能被判为冗余消除。volatile 字段的冗余存储不能消除。此情形常因两存储间隔许多代码，或方法内联把构造器初始化与随后更新纳入同一编译单元而出现。
- 死代码消除（dead code elimination）：①局部变量死存储——因 Sea-of-Nodes IR 特性无额外代价，无节点依赖旧值则该计算被消除，部分路径死存储用 Phi 节点处理；②不可达分支消除——删除任何路径都到不了的分支，精简数据流、减少编译时间和机器码大小。不可达分支常由内联、常量传播、基于 profile 的优化产生。

**面试高频问答**
- **Q：** 即时编译器怎么优化重复的字段读取？
  **A：** 沿控制流缓存字段值，遇到对同一字段的后续读取直接用缓存值替换，省去内存访问。前提是两者之间没有方法调用、内存屏障或其他可能写该字段的节点，否则保守舍弃缓存。
- **Q：** 为什么 `while(a){}` 不加 volatile 会变死循环，加了就不会？
  **A：** 不加 volatile 时字段读取优化把 `a` 替换为存储的常量 true，循环里不再重读内存，成死循环。加 volatile 后编译器在访问前后插入内存屏障节点，阻止用缓存值，强制每轮用内存读指令取 a 的最新值，故能感知其他线程的修改。
- **Q：** 冗余字段存储在什么条件下能消除？
  **A：** 同一字段先后存两次、中间没有对它的读取、没有方法调用、没有内存屏障，则第一次存储被消除。即使中间有读取，也可能靠读取优化把读转成常量后再判定第一次存储冗余。volatile 字段不能消除。
- **Q：** 死代码消除有哪两种形式？
  **A：** 一是死存储消除（局部变量存了没被依赖使用，含部分路径死存储，靠 Sea-of-Nodes IR 转换零代价完成）；二是不可达分支消除（删除任何路径都到不了的分支）。后者常由内联、常量传播、profile 优化产生，能减少编译时间与机器码体积。
- **Q：** `int t = x/y; t = x+y; return t;` 里的除法能被死代码消除吗？
  **A：** 不能。虽然 t 的第一个值 x/y 没被使用，看似死存储，但除法可能抛除零异常（副作用），即时编译器需判断除数是否为 0，不能直接删掉除法运算。

## 25 | 循环优化

**一句话核心**：即时编译器通过循环无关代码外提、循环展开、循环判断外提、循环剥离等优化，在不改变语义的前提下尽量减少循环内重复执行的工作量。

**关键知识点**
- 循环无关代码（Loop-invariant Code）外提：把循环中值不变的表达式（如 `x*y`）和内存访问（如 `a.length`）提到循环之前，避免每次迭代重复计算。借助 Sea-of-Nodes IR + 节点调度实现：这类浮动节点不依赖循环体数据，会被调度到循环前的基本块。
- 循环无关"检测"外提：`iaload` 隐含 null 检测和数组下标范围检测。null 检测涉及控制流依赖，无法靠节点调度外提，而是通过独立优化"循环预测"（Loop Prediction，`-XX:+UseLoopPredicate`）实现——在循环前插入检测代码，命中则去优化（deoptimize），从而消除循环内的检测。
- 下标范围检测虽是循环有关（依赖循环变量 i），但也能被循环预测外提，只需把检测条件转换为对循环变量最大值 IMAX 的判断。
- 循环展开（Loop Unrolling）：在循环体内重复多次迭代、相应减少循环次数，以空间换时间，增大循环体换取更多优化机会。C2 只对计数循环（Counted Loop）做展开。默认展开上限 60 次（`-XX:LoopUnrollLimit`）。
- 计数循环 4 个条件：①有唯一的基于计数器的循环出口；②计数器类型为 int/short/char（不能是 byte/long/float/double）；③每次迭代增量为常数；④循环上/下限是循环无关值。
- 完全展开（Full Unroll）：循环次数固定且很小（如 ≤3 次）时，整个循环被展开为若干顺序执行的循环体，去掉循环判断。
- 循环判断外提（Loop Unswitching）：把循环内的 if 语句提到循环外，在 if 两个分支中各放一份循环。区别于检测外提——它针对普通分支逻辑而非抛异常。
- 循环剥离（Loop Peeling）：将循环的前几个或后几个特殊迭代剥离出去，使主循环体规律性更明显，便于进一步优化。

**面试高频问答**
- **Q：** 什么是循环无关代码外提？靠什么实现？
  **A：** 把循环中值不变的表达式提到循环外，避免重复计算。普通表达式靠 Sea-of-Nodes IR 的节点调度实现；带控制流依赖的检测（如 null 检测、下标越界检测）靠独立的"循环预测"优化，在循环前插入检测并在命中时去优化。
- **Q：** C2 什么样的循环才能被展开？
  **A：** 只有计数循环。需满足唯一计数器出口、计数器为 int/short/char、增量为常数、上下限循环无关。
- **Q：** 循环展开的代价是什么？
  **A：** 增加代码冗余、机器码变长；但循环体增大也带来更多优化机会（如触发向量化）。默认展开上限 60，超大循环不展开。
- **Q：** 强度削减/进一步优化的例子？
  **A：** byte 数组复制循环展开后，可把 4 个连续 byte 的赋值合并成一次读写（下一讲的向量化）。

## 26 | 向量化

**一句话核心**：向量化借助 CPU 的 SIMD 指令用单条指令操控多组数据，是 CPU 指令级并行；HotSpot 通过 SIMD intrinsic（点覆盖）和自动向量化（条件苛刻）两种方式利用它。

**关键知识点**
- SIMD（单指令流多数据流）：单条指令批量计算一个向量。把宽寄存器里的值看成多个整数/浮点数组成的向量批量运算，如 `PADDB/PADDW/PADDD/PADDQ` 分别做 byte/short/int/long 的向量加法。
- 向量寄存器演进：SSE 引入 128 位 XMM；AVX（2011）升级为 256 位 YMM（原 XMM 指令用其低 128 位）；AVX512 升级为 512 位 ZMM。数组连续存储使得一次读写可批量搬运多个元素。
- 效果：原本 `c.length` 次加法，用 PADDD 后理论上只需 `c.length/4` 次向量加法（实际受缓存行对齐等因素影响，只有中间部分元素能向量化）。
- 不同 CPU 支持的 SIMD 指令集不同。C++ 用 Intel 的 intrinsic 方法编译期选指令；Java 字节码平台无关，无法直接用，但 JIT 在运行时已知目标 CPU 指令集，可动态选择。执行到不支持的指令会触发 sigill 信号。
- 方式一 HotSpot intrinsic：调用特定方法时替换为 SIMD 高效实现，语义比单条 SIMD 指令复杂，由虚拟机开发者定制、性能优越，但屈指可数（`System.arraycopy`、`Arrays.copyOf`、`Arrays.equals`、Java 9 的 `Arrays.compare`/`Arrays.mismatch`、`String.indexOf` 等）。属于"点覆盖"，只有用到对应语义才受益。
- 方式二 自动向量化（Auto Vectorization / SLP）：针对能展开的计数循环，在循环展开后把不同迭代的运算合并为向量运算。
- 自动向量化条件（苛刻）：循环变量增量为 1、不能是 long；迭代间最好无数据依赖（如 `a[i]=a[i-1]` 不行）；循环体内无分支跳转；不要手工展开（手工展开会导致 C2 无法自动展开进而无法向量化）。
- C2 支持的整数向量操作有限：向量加/减、按位与或异或、批量移位和乘法、向量点积（需多条 SIMD 指令，不高效）。OpenJDK Panama 项目引入可控的 Vector API 抽象来扩大覆盖面。

**面试高频问答**
- **Q：** 什么是向量化/SIMD？
  **A：** 用单条指令同时对一组数据做相同运算（把宽寄存器当向量），是 CPU 指令级并行；如 PADDD 一次完成 4 个 int 的加法。
- **Q：** HotSpot 有哪两种向量化方式？各自局限？
  **A：** ①SIMD intrinsic：对特定方法替换为 SIMD 实现，性能好但只覆盖少数方法（点覆盖）；②自动向量化：JIT 对展开后的计数循环自动合并为向量运算，但触发条件苛刻，覆盖面窄。
- **Q：** 为什么不能手工循环展开？
  **A：** 手工展开会让 C2 无法识别并自动展开循环，从而无法进行自动向量化。应把展开交给 JIT。

## 27 | 注解处理器

**一句话核心**：注解处理器是接入 javac 的插件，在编译期处理注解，可定义编译规则、修改 AST（不推荐）或生成新源代码（如 Lombok、JMH）。

**关键知识点**
- 注解生命周期由 `@Retention` 决定：SOURCE（只在源码，如 `@Override`）、CLASS（源码+字节码）、RUNTIME（源码+字节码+运行时，可反射读取）；`@Target` 限定可标注的结构。
- Java 编译三步：①解析源文件生成抽象语法树（AST）；②调用已注册的注解处理器；③生成字节码。若第 2 步生成了新源文件，则重复第 1、2 步，每次重复称一"轮"（Round），直到不再生成新源码，再进入第 3 步。
- 所有注解处理器实现 `Processor` 接口，四个关键方法：`init`（初始化，因用反射创建实例故不用构造器、需无参构造器）、`getSupportedAnnotationTypes`、`getSupportedSourceVersion`、`process`（核心处理方法）。
- `AbstractProcessor` 抽象类已实现前三个方法，子类可用 `@SupportedAnnotationTypes`、`@SupportedSourceVersion` 声明，只需重写 `process`。
- 处理时操作各种 Element：`TypeElement`（类/接口）、`VariableElement`（字段/局部变量/enum 常量）、`ExecutableElement`（方法/构造器）、`PackageElement`（包），通过 `getEnclosedElements` 等 API 遍历。
- 注册方式两种：①`javac -processor 全限定类名`；②打进 jar 并在 `META-INF/services/javax.annotation.processing.Processor` 中声明，javac 自动发现注册。
- 三大用途：①定义编译规则并检查源文件；②修改已有源码（实为改 AST，用内部 API，随版本失效，不推荐）；③生成新源文件（常用，通过 `Filer.createSourceFile` + `PrintWriter` 写出）。jcstress、JMH 都用此生成测试代码。
- Lombok 原理：自定义注解 + 在编译期修改 AST 插入 getter/setter 等节点，使用了 Java 编译器内部 API，因此可能随 JDK 版本升级而失效。

**面试高频问答**
- **Q：** 注解处理器是什么时候、干什么的？
  **A：** 编译期（javac 编译第 2 步）以插件形式运行，处理源码中的注解，可校验编译规则、改 AST、或生成新源码。运行时通过反射解析注解则是另一条路径。
- **Q：** Lombok 的 @Getter/@Setter 原理？
  **A：** 通过注解处理器在编译期修改抽象语法树，插入对应方法节点，使生成的字节码带有 getter/setter。依赖内部 API，有版本兼容风险。
- **Q：** 编译期生成代码 vs 运行时字节码工具（如 CGLIB）区别？
  **A：** CGLIB 等运行时生成会影响启动性能，峰值性能无差别；注解处理器编译期生成对不熟字节码者更易用，但字节码工具能力更强。
- **Q：** 注解处理器生成新源码为什么不会死循环？
  **A：** 编译按"轮"进行，新生成的源码触发下一轮解析+处理，直到某一轮不再产生新源码就停止进入生成字节码阶段。

## 28 | 基准测试框架 JMH（上）

**一句话核心**：手写 benchmark 极易被 JIT 优化、操作系统和硬件因素误导；JMH 是 OpenJDK 出品、内置控制 JIT 优化能力的性能基准测试框架，用 `@Benchmark` 标注测试方法即可。

**关键知识点**
- 手写 benchmark 的坑（JVM 层）：循环优化会把 `while(i<10^9)i++` 直接优化成 `return 10^9`（基于计数循环，非循环展开）；解释执行与 JIT 代码混杂导致数据含义不明；冷循环未触发 OSR 编译。把 int 改 long 可"避免"该循环优化。
- 死代码消除 / 常量折叠：测试代码若结果没被使用，会被 JIT 当死代码消除掉；常量表达式会被常量折叠。
- 硬件/OS 层的坑：电源管理动态调 CPU 频率（温度升高降频）、CPU 缓存与数据本地性、false sharing（多线程写同一缓存行不同部分）、分支预测器、超线程（两线程排到同一物理核上性能差）。
- JMH 面向 ns/μs/ms/s 级测试，由很多 JIT 开发者参与，内置控制 JIT 优化和降低外部干扰的能力，让开发者只关注业务逻辑。但 JMH 仍不能完美消除偏差，输出会打印 "The numbers below are just data" 的忠告，不要轻信数据乱下结论。
- 生成项目：用 maven archetype（`jmh-java-benchmark-archetype`）生成模板，含 pom.xml 和 `MyBenchmark.java`。核心是 `@Benchmark` 注解，标注的方法就是测试方法。
- JMH 用注解处理器生成真正的测试代码：`mvn compile` 在 `target/generated-sources` 生成 `MyBenchmark_jmhType`（继承链 jmhType→B3→B2→B1→MyBenchmark，B1/B3 各填 256 个 boolean 字段防 false sharing）和 `MyBenchmark_testMethod_jmhTest`（真正被 OSR 编译的测试代码）。
- `mvn package` 打成可执行 `benchmarks.jar`（入口 `org.openjdk.jmh.Main`），含 BenchmarkList（测试配置）、CompilerHints（`-XX:CompileCommandFile` 内容，强制内联 testMethod、禁止内联 stub）。
- 结果关键指标：Score（平均吞吐量 ops/s）和 Error（误差范围）。

**面试高频问答**
- **Q：** 为什么手写 benchmark 不准？
  **A：** ①JIT 会做循环优化/死代码消除/常量折叠，把测试代码"优化没了"；②解释执行与编译代码混杂、未预热；③电源管理降频、CPU 缓存、false sharing、分支预测、超线程等硬件/OS 因素干扰。
- **Q：** 为什么 `while(i<10^9)i++` 计时得出的频率高得离谱？
  **A：** JIT 基于计数循环把整个循环优化成直接 `return 10^9`，根本没循环，所以耗时极短、看似频率超物理极限。
- **Q：** JMH 是什么？解决什么？
  **A：** OpenJDK 的 Java 微基准测试框架，内置控制 JIT 优化、降低 OS/硬件干扰的机制，用 `@Benchmark` 标注方法，注解处理器自动生成测试代码和配置，让开发者专注业务逻辑。

## 29 | 基准测试框架 JMH（下）

**一句话核心**：JMH 通过 @Fork 隔离虚拟机、@Warmup/@Measurement 控制预热与测量、@State/@Setup/@TearDown 管理状态、Blackhole 防死代码消除，来获得可靠的性能数据。

**关键知识点**
- `@Fork(n)`：Fork 出 n 个新 JVM 分别测试再平均。目的：获得干净 JVM 环境（避免类型 profile 污染影响内联、避免无关子类影响完全内联），并抵消 TLAB 大小、偏向/轻量锁、并发数据结构等不确定性带来的 run-to-run 差异。
- `@BenchmarkMode`：指定数据格式，如 `Mode.Throughput`（吞吐量 ops/s，默认）、`Mode.AverageTime`（单次操作平均耗时）等。
- 预热迭代（Warmup）vs 测试迭代（Measurement）：预热为把 JVM 带到稳定状态——不仅要 JIT 编译成机器码，还要各种自适配优化（TLAB 大小、Eden/Survivor/老年代大小）稳定。注意可能存在多个"中间稳定状态"（如 Nashorn 跑 JS），自动检测稳定态不可靠，需人工定次数。
- `@Warmup(iterations, time, timeUnit, batchSize)`：预热迭代次数、每次迭代时长、每次操作调用测试方法的次数。`@Measurement` 参数相同；测试迭代越多越精确。
- `@State`：标注程序状态类（需无参构造器、内部类须静态）。Scope：`Benchmark`（整个 JVM 共享）、`Thread`（线程私有）、`Group`（线程组私有，JMH 自定义概念）。状态类作为测试方法参数传入，由 JMH 负责构造注入。
- `@Setup`/`@TearDown`：测试前初始化/测试后校验，必须是状态类的方法。Level：`Trial`（整个测试前后）、`Iteration`（每次迭代前后）、`Invocation`（每次调用测试方法前后，会影响精度）。
- 即时编译控制：`@CompilerControl` 控制方法内联与否。
- **Blackhole**：`bh.consume(...)` 防止 JIT 把传入值当死代码消除；但不阻止对传入值本身的计算优化（如 `consume(3+4)` 仍会常量折叠成 7，但 7 不会被消除）。还有静态 `Blackhole.consumeCPU(long)` 消耗与参数线性相关的 CPU 时间。

**面试高频问答**
- **Q：** JMH 为什么要 Fork 新 JVM？
  **A：** 获得干净虚拟机环境，避免类型 profile 污染、无关子类影响内联决策，并通过多 Fork 平均抵消 TLAB、锁算法等不确定性，使误差更小。
- **Q：** 为什么要分预热迭代和测试迭代？
  **A：** 预热是把 JVM 带到稳定状态：JIT 编译完成 + 各种自适配优化（TLAB、堆各区大小）稳定后再记录数据才可靠。
- **Q：** Blackhole 的作用？能防常量折叠吗？
  **A：** `consume` 防止返回值/中间值被死代码消除；但不能阻止对参数表达式本身的常量折叠等计算优化。要避免常量折叠需让输入依赖运行时状态（如 @State）。
- **Q：** @State 有什么用？
  **A：** 定义测试所需的程序状态（如请求上下文），由 JMH 构造并作为参数注入测试方法；配合 @Setup/@TearDown 做初始化和校验，Scope 控制共享范围。

## 30 | JVM 监控诊断工具（命令行篇）

**一句话核心**：JDK 命令行工具各司其职——jps 列进程、jstat 看 GC/类加载/JIT 数据、jmap dump 堆、jstack dump 线程并检测死锁、jinfo 看改参数，jcmd 是能替代除 jstat 外全部工具的瑞士军刀。

**关键知识点**
- **jps**：列出所有 Java 进程（PID+主类名）。`-l` 打印模块/包名、`-v` 打印 JVM 参数、`-m` 打印传给主类的参数。关闭 `UsePerfData`（`-XX:-UsePerfData`）的进程 jps/jstat 探测不到。
- **jstat**：打印目标进程性能数据。子命令 `-class`（类加载）、`-compiler`/`-printcompilation`（即时编译）、`-gc*` 系列（垃圾回收）。可配 `interval count` 周期打印，`-t` 加进程启动时间列。
- jstat 用途：GCT/运行时间比值判断堆压力（>20% 压力大，>90% 随时 OOM）；连续取多组 OU（老年代已用）最小值，若持续上涨则可能内存泄漏。G1 下 S0C/S0U 为 0，Survivor 逻辑上只有一个（用 S1C/S1U 记总量）。
- **jmap**：分析堆对象。`-histo`（各类实例数/占用内存排序，`:live` 只算存活）、`-dump`（导出堆快照，常用 `jmap -dump:live,format=b,file=xxx.bin`，format=b 与 hprof/HeapDumpOnOutOfMemoryError 格式一致，可被 MAT 等 GUI 分析）、`-clstats`、`-finalizerinfo`。jmap 需到安全点，可能被长时间未到安全点的线程卡住等待；jstat 不会（GC 主动把摘要数据写到固定位置）。
- **jinfo**：查看目标进程的 `-X`/`-XX`/`-D` 参数；还能改 manageable 类参数，如 `jinfo -flag +HeapDumpAfterFullGC <PID>`。manageable 参数可用 `java -XX:+PrintFlagsFinal -version | grep manageable` 查看。
- **jstack**：打印各线程栈轨迹、线程状态、持有/请求的锁，并自动检测死锁（输出 "Found one Java-level deadlock"，标出互相等待的线程和锁）。
- **jcmd**：瑞士军刀，可替代除 jstat 外的所有命令，还含 JFR 相关子命令（下一讲）。
- jmap/jinfo/jstack/jcmd 依赖 JVM 的 Attach API，只能监控本地进程；开启 `-XX:+DisableAttachMechanism` 后这些命令失效（可用于防被监控）。

**面试高频问答**
- **Q：** 如何 dump 堆和线程？
  **A：** 堆：`jmap -dump:live,format=b,file=xxx.bin <PID>`（或 jcmd GC.heap_dump）；线程：`jstack <PID>`（或 jcmd Thread.print），format=b 的堆文件可用 MAT 分析。
- **Q：** 如何排查内存泄漏（命令行）？
  **A：** 用 jstat 连续多次取 OU（老年代已用）最小值，隔较长时间取多组，若持续上涨说明无法回收对象在增加，疑似泄漏；再用 `jmap -dump` 导出堆用 MAT 分析。
- **Q：** 如何排查死锁？
  **A：** `jstack <PID>`，它会打印各线程状态（BLOCKED）、持有和等待的锁，并自动报告 "Found N Java-level deadlock" 及互相等待关系。
- **Q：** jstat 和 jmap 谁会被卡住，为什么？
  **A：** jmap 要让所有线程停在安全点，若某线程长时间不到安全点会一直等待；jstat 直接读 GC 主动写入固定位置的摘要数据，不会卡。
- **Q：** 各工具一句话用途？
  **A：** jps 列 Java 进程、jstat 看类加载/JIT/GC 数据、jmap 统计并 dump 堆、jinfo 看改参数、jstack dump 线程查死锁、jcmd 万能替代（除 jstat）。

## 31 | JVM 监控诊断工具（GUI 篇）

**一句话核心**：eclipse MAT 解析堆快照（直方图 + 支配树）定位内存泄漏；JMC + JFR 以极低开销（默认 <1%）收集 JVM 运行事件，适合生产环境性能剖析。

**关键知识点**
- **eclipse MAT**：解析 jmap 导出的堆二进制快照（也能自己借 Attach API/jmap 抓取）。
- MAT 两种内存计量：Shallow heap（对象自身占用）；Retained heap（对象被回收时能连带释放的总内存 = 自身 + 仅经它可达的其他对象）。
- MAT 两大视图：直方图（histogram，各类实例数 + Shallow/Retained heap，可排序、可按超类/类加载器/包分组）；支配树（dominator tree，按 Retained heap 排列，快速找出"回收它能释放最多内存"的对象）。注意：引用字段关系 ≠ 支配树父子关系。
- 支配概念：从入口到 b 的所有路径都经过 a 则 a 支配 b；堆对象图以 GC Roots 为入口。选中对象可用 "Path To GC Roots" 反查引用链，用于定位泄漏根因。MAT 还能自动匹配常见泄漏模式并报告。
- **JMC（Java Mission Control）**：JVM 性能监控 GUI，含 JMX Console（访问 MXBeans）和 **JFR（Java Flight Recorder）**。JFR 开销极低（默认 <1%，profile 配置约 2%），直接访问 JVM 内部数据、不影响优化，适合生产满负荷环境。Java 11 起 JFR 开源（之前是 Commercial Feature，需 `-XX:+UnlockCommercialFeatures`）。
- JFR 记录 Java 层事件（线程、锁）和 JVM 内部事件（对象分配、GC、JIT）。四类事件：瞬时事件（关心是否发生，如异常/线程启动）、持续事件（关心时长，如 GC）、计时事件（超阈值的持续事件）、取样事件（周期取样，如方法抽样找热点方法）。
- JFR 方法抽样比基于 JVMTI GetAllStackTraces 的工具更精确，因为它不依赖安全点机制（后者栈轨迹总在安全点上，结论未必准）。
- JFR 三种启用方式：①`-XX:StartFlightRecording=`（可配 delay/duration/filename/settings/maxage/maxsize/dumponexit）；②`jcmd <PID> JFR.start/JFR.dump/JFR.stop`；③JMC 的 JFR 插件 GUI。配置文件 default.jfc（Continuous）和 profile.jfc（Profiling，事件更多开销更大）。

**面试高频问答**
- **Q：** Shallow heap 和 Retained heap 区别？
  **A：** Shallow 是对象自身占的内存；Retained 是回收该对象时能连带回收的总内存（自身 + 仅通过它可达的对象）。找泄漏看 Retained heap。
- **Q：** 如何用 MAT 排查内存泄漏？
  **A：** 加载 jmap dump 的堆快照，用直方图/支配树按 Retained heap 找占用最大的对象，再用 "Path To GC Roots" 反查引用链定位是谁一直持有它导致无法回收；MAT 还会自动报告可疑泄漏。
- **Q：** JFR 为什么适合生产环境？
  **A：** 开销极低（默认 <1%），直接访问 JVM 内部数据且不影响 JIT 优化；方法抽样不依赖安全点，结果比 JVMTI 类工具更精确。
- **Q：** 如何排查 CPU 飙高/找热点方法？
  **A：** 用 JFR 的方法抽样（Method Sampling）周期性统计各线程栈轨迹，反复出现的方法即热点方法；JMC 可视化编译/GC/线程等子系统事件辅助定位。
- **Q：** JFR 有哪几种启用方式？
  **A：** ①启动参数 `-XX:StartFlightRecording=`；②运行中用 `jcmd <PID> JFR.start/dump/stop`；③JMC GUI 的 JFR 插件。

## 32 | JNI的运行机制

**一句话核心**：JNI（Java Native Interface）是Java虚拟机让Java代码与C/C++等本地代码互相调用的机制，native方法通过命名规范或主动注册链接到C函数，代价是有额外的性能开销。

**关键知识点**
- native方法：标记为native、没有方法体的方法（如`Object.hashCode`、`wait`、`clone`），由JVM通过JNI转调对应的C函数实现。
- native方法两种链接方式：①按默认命名规范命名C函数，由JVM自动链接（`javac -h`可自动生成符合规范的头文件，前缀`Java_`+包名+方法名，重载时追加方法描述符）；②在C代码中用`RegisterNatives` API主动链接，对函数名无要求，通常放在`registerNatives`里并在类初始化时调用。
- C代码通过`System.loadLibrary`加载动态链接库（前缀`lib`，扩展名`.so`/`.dylib`），可用`-Djava.library.path`指定库路径。
- JNIEnv：聚合了所有JNI函数指针的线程私有数据结构，每个线程一份，不能跨线程共享（HotSpot中内嵌在Java线程结构里，甚至从其地址反推线程地址）。作用：提供命名空间、允许JVM替换JNI函数实现（如带/不带参数检查的快慢版本）。
- JNI把Java类型映射为C可用的数据结构：基本类型对应`jint`/`jlong`等；引用类型有继承关系（`jobject`为根，下有`jclass`/`jstring`/`jarray`等）。访问字段类似反射：先拿FieldID再取值。
- JNI异常处理特殊：JNI函数出错时JVM生成异常实例并缓存，但不会跳转，会继续执行后续C代码。必须用`ExceptionOccurred`检查、`ExceptionClear`清除，否则返回Java层才抛出且中间数据可能错误。
- 局部引用（Local Reference）与全局引用（Global Reference）：都能阻止GC回收被引用对象。局部引用（传入参数+大部分JNI函数返回值）在native方法返回后失效，不能缓存；需长期持有要用`NewGlobalRef`转全局引用，用`DeleteGlobalRef`释放。
- HotSpot用句柄（handle，指针的指针）实现引用，GC移动对象时只改句柄指向的指针值，句柄本身不变。局部引用句柄存于本地方法栈帧（参数）或线程私有句柄块（C中新建的）。
- JNI性能开销来源：进入C函数时对引用参数句柄化、调整参数位置（C与Java传参方式不同）、返回时清理线程私有句柄块。

**面试高频问答**
- **Q：** native方法是如何链接到C实现的？
  **A：** 两种方式。一是按JNI默认命名规范（`Java_`+全限定名，重载加描述符）命名C函数由JVM自动链接，`javac -h`可生成头文件；二是在C里用`RegisterNatives`主动注册，通常通过`registerNatives`方法在类静态初始化时完成。
- **Q：** JNI里的局部引用和全局引用有什么区别？
  **A：** 都能阻止GC回收对象。局部引用（传入参数、多数JNI函数返回值）在native方法返回后就失效，不能缓存或跨线程用；要长期持有须用`NewGlobalRef`升级为全局引用，并用`DeleteGlobalRef`手动释放。
- **Q：** 为什么JNI调用有性能开销？
  **A：** 进入C函数时要把引用类型参数句柄化、按C调用约定调整参数位置，返回时要清理线程私有句柄块；且访问对象都要通过句柄（多一层解引用）。这些额外工作构成开销，所以频繁的JNI调用应尽量减少或批量化。
- **Q：** JNIEnv为什么是线程私有的？
  **A：** 它内嵌在Java线程结构中，部分JVM代码会从其地址反推线程地址；跨线程共享会导致线程识别错误，破坏JNI函数正确性。同时它提供独立命名空间并允许JVM替换函数实现（快慢版本切换）。

## 33 | Java Agent与字节码注入

**一句话核心**：Java Agent通过premain/agentmain入口和Instrumentation API拦截类加载事件并修改字节码，是APM监控、热部署、AOP、profiler等工具的底层实现机制。

**关键知识点**
- 两种启动方式：①启动时`-javaagent:xxx.jar`，执行`premain`方法（在main之前），MANIFEST需配`Premain-Class`；②运行时通过Attach API（`VirtualMachine.attach(pid).loadAgent`）远程加载，执行`agentmain`方法，MANIFEST配`Agent-Class`。premain参数是单个String（不是String[]）。
- 可加载多个Agent，JVM按`-javaagent`定义顺序或attach顺序依次执行。
- Instrumentation机制：`premain`/`agentmain`可多接一个`Instrumentation`参数，用`addTransformer`注册`ClassFileTransformer`，重写`transform`方法拦截类加载。
- `transform`接收正在加载类的字节码byte数组，返回修改后的数组；返回null或抛异常则用原字节码。这就是字节码注入（bytecode instrumentation）的基础。
- 字节码工程框架：ASM（`ClassReader`/`ClassNode`/`MethodNode`/`ClassWriter`，tree包面向对象操作）、Javassist、ByteBuddy等，用于往被加载类里插入字节码。
- redefine与retransform（针对已加载的类）：redefine直接用用户提供的byte数组替换原字节码（危险，一般用于修复）；retransform重新调用所有已注册transformer。retransform两大用途：注入premain前已加载未拦截的类；移除部分注入后从原始字节码重新注入。
- 底层实现：Java Agent通过JVMTI（C agent）实现，JVMTI是事件驱动接口，在`Agent_OnLoad`入口注册事件钩子（如`ClassFileLoadHook`拦截类加载）。
- 基于字节码注入的profiler：在特定行为（如`new`字节码）周围注入对运行时类方法的调用来统计。三大坑：①注入JDK核心类可能造成死循环递归（如给`println`注入`println`），需用线程私有标识位区分应用/注入上下文；②命名空间冲突（Agent依赖的ASM版本与应用不同），需自定义类加载器隔离；③观察者效应——注入本身影响逃逸分析、方法内联决策，收集的数据只反映"被注入状态"而非真实运行状态。
- AOP（面向方面编程）：核心概念切入点pointcut、通知advice、连接点joinpoint。字节码注入是AOP的一种实现方式（如AspectJ）。

**面试高频问答**
- **Q：** premain和agentmain有什么区别？
  **A：** premain在main方法之前执行，通过启动参数`-javaagent`加载，MANIFEST配`Premain-Class`；agentmain通过Attach API在运行时动态加载到已启动的JVM，执行时机取决于attach时机，MANIFEST配`Agent-Class`。二者都可接收Instrumentation参数。
- **Q：** 字节码注入是怎么实现的？有哪些框架？
  **A：** 通过Instrumentation注册ClassFileTransformer拦截类加载，在transform里拿到类的byte数组、用ASM/Javassist/ByteBuddy修改后返回。底层依赖JVMTI的ClassFileLoadHook事件。retransform可对已加载类重新注入。
- **Q：** APM工具（如SkyWalking）为什么能无侵入监控？
  **A：** 用Java Agent在类加载时通过字节码注入，在目标方法入口/出口织入埋点代码收集调用链、耗时等，无需改动业务源码。这本质是AOP，切入点是方法，通知是埋点逻辑。
- **Q：** 用字节码注入做profiler有什么坑？
  **A：** ①注入JDK类易死循环（需线程私有标识位区分上下文）；②ASM等依赖库版本冲突（需自定义类加载器隔离命名空间）；③观察者效应，注入会干扰逃逸分析和内联，统计到本不会发生的操作，数据只反映被注入状态。

## 34 | Graal：用Java编译Java

**一句话核心**：Graal是一个用Java编写的即时编译器，通过JVMCI接口与JVM解耦交互，可替换C2，其激进的投机优化使编译结果性能（尤其Scala）优于C2。

**关键知识点**
- Graal是用Java写的JIT编译器，从Java 9起作为实验性即时编译器集成进JDK，是GraalVM的基石。
- 启用：`-XX:+UnlockExperimentalVMOptions -XX:+UseJVMCICompiler`，启用后替换C2响应编译请求。
- 即时编译器与JVM的三方面交互：①响应编译请求；②获取元数据（类/方法/字段）和profile；③把生成的机器码部署到code cache。
- JVMCI（JVM Compiler Interface）：把上述三个功能抽象成Java接口，让JVM与Graal解耦——JVMCI版本不变时，只需替换Graal的jar/jmod即可升级编译器，无需重编译整个JVM。JVMCI还允许Java程序直接调用Graal编译指定方法（Graal单元测试、Truffle都基于此）。
- Graal vs C2：Graal用Java写（模块化、易维护），C2用C++写；充分预热后二者峰值性能相当（相同优化则编译结果相同）；解释执行Graal只减慢编译速度不影响结果性能；C2新优化易移植到Graal，反之难（Graal的部分逃逸分析至今未移植到C2）。
- 性能：Java程序Graal优势不明显，Scala程序Graal性能优约10%（Twitter生产环境部署获约11%提升，EE版可达20%）。
- Graal实现：分前端（HIR，Sea-of-Nodes IR，平台无关优化如内联，由可配置的优化阶段HighTier/MidTier/LowTier组成）和后端（LIR，平台相关优化如寄存器分配+机器码生成）。
- 投机性优化：基于假设（assumption），假设失败时用去优化（deoptimization）切回解释执行，必要时废弃机器码重新收集profile再编译。Graal比C2更激进，支持自定义假设直接关联去优化节点。
- intrinsic方法：Graal在解析字节码时把匹配的方法调用替换为内部方法或特殊节点（如`Arrays.equals`替换为数组比较特殊节点），简化图、利于其他优化。
- 启动性能问题：Graal本身也需被即时编译，会抢占应用编译资源，所以启动性能较差（解决方案见SubstrateVM篇AOT编译Graal）。

**面试高频问答**
- **Q：** Graal编译器是什么？和C2有何区别？
  **A：** Graal是用Java写的JIT编译器，从Java 9起实验性集成，通过JVMCI与JVM交互，可替换C2。区别：Graal用Java（模块化易维护），C2用C++；充分预热后峰值性能相当，但Graal投机优化更激进，对Scala/lambda等新特性更友好，Scala程序快约10%。
- **Q：** JVMCI是什么？有什么作用？
  **A：** JVM Compiler Interface，把即时编译器与JVM交互的三大功能（响应编译请求、获取元数据/profile、部署机器码）抽象成Java接口，使Graal与JVM解耦，升级编译器只需换jar包无需重编JVM；还允许Java程序直接调用Graal编译方法。
- **Q：** 为什么用Java写的Graal不会比C++写的C2慢？
  **A：** 充分预热后Graal自身热点代码已被即时编译成机器码，执行速度不亚于静态编译的C++；即便解释执行Graal也只是减慢编译速度，不影响编译结果性能。相同优化手段下二者编译结果一致，峰值性能相同。

## 35 | Truffle：语言实现框架

**一句话核心**：Truffle是用Java写的语言实现框架，开发者只需实现AST解释器，靠Partial Evaluation和节点重写就能借Graal获得高性能，并支持零开销的多语言Polyglot互操作。

**关键知识点**
- 传统实现新语言：写完整编译器（前端词法/语法/类型检查+后端优化/代码生成），或编译成已知中间形式（如Scala/Kotlin→Java字节码复用JVM优化）；解释型语言（JS/Ruby/Python）则需解释器+虚拟机组件（JIT、GC）。
- Truffle目标：开发新语言只需用Java实现词法分析、语法分析、AST解释执行器，即可复用Truffle提供的即时编译、GC等运行时优化。基于Java故可跑在任意JVM上；跑在带Graal的JVM上时会调用Graal API主动触发对AST的即时编译。
- Oracle Labs已实现的Truffle语言：JavaScript(Graal.js)、Ruby、R(FastR)、Python、Sulong（解释LLVM bitcode，可跑C/C++）。
- Partial Evaluation（部分求值）：把程序P的输入分为编译时已知常量IS和未知ID，将`P:I->O`特化为`P':ID->O`。Truffle把解释器当P、把用Truffle语言写的程序（其AST）当作编译时常量IS，特化出代表该程序执行的Java代码，再交给Graal编译成机器码。过程中不断内联（直到遇到`@TruffleBoundary`注解），最终把AST解释执行内联成等价的直接Java运算。
- 节点重写（node rewriting）：动态语言变量类型运行时才确定（如`+`可能整数/浮点/字符串加）。Truffle解释器收集每个AST节点的类型profile，即时编译时按profile做特化（当成某种具体类型），类型不匹配时调Graal去优化API退回解释执行、重新收集类型再编译——本质是投机性优化。据统计JS/Ruby方法80%在5次调用后类型稳定，99%在19次后稳定。
- Polyglot（多语言）：同一段代码混用多种语言。Truffle语言间能共用对象（不复制、不序列化），通过Polyglot API解析外来对象，切换语言开销极小甚至零开销。

**面试高频问答**
- **Q：** Truffle是什么？它解决了什么问题？
  **A：** 用Java写的语言实现框架。开发者只需实现语言的词法/语法分析和AST解释器，就能复用Truffle+Graal提供的即时编译、GC等运行时优化，避免为每门新语言重复造JIT/GC轮子，同时获得高性能。
- **Q：** Partial Evaluation（部分求值）在Truffle里起什么作用？
  **A：** 把AST解释器当程序P、把待运行程序的AST当编译时常量，特化并内联出一段等价的Java代码（相当于该程序的专用执行逻辑），再由Graal即时编译成机器码，从而把"解释执行AST"变成"执行机器码"，实现高性能。
- **Q：** GraalVM的Polyglot为什么开销小？
  **A：** Truffle语言间通过Polyglot API直接共用内存中的同一对象，无需复制或序列化/反序列化，切换语言时几乎零开销。

## 36 | SubstrateVM：AOT编译框架

**一句话核心**：SubstrateVM是GraalVM的AOT编译框架，基于封闭世界假设把Java程序提前编译成带独立精简运行时的native image，实现极快启动和极低内存开销。

**关键知识点**
- AOT（Ahead-Of-Time）编译：在程序运行之前把字节码转成机器码（对立于运行时的JIT），产物可以是动态共享库或独立可执行文件。优点：无需运行时耗CPU做即时编译、启动即达理想性能；缺点：拿不到运行时信息，无法做基于类层次分析的完全虚方法内联或基于profile的投机优化，影响峰值性能。
- jaotc（Java 9实验性AOT工具）：借Graal把class编译成机器码存入动态共享库，启动时用`-XX:AOTLibrary`加载；机器码作用机理同JIT，可去优化回解释执行；用类指纹（class fingerprinting）保存类摘要，运行时比对，不匹配则舍弃AOT机器码。典型应用：AOT编译java.base（核心基础类）避免解释执行惩罚。
- SubstrateVM与jaotc的两点区别：①脱离HotSpot，拥有独立精简运行时（含异常处理、同步、线程管理、内存管理/GC、JNI）；②要求目标程序封闭（不能动态加载类库），基于封闭世界假设静态分析推算所有虚方法调用目标，把所有可能执行到的方法纳入编译，从而无需解释执行器。
- 两部分：native image generator（本身是Java程序，用Graal做真正的AOT编译逻辑）+ SubstrateVM运行时（精简运行时，AOT后的程序跑在其上）。
- 编译流程：native image generator用指针分析（points-to analysis）从程序入口出发探索所有可达代码，同时执行初始化代码，生成可执行文件时把已初始化的堆保存为堆快照（heap snapshot），使程序直接从目标程序开始运行，无需重复JVM初始化。
- 适用：主要用于JVM语言（Java/Scala/Kotlin）的AOT；Truffle语言实现本质是Java程序且用到的类编译时已知，也适合跑在SubstrateVM上（但不AOT编译用Truffle语言写的程序）。
- 启动时间/内存：Hello World对比——C程序<10ms/<500KB，HotSpot Java 40ms/24MB，SubstrateVM Java与C持平/约850KB。得益于堆快照+无需额外初始化。轻量特性适合嵌入其他系统（如嵌入Oracle数据库跑存储过程MLE）。
- 封闭世界假设的代价：反射、动态代理、JNI、资源加载等都需要额外配置（配置文件）告知，因为静态分析看不到运行时动态行为。
- Metropolis项目：OpenJDK的"Java-on-Java"目标（用Java开发JVM）。第一个子项目用SubstrateVM把整个Graal编译器AOT编译成机器码，使Graal运行时不再需要被即时编译、不抢占应用编译CPU资源、用独立堆空间不干扰应用堆，从而大幅提升启动性能。

**面试高频问答**
- **Q：** AOT编译和JIT编译有什么区别？各自优缺点？
  **A：** JIT在运行时把字节码编成机器码、可用运行时profile做投机优化（峰值性能高但启动慢、耗CPU）；AOT在运行前提前编译（启动即达理想性能、省CPU），但拿不到运行时信息，无法做完全虚方法内联和基于profile的投机优化，峰值性能可能受限。
- **Q：** GraalVM native image（SubstrateVM）为什么启动快、内存省？
  **A：** SubstrateVM有独立精简运行时，AOT编译时已通过指针分析把可达代码全部编译为机器码，并把初始化好的堆存成堆快照，程序直接从入口运行，省去了HotSpot执行main前的一系列初始化，所以启动时间和内存都远小于HotSpot（接近C程序水平）。
- **Q：** 什么是封闭世界假设？它带来什么限制？
  **A：** SubstrateVM假设目标程序是封闭的、不能动态加载类库，从而能静态分析出所有可能执行的方法全部AOT编译。代价是反射、动态代理、JNI、资源加载等运行时动态行为无法被静态分析发现，必须通过配置文件显式声明，否则运行时会失败。
- **Q：** SubstrateVM有独立运行时，包含哪些组件？
  **A：** 脱离HotSpot，自带异常处理、同步、线程管理、内存管理（垃圾回收）和JNI等组件，AOT编译后的程序跑在这个精简运行时之上。

## 附 | 常用工具介绍

**一句话核心**：javap反编译查阅字节码，Code Tools（ASMTools/JOL）分别用于字节码汇编反汇编和查看对象内存布局，ASM是广泛使用的字节码分析与修改框架。

**关键知识点**
- javap：把class文件反汇编成可读格式。常用选项：`-p`打印私有成员，`-v`打印全部信息，`-c`只看方法字节码，`-g`（编译时`javac -g`）保留局部变量表。
- javap -v 输出四大块：①基本信息（class版本号——Java 8为52、Java 10为54、访问权限、this/super类、字段/方法数等）；②常量池（存常量和符号引用，项间可互相引用）；③字段区域（类型描述符、访问权限，final静态基本/字符串字段还含常量值）；④方法区域（Code含操作数栈stack、局部变量locals、参数数args_size最大值，字节码带BCI偏移量，异常表Exception table用偏移量定位处理器范围/起点/异常类型any为任意，LineNumberTable源码到字节码映射，LocalVariableTable和StackMapTable为调试/验证信息）。
- 版本号不兼容：新JRE的javac编译的class在旧JRE上跑会抛`UnsupportedClassVersionError`。
- Code Tools项目小工具：ASMTools（字节码汇编/反汇编器`jdis`反汇编、`jasm`汇编，可手改字节码生成编译器无法直接产出的类，如把`iconst_1`改`iconst_2`）；JOL（`internals`/`estimates`查看对象内存分布，如`java.util.HashMap`）。
- ASM：字节码分析及修改框架，广泛用于Groovy/Kotlin编译器、Cobertura/JaCoCo覆盖率工具、Java 8 Lambda适配器类动态生成等。可生成新class或修改已有class。核心用访问者模式：`ClassReader`读原始字节转为访问请求，`ClassWriter`生成字节数组，中间插入自定义`ClassVisitor`/`MethodVisitor`委派转发实现增删改。辅助类`ASMifier`接收class输出生成它的代码，便于快速上手。

**面试高频问答**
- **Q：** 怎么查看一个类的字节码？javap常用选项？
  **A：** 用`javap`。`-c`看方法字节码，`-v`看全部信息（常量池、版本号、栈/局部变量、异常表等），`-p`包含私有成员，配合`javac -g`可保留局部变量表。
- **Q：** ASM是什么？用在哪里？
  **A：** ASM是字节码分析与修改框架，用访问者模式（ClassReader读→自定义Visitor转发增删改→ClassWriter写）操作字节码。广泛用于Groovy/Kotlin编译器、JaCoCo等覆盖率工具、字节码注入监控工具、Java 8 Lambda适配器类生成等。
- **Q：** JOL工具有什么用？
  **A：** JOL（Java Object Layout）用于查看JVM中对象的内存分布（对象头、字段排布、对齐填充等），常用于分析对象占用内存，命令如`java -jar jol-cli internals java.util.HashMap`。
