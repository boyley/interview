# 06 · AI 工程化落地 速答（AI Engineering）

> 面向 Java/后端程序员的 AI 应用**工程化落地**：框架选型、五大工程挑战治理（慢/贵/不确定/会瞎编/不安全）、可观测、测试、成本控制、内容审核。**核心主线：把 LLM 调用当成一个"慢、贵、不稳定、会瞎编的外部 RPC 依赖"，复用后端成熟的缓存/限流/熔断/降级/超时/可观测那套治理**——这是后端程序员相比算法岗最能体现价值的一篇。覆盖 ~16 个高频问。深度实现见 [`../../ai-learning`](../../ai-learning)（spring-ai-learning `15-model-evaluation`、`16-observability-testing`、`20-moderation`；框架对比文档）。

## 🔥 高频必背（Top 16）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | AI 应用的五大工程挑战？ | **慢、贵、不确定、会瞎编、不安全**——面试万能框架，任何 AI 功能都从这五点谈治理。 |
| 2 | 一句话概括后端做 AI 工程的思路？ | 把 LLM 当**慢/贵/不稳定的外部依赖（RPC）**，复用缓存/限流/熔断/降级/超时/可观测那套治理。 |
| 3 | Java 主流 AI 框架怎么选？ | **Spring 栈 → Spring AI**（官方、Starter 自动配置、ChatClient）；对标 Python LangChain / 声明式 → **LangChain4j**；要国产模型/多智能体 → **Spring AI Alibaba**。 |
| 4 | 慢怎么治？ | **流式 Streaming**（降首字延迟）+ 异步 + **超时控制** + 限 `max_tokens` + 简单任务走小模型。 |
| 5 | 贵怎么治？ | **语义缓存/结果缓存** + 限流 + **分级路由**（简单请求走便宜小模型）+ 精简 Prompt/RAG 只塞相关 + 限 `max_tokens` + 成本监控告警。 |
| 6 | 不确定（输出格式不稳）怎么治？ | **结构化输出**（要 JSON）+ **反序列化校验** + **失败重试** + 低 temperature。 |
| 7 | 会瞎编（幻觉）怎么治？ | **RAG 提供事实依据** + 引用溯源 + **事实性评估**兜底 + 低温度。详见 [`03-rag`](03-rag.md)。 |
| 8 | 不安全怎么治？ | **Prompt 注入防护** + **内容审核 Moderation** + **护栏 Guardrails** + 工具权限最小化 + **PII 脱敏**。 |
| 9 | 什么是语义缓存（Semantic Cache）？ | 用 **Embedding 相似度**匹配"语义相同但措辞不同"的历史问答，命中就不调模型——比精确匹配缓存命中率高得多，省钱提速。 |
| 10 | 什么是分级路由（Model Routing）？ | 按任务难度**分流到不同模型**：简单/高频请求走便宜小模型，复杂请求才上大模型——省钱大杀器。 |
| 11 | AI 应用怎么做可观测（Observability）？ | 把 LLM 当外部依赖监控：**Token 用量、延迟、成功率、成本**四大指标，Micrometer/Prometheus + 分布式 tracing。 |
| 12 | AI 输出不确定，怎么写测试？ | **不做精确匹配**，改断言"**包含关键点 / 格式正确 / 不含违规**"；质量评估用 **LLM-as-a-Judge**；维护**金标准集回归**。 |
| 13 | 什么是 LLM-as-a-Judge？ | **用另一个（通常更强的）模型给输出打分**（相关性/事实性/有害性），把主观质量评估自动化。 |
| 14 | 模型挂了/超时怎么办？ | 当外部依赖做：**超时 + 重试（注意幂等）+ 熔断降级**（返回兜底/缓存）+ **故障转移**（多模型/多供应商备份）。 |
| 15 | 怎么防限流（RPM/TPM 配额）被打爆？ | 客户端限流（令牌桶）+ 请求排队 + 分级路由分散压力 + 多 Key/多供应商负载均衡。 |
| 16 | 内容审核（Moderation）怎么做？ | **输入输出双向过滤**：入口拦 Prompt 注入/违规输入，出口过有害/敏感/PII 内容，用 Moderation API 或规则护栏。 |

## 🧩 五大工程挑战 × 治理手段（必画一张对照表）

> 面试官问"你怎么把 AI 做上生产？"——直接甩这张表，每一行都能展开，立刻显得做过真实项目。

| 工程挑战 | 具体表现 | 工程治理手段（后端视角） |
|---|---|---|
| 🐢 **慢** | 自回归逐 Token 生成，输出越长越慢；首字延迟高 | **流式 Streaming**（SSE 打字机、降首字延迟）+ 异步化 + **超时控制**（别让线程干等）+ 限 `max_tokens` + 简单任务分流小模型 |
| 💰 **贵** | 按 Token 计费（输入+输出），高并发下成本爆炸 | **语义缓存/结果缓存**（命中不调模型）+ **分级路由**（小模型扛简单请求）+ 精简 Prompt/RAG 只塞相关片段 + 限 `max_tokens` + **成本监控告警** |
| 🎲 **不确定** | 同样输入输出不同、格式漂移、JSON 解析失败 | **结构化输出**（Spring AI `entity()`）+ **反序列化校验** + **失败重试** + 低 `temperature`（抽取/分类场景） |
| 🤥 **会瞎编** | 幻觉：编造假事实/假 API/假引用 | **RAG 提供事实依据**（最有效）+ **引用溯源** + **事实性评估**（LLM-as-a-Judge / 规则）兜底 + 低温度 + "不知道就说不知道" |
| 🔓 **不安全** | Prompt 注入、越权、有害内容、泄露 PII | **Prompt 注入防护**（隔离用户输入）+ **内容审核 Moderation**（输入输出双向）+ **护栏 Guardrails** + **工具权限最小化** + **PII 脱敏** |

> 记忆口诀：**慢→流式、贵→缓存分流、乱→结构化校验、编→RAG评估、险→审核护栏**。

## 📌 展开速答

**Q：Spring AI、LangChain4j、Spring AI Alibaba 到底怎么选？**
三者都是 Java 生态调 LLM 的框架，定位不同：
- **Spring AI**：Spring 官方出品，**Starter 自动配置**、`ChatClient` 流式 API、Advisor（拦截器）、与 Spring Boot/生态无缝。**你是 Spring 栈就选它**，学习成本最低、最"Spring 味"。
- **LangChain4j**：对标 Python 的 LangChain，**原生 API 更贴近底层**，`@AiService` **声明式**（像 Feign 一样定义接口就能调模型）。想要 LangChain 那套心智模型、或非 Spring 项目可选。
- **Spring AI Alibaba**：阿里出品，深度集成 **DashScope/通义千问**等国产模型，主打 **Graph 多智能体**编排。**要用国产大模型、或做多智能体系统**时选它。
一句话选型：**Spring 栈默认 Spring AI；要多智能体/国产模型上 Alibaba；喜欢声明式/LangChain 风格用 LangChain4j。**

**Q：为什么说"把 LLM 当外部依赖治理"是后端程序员的最大加分点？**
因为 LLM 调用本质就是一次**慢、贵、可能失败、结果不稳定的远程调用**——这和后端天天治理的 RPC/HTTP 依赖是同一类问题。你熟悉的那套高可用手段可以**原样复用**：超时（别让线程无限等）、重试（注意幂等）、熔断降级（模型挂了返回缓存/兜底文案）、限流（对齐供应商 RPM/TPM 配额）、故障转移（主模型挂了切备用模型/供应商）、可观测（当外部依赖埋点）。**答题时把这些后端术语和 AI 场景对应上，立刻和只会"调 API 跑 Demo"的候选人拉开差距。**（同一套思路见系统设计库 [`../03-system-design/06-high-availability.md`](../03-system-design/06-high-availability.md)、[`07-high-concurrency.md`](../03-system-design/07-high-concurrency.md)。）

**Q：AI 应用的成本控制，成体系地怎么答？**
分四层往下压，能背出这套就显得做过真实项目的成本优化：
1. **压次数**——**语义缓存/结果缓存**：高频重复问答直接命中，不调模型；语义缓存比精确匹配命中率高得多。
2. **压模型单价**——**分级路由**：简单/高频请求走便宜小模型，复杂任务才上大模型；甚至先用规则/小模型判定难度再路由。
3. **压 Token**——精简 Prompt、**RAG 只塞相关片段别塞全文**、限制 `max_tokens`、压缩历史。
4. **可观测兜底**——**Token 用量 + 成本按用户/接口维度统计 + 超预算告警**，避免某个功能悄悄烧钱。
`成本 ≈ (输入Token + 输出Token) × 调用次数 × 单价`，四层正好对应式子里的每个因子。详见 [`01-llm-basics`](01-llm-basics.md) 成本估算。

**Q：AI 输出不确定，这功能到底怎么测？**
核心思路：**放弃精确匹配（`assertEquals` 整段文本必挂），改测"性质"**：
- **格式断言**：输出能成功反序列化成目标 JSON / schema 校验通过。
- **包含断言**：输出**包含关键点**（关键实体、必答要素），或**不包含违规内容**（敏感词、拒答标志）。
- **LLM-as-a-Judge**：主观质量（相关性、事实性、语气）交给**另一个更强的模型打分**，设阈值判定通过。
- **金标准集回归**：维护一批"问题→期望要点"样本，改 Prompt/换模型后**跑回归对比通过率**，防止改一处崩一片。
- **控制随机性**：测试时 `temperature=0` 让输出尽量稳定，降低 flaky。
深度见 spring-ai-learning `15-model-evaluation`、`16-observability-testing`。

**Q：可观测（Observability）具体监控哪些指标、用什么？**
把 LLM 调用当一个**外部依赖 span** 埋点，四大类指标：
- **Token 用量**：输入/输出 Token 数（直接关联成本，按接口/用户维度聚合）。
- **延迟**：首字延迟（TTFT）+ 总耗时（P95/P99，流式尤其关注首字）。
- **成功率/错误率**：超时、限流 429、内容审核拦截、解析失败各自的比例。
- **成本**：Token × 单价折算，做**预算告警**。
技术栈：**Micrometer + Prometheus + Grafana**（指标）、**分布式 tracing**（把 LLM 调用串进请求链路，Spring AI 内置 Observation 支持）。一句话：**你怎么监控数据库/下游服务，就怎么监控 LLM。**

**Q：Prompt 注入、内容审核、护栏，安全这块怎么答全？**
分**输入侧**和**输出侧**双向防护：
- **输入侧**：**Prompt 注入防护**——把用户输入和系统指令**明确隔离**（分隔符/角色分离/输入不拼进 System），警惕"忽略以上指令"类攻击；入口做**内容审核**拦违规输入。详见 [`02-prompt-engineering`](02-prompt-engineering.md) 注入防护。
- **输出侧**：**Moderation** 过滤有害/敏感内容；**PII 脱敏**（手机号/身份证/邮箱返回前打码）；**护栏 Guardrails** 校验输出是否越界（是否泄露系统 Prompt、是否答了不该答的）。
- **工具/Agent 侧**：**工具权限最小化**——给 Agent 的工具只开必要权限（只读优先、危险操作要人工确认），防止被诱导执行越权操作。详见 [`05-agent`](05-agent.md) 兜底。
深度见 spring-ai-learning `20-moderation`。

**Q：语义缓存和普通缓存有什么区别？为什么 AI 场景要用它？**
普通缓存按 **key 精确匹配**（"北京天气"≠"北京今天天气怎么样"，两次都会调模型）。**语义缓存**把问题转成 **Embedding 向量**，用**相似度检索**历史问答，语义相近就命中——大幅提高命中率、省钱提速。代价：有"语义相近但答案实则不同"的**误命中风险**，所以要设**相似度阈值**，对**实时性强/个性化强**的请求（如"我的订单""今天股价"）要**跳过缓存**。本质是"带索引的相似度查询"，和向量检索同一套底层，见 [`07-vector-embedding`](07-vector-embedding.md)。

**Q：熔断降级在 AI 场景具体降成什么？**
LLM 供应商抖动/限流/宕机时不能让整个功能崩，降级策略从优到劣：①**切备用模型/供应商**（故障转移，多供应商是 AI 高可用标配）②**返回缓存/语义缓存的近似答案** ③**降级到规则/模板兜底回复**（"当前咨询量大，请稍后重试"）④**跳过 AI 增强、回退到无 AI 的基础功能**。关键是**别让 LLM 成为单点**——它是你系统里最不稳定的那个依赖。

## ⚠️ 易错 / 反问加分

- ✅ **加分（最强）**：全程用"**把 LLM 当慢/贵/不稳定的外部 RPC 依赖**"贯穿，把缓存/限流/熔断/降级/超时/可观测这些后端术语和 AI 场景一一对应——这是后端程序员答 AI 工程题的天花板。
- ✅ **加分**：五大挑战×治理那张表能默画出来，任何 AI 功能都能套这个框架谈落地。
- ✅ **加分**：成本控制答出"**缓存 + 分级路由 + 精简上下文 + 限输出 + 监控告警**"成体系，而不是只说一句"加缓存"。
- ✅ **加分**：测试题主动提 **LLM-as-a-Judge + 金标准集回归**，比只说"断言包含关键词"高一档。
- ⚠️ **坑**：别说"AI 输出没法测所以不测"——要答"**测性质不测精确文本**"，否则暴露没落地过。
- ⚠️ **坑**：重试要提一句"**注意幂等**"——LLM 调用重试没有副作用，但若重试的是"下单/发消息"类**带工具调用**的请求，重复执行会出事。
- ⚠️ **坑**：语义缓存别忘了说**误命中风险和跳过缓存的场景**（个性化/实时请求），只说"能省钱"会被追问穿。
- ⚠️ **坑**：框架选型别只背名字，要能说**各自定位和选型建议**（Spring 栈选 Spring AI、多智能体/国产模型选 Alibaba、声明式选 LangChain4j）。
