# 04 · 工具调用 & MCP 速答（Function Calling & MCP）

> 面向 Java/后端程序员：让大模型**跳出"只会聊天"**，去查库、调 API、算数、读实时数据的核心能力。覆盖 ~15 个高频问：Function Calling 本质 / 工具定义 / 多工具与多轮编排 / MCP 协议 / Client-Server 架构 / 选错工具与安全兜底。深度实现见 [`../../ai-learning/spring-ai-learning`](../../ai-learning/spring-ai-learning)（`08-tool-calling`、`14-mcp`、`19-mcp-client`）。

## 🔥 高频必背（Top 14）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | Function/Tool Calling 是什么？ | 让模型**根据工具定义，输出"该调哪个函数+参数"的结构化 JSON**；模型**不执行代码**，由你的程序执行后把结果喂回模型再生成答复。 |
| 2 | 模型真的会执行代码吗？ | **不会**。模型只做"决策"——决定调什么、传什么参；真正的执行（查库/调 API）是**你的程序干的**。这是最爱考的点。 |
| 3 | 为什么需要工具调用？ | 补模型三大短板：**不知道实时信息**（天气/股价/库存）、**不会精确计算**、**不能操作外部系统**（下单/发邮件/查数据库）。 |
| 4 | 一个工具怎么定义？ | 三件套：**名称 + 描述（给谁用、干什么） + 参数 JSON Schema**。模型靠这些"望文生义"选工具，描述写烂就选错。 |
| 5 | 工具描述为什么关键？ | 模型**只能看文字描述来判断**该不该调、怎么调；描述含糊/重叠 → 选错工具或不调。相当于给同事写清楚的接口文档。 |
| 6 | 什么是多轮工具调用？ | 一次任务可能要调**好几次**：模型调 A→拿结果→发现还需调 B→再拿结果→才生成最终答复。是**多轮循环**，不是一次性。 |
| 7 | 并行工具调用（Parallel）？ | 模型一轮里可**同时请求多个互不依赖的工具**（如同时查天气+查汇率），程序并发执行后一起回传，省往返。 |
| 8 | 多工具怎么选？ | 你把一批工具的 schema 一起给模型，模型**按描述匹配意图**自主选 0/1/多个；工具太多会"选择困难"，需精简或分组。 |
| 9 | MCP 是什么？ | **Model Context Protocol**，Anthropic 提出的**标准协议**，规范"模型/Agent 如何接入外部工具、数据、资源"。类比 **USB-C / JDBC**——统一接口，工具可插拔。 |
| 10 | MCP 解决什么痛点？ | 没有它，每个模型/框架都要**为每个工具重写一遍集成**（N×M）。MCP 定标准后，工具**写一次，处处可用**，跨模型跨框架复用。 |
| 11 | MCP 架构长啥样？ | **MCP Client**（宿主应用/Agent 侧）↔ **MCP Server**（暴露工具/资源的一方），走标准协议通信。Server≈微服务对外暴露 API。 |
| 12 | Function Calling 和 MCP 什么关系？ | **互补不冲突**：Function Calling 是"模型级"能力（模型决定调用）；MCP 是"集成级"协议（工具如何被标准暴露/接入）。MCP Server 的工具**最终仍靠 Function Calling 被调**。 |
| 13 | 模型编造了不存在的参数怎么办？ | **参数幻觉**。靠 **JSON Schema 强约束 + 服务端二次校验（非法参数直接拒）+ 必填/枚举限制 + 让模型缺参时反问**，别裸信模型给的参数。 |
| 14 | 工具执行失败/超时怎么兜底？ | 给模型**结构化错误回传**（让它重试或换方案）+ 设超时/重试上限 + 降级返回兜底话术 + 最大工具调用轮数，防死循环。 |

## 🔄 Function Calling 工作流程（必画）

> 一句话主线：**模型只"点单"（决定调什么、传什么参），厨房（你的程序）才"炒菜"（真正执行）**。

```
用户提问：「北京现在天气怎么样？」
        │
        ▼
┌──────────────────────────────────────────┐
│  你的程序 → LLM                            │
│  发送：用户问题 + 可用工具定义列表          │
│  工具定义 = 名称 + 描述 + 参数 JSON Schema │
└──────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────┐
│  LLM 判断：需要调工具（不是直接答）         │
│  ★ 不执行代码，只返回结构化 JSON：          │
│    { "tool": "getWeather",                 │
│      "args": { "city": "北京" } }          │
└──────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────┐
│  ★ 你的程序真正执行 getWeather("北京")     │
│    → 调气象 API → 拿到 "晴, 28℃"           │
└──────────────────────────────────────────┘
        │  把工具结果作为一条消息回传 LLM
        ▼
┌──────────────────────────────────────────┐
│  LLM 拿到结果，生成自然语言最终答复：       │
│  「北京现在晴，气温 28℃。」                 │
│  （若还缺信息 → 再发起下一轮工具调用，循环）│
└──────────────────────────────────────────┘
        │
        ▼
     返回用户
```

**记住三句话**：①模型不执行代码，只**决定调什么**；②真正执行的是**你的程序**；③结果要**回传模型**再生成答复，可**多轮循环**。后端类比：**模型帮你决定发哪个 RPC、传什么参，但 RPC 是你的服务真正发出去的**。

## 📌 展开速答

**Q：Function Calling 的本质到底是什么？一句话讲透。**
本质是**让模型把"自然语言意图"翻译成"结构化的函数调用指令"**。你事先把一批工具的 `名称/描述/参数 Schema` 告诉模型，模型读懂用户意图后，**不自己动手**，而是输出一段 JSON 说"该调 `getWeather`、参数 `city=北京`"。你的程序按这个 JSON 执行真实逻辑，再把结果回传，模型据此说人话。整个过程是**模型决策 + 程序执行**的多轮协作。后端视角：**它约等于让 AI 帮你决定发起哪个 RPC/接口调用**，schema 就是接口契约（≈OpenAPI/接口定义）。

**Q：工具（Tool）到底怎么定义？为什么描述这么重要？**
三要素：**名称**（唯一标识）、**描述**（这个工具是干什么的、什么场景用——给模型看的说明书）、**参数 JSON Schema**（每个参数名、类型、是否必填、枚举/取值范围、含义）。模型**没有源码、只能看这段文字**来判断"该不该调、调哪个、每个参数填什么"。所以描述必须**清晰、无歧义、工具间别重叠**——描述写烂，模型就选错工具或漏调。Spring AI 里用 `@Tool` 注解 + `@ToolParam` 描述参数，框架自动生成 Schema。这跟你给下游同事写清楚一份接口文档是一回事。

**Q：多轮工具调用和并行工具调用怎么区别？**
**多轮（串行/链式）**：任务有依赖，模型调 A 拿到结果后，才知道下一步要调 B（如先"查用户 ID"再"用 ID 查订单"），一轮轮循环直到信息够了才作答。**并行（Parallel）**：一轮里有多个**互不依赖**的工具需求（如同时"查北京天气"+"查上海天气"），模型一次性返回多个调用请求，程序**并发执行**后一起回传，减少往返轮次、降延迟。工程上要设**最大轮数上限**防止模型反复调工具陷入死循环。

**Q：MCP（Model Context Protocol）是什么？为什么类比 USB-C / JDBC？**
MCP 是 Anthropic 提出的**开放标准协议**，专门规范"**模型/Agent 如何接入外部工具、数据源、资源**"。没有它之前，每个模型（Claude/GPT/…）× 每个框架 × 每个工具，都要各写各的对接代码，是 **N×M 的重复集成地狱**。MCP 定了统一接口后，工具方按协议实现一次，任何支持 MCP 的宿主都能直接用——**工具可插拔、跨模型跨框架复用**。所以类比 **USB-C**（统一物理接口，任何设备都能插）、**JDBC/ODBC**（统一数据库访问接口，换数据库不用改上层代码）。一句话：**MCP 是"AI 接工具"的统一驱动标准**。

**Q：MCP 的架构？Client 和 Server 分别是谁？**
两个角色：**MCP Client**——在宿主应用/Agent 侧，负责发现和调用工具（如你的 Spring AI 应用、Claude Desktop）；**MCP Server**——由**暴露工具/数据/资源的一方**实现，把能力按 MCP 协议标准对外提供（如一个封装了"查数据库/操作文件系统/调内部 API"的 Server）。两者通过标准协议通信（stdio 本地进程 或 HTTP/SSE 远程）。后端类比：**MCP Server ≈ 一个对外暴露 API 的微服务，MCP Client ≈ 调它的消费方**，只不过接口格式由 MCP 协议统一规定。Spring AI 提供 MCP Client/Server 两侧的 starter。

**Q：Function Calling 和 MCP 是竞争关系吗？该用哪个？**
**不是竞争，是互补的两层**。Function Calling 是**模型级能力**——解决"模型怎么决定要调工具、输出什么调用指令"。MCP 是**集成级协议**——解决"工具怎么被标准化地暴露、发现、接入"。二者串起来：MCP Server 把工具按标准暴露 → MCP Client 拉到工具列表 → 交给模型 → **模型用 Function Calling 决定调哪个** → Client 执行（实际转发给 Server）→ 结果回传模型。所以**MCP Server 暴露的工具，最终仍是通过 Function Calling 被模型调用的**。选择上：临时/少量工具直接用框架的 `@Tool` 即可；要**跨应用复用、生态化、可插拔**就上 MCP。

**Q：工具调用典型能干什么？举几个业务例子。**
补上模型"不知道实时、不会精算、不能操作外部系统"的短板：**查数据库**（"查一下用户 A 最近订单"）、**调内部/第三方 API**（查物流、查库存、发短信/邮件）、**实时信息**（天气、股价、汇率、新闻）、**精确计算**（模型算数会错，交给代码算）、**执行动作**（创建工单、下单、改配置）。这也是 **Agent 能"干活"的底座**——Agent 就是靠反复的工具调用来完成多步任务，见 [`05-agent`](05-agent.md)。

## ⚠️ 易错 / 反问加分

- ⚠️ **别说"模型自己执行了函数/调了 API"**——模型**只输出该调什么的 JSON，执行永远是你的程序**。答错这条直接暴露没做过工具调用。
- ⚠️ **别把 MCP 说成"另一种 Function Calling"或"取代 Function Calling"**——它是**协议/集成标准**，工具最终还是靠 Function Calling 被模型调。二者是不同层次、互补。
- ⚠️ **参数幻觉要主动提**：模型会**编造不存在或非法的参数**。兜底=JSON Schema 强约束 + **服务端二次校验拒绝非法参数** + 枚举/必填限制 + 缺参让模型反问，别裸信。
- ⚠️ **工具选错常因描述烂**：工具描述含糊或功能重叠，模型就选错/漏调。优化描述、精简工具数量、按场景分组；**工具太多本身会降低选择准确率**。
- ✅ **加分——安全权限最小化**：工具是模型伸向真实系统的手，**绝不能给"删库/转账/删文件"这类高危工具裸奔权限**。要做**权限最小化、危险操作加人工确认、参数白名单校验、审计日志**。（见 [`06-ai-engineering`](06-ai-engineering.md) 安全与兜底）
- ✅ **加分——失败兜底成体系**：工具超时/报错时，**把结构化错误回传模型**让它重试或换方案，同时设**重试上限 + 最大工具轮数 + 降级话术**，防止模型反复调工具烧钱/死循环。
- ✅ **加分——映射回后端熟词**：Function Calling≈**让 AI 发起 RPC**、工具 Schema≈**接口定义/OpenAPI 契约**、MCP≈**JDBC/USB-C 统一驱动标准**、MCP Server≈**对外暴露 API 的微服务**。能这样类比，面试官立刻觉得你"用后端脑子理解了 AI"。
