# 07 · 向量与向量库 速答（Embedding & Vector Store）

> 面向 Java/后端程序员的**向量语义检索基础**：Embedding、维度、相似度度量、向量库选型、ANN 索引、混合检索。覆盖 ~14 个高频问，全部「能背出口」。深度实现见 [`../../ai-learning/spring-ai-learning`](../../ai-learning/spring-ai-learning)（`09-embedding`、`10-vector-store`）。

## 🔥 高频必背（Top 14）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | 什么是 Embedding（向量）？ | 用模型把**文本/图片编码成固定长度的高维浮点向量**（如 1536 维），语义越相近向量距离越近；类比后端"把对象编码成可比较的特征值"。是 RAG 语义检索的地基。 |
| 2 | 向量维度（Dimension）是什么？ | 一个向量里浮点数的个数（如 OpenAI `text-embedding-3-small` = 1536 维）；**维度由 embedding 模型固定决定**，越高表达力越强但越占存储和计算。 |
| 3 | Embedding 模型怎么选？ | 看**维度/效果/价格/语种**权衡；铁律：**同一套数据的入库和查询必须用同一个 embedding 模型**，换模型维度和语义空间都变，向量不可比。 |
| 4 | 怎么衡量两个向量像不像？ | 三种度量：**余弦相似度 Cosine（最常用，看方向）**、点积 Dot product、欧氏距离 L2（看直线距离）。 |
| 5 | 为什么 Cosine 最常用？ | 它**只看向量方向不看长度**，不受文本长短/向量模长影响，纯比"语义方向"；本质是**归一化后的点积**（向量都归一化后 cosine=dot=L2 单调等价）。 |
| 6 | 什么是向量数据库？ | 专门**存海量向量 + 高效做相似度 Top-K 检索**的存储；类比"带相似度索引的专用数据库"。 |
| 7 | 为什么不能暴力全量比对？ | 全量算相似度是 O(N)，百万级向量每次查询都慢到不可用；所以要用**ANN 近似最近邻索引**，牺牲一点精度换几个数量级的速度。 |
| 8 | 什么是 ANN？主流索引有哪些？ | 近似最近邻（Approximate Nearest Neighbor）：不保证找到绝对最近，但极快。主流 **HNSW（图索引，最主流、快、精度高）、IVF（倒排+聚类）**；类比"用索引换精确性的近似查询"。 |
| 9 | 向量库 vs 传统数据库？ | 传统库是**精确匹配**（`=`、`like`、B+树查确定值）；向量库是**语义近似匹配**（相似度 Top-K + ANN 索引）；一个查"值相等"，一个查"意思相近"，互补常混用。 |
| 10 | 向量库选型对比？ | **pgvector**（PG 扩展，中小量/已有 PG 首选）、**Milvus**（专业级、海量高性能）、**Redis**（RediSearch，已有 Redis 顺手）、**Chroma**（轻量/原型）、**Elasticsearch**（混合检索强）、内存版（学习/小规模）。 |
| 11 | 什么是混合检索 Hybrid Search？ | **向量检索（语义相近）+ 关键词 BM25（精确命中词）** 结果融合，召回更全，兼顾"意思对"和"关键词命中"（细节见 03-rag）。 |
| 12 | Top-K 和相似度阈值是什么？ | Top-K = 返回最相似的前 K 条；阈值 = 相似度低于某值就丢弃（防召回垃圾）。两者都要**按业务调参**，平衡召回率 vs 精确率。 |
| 13 | 什么是归一化（Normalize）？ | 把向量缩放到单位长度（模=1）；归一化后点积就等于 cosine，很多库靠它统一度量、加速计算。 |
| 14 | 换了 embedding 模型要做什么？ | **全量重新 embedding + 重建索引**；新旧模型维度/语义空间不同，老向量作废，不能混着查。 |

## 📌 展开速答

**Q：Embedding 到底解决了什么问题？用后端的话怎么理解？**
计算机没法直接比较两段文本"意思像不像"。Embedding 就是用模型把文本**编码成一个固定长度的高维浮点数组**（向量），让"语义相似"变成"向量在空间里距离近"这个可计算的数学问题。类比后端里给对象算 hash / 提取特征做比较——只不过这里的"特征"是模型学出来的语义特征，相近的内容自然聚在一起。有了它，才能做"按意思检索"而不是"按关键词检索"，这是 RAG 的第一步。

**Q：Cosine、Dot、L2 三种度量怎么选？**
最常用 **Cosine（余弦）**，因为它只看方向不看长度——一段短文和一段长文只要意思一致，方向就接近，不会因为文本长短把结果带偏。**Dot（点积）** 同时受方向和模长影响，适合模长本身有意义的场景。**L2（欧氏距离）** 看两点直线距离。实践中很多库先把向量**归一化**成单位长度，这时 Cosine、Dot、L2 排序结果等价，选哪个主要看库支持和习惯。记住一句：**Cosine = 归一化后的点积**。

**Q：ANN / HNSW 为什么必须要？原理一句话？**
向量检索本质是"在 N 个向量里找和查询最像的 K 个"。暴力算是 O(N)，百万级就扛不住。**ANN（近似最近邻）** 放弃"绝对精确"，用索引结构把搜索范围缩小到极小一块，速度提升几个数量级，代价是偶尔漏掉真正的最近邻（召回率<100%）。**HNSW** 是主流：一种**多层图索引**，像跳表一样从粗到细逐层逼近目标，快且精度高，但吃内存。**IVF** 先把向量聚类分桶，查询只搜最近的几个桶。核心权衡就是**精度 vs 速度 vs 内存**三选二。

**Q：pgvector、Milvus、Redis、Chroma、ES 分别什么时候用？**
- **pgvector**：Postgres 扩展，数据量中小、已经在用 PG——首选，省一套中间件、能和业务表 JOIN。
- **Milvus**：专业向量库，海量向量 + 高性能 + 分布式，数据量大/QPS 高时上。
- **Redis（RediSearch）**：已有 Redis、想低延迟、量不大时顺手用。
- **Chroma**：轻量、装即用，原型和小项目 / 本地开发。
- **Elasticsearch**：本来就用 ES 做全文检索，且要**混合检索**（BM25+向量）时天然合适。
- **内存向量库**：学习、小规模、Demo；进程重启数据没了，别上生产。
一句话选型逻辑：**已有什么就优先复用（PG/Redis/ES），量真大再上专业库（Milvus）。**

**Q：向量库和 MySQL 到底区别在哪？会不会替代？**
不替代，是**互补**。MySQL 靠 B+树做**精确/范围匹配**（`id=`、`age between`），查的是"值相等或落在区间"；向量库靠 ANN 索引做**语义近似匹配**，查的是"意思最接近的 Top-K"，结果带相似度分数、不保证绝对精确。真实系统常**混着用**：先用传统条件过滤（租户、时间、类目），再在候选集里做向量相似检索，这叫带元数据过滤的向量检索。

**Q：向量检索里怎么平衡召回率和精确率？**
两个旋钮：**Top-K** 调大→召回更多但可能混进不相关；**相似度阈值** 调高→更精确但可能漏掉。加上 **ANN 索引参数**（如 HNSW 的 `ef_search`）也在拿精度换速度。工程做法：先给合理 Top-K（如 5~10）+ 阈值兜底过滤，再用 **Rerank 重排**或**混合检索**补召回质量（见 [`03-rag`](03-rag.md)）。能说出"召回率 vs 精确率要按业务调、不是固定值"就显得做过。

## ⚠️ 易错 / 反问加分

- ⚠️ **别混用 embedding 模型**——入库和查询、甚至同一批数据换了模型，向量就不可比；换模型必须**全量重建索引**。这是最常见的生产事故。
- ⚠️ **别说"向量库能替代 MySQL"**——一个精确匹配一个语义近似，是互补关系；答成替代直接暴露没理解。
- ⚠️ **维度必须一致**——库里向量和查询向量维度不同直接报错；1536 的模型出来的向量塞不进给 768 建的表。
- ✅ **加分**：把 Cosine 说成"归一化后的点积"，并点明"归一化后三种度量排序等价"，比只背"cosine 看夹角"专业。
- ✅ **加分**：主动讲 ANN 的**精度 vs 速度 vs 内存**权衡，并点名 HNSW 是图索引、吃内存但快，显示你调过参。
- ✅ **加分**：谈选型时先说"优先复用已有的 PG/Redis/ES，量大再上 Milvus"，体现工程 trade-off 而非堆名词。
- ✅ **加分**：提一句**带元数据过滤的向量检索**（先传统条件过滤再向量召回）+ **混合检索**，把"会调向量库"升级成"会设计检索链路"。
