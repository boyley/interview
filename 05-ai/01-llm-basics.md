# 01 · LLM 应用基础 速答（LLM Basics for Developers）

> 面向 Java/后端程序员的大模型**应用基础**：Token、上下文窗口、采样参数、幻觉、流式、成本、多模态。覆盖 ~18 个高频问，全部「能背出口」。深度实现见 [`../../ai-learning/spring-ai-learning`](../../ai-learning/spring-ai-learning)。

## 🔥 高频必背（Top 15）

| # | 问题 | 一句话答 |
|---|---|---|
| 1 | 什么是 Token？ | 模型处理文本的**最小单位**（不是字/词，是子词片段）；一个汉字≈1~2 Token，英文单词≈1~1.5 Token；**计费和限长都按 Token 算**。 |
| 2 | 什么是上下文窗口（Context Window）？ | 模型一次能"看到"的**最大 Token 数**（输入+输出合计），如 128K；超了最早的内容会被截断/放不进。 |
| 3 | 上下文窗口满了怎么办？ | ①滑动窗口只留最近 N 轮 ②**摘要压缩**历史 ③**RAG** 把长知识外置检索 ④关键信息前置。 |
| 4 | temperature（温度）是什么？ | 控制**随机性/创造性**：低（0~0.3）更确定、适合抽取/分类/代码；高（0.7~1）更发散、适合创意写作。 |
| 5 | Top-p / Top-k 是什么？ | 另一种采样控制：Top-p（核采样）只从累积概率前 p 的词里选；Top-k 只从概率最高的 k 个里选。一般调 temperature 或 Top-p 二选一。 |
| 6 | 什么是幻觉（Hallucination）？ | 模型**一本正经地编造**看似合理实则错误的内容（假事实/假 API/假引用）。是应用落地第一大风险。 |
| 7 | 幻觉怎么治？ | **RAG 给依据** + 降 temperature + 要求**引用溯源** + 结构化输出约束 + 输出后**校验/评估兜底** + 提示"不知道就说不知道"。 |
| 8 | 什么是流式输出（Streaming）？ | 模型**边生成边逐 Token 返回**（SSE），前端打字机效果；提升"首字延迟"体验，不用等整段生成完。 |
| 9 | 为什么大模型响应慢？怎么优化？ | 自回归**逐 Token 生成**，输出越长越慢。优化：流式返回、异步、控制 `max_tokens`、缓存、简单任务用小模型分流。 |
| 10 | 大模型调用为什么贵？怎么省？ | 按 Token 计费（输入+输出）。省：**缓存**重复问答、**RAG 只塞相关片段**别塞全文、压缩 Prompt、小模型/大模型**分级路由**、限制 `max_tokens`。 |
| 11 | 什么是多模态（Multimodality）？ | 模型能同时理解**文字+图片+音频**等多种输入（如图文一起问）。 |
| 12 | System / User / Assistant 三种角色？ | System=设定人设/规则（最高优先级）；User=用户输入；Assistant=模型历史回复。多轮对话就是这组消息的累积。 |
| 13 | 大模型是无状态的吗？多轮对话怎么记住？ | **是无状态的**——模型本身不记忆，每次请求要把**历史消息一起再发一遍**；"记忆"是应用层维护对话历史（ChatMemory）实现的。 |
| 14 | 结构化输出怎么保证？ | 让模型返回 JSON：Prompt 里给 schema/示例 + 用框架的结构化输出（Spring AI `entity()`）+ 输出后**反序列化校验、失败重试**。 |
| 15 | LLM 和传统 NLP/规则的区别？ | LLM 是**通用生成式**、零样本即可干多任务、无需为每个任务训练；代价是不确定、慢、贵、会幻觉，要靠工程驯服。 |

## 📌 展开速答

**Q：Token 到底怎么算？为什么面试老强调它？**
Token 是模型的计价与限长单位。分词器（Tokenizer）把文本切成子词，中文一个字通常 1~2 Token，英文一个词约 1~1.5 Token，标点也算。**它决定三件事**：①**费用**（输入+输出 Token 数×单价，通常输出更贵）②**上下文窗口上限**（塞不下就截断）③**延迟**（输出 Token 越多越慢）。所以工程上"省钱、防超长、提速"本质都是"省 Token"。

**Q：上下文窗口越大越好吗？**
不完全。大窗口能塞更多背景，但：①塞得越多**越贵越慢**；②有"**Lost in the Middle**"现象——放中间的信息容易被忽略，关键内容应放开头或结尾；③大窗口不等于能替代 RAG，无关内容塞进去反而干扰。实践是**只塞相关的（RAG 检索）**，而不是把全部文档硬塞进大窗口。

**Q：temperature 设 0 就完全确定（每次一样）吗？**
接近确定但**不保证 100% 可复现**。temperature=0 时取概率最高的词（贪心），但受浮点计算、并行、模型版本、后端实现影响，仍可能有细微差异。要"稳定抽取/分类"就用低温 + 结构化输出 + 校验，别假设绝对一致。

**Q：幻觉能彻底消除吗？**
**不能彻底消除，只能压低到可接受**。它源于模型"按概率续写"的本质，不是 bug。工程上是分层兜底：RAG 提供事实依据（最有效）→ 低温度减少发散 → 强制引用来源可溯源 → 结构化约束输出格式 → 用另一个模型/规则做**事实性评估**（见 [`06-ai-engineering`](06-ai-engineering.md) 评估）→ 关键场景加人工确认。

**Q：什么时候用流式，什么时候不用？**
面向用户的**对话/长文本生成**用流式（改善体验、降首字延迟）。但**需要拿到完整结果再处理**的场景不适合，比如：要解析成 JSON 对象、要做工具调用、要对整段结果校验/审核——这些得等生成完整。

**Q：怎么给一个 AI 功能做成本估算？**
`成本 ≈ (输入Token + 输出Token) × 调用次数 × 单价`。先压输入（RAG 只塞相关片段、精简 Prompt、缓存），再压输出（限 `max_tokens`），再压次数（结果缓存、防重复），最后**分级路由**（简单请求走便宜小模型，难的才上大模型）。这套能答出来就显得做过真实项目。

## ⚠️ 易错 / 反问加分

- ⚠️ **别说"模型能记住对话"**——模型无状态，记忆是应用层每次把历史重发实现的。答错这条直接暴露没做过。
- ⚠️ **别把 Token 当成"字数"**——中英文、标点、代码的 Token 换算不同；面试官可能追问"1 万字文档大概多少 Token"，答"中文约 1.5~2 万 Token"更专业。
- ✅ **加分**：把参数和场景挂钩——"抽取/分类/代码用低温 + 结构化输出，创意用高温"，比只解释参数含义强。
- ✅ **加分**：主动提"Lost in the Middle"和"上下文不是越大越好"，显示你踩过长上下文的坑。
- ✅ **加分**：谈任何能力都带一句工程治理（慢→流式、贵→缓存分流、瞎编→RAG+评估），把"会调 API"升级成"会落地生产"。
