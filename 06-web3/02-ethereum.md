# 02 · 以太坊核心 速答（Ethereum）

> 以太坊 = 一台全球共享的「世界计算机」，比特币记账、它跑程序（智能合约）。覆盖 13 个高频考点。

## 🔥 高频必背（Top 6）

| # | 考点 | 一句话背诵 |
|---|------|-----------|
| 1 | ETH vs BTC | 比特币是「电子现金账本」（转账为主）；以太坊是「图灵完备的智能合约平台」，能跑任意逻辑的程序。 |
| 2 | 账户模型 | 以太坊用「账户模型」（余额直接记在账户上）；比特币用「UTXO 模型」（余额是未花费输出的集合）。 |
| 3 | 两类账户 | EOA（私钥控制、能主动发交易）vs 合约账户（代码控制、被动被调用、无私钥）。 |
| 4 | Nonce | 每个账户的交易计数器，从 0 递增：① 防重放攻击 ② 保证交易按顺序执行、不可跳号。 |
| 5 | Gas 机制 | Gas = 计算资源计价单位，防止死循环/滥用。矿工/验证者按消耗的 Gas 收费；不足则 out of gas 回滚，但已消耗的 Gas 不退。 |
| 6 | EVM | 以太坊虚拟机，一个基于栈的字节码执行引擎，所有节点跑同一份字节码 → 得到同一状态（确定性）。 |

## 📌 展开速答

### ① 以太坊 vs 比特币（图灵完备 / 智能合约）

| 维度 | 比特币 BTC | 以太坊 ETH |
|------|-----------|-----------|
| 定位 | 数字黄金 / 电子现金 | 世界计算机 / 智能合约平台 |
| 脚本 | 非图灵完备（无循环，故意受限） | 图灵完备（可写任意逻辑） |
| 记账模型 | UTXO | 账户模型 |
| 出块 | ~10 分钟 | ~12 秒 |
| 核心能力 | 转账 | 转账 + 部署/执行合约（DeFi/NFT/DAO） |

一句话：**图灵完备**意味着 EVM 能表达任意计算（有循环、条件、状态），所以能承载 DeFi、NFT 这些复杂应用；Gas 就是用来约束「图灵完备可能带来的死循环」的经济手段。

### ② 账户模型：EOA vs 合约账户，UTXO vs 账户

**两类账户对比：**

| | EOA 外部账户 | 合约账户 |
|---|-------------|---------|
| 控制方式 | 私钥 | 合约代码 |
| 能否主动发交易 | 能 | 不能（只能被调用触发） |
| 有无代码 | 无 | 有（部署时写入） |
| 有无 Nonce | 有 | 有（创建合约时用） |
| 地址来源 | 公钥推导 | 部署时由 创建者地址+nonce 计算 |

**UTXO vs 账户模型：**

```
UTXO（比特币）：钱包 = 一堆「未花费的硬币」，转账要凑硬币、找零
   余额 = 所有属于你的 UTXO 之和（需遍历）

账户模型（以太坊）：钱包 = 银行账户，直接记一个数字余额
   转账 = 扣 A 加 B，天然适合合约维护状态
```

- 账户模型优点：状态直观、适合智能合约（合约要持久保存变量）。
- UTXO 优点：并行验证好、隐私略强、无重放/顺序问题。
- 以太坊为「顺序问题」引入了 **Nonce** 来补偿账户模型的缺点。

### ③ Nonce 作用

- 定义：账户已发出的交易数量，从 0 开始，每笔交易 +1。
- 作用一 **防重放**：同一笔已上链的交易 nonce 已用过，再广播会被拒，无法重复扣款。
- 作用二 **定顺序**：nonce 必须连续递增，节点严格按 nonce 顺序打包执行，避免乱序。
- 卡住场景：若 nonce=5 的交易一直 pending（gas 太低），nonce=6、7 都会被卡在后面 → 需要用「相同 nonce + 更高 gas」替换掉。

### ④ 交易结构

| 字段 | 含义 |
|------|------|
| from | 发送方地址（由签名恢复，不显式写） |
| to | 接收方；为空则表示「创建合约」 |
| value | 转账金额（单位 Wei） |
| data | 调用数据：调合约时是「函数选择器 + 参数」，部署时是合约字节码 |
| nonce | 发送方交易计数 |
| gasLimit / maxFeePerGas / maxPriorityFeePerGas | Gas 相关（见下） |
| v, r, s | 签名 |

一句话：**转账**看 value，**调合约**看 data，**创建合约**是 to 为空 + data 放字节码。

### ⑤ Gas 机制（重点）

**为什么要 Gas？** 图灵完备 → 可能写死循环；每条指令收 Gas，钱花光就停 → 既防资源滥用，又给出块者付计算报酬。

- **gasLimit**：本交易最多愿意消耗的 Gas 上限（普通转账固定 21000）。防止意外消耗无限 Gas。
- **实际费用 = 实际消耗 Gas × 每单位 Gas 价格**。

**旧模型（EIP-1559 前）：** `费用 = gasUsed × gasPrice`，用户竞价 gasPrice，费用全给矿工，拥堵时体验差。

**EIP-1559（当前主流）：** 每单位 Gas 价格拆成两部分：

```
每 Gas 价格 = baseFee（基础费，协议算出，被销毁 🔥）
            + priorityFee（小费，给验证者，用户设 maxPriorityFeePerGas）

用户设 maxFeePerGas = 愿付上限
实付 ≈ baseFee + min(priorityFee, maxFee - baseFee)
退款 = (maxFee - 实付) × gasUsed
```

- baseFee 随区块拥堵程度自动升降（目标是每块 50% 满），且**被销毁**（通缩来源）。
- priorityFee 是给验证者的「插队小费」。

**Gas 不足会怎样（out of gas）？**
- 交易执行中 Gas 耗尽 → 抛 `out of gas`，**所有状态变更回滚**（就像没发生）。
- 但 **已消耗的 Gas 不退**，仍要付费（因为节点确实做了计算）。
- 记忆点：**状态回滚，Gas 照扣。**

### ⑥ EVM 是什么

- Ethereum Virtual Machine，以太坊虚拟机。
- **基于栈的字节码执行引擎**：Solidity 编译成字节码（opcode），EVM 逐条执行。
- 特点：① 确定性（同输入 → 同结果，所有节点一致）② 沙箱隔离 ③ 每条 opcode 有固定 Gas 成本。
- 256 位字长（为哈希/椭圆曲线运算优化）。

### ⑦ 状态与状态树（世界状态 / MPT）

- **世界状态 World State**：所有账户地址 → 账户状态（余额、nonce、code、storage）的映射；区块链的「当前快照」。
- 存储结构 **MPT（Merkle Patricia Trie，默克尔帕特里夏树）**：
  - Merkle 树 → 任意数据变了，根哈希就变 → 可高效验证 & 防篡改。
  - Patricia Trie（压缩前缀树）→ 按 key 高效查找/更新。
- 区块头存三棵树的根：**stateRoot（状态树）、transactionsRoot（交易树）、receiptsRoot（收据树）**。
- 好处：轻节点只拿区块头 + 少量证明就能验证某账户余额（默克尔证明）。

### ⑧ 以太坊网络（主网 / 测试网）

| 网络 | 用途 |
|------|------|
| Mainnet 主网 | 真实资产、真金白银 |
| Sepolia | 主流测试网，测试合约（水龙头领测试币，无价值） |
| Holešky/Hoodi | 面向质押/验证者的测试网 |

背诵：开发先在 **Sepolia** 测，没问题再上主网。（Goerli 已弃用。）

### ⑨ L1 vs L2（扩容 / Rollup）

**为什么要扩容？** 主网（L1）TPS 低（~15）、Gas 贵 → 无法承载大规模应用。

- **L1**：以太坊主网本身，负责最终安全与结算。
- **L2**：建在 L1 之上的链，把大量交易在链下/L2 执行，再把结果压缩后提交回 L1 → 便宜、快，同时继承 L1 安全性。
- **Rollup**（主流 L2 方案）：把一批交易「卷起来」批量提交到 L1。

| Rollup 类型 | 一句话 |
|-------------|--------|
| Optimistic（乐观）| 默认交易有效，留「挑战期」让人举报欺诈（欺诈证明）。代表 Arbitrum、Optimism。 |
| ZK（零知识）| 用零知识证明数学上证明交易正确，无需挑战期、提现快。代表 zkSync、StarkNet。 |

记忆：**Optimistic「先信任后挑战」，ZK「用密码学直接证明」。**

### ⑩ 单位换算（Wei / Gwei / Ether）

```
1 Ether  = 10^9  Gwei = 10^18 Wei
1 Gwei   = 10^9  Wei
```

- **Wei**：最小单位，链上一切数值的真实单位。
- **Gwei**：常用于表示 Gas 价格（如 30 Gwei）。
- **Ether**：面向人的展示单位。

### ⑪ 区块 Gas Limit

- 每个区块能容纳的 Gas 总量上限（当前约 3000 万，动态可微调）。
- 作用：限制单个区块的计算/存储量 → 保证出块时间稳定、节点跟得上。
- 与「交易 gasLimit」区别：**交易 gasLimit** 是单笔上限；**区块 Gas Limit** 是整块所有交易之和的上限。
- EIP-1559 下，区块目标用量 = Gas Limit 的一半，超过则 baseFee 上涨。

## ⚠️ 易错 / 反问加分

- **gasLimit ≠ gasPrice**：前者是「最多用多少 Gas（量）」，后者是「每单位 Gas 多少钱（价）」。费用 = 量 × 价。
- **out of gas 回滚但扣费**：很多人以为回滚就退钱，错。状态回滚，Gas 不退。
- **baseFee 被销毁**：EIP-1559 后 baseFee 不给验证者，是销毁的，这是 ETH 通缩机制的核心。
- **合约账户没有私钥**：不能主动发起交易，只能被 EOA（或另一合约）调用触发。
- **Nonce 是账户级别的**，不是全局的；每个 EOA 各自维护自己的 nonce。
- **UTXO vs 账户** 常被追问：账户模型天然适合合约维护状态，UTXO 难以表达「持久变量」。
- **图灵完备 ≠ 无限制**：EVM 图灵完备，但 Gas + 区块 Gas Limit 在经济上约束了执行，避免真正无限循环卡死网络。
- 反问加分：EIP-1559 让费用「更可预测」而非「更便宜」；拥堵时 baseFee 仍会飙升。
- 反问加分：为什么 256 位？→ 为 Keccak-256 哈希和椭圆曲线运算对齐，减少大数运算开销。
